## 大白话解释

项目演示最怕两件事：

```text
一是主链路现场跑不通；
二是答错后说不清为什么。
```

所以演示前不能只靠“我昨天跑过”。
要准备一套固定检查：

- 索引是否存在；
- 成功问题是否能返回答案和 citations；
- 无答案问题是否能明确拒答；
- 空问题、top_k 越界这类参数错误是否有清楚提示；
- 输出里是否有 request_id，方便排查。

这样演示时就算遇到 bad case，也能解释这是资料、检索、参数还是边界条件的问题。

---

## 生产实际

真实公司里，一个 RAG 项目上线或演示前，通常要准备三类检查：

| 类型 | 检查内容 | 目的 |
|------|----------|------|
| 成功路径 | 典型问题、正确引用、稳定响应 | 证明主链路可用 |
| 边界路径 | 无答案、低置信度、空输入、参数越界 | 证明系统不胡编、不崩溃 |
| 排查路径 | request_id、citations、latency、日志 | 出问题后能定位原因 |

面试演示时，不要只展示“最好看的回答”。
更好的讲法是：

1. 先展示成功样例；
2. 再展示 citations 和 request_id；
3. 主动展示一个无答案或 bad case；
4. 说明后续怎么用 query rewrite、rerank、缓存、安全控制继续优化。

这会让项目看起来更接近生产系统，而不是一次性 Demo。

## 常见演示风险

| 风险 | 表现 | 处理方式 |
|------|------|----------|
| 索引未生成 | API health 显示 index_ready=false | 演示前先跑 Day 17 入库 |
| 成功样例不稳定 | 召回不到预期文档 | 固定演示问题并跑回归 |
| 问题太泛 | citations 杂乱，答案不聚焦 | 演示问题要具体，能命中已入库资料 |
| 无答案时胡编 | 没资料也强行回答 | 返回 cannot_answer_reason |
| 参数错误不清楚 | 空问题或 top_k 越界报错难懂 | 依赖 Pydantic/FastAPI 校验并整理错误提示 |
| 没有排查字段 | 答错后不知道看哪里 | 返回 request_id、citations、retrieved_chunks、latency |

## 面试沉淀

Q055：你会怎么准备一个 RAG 项目的 5 分钟演示？

### 回答

我会按“目标、架构、成功样例、引用排查、边界和后续优化”来组织。
先用一句话说明项目解决什么问题，比如把本地知识库封装成可调用的问答 API。
然后讲离线入库链路：文档清洗、切分、embedding、索引和 metadata。
再讲在线问答链路：用户问题、检索、重排、上下文、生成、citations 和 request_id。
演示时先跑 `/health`，确认索引存在，再跑一个固定成功问题，展示答案、引用和置信度。
最后主动展示一个无答案或 bad case，说明系统如何拒答，以及后续如何通过 query rewrite、rerank、评测集和缓存继续优化。

Q056：RAG 项目演示前应该做哪些稳定性检查？

### 回答

演示前要检查三类链路。
第一是前置条件，例如索引文件是否存在、依赖是否可用、固定回归问题是否能跑完。
第二是成功路径，例如典型问题能返回 answer、citations、request_id、confidence 和 latency。
第三是边界路径，例如空问题、top_k 越界、无相关资料时是否有清楚错误提示或 cannot_answer_reason。
还要准备一份固定演示脚本和检查报告，避免现场临时选问题导致结果不稳定。

Q057：为什么演示时要主动讲 bad case？

### 回答

主动讲 bad case 能体现你理解的是生产系统，而不是只会展示顺利样例。
RAG 项目一定会遇到资料缺失、召回不准、引用错误、权限边界和成本问题。
如果只展示成功样例，面试官很难判断你是否知道系统边界。
主动展示 bad case，并说明它是资料问题、检索问题、prompt 问题还是权限问题，
再给出 query rewrite、rerank、评测集、拒答和监控等改进方案，会更像真实工程复盘。

保证索引能够使用，固定回归问题能够回答

演示前先检查索引和固定回归问题，是为了保证基础链路可用，不在演示现场暴露低级故障。
还可以补充：固定问题能验证改动没有破坏核心能力。

参考答案：
RAG 项目演示前要先检查索引和固定回归问题，是因为索引不可用时，后面的检索、引用和答案都会失败。
固定回归问题可以验证核心成功路径是否稳定，比如知识入库、top-k 检索、citations 和答案生成是否正常。
这样能避免演示时才发现索引缺失、资料未入库、引用为空或接口报错。

回答评价：
回答正确。

### 4. 空问题、top_k 越界这类参数错误应该怎么处理？
  我的回答：

拒绝回答，解释澄清建议权限过滤和审计，记录 requst id 方便后续问题追踪

回答评价：
方向部分正确，但需要区分参数错误和业务拒答。
空问题、top_k 越界属于请求参数校验问题，通常应该返回明确的 4xx 错误；
不是走普通 RAG 拒答，也不一定涉及权限过滤。

参考答案：
空问题、top_k 越界这类问题应该在 API 入参校验阶段处理。
空 question 应返回清楚的参数错误，比如 question 不能为空；
top_k 小于最小值或超过最大值时，应返回合法范围说明。
这类错误要有统一错误结构、错误码、request_id 和日志记录。
它们不应该进入检索和模型生成阶段，否则会浪费成本，也会让错误变得难排查。

### 5. 如果演示时答案不理想，应该怎么定位问题？
  我的回答：

追踪requst id，定位问题，

回答评价：
方向正确，但偏短。
request_id 是排查入口，还要沿着问题、改写、检索结果、citations、上下文、prompt、模型输出和日志逐层定位。

参考答案：
如果演示时答案不理想，应该先用 request_id 找到完整链路日志。
然后按层排查：
看用户问题是否清楚，query rewrite 是否改偏；
看 expected source 是否进入 top-k；

最后看模型是否没有基于引用回答。
这样才能判断问题在资料、检索、排序、引用、生成还是接口稳定性。
### B. 前两天核心回顾

### 6. [Day 25] chunk metadata 里为什么要有权限标签？
  我的回答：

进行权限过滤，防止进入上下文，防止上下文注入

回答评价：
方向正确，但表述偏短。
权限标签的核心是让系统按用户角色、业务域和安全等级过滤 chunk；
不是主要为了防“上下文注入”，而是防无权限资料进入上下文、答案和 citations。

参考答案：
chunk metadata 里要有权限标签，是因为 RAG 检索和引用的最小单位通常是 chunk。
如果 chunk 没有 allowed_roles、security_level、business_domain 等信息，
系统就无法判断当前用户是否能看这段资料。
权限标签可以用于检索过滤、上下文构造、citations 展示和审计回放，
防止无权限资料进入 LLM，也防止引用来源泄露内部信息。

### 7. [Day 25] 为什么 RAG 不能只靠 prompt 防止敏感信息泄露？
  我的回答：

不能代替权限层来进行权限过滤，模型可能上下文注入，推断出权限之外的信息，获取路径，是否存在，文件名等

回答评价：
回答正确。
你覆盖了 prompt 不能代替权限层、模型可能推断权限外信息，以及路径、文件名和资料是否存在也可能泄露。

参考答案：
不能只靠 prompt 防止敏感信息泄露，因为 prompt 只是对模型的行为约束，不是权限系统。
如果敏感 chunk 已经进入上下文，模型可能直接引用、总结或推断出敏感内容。
用户也可能通过 prompt 注入诱导模型忽略规则。
生产里要在模型之前完成数据分级、入库脱敏、权限标签、检索过滤和上下文过滤。
prompt 只能作为最后一层提醒，不能作为唯一边界。

### 8. [Day 26] 为什么上下文不是越长越好？
  我的回答：

浪费资源，代入噪声

回答评价：
回答正确但偏短。
上下文过长除了浪费 token，还会增加延迟、挤占真正相关资料位置，并提高引用错误和幻觉风险。

参考答案：
上下文不是越长越好。
太长会增加 input token 成本和模型延迟，也会把无关 chunk 带进 prompt。
无关资料越多，模型越容易被干扰，引用错误来源，甚至基于噪声生成错误答案。
生产里通常用 rerank、去重、权限过滤和 token budget，只保留少量最相关、最可信的 chunk。

### 9. [Day 26] RAG 缓存 key 为什么不能只用用户问题文本？
  我的回答：

用户权限，版本，业务域不同，导致回答错误，模糊，越权等问题

回答评价：
回答正确。
你说到了用户权限、版本和业务域不同会导致错误和越权。
还可以补充 prompt 版本、模型版本和 TTL，否则缓存会返回过期答案。

参考答案：
RAG 缓存 key 不能只用用户问题文本。
同一个问题在不同用户角色、权限范围、业务域、知识库版本、prompt 版本和模型版本下，
可能对应不同的可见资料和不同答案。
如果只按问题文本缓存，可能把高权限用户的答案返回给低权限用户，造成越权；
也可能在文档更新后返回旧答案。
所以缓存 key 至少要包含规范化问题、用户角色、业务域、知识库版本、prompt 版本和模型版本。
