## 大白话解释

NL2SQL 不是把一句话直接变 SQL。
更稳的做法是先把一句话拆成“SQL 需要的零件”。

比如用户问：

```text
上周每个渠道的授信通过率是多少？
```

系统应该先解析出：

```json
{
  "metric": "approval_rate",
  "group_by": ["channel"],
  "time_range": "last_week",
  "query_type": "group_by"
}
```

再结合 Day 29 的 Schema Catalog，才知道应该查哪张表、哪个字段、怎么聚合、用哪个时间字段。

## 生产实际

真实公司的 NL2SQL 链路通常会把“问题解析”放在 SQL 生成之前。

| 环节 | 作用 |
|------|------|
| 问题规范化 | 去掉口语噪声、统一大小写、识别同义词 |
| 指标抽取 | 把授信申请量、授信通过率、放款金额、逾期率映射到标准指标 |
| 维度抽取 | 判断是否按渠道、城市、信贷产品、风险等级、逾期账龄等维度分组 |
| 时间解析 | 把昨天、上周、最近 7 天、本月转成结构化时间条件 |
| 过滤抽取 | 抽出城市=北京、申请编号=A123、风险等级=A、审批状态=通过等条件 |
| 风险识别 | 发现敏感字段、缺少时间范围、问题不清楚等风险 |

这一步做得好，后续 SQL 生成会稳定很多。
这一步做不好，模型即使语法写对，也可能选错字段、漏掉时间、算错口径。

---

## 常见坑

| 问题 | 表现 | 处理方式 |
|------|------|----------|
| 指标同义词没管 | “授信通过率”和“审批通过率”被当成不同指标 | 建指标别名表 |
| 时间解析不清 | “最近”不知道是几天 | 需要默认规则或追问 |
| 维度和过滤混淆 | “北京用户”既可能是过滤，也可能是分组 | 根据“每个/按/分”判断 group by |
| 表粒度没考虑 | 申请明细和渠道日汇总表混用 | 结合 Schema Catalog 校验 |
| 敏感字段没拦截 | 用户要求导出手机号 | 解析阶段加 risk_flags |
| 缺少时间范围 | 查询大表全量数据 | 设置默认时间或要求补充 |

## 工程取舍

Day 30 先用规则解析，不急着接 LLM。

原因：

- 指标别名、维度名、时间词、TopN 词很多是确定性规则；
- 规则输出稳定，方便测试；
- 后续接 LLM 时，也应该让 LLM 输出同样的结构化字段；
- 有规则 baseline，才能评估 LLM 是否真的变好。

生产里常见做法是规则和模型结合：

```text
确定性实体用规则和字典抽取；
复杂表达和模糊意图再交给 LLM 解释；
最终都落到结构化 parse_result。
```

## 面试沉淀

Q068：NL2SQL 里为什么要先做问题解析？

### 回答

NL2SQL 先做问题解析，是为了把用户自然语言拆成 SQL 生成需要的结构化信息。
用户问题里通常混合了指标、维度、时间、过滤条件、排序、TopN 和业务口径。
如果直接让模型生成 SQL，模型容易漏掉时间范围、选错表字段、混淆指标口径或生成高成本查询。
问题解析会先抽取 metric、dimensions、time_range、filters、query_type 和 risk_flags，
再结合 Schema Catalog 选择候选表和字段。
这样后续 SQL 生成更可控，也更容易做权限校验、成本校验和错误排查。

Q069：自然语言时间在 NL2SQL 里怎么处理？

### 回答

自然语言时间必须转成结构化时间条件。
比如“昨天”“上周”“最近 7 天”“本月”都要解析成 start_date、end_date、granularity 和 time_field。
生产里不能只把原始时间词塞进 prompt，因为不同业务可能有不同时间口径，
比如申请提交时间、审批完成时间、放款成功时间、还款到期日、分区日期和事件发生时间。
如果时间不清楚，要使用业务默认规则或向用户追问。
同时，大表查询必须限制时间范围，避免生成全表扫描 SQL。

Q070：指标抽取和维度抽取有什么区别？

### 回答

指标抽取关注“要算什么”，比如授信申请量、授信通过率、放款金额、逾期率。
维度抽取关注“按什么看”，比如渠道、城市、信贷产品、日期、风险等级、逾期账龄。
在 SQL 里，指标通常对应聚合字段或聚合表达式，维度通常对应 group by、where 或展示字段。
两者不能混淆。比如“每个渠道的授信通过率”，授信通过率是指标，渠道是分组维度。
如果把维度当指标，SQL 聚合会错；如果把过滤条件当分组维度，结果粒度也会错。
