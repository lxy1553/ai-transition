## 大白话解释

NL2SQL 不是把一句话直接变 SQL。更稳的做法是先把一句话拆成“SQL 需要的零件”。

比如用户问：

```text
上周每个渠道的授信通过率是多少？
```

系统应该先解析出：

```json
{
  "metric": "approval_rate",
  "group_by": ["channel"],
  "time_range": "last_week",
  "query_type": "group_by"
}
```

再结合 Day 29 的 Schema Catalog，才知道应该查哪张表、哪个字段、怎么聚合、用哪个时间字段。

## 生产实际

真实公司的 NL2SQL 链路通常会把“问题解析”放在 SQL 生成之前。

- 表格行 1
  - 环节：问题规范化
  - 作用：去掉口语噪声、统一大小写、识别同义词
- 表格行 2
  - 环节：指标抽取
  - 作用：把授信申请量、授信通过率、放款金额、逾期率映射到标准指标
- 表格行 3
  - 环节：维度抽取
  - 作用：判断是否按渠道、城市、信贷产品、风险等级、逾期账龄等维度分组
- 表格行 4
  - 环节：时间解析
  - 作用：把昨天、上周、最近 7 天、本月转成结构化时间条件
- 表格行 5
  - 环节：过滤抽取
  - 作用：抽出城市=北京、申请编号=A123、风险等级=A、审批状态=通过等条件
- 表格行 6
  - 环节：风险识别
  - 作用：发现敏感字段、缺少时间范围、问题不清楚等风险

这一步做得好，后续 SQL 生成会稳定很多。这一步做不好，模型即使语法写对，也可能选错字段、漏掉时间、算错口径。

## 常见坑

- 表格行 1
  - 问题：指标同义词没管
  - 表现：“授信通过率”和“审批通过率”被当成不同指标
  - 处理方式：建指标别名表
- 表格行 2
  - 问题：时间解析不清
  - 表现：“最近”不知道是几天
  - 处理方式：需要默认规则或追问
- 表格行 3
  - 问题：维度和过滤混淆
  - 表现：“北京用户”既可能是过滤，也可能是分组
  - 处理方式：根据“每个/按/分”判断 group by
- 表格行 4
  - 问题：表粒度没考虑
  - 表现：申请明细和渠道日汇总表混用
  - 处理方式：结合 Schema Catalog 校验
- 表格行 5
  - 问题：敏感字段没拦截
  - 表现：用户要求导出手机号
  - 处理方式：解析阶段加 risk_flags
- 表格行 6
  - 问题：缺少时间范围
  - 表现：查询大表全量数据
  - 处理方式：设置默认时间或要求补充

## 工程取舍

Day 30 先用规则解析，不急着接 LLM。

原因：

- 指标别名、维度名、时间词、TopN 词很多是确定性规则；
- 规则输出稳定，方便测试；
- 后续接 LLM 时，也应该让 LLM 输出同样的结构化字段；
- 有规则 baseline，才能评估 LLM 是否真的变好。

生产里常见做法是规则和模型结合：

```text
确定性实体用规则和字典抽取；
复杂表达和模糊意图再交给 LLM 解释；
最终都落到结构化 parse_result。
```

## 面试沉淀

Q068：NL2SQL 里为什么要先做问题解析？

### 回答

NL2SQL 先做问题解析，是为了把用户自然语言拆成 SQL 生成需要的结构化信息。用户问题里通常混合了指标、维度、时间、过滤条件、排序、TopN 和业务口径。
如果直接让模型生成 SQL，模型容易漏掉时间范围、选错表字段、混淆指标口径或生成高成本查询。
问题解析会先抽取 metric、dimensions、time_range、filters、query_type 和 risk_flags，再结合 Schema
Catalog 选择候选表和字段。 这样后续 SQL 生成更可控，也更容易做权限校验、成本校验和错误排查。

Q069：自然语言时间在 NL2SQL 里怎么处理？

### 回答

自然语言时间必须转成结构化时间条件。比如“昨天”“上周”“最近 7 天”“本月”都要解析成 start_date、end_date、granularity 和
time_field。 生产里不能只把原始时间词塞进 prompt，因为不同业务可能有不同时间口径，比如申请提交时间、审批完成时间、放款成功时间、还款到期日、
分区日期和事件发生时间。 如果时间不清楚，要使用业务默认规则或向用户追问。同时，大表查询必须限制时间范围，避免生成全表扫描 SQL。

Q070：指标抽取和维度抽取有什么区别？

### 回答

指标抽取关注“要算什么”，比如授信申请量、授信通过率、放款金额、逾期率。维度抽取关注“按什么看”，比如渠道、城市、信贷产品、日期、风险等级、逾期账龄。
在 SQL 里，指标通常对应聚合字段或聚合表达式，维度通常对应 group by、where 或展示字段。两者不能混淆。
比如“每个渠道的授信通过率”，授信通过率是指标，渠道是分组维度。如果把维度当指标，SQL 聚合会错；如果把过滤条件当分组维度，结果粒度也会错。

### 7. [Day 29] NL2SQL 为什么不能直接把用户问题交给模型生成 SQL？
我的回答：
模型可能会猜测编造问题，也可能生成权限之外的数据

回答评价：
回答正确。你抓住了模型幻觉和权限越界两个关键风险。还可以补充：模型也可能选错表字段、漏掉时间范围、混淆指标口径或生成高成本 SQL。

参考答案：
NL2SQL 不能直接把用户问题交给模型生成 SQL，因为模型不知道真实数据库里有哪些表、字段、指标口径、表粒度、权限规则和分区字段。
如果没有 Schema Catalog 和校验约束，模型可能编造字段，选错表，漏掉时间条件，把不同业务口径混在一起，甚至生成越权查询或高成本全表扫描。
生产里应该先做问题分类、Schema 匹配和问题解析，再生成只读 SQL，并经过 SQL 校验后才能执行。

### 8. [Day 29] Schema Catalog 在 NL2SQL 里解决什么问题？
我的回答：
表名，维度，时间，权限，条件，指标等

回答评价：
回答正确。你列出了 Catalog 的核心内容。再补充一句会更完整：它的价值是把 SQL 生成限制在可信表字段、指标口径和权限边界内。

参考答案：
Schema Catalog 在 NL2SQL 里解决的是“系统能查哪些数据、字段是什么意思、指标怎么算、谁能查”的问题。
它通常包含表名、表描述、业务域、表粒度、字段名、字段类型、字段含义、指标、维度、时间字段、分区字段、权限等级和常见问题示例。
有了 Schema Catalog，SQL 生成器才能只使用可信的表和字段，减少模型编造，也方便后续做权限校验、字段白名单校验、成本控制和审计。
