## 大白话解释

离线数据仓库分层，本质上是在回答一个问题：

```text
不同用途的数据，应该放在什么位置，给谁用，怎么防止乱用？
```

ODS 像“原始收件箱”，业务系统怎么来就先怎么收。
DWD 像“清洗后的明细账”，把原始数据修正成统一字段、统一状态、统一粒度。
DWS 像“主题汇总账”，按授信、放款、还款、逾期等主题提前算好常用汇总。
ADS 像“业务看板和日报”，直接面向经营分析、管理驾驶舱和高频指标问答。

Agent 如果不理解这些层级，就会出现两个风险：

- 明明只是问“昨天通过率”，却去扫授信申请明细；
- 明明涉及客户手机号、身份证号、银行卡号，却没有权限校验就生成 SQL。

所以 Day 47 的核心不是“会说 ODS/DWD/DWS/ADS”，而是能说清楚：

```text
Agent 查离线指标时，为什么要优先查 ADS/DWS，
什么时候才允许查 DWD，
为什么 ODS 默认不能作为普通问答入口。
```

## 生产实际

金融信贷公司里的离线仓库通常会接入多个系统：

| 系统 | 原始数据 | 常见用途 |
|------|----------|----------|
| 授信申请系统 | 申请单、客户信息、渠道、产品 | 申请量、通过率、渠道分析 |
| 风控决策系统 | 策略命中、审批结果、风险等级 | 拒绝率、策略命中率、风险监控 |
| 放款系统 | 放款单、放款金额、放款状态 | 放款成功率、放款金额趋势 |
| 还款系统 | 还款流水、扣款结果、失败原因 | 还款成功率、失败原因分析 |
| 贷后系统 | 逾期天数、账龄、催收状态 | 逾期率、M1/M2、催收效果 |

这些数据不会直接丢给 Agent。
生产里通常先按层加工：

| 层级 | 作用 | Agent 查询建议 |
|------|------|----------------|
| ODS | 保留业务系统原始数据 | 默认不查，只用于排查上游来源 |
| DWD | 清洗后的业务明细事实 | 有权限、精确过滤、带 limit 时才能查 |
| DWS | 按主题和维度汇总 | 适合渠道、产品、日期维度分析 |
| ADS | 面向看板和日报的应用指标 | 适合经营总览和高频指标问答 |

例如用户问：

```text
昨天信贷整体申请量、审批通过率和放款金额是多少？
```

Agent 应该优先查 `ads_credit_daily_metrics`。
如果用户问：

```text
近 7 天各渠道授信通过率趋势。
```

Agent 应该优先查 `dws_credit_apply_channel_1d`。
如果用户问：

```text
导出昨天所有申请人的手机号和身份证号。
```

Agent 不应该查任何层，应该直接走权限阻断和审计。

## Agent 离线路由标准

Agent 查询离线指标时，可以按这条路线：

```text
intent_classifier
-> offline_layer_router
-> schema_catalog
-> offline_sql_generator
-> sql_validator
-> offline_query_executor
-> result_interpreter
-> audit_logger
```

每一步的责任：

| 步骤 | 作用 | 失败时怎么办 |
|------|------|--------------|
| 意图识别 | 判断是离线指标、明细、口径、血缘还是敏感导出 | 意图不清就澄清 |
| 分层路由 | 判断优先查 ADS、DWS 还是 DWD | 无合适层就拒答或转口径问答 |
| Schema Catalog | 找到表、字段、粒度、分区和权限 | 找不到表就资料不足 |
| SQL 生成 | 生成只读候选 SQL | 不能直接执行 |
| SQL 校验 | 检查分区、limit、敏感字段、权限和成本 | 不通过就阻断 |
| 查询执行 | 只执行已校验 SQL | 超时或分区缺失要返回异常 |
| 结果解释 | 把结果解释成业务语言 | 不能编造 SQL 没查出的原因 |
| 审计记录 | 记录问题、用户、路线、SQL 和结果 | 金融场景必须保留 |

## 常见坑

| 类型 | 可能的问题 | 生产处理方式 |
|------|------------|--------------|
| 乱查 ODS | Agent 直接查原始客户明细 | ODS 默认禁止作为普通问答入口 |
| 明细滥用 | 普通指标问题扫描 DWD 明细 | 指标优先 ADS/DWS，DWD 只做授权明细 |
| 粒度错误 | 用日汇总表回答单笔明细问题 | Schema Catalog 写清表粒度 |
| 口径错误 | 把通过率直接 avg | 指标字典写清分子、分母和过滤条件 |
| 缺少分区 | SQL 没有 dt 条件导致全表扫描 | SQL Validator 强制分区过滤 |
| 权限越界 | 查询手机号、身份证号、银行卡号 | 权限标签、安全阻断和审计 |

## 工程取舍

### 取舍一：为什么总览指标优先 ADS？

ADS 是面向业务看板和日报的应用层，指标已经按固定口径加工好。
Agent 查询 ADS，可以减少 SQL 复杂度、降低扫描成本，也更容易保证口径一致。

如果每次都从 DWD 明细临时聚合，容易出现分区漏写、过滤条件不一致、重复计数和成本过高。

### 取舍二：为什么维度分析常用 DWS？

DWS 保留了主题维度，例如渠道、产品、风险等级、日期。
它比 ADS 更适合做趋势和对比，又比 DWD 明细更安全、更低成本。

例如“近 7 天各渠道授信通过率趋势”，DWS 是更合适的入口。

### 取舍三：为什么不能让 Agent 默认查 ODS？

ODS 是原始数据层，字段未经统一清洗，通常包含敏感明细。
Agent 默认查 ODS 会带来三类风险：口径不稳定、权限风险高、查询成本不可控。

ODS 更适合数据工程排查上游问题，不适合作为业务用户问答入口。

## 面试沉淀

Q103：金融信贷离线仓库为什么要分 ODS、DWD、DWS、ADS？
重要程度：5/5

### 回答

金融信贷离线仓库分层，是为了把原始数据、明细事实、主题汇总和应用指标分开管理。
ODS 保留业务系统原始数据，DWD 负责清洗成统一粒度的明细事实，DWS 按主题和维度提前汇总，
ADS 面向日报、看板和高频业务查询提供稳定指标。

如果不分层，Agent 或数据分析人员会直接面对大量原始表，很容易选错表、混淆口径、扫大表、重复计数，
还可能访问客户手机号、身份证号、银行卡号等敏感字段。
分层后，经营总览优先查 ADS，维度趋势优先查 DWS，授权明细才查 DWD，ODS 默认只用于上游排查。

在 Agent 数据问答里，分层还能降低 NL2SQL 难度。
Schema Catalog 可以告诉 Agent 每张表的层级、粒度、分区、权限和适用问题，
让 Agent 先选对层，再生成 SQL、校验 SQL、执行查询和记录审计。

Q104：Agent 查询离线指标时，为什么应该优先选择 ADS 或 DWS？
重要程度：5/5

### 回答

Agent 查询离线指标时优先选择 ADS 或 DWS，是因为这两层更接近业务指标。
ADS 面向日报、看板和经营总览，指标口径已经固定；DWS 面向主题汇总和维度分析，
适合按渠道、产品、日期、风险等级做趋势和对比。

如果 Agent 默认从 DWD 或 ODS 明细临时聚合，风险会明显增加。
第一，SQL 更复杂，容易漏分区、漏过滤条件或重复计数；
第二，扫描成本更高；
第三，更容易碰到敏感字段和权限边界；
第四，指标口径可能和业务看板不一致。

所以生产级离线指标 Agent 应该先通过意图识别和分层路由判断问题粒度：
总览走 ADS，维度趋势走 DWS，授权明细才走 DWD，敏感导出直接阻断。
之后再进入 Schema Catalog、SQL 生成、SQL 校验、执行、解释和审计。
