## 大白话解释

RAG 测试集就是一批固定问题，每次改检索、chunk、query rewrite 或 prompt 后都用它们再跑一遍。

没有测试集时，优化很容易靠感觉：

```text
这个问题好像答得更好了。
```

有测试集后，就能看清楚：

```text
20 条问题里，正确资料进入 top-k 的有多少条？
正确资料平均排第几？
哪些问题变好了，哪些问题变差了？
```

---

## 生产实际

真实公司里，RAG 项目上线前通常要准备评测集。它至少覆盖：

- 高频业务问题
- 精确实体问题，比如表名、字段名、错误码
- 口语化问题
- 多意图问题
- 知识库没有答案的问题
- 权限或敏感边界问题
- 容易幻觉的问题
- 已知 bad case

每条样本不应该只有 question，还要有期望来源、问题类型和是否应该回答。

推荐字段：

| 字段 | 作用 |
|------|------|
| `id` | 样本编号 |
| `question` | 用户原始问题 |
| `question_type` | 问题类型 |
| `expected_sources` | 应该命中的资料来源 |
| `should_answer` | 是否应该回答 |
| `expected_answer_points` | 标准答案要点 |
| `risk_tags` | 评测风险标签 |

---

## 常见坑

| 问题 | 影响 | 处理方式 |
|------|------|----------|
| 只放简单问题 | 评测虚高 | 加入口语、边界和 bad case |
| 只看最终答案 | 不知道错在检索还是生成 | 分开看 retrieval 和 generation |
| 没有期望引用 | citations 无法校验 | 每条样本写 expected_sources |
| 测试集经常变 | 前后结果不可比 | 固定核心集，新增样本要分版本 |
| 只追求命中率 | 噪声可能变多 | 同时看 hit_rank 和无关召回 |

---

## 工程取舍

第一版评测先做检索评估，不急着做复杂自动打分。

原因是 RAG 里很多错误首先来自检索阶段：

```text
正确资料没进 top-k
-> 后面 prompt 再好也答不准
```

等检索评估稳定后，再加入生成答案评估、引用准确率、拒答准确率和人工评分。

## 面试沉淀

Q043：RAG 测试集应该怎么构建，评估哪些指标？

### 回答

RAG 测试集要覆盖真实用户会问的问题，而不是只放几个简单样例。每条样本至少要包含
问题、期望答案要点、期望引用来源、问题类型和是否应该拒答。评估时要分层看：
检索层看正确资料是否进入 top-k、排名是否靠前；生成层看答案是否正确、是否基于上下文、
是否有幻觉；引用层看 citations 是否指向正确文档；工程层看延迟、失败率和成本。
测试集还要包含边界问题，比如知识库没有答案、权限不足、资料冲突、用户表达模糊和精确表字段匹配。
每次改 chunk、top-k、query rewrite、rerank 或 prompt 后都跑同一批样本，这样才能知道优化是否真的有效。

### 1. RAG 测试集应该怎么构建？

我的回答：
尽量覆盖所有用户可能提问问题的各个角度，
问题要覆盖高频问题， sql 问题，精确字段或者表名，边界问题，权限问题，敏感问题，bad case。
而且不仅要包含问题本身，还有问题类型，期望回答的关键词，是否拒答，expected source
要用同一个 bad case 和测试集反复测试

回答评价：
方向正确，覆盖了高频问题、边界问题、敏感问题和 bad case；
字段上建议把“问题引用来源”明确说成 expected_sources；
“置信度”不是测试集必填字段，更核心的是 expected_answer_points、should_answer、question_type 和 expected_sources。

参考答案：
RAG 测试集要覆盖真实用户会问的问题；
不能只放几个容易命中的成功样例。
每条样本至少应该包含 question、question_type、expected_sources、should_answer、expected_answer_points 和 risk_tags；
这样才能同时评估检索、引用、拒答和边界风险。
样本类型要覆盖高频问题、精确表字段问题、口语化问题、多意图问题、bad case、无答案问题、权限问题和敏感问题；
核心测试集要保持稳定，新增样本可以分版本管理。

### 2. 为什么评估 RAG 不能只看最终答案？

我的回答：
最终答案有可能命中率高置信度高，但不一定是正确答案

回答评价：
方向对，但还不够完整；
需要补充 RAG 错误可能来自资料、检索、引用、生成或工程环节；
只看最终答案无法定位到底是 expected source 没召回，还是引用正确但模型生成错。

参考答案：
RAG 的最终答案是多个环节叠加后的结果；
只看答案无法判断问题出在哪里。
错误可能来自知识库资料缺失、chunk 切分不合理、query rewrite 改写偏移、top-k 太小、metadata 过滤误伤、rerank 排序错误；
也可能来自模型生成时没有忠实依据。
所以要拆开评估；
先看 expected_sources 是否进入 top-k；
再看 citations 是否正确；
最后看答案是否基于上下文。

### 3. expected_sources 在测试集中有什么作用？

我的回答：
用来判断是否命中，提高召回质量

回答评价：
方向正确，但偏短；
需要补充 expected_sources 可以计算 hit、hit_rank 和 citations 准确性；
它能帮助判断正确资料是否进入 top-k，以及错误发生在检索还是生成阶段。

参考答案：
expected_sources 是每条测试问题应该命中的资料来源；
它用来判断检索是否找到了正确依据。
没有 expected_sources，系统返回几个看起来相关的 chunk，也很难客观判断是否真的召回了正确资料。
它可以用于计算 hit_rate、hit_rank 和 citations 准确性；
如果 expected source 没进入 top-k，说明检索或知识库侧有问题；
如果进入但排名靠后，说明排序或 rerank 需要优化；
如果引用正确但答案错，才重点看 prompt 或模型生成。

### 4. RAG 评估应该看哪些指标？

我的回答：
命中率，命中排名，引用率，拒单率，延迟，失败率和成本


回答评价：
方向正确，已经覆盖命中率、命中排名、引用、拒答、延迟、失败率和成本；
建议把“拒单率”改成“拒答准确率”；
还可以补生成层的答案正确率、是否基于上下文、是否有幻觉。

参考答案：
RAG 评估要分层看指标；
检索层看 hit_rate、avg_hit_rank、top-k 是否包含 expected_sources；
引用层看 citations 是否指向正确文档、chunk 和位置；
生成层看答案正确率、是否基于上下文、是否有幻觉；
拒答层看无答案、无权限、敏感问题是否能正确拒答；
工程层看延迟、失败率、超时率、成本和 token 消耗。
这些指标要一起看；
不能只追求某一个指标变高。

### 5. 测试集中为什么要包含 bad case 和拒答问题？

我的回答：
 bad case 方便后续排查问题，用同一个测试集测试 bad case 看是否有改变，提高召回率，
拒答问题有些问题触发边界，权限，敏感问题，保证合法合规

防止同类型问题反复出现，防止没有权限敏感场景下编造答案

回答评价：
方向正确；
已经覆盖 bad case 复盘和拒答边界；
可以再补一句：bad case 用来防止同类问题反复出现；
拒答问题用来验证系统不会在无依据、无权限或敏感场景下编造答案。

参考答案：
测试集中必须包含 bad case；
因为系统最需要持续改进的，往往不是成功样例，而是反复失败的问题。
bad case 可以帮助定位是资料缺失、chunk 切分不合理、query 表达不清、top-k 不合适、metadata 过滤误伤；
也可以继续判断是 rerank 或生成出了问题。
测试集中也要包含拒答问题；
因为生产 RAG 不能在知识库没有依据、用户无权限、问题超范围或涉及敏感信息时编造答案。
拒答样本可以验证系统边界是否稳定。

### 6. 每次改 chunk、top-k、query rewrite 或 prompt 后，为什么要跑同一批测试集？

我的回答：
看后续改变是否有效，是否提高了置信率，召回率

回答评价：
方向正确；
建议把“置信率”改成更准确的“命中率、命中排名、引用准确率和答案正确率”；
核心原因是同一批测试集才能做前后对比，判断改动是整体变好，还是只让少数样例变好但引入新退化。

参考答案：
每次改 chunk、top-k、query rewrite、rerank 或 prompt 后，都要跑同一批测试集；
因为只有测试集固定，前后结果才可比较。
如果每次都换问题，就无法判断改动是真的提升了系统，还是只碰巧让某几个样例变好。
固定测试集可以观察 hit_rate、hit_rank、引用准确率、拒答准确率和答案正确率是否变化；
也能发现新策略是否引入了退化。
生产里通常会把测试集分成核心回归集和扩展样本集；
核心集用于稳定对比，扩展集用于持续发现新问题。
