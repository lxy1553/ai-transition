## 大白话解释

query rewrite 就是“把用户的话翻译成检索系统更容易理解的话”。

用户可能会问：

```text
这个为什么要带来源？
```

但知识库里写的是：

```text
RAG 问答必须返回 citations、引用来源、chunk_id 和可追溯信息。
```

如果直接拿用户原话去检索，系统可能找不到正确资料。rewrite 会把问题补全成：

```text
RAG 为什么要返回引用来源 citations 可追溯 chunk_id？
```

这样检索更容易命中 Day 18 的引用来源笔记。

## 生产实际

生产 RAG 里，用户问题通常不标准：

- 省略主语，比如“这个怎么处理？”
- 使用口语，比如“问不出来怎么办？”
- 使用同义词，比如“资料出处”和“引用来源”
- 中英文混用，比如 citation、source、chunk
- 一个问题里有多个意图
- 带上下文指代，比如“上面那个指标怎么算？”

query rewrite 通常接在用户提问之后、正式检索之前：

```text
用户问题
-> 问题清洗
-> query rewrite
-> 向量检索 / 关键词检索
-> metadata 过滤
-> rerank
-> 构造上下文
-> 生成答案和 citations
```

真实公司里 rewrite 可以用规则、词典、历史搜索日志，也可以用 LLM。
但无论哪种方式，都要记录 original query 和 rewritten query，方便排查。

---

## 常见坑

| 问题 | 影响 | 处理方式 |
|------|------|----------|
| 改写改变用户原意 | 召回看似相关但答错问题 | 保留原始 query，一起检索或做对照 |
| 加太多关键词 | 引入噪声，top-k 被稀释 | 控制扩展词数量 |
| 丢掉精确实体 | 表名、字段名、错误码检索失败 | 精确词必须原样保留 |
| 只看命中率 | 可能牺牲排序和准确性 | 同时看 hit、hit_rank、引用相关性 |
| 没有日志 | bad case 无法复盘 | 记录 original、rewritten、results、request_id |

---

## 工程取舍

第一版不要直接上复杂 LLM rewrite。

更稳的路线是：

```text
规则词典改写
-> 保存 original query 和 rewritten query
-> 用固定测试集评估命中率
-> 再考虑多 query、LLM rewrite 和 rerank
```

原因是规则改写可解释、可测试、成本低。等你能看懂哪些问题被改好了、
哪些问题被改坏了，再接 LLM rewrite 才有判断依据。

## 面试沉淀

Q032：Query rewrite 在 RAG 里解决什么问题，生产里怎么控制风险？

### 回答

Query rewrite 解决的是用户表达和知识库表达不一致的问题。用户可能说得很口语、
省略上下文、中英文混用，或者一个问题里包含多个意图；而知识库通常使用标准术语、
表名、字段名和指标名。改写可以补全省略、加入同义词、保留精确实体，或者把复杂问题
拆成多个检索 query，从而提升召回质量。但生产里不能让改写随意改变用户意图。
工程上要保留原问题和改写问题同时检索，日志记录 original query、rewritten query、
召回结果和最终引用；对表名、字段名、错误码这类精确词要原样保留；多查询数量要受控，
避免延迟和噪声过高。是否启用改写要通过测试集验证，而不是凭感觉判断。

## 今日复盘

今天要记住三句话：

1. query rewrite 是为了让检索更容易命中正确资料。
2. 改写必须保留用户原意，不能为了命中而乱加词。
3. rewrite 是否有效，要用固定测试集和 bad case 验证。

### 3. query rewrite 有哪些常见方式？
  我的回答：

关键词的提取，同义词转义，拆分语句

回答评价：
方向正确，覆盖了关键词、同义词和拆分；可以再补业务术语补全、上下文补全、精确实体保留、多 query 和 LLM 改写。

参考答案：
常见方式包括规则词典改写、同义词扩展、业务术语补全、上下文补全、精确实体保留、多 query 拆分和 LLM 改写。
规则词典适合高频稳定的业务词；
同义词扩展适合用户表达和文档表达不一致；
上下文补全适合“这个怎么算”这类指代问题；
多 query 适合一个问题包含多个检索方向；LLM 改写适合复杂口语表达。

### 4. rewrite drift 是什么，为什么危险？
  我的回答：

重写曲解，把用户的问题曲解，导致问题回答偏离错误，

回答评价：
正确，已经抓住“曲解原意”和“回答偏离”；可以再补它为什么危险：召回结果看似相关但实际回答了错误问题。

参考答案：
rewrite drift 指 query rewrite 改变了用户原意，导致系统检索到看似相关但实际错误的资料。
它危险的地方在于，召回结果可能分数很高、答案看起来也合理，但回答的不是用户真正想问的问题。
生产里 rewrite drift 会造成错误回答、错误引用和用户信任下降，严重时还可能触发权限或合规风险。


### 5. 生产里如何控制 query rewrite 的风险？
  我的回答：

避免曲解问题原意

回答评价：
方向正确但不够工程化；
需要补保留 original query、限制扩展词、保留精确实体、记录日志、用测试集验证和必要时澄清/拒答。

参考答案：
生产里要保留 original query 和 rewritten query，必要时两者同时检索；对表名、字段名、错误码、
产品名等精确实体要原样保留；限制扩展词和多 query 数量，避免引入噪声；记录 rewritten query、
召回结果、最终引用和 request_id，方便 bad case 复盘；对低置信度或意图不清的问题，可以选择澄清或拒答。
是否启用改写要通过固定测试集验证，而不是凭感觉判断。

### 6. 怎么判断 query rewrite 是否真的改善了召回质量？
  我的回答：

多用测试数据测试，试试每次返回是否是想要的答案

回答评价：
方向正确，已经说到用测试数据验证；
可以再补具体指标：expected source 是否进入 top-k、hit_rate、hit_rank、citations 相关性和噪声是否增加。

参考答案：
要用固定测试集评估，而不是只看一次答案是否变好。
每条样本应该包含原始问题、期望来源、问题类型和必要的标准答案要点。
评估时先看 expected source（预期来源） 是否进入 top-k，再看命中排名是否更靠前，同时观察无关 chunk 是否增加。
如果 rewritten query 提高了 hit_rate 和 avg_hit_rank，并且 citations（引用来源） 更相关，说明有正向收益。
如果命中率提高但噪声也明显增加，就要结合 rerank、限制扩展词或调整规则。
