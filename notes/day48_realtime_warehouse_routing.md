## 大白话解释

离线仓库像“每天结一次账”，适合问昨天、上周、上月的稳定指标。
实时仓库像“现场监控屏”，适合问当前、近 5 分钟、近 10 分钟、是否异常、是否告警。

实时仓库里最重要的几个概念：

| 概念 | 大白话解释 | 信贷例子 |
|------|------------|----------|
| 实时事件 | 业务刚发生的一条状态变化 | 用户提交授信申请、风控规则命中、还款失败 |
| 事件时间 | 业务真实发生的时间 | 用户实际提交申请的时间 |
| 处理时间 | 系统处理这条事件的时间 | Flink 收到并计算这条事件的时间 |
| 窗口 | 聚合实时事件的时间范围 | 近 5 分钟申请量、近 10 分钟拒绝率 |
| 延迟 | 数据到达或计算落后的程度 | 实时链路延迟 20 分钟，当前指标不可信 |

Agent 如果不理解这些概念，就会犯两个典型错误：

- 用昨天离线日报回答“近 5 分钟是否异常”；
- 实时链路已经延迟，却仍然说“当前正常”。

## 生产实际

金融信贷实时仓库常见链路：

```text
业务系统事件
-> Kafka / Event Bus
-> Flink 实时计算
-> 实时指标状态表
-> 实时告警表
-> Agent 实时指标工具 / 告警工具
-> 延迟检查
-> 结果解释和审计
```

典型事件：

| 事件 | 用途 | 常见实时指标 |
|------|------|--------------|
| 授信申请提交 | 监控申请流量和渠道突增 | 近 5 分钟申请量 |
| 风控决策完成 | 监控拒绝率和策略命中 | 近 10 分钟拒绝率 |
| 还款失败 | 监控扣款异常和渠道故障 | 近 5 分钟还款失败数 |

真实公司里，实时指标不仅要有数值，还要有状态：

- 当前窗口是什么；
- 数据基于事件时间还是处理时间；
- 当前链路延迟多少；
- 是否超过延迟阈值；
- 是否触发告警；
- 告警证据来自哪里。

如果这些状态缺失，Agent 就不能给出确定答案。

## Agent 实时路由标准

Agent 看到这些信号时，应该优先进入实时路线：

| 信号 | 例子 | 路由 |
|------|------|------|
| 近几分钟 | 近 5 分钟申请量是否异常 | 实时指标工具 |
| 当前状态 | 当前拒绝率是否突增 | 实时指标工具 |
| 告警 | STR_BLACKLIST 策略是否触发告警 | 告警工具 |
| 事件流 | 实时风控事件流 | 权限校验或安全阻断 |
| 延迟 | 实时链路延迟 20 分钟 | 延迟检查后异常/降级 |

标准工具路线：

```text
intent_classifier
-> realtime_router
-> realtime_metric_catalog / alert_query_tool
-> realtime_metric_tool
-> delay_checker
-> result_interpreter
-> audit_logger
```

关键规则：

- 实时问题不能走 `offline_sql_generator` 和 `offline_query_executor`；
- 缺少窗口、业务线、渠道、策略等必要条件时，要先澄清；
- 延迟超过阈值时，最终状态应是 `execution_failed` 或明确降级；
- 告警解释只能基于告警等级、触发规则、窗口指标和证据来源；
- 实时事件流里的客户明细仍然受权限和敏感字段控制。

## 常见坑

| 类型 | 可能的问题 | 生产处理方式 |
|------|------------|--------------|
| 离线实时混用 | 近 5 分钟问题走离线日报 | 实时信号强制走实时路由 |
| 缺窗口 | “看实时拒绝率”没有窗口和业务线 | 先澄清，不直接查询 |
| 忽略延迟 | 链路延迟 20 分钟仍说当前正常 | delay_checker 返回异常或降级 |
| 编造告警原因 | 只看到拒绝率升高就说某策略导致 | 只基于告警证据解释 |
| 敏感导出 | 导出实时事件流客户手机号 | 权限阻断和审计 |
| 状态不结构化 | 工具只返回数字，不返回延迟状态 | 工具输出必须包含窗口、时间和延迟 |

## 工程取舍

### 取舍一：为什么实时问题不能走离线 SQL？

离线 SQL 查的是已经沉淀好的日级或批量指标。
实时问题关心的是当前窗口状态，例如近 5 分钟、近 10 分钟。
用离线 SQL 回答实时问题，即使 SQL 能跑通，业务上也是错的。

### 取舍二：为什么实时工具要返回延迟状态？

实时指标的可信度取决于链路是否及时。
如果实时链路延迟超过阈值，数值可能是过期的。
因此工具不能只返回一个数字，还要返回窗口、事件时间、处理时间、延迟和是否可用。

### 取舍三：为什么告警解释不能编造原因？

告警通常只能说明“某个指标超过阈值”。
真正原因可能来自策略上线、渠道流量变化、系统故障或外部攻击。
Agent 只能基于告警规则、窗口指标和证据来源解释事实，不能编造业务原因。

## 面试沉淀

Q106：实时仓库和离线仓库在 Agent 数据问答里有什么区别？

### 回答

离线仓库面向稳定的历史指标和经营分析，通常按天或批次加工数据。
Agent 查询昨日、上周、上月的授信通过率、放款金额、逾期率时，应该走离线 ADS 或 DWS。
实时仓库面向当前状态、分钟级窗口、实时特征和告警，适合回答近 5 分钟申请量、当前拒绝率是否异常、
实时还款失败数是否突增这类问题。

两者最大的区别不是表名不同，而是时间语义不同。
离线指标强调分区、批次、指标口径和稳定性；实时指标强调事件时间、处理时间、窗口、延迟和链路状态。
如果 Agent 把实时问题路由到离线 SQL，即使能查出数字，也可能业务上错误。

生产级 Agent 应该先识别问题是离线指标还是实时状态。
离线问题走 Schema Catalog、SQL 生成、SQL 校验和离线执行；
实时问题走实时指标工具或告警工具，并检查窗口和延迟状态。
敏感明细和实时事件流导出都必须经过权限控制和审计。

Q107：Agent 查询实时指标时为什么必须检查窗口和延迟？

### 回答

实时指标的含义依赖窗口。
“申请量是否异常”如果没有说明近 5 分钟、近 10 分钟、渠道或产品线，就无法判断应该查哪个实时指标。
所以缺少窗口或关键维度时，Agent 应该先澄清，而不是直接查询。

实时指标的可信度还依赖延迟。
如果实时链路延迟 20 分钟，那么当前窗口的数据可能不可用或已经过期。
这时 Agent 不能给出“当前正常”这类确定结论，而应该返回延迟说明、降级或 `execution_failed`。

工程上，实时工具的输出不能只有数值，还要包含窗口、事件时间、处理时间、延迟分钟数、延迟阈值和可用状态。
结果解释层只能基于这些结构化状态解释，不能自行编造实时原因。

## 术语更新

今天新增或强化这些术语：

- 实时事件：业务系统持续产生的状态变化记录。
- 事件时间：业务事件真实发生的时间。
- 处理时间：系统实际处理事件的时间。
- 实时窗口：实时指标聚合事件的时间范围。
- 实时延迟：实时链路处理落后于事件发生的程度。
- 实时路由：Agent 判断问题应该走实时指标工具、告警工具、澄清还是阻断的过程。

### 2. 什么是事件时间，什么是处理时间？

  参考答案：
