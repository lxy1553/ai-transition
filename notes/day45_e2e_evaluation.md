# Day 45 - Agent 评测：离线/实时仓库评测样例

## 今日目标

Day 43 把 Agent 工作流画清楚，Day 44 把工具选择、前置条件和失败回退规则写清楚。
从 Day 45 开始，学习主线升级为“金融信贷离线/实时仓库 + Agent”。
今天先做评测体系：用固定问题集检查仓库 Agent 的完整链路是否真的按预期工作。

今天要掌握：

- 端到端评测不是只测某个函数，而是测“用户问题到最终回答”的完整链路；
- 仓库 Agent 评测必须同时覆盖离线指标、实时指标、口径解释、血缘追溯、权限阻断和系统异常；
- 每条样例要写清预期状态、必须调用的工具、禁止调用的工具和失败类别；
- 正确拒答、要求补充条件、识别实时延迟和阻断敏感明细，都算质量的一部分；
- 失败要分类统计，方便 Day 46 做错误治理，而不是凭感觉改 Prompt。

## 大白话解释

仓库 Agent 端到端评测，就是把系统当成一个完整产品来测。

不要只问：

```text
SQL 生成函数能不能生成 SQL？
实时查询工具能不能返回一个数？
RAG 能不能召回几段口径文档？
```

而是问：

```text
用户真的问了一个信贷业务问题后，
系统有没有判断这是离线指标、实时状态、口径解释、血缘追溯还是敏感明细？
有没有选对工具？
离线 SQL 有没有经过校验？
实时指标有没有检查窗口和延迟？
口径解释有没有引用？
该拒答时有没有拒答？
最终回答能不能被业务接受和审计回放？
```

一条样例可能单个工具都没报错，但整体仍然失败。
例如“近 5 分钟授信申请量是否异常”如果被路由到离线日报表，SQL 也许能跑通，
但业务上是错的，因为实时窗口问题不能用离线日指标回答。

## 生产实际

真实公司里，金融信贷数据问答 Agent 上线前要准备固定评测集。
样例来源通常包括业务常见问题、历史 bad case、权限边界、异常工单、实时告警和安全审计要求。

Day 45 的评测集要覆盖：

| 场景 | 要验证什么 | 正确表现 |
|------|------------|----------|
| 离线指标查询 | 是否查对 ADS/DWS 表、时间、维度和口径 | 走 Catalog、SQL 校验、离线查询和解释 |
| 实时指标查询 | 是否识别窗口、事件时间、处理时间和延迟 | 走实时指标工具，不误走离线 SQL |
| 实时告警解释 | 是否基于告警证据解释 | 只解释事实和证据，不编业务原因 |
| 指标口径问答 | 是否基于指标字典或 RAG 回答 | 返回引用，资料不足时拒答 |
| 血缘追溯 | 是否能追来源表、事件流、加工任务和下游影响 | 走血缘工具 |
| 敏感数据导出 | 是否拦截手机号、身份证、住址、联系人等字段 | 安全阻断并记录审计 |
| 条件缺失 | 是否要求补时间、窗口、产品或业务域 | 不进入查询执行 |
| 系统异常 | 分区缺失、实时延迟、数据库超时时怎么处理 | 分类返回错误，不编造结果 |

生产里的评测结果应该沉淀成趋势。
每次改 Prompt、Schema Catalog、工具路由、SQL 校验、实时状态工具、RAG 检索或结果解释，
都要跑同一批评测集。如果新增能力让老样例退化，就不能只看新功能演示成功。

## Day 45 评测标准

今天的本地练习使用 24 条样例。
每条样例都包含：

- 业务问题；
- 业务类别；
- 预期最终状态；
- 预期失败类别；
- 必须调用的工具；
- 禁止调用的工具；
- 是否需要引用；
- 评测说明。

成功不等于“所有问题都回答”。

```text
离线指标查询 -> answered
实时状态查询 -> answered
敏感导出 -> safely_blocked
缺少时间范围或实时窗口 -> clarification_required
口径资料不足 -> insufficient_evidence
分区缺失或实时延迟 -> execution_failed
不支持的问题 -> unsupported
```

这样设计的原因是：生产 AI 应用的质量不只来自会回答，还来自会停下。
尤其在金融信贷场景里，越权、敏感、无依据、高成本和实时延迟问题必须被准确识别。

## 常见坑

| 类型 | 可能的问题 | 生产处理方式 |
|------|------------|--------------|
| 样例偏差 | 测试集全是成功问题 | 加入拒答、权限、异常、实时延迟和边界问题 |
| 判定模糊 | 只靠人工看答案顺不顺 | 写清预期状态、必需工具、禁用工具和引用要求 |
| 离线实时混用 | 实时 5 分钟问题走离线日报表 | 意图识别必须区分离线指标和实时状态 |
| 只测单点 | SQL 生成通过就算成功 | 检查从意图到审计的完整链路 |
| 忽略拒答 | 把拒答当成失败 | 对敏感、无依据和越权问题，正确拒答算成功 |
| 不做归因 | 只看通过率 | 统计失败类型，定位是路由、校验、引用还是实时状态问题 |
| 不回归 | 修完一个 bad case 后不测旧问题 | 固定评测集，每次改动后全量回归 |

## 工程取舍

### 取舍一：为什么先做固定评测集？

固定评测集能让改动有可比较的基准。
如果每次都临时换问题，就很难判断系统是真的变好，还是刚好命中新样例。

学习阶段先用 20 到 30 条高质量样例，比追求上百条低质量样例更有价值。
样例要覆盖业务主链路和高风险边界，后续再从真实 bad case 中持续扩充。

### 取舍二：为什么评测里要包含“应该失败”的问题？

因为生产系统不能只会回答。
金融信贷系统遇到敏感字段、越权表、无引用政策、高成本查询、写操作、实时链路延迟时，
正确行为就是拒答、阻断、要求补充、返回异常或转人工。

如果测试集只包含成功问题，系统可能在演示里很好看，但上线后最容易在安全、合规和实时边界上出问题。

### 取舍三：为什么要统计失败占比？

失败占比能告诉你下一步该修哪里。
如果失败来自 `missing_citation`，优先修 RAG 召回和无依据拒答；
如果失败来自 `forbidden_tool_used`，优先修工具路由和前置条件；
如果失败来自实时延迟误判，优先修实时状态校验和异常分类。

这样 Day 46 的错误治理就不是凭感觉改 Prompt，而是按数据修复。

## 面试沉淀

Q096：生产级 AI 应用为什么需要端到端评测，而不是只测单个 Prompt 或工具？

### 回答

生产级 AI 应用需要端到端评测，因为真实用户关心的是完整链路结果，
而不是某个 Prompt、检索器、SQL 生成器或实时查询工具单独表现。
一个金融信贷仓库 Agent 可能每个单点工具都能跑通，但整体仍然出错。
例如实时问题被路由到离线 SQL、缺少时间范围却继续查库、口径解释没有引用、SQL 生成后跳过校验、
实时链路延迟却仍然回答正常，都会导致生产风险。

端到端评测应该从用户问题开始，检查意图识别、工具路由、Schema Catalog、RAG 检索、
SQL 生成、SQL 校验、离线查询、实时查询、告警解释、结果解释、安全阻断和审计记录。
每条样例都要有明确成功判定，包括预期最终状态、必须调用的工具、禁止调用的工具、
是否需要引用和失败分类。
对敏感导出、越权访问、资料不足、分区缺失、实时延迟和系统异常来说，
正确拒答或正确降级也应该算成功。

在金融信贷场景里，端到端评测尤其重要，因为系统可能接触授信、风控、放款、还款、
逾期、实时风控告警和客户敏感信息。
固定评测集可以持续回归，帮助团队判断每次 Prompt、规则、Schema、检索或工具路由改动
是否真的提升质量，是否引入新的安全、合规或实时风险。

Q100：金融信贷仓库 Agent 的端到端评测集应该怎么设计？

### 回答

金融信贷仓库 Agent 的端到端评测集不能只放“问指标并回答”的成功样例。
它至少要覆盖离线指标、实时指标、告警解释、指标口径、血缘追溯、权限阻断、缺参澄清和系统异常。
每条样例都要写清用户问题、业务类别、预期状态、必须调用的工具、禁止调用的工具、
是否需要引用、是否允许执行 SQL、是否要检查实时延迟，以及失败后应该归到哪类责任层。

离线指标样例要检查 Schema Catalog、分区字段、指标口径、SQL 生成、SQL 校验、执行和结果解释。
实时指标样例要检查窗口、事件时间、处理时间、延迟状态和实时工具路由。
同时，评测集必须包含应该失败的问题，例如敏感导出、无权限明细、缺少时间范围、口径资料没有引用、
分区缺失、实时链路延迟和数据库超时。
这些样例的正确结果不是硬答，而是安全阻断、要求补充条件、资料不足或系统异常兜底。

Q101：实时仓库问题进入 Agent 评测时要重点检查哪些风险？

### 回答

实时仓库问题进入 Agent 评测时，首先要检查工具路由风险。
只要用户问题出现近 5 分钟、近 10 分钟、实时拒绝率、实时申请量、告警、事件流、链路延迟这类信号，
Agent 就应该优先走实时指标工具或告警工具，而不是生成离线 SQL 查日报表。

其次要检查窗口和延迟风险。
实时指标必须说明统计窗口、事件时间、处理时间、延迟阈值和当前链路状态。
如果实时链路延迟 20 分钟，系统不能给出“当前正常”这类确定结论，
而应该提示数据延迟、返回系统异常或降级到可解释状态。

还要检查告警解释边界和权限边界。
告警解释只能基于告警等级、触发规则、窗口指标和证据来源，不能编造业务原因。
实时事件流里的客户明细、设备号、手机号、身份证号等敏感字段仍然要走权限校验或安全阻断。

## 术语更新

今天新增或强化这些术语：

- 端到端评测：从用户问题到最终答案检查完整链路表现。
- 成功判定标准：判断一条评测样例是否通过的具体规则。
- 失败分类：把失败归入路由错误、安全错误、引用缺失、系统异常、实时延迟等类别。
- Bad Case：系统表现不符合预期的样例，用来驱动后续修复。
- 回归评测：每次改动后用固定评测集重新验证，避免旧能力退化。
