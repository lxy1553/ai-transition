## 常见坑

| 类型 | 可能的问题 | 生产处理方式 |
|------|------------|--------------|
| 多脚本孤立 | 无统一入口和复用 | 抽主流程和模块边界 |
| 配置散落 | 模型、路径、阈值难维护 | 配置集中管理 |
| 工具重复实现 | 审计和错误码不一致 | 工具注册表统一约束 |
| 强依赖模型 | 本地演示和回归不稳定 | 保留规则和本地 fallback |
| README 太轻 | 面试时讲不清项目全貌 | 用架构和流程写清边界 |

## 工程取舍

### 取舍一：为什么要保留本地 fallback？

因为这个项目既是学习仓库，也是作品集。
如果所有能力都强依赖外部模型和在线环境，演示稳定性和回归确定性都会变差。

### 取舍二：为什么集成前先定模块边界？

因为 Day 59 之后要进入演示、面试和投递阶段。
如果此时还在把逻辑随意塞进 `main.py`，后面很难讲清楚架构和取舍。

## 面试沉淀

Q135：为什么综合项目必须提供统一入口，而不是保留多个独立脚本？
重要程度：5/5

### 回答

因为面试和生产都看重系统边界，而不是零散能力清单。
统一入口意味着意图识别、权限校验、工具路由、结果校验和审计逻辑被真正整合，而不是靠人工说明把多个脚本拼成一个故事。

Q136：为什么综合项目要保留本地 fallback？
重要程度：5/5

### 回答

因为本地 fallback 可以保证演示稳定、回归可重复、成本可控，也能让你把确定性规则和模型能力分开讲清楚。
对于 SQL 安全、权限拦截、延迟判断这类关键逻辑，本来就不应该完全依赖模型。

## 术语更新

- 统一入口：对外暴露单一调用入口，内部再做路由和编排。
- fallback：主方案失败或不适用时的降级方案。
- 模块边界：系统中各组件的职责、输入输出和依赖范围。
- 综合项目 README：既说明运行方式，也说明生产映射、风险和架构。

### 1. 为什么综合项目不能只是多个脚本并列存在？
  重要程度：5/5
  我的回答：
独立的脚本都是各个独立的脚本功能，综合的项目需要统一的能力，
意图识别，工具路由，权限校验，结果校验，审计审核的能力是
整合在一起的，而不是靠人工说明把多个脚本拼成一个故事

  评分：8/10。理解方向正确。建议补充具体痛点——配置散落难维护、
  工具重复实现导致审计/错误码不一致、
  面试官问模块间如何协作时没法回答。

  参考答案：
  多个独立脚本的问题在于：无统一入口和复用，配置散落难以维护，
  工具重复实现导致审计和错误码不一致，
  用户（或面试官）也无法直观理解这些能力如何协同工作。
  统一入口意味着意图识别、权限校验、工具路由、结果校验
  和审计逻辑被真正整合成一个主流程，而不是靠人工说明
  把多个脚本拼成一个故事。这才是"一个系统"而不是
  "一堆脚本拼成的故事"。
### 2. 统一入口的最小主流程应该包含哪些节点？
  重要程度：5/5
  我的回答：
意图识别，权限校验，工具路由，结果校验，审计审核

  评分：6/10。有遗漏。你列的 5 个节点是对的，但缺少
  工具执行、回答生成和 trace/eval。流程到工具路由就断了，
  没有实际执行和产出的环节。

  参考答案：
  最小主流程应为：用户问题 → 意图识别 → 权限校验 →
  工具路由 → 工具执行 → 结果校验 → 回答生成 →
  审计落库 → trace/eval。每个节点只做一件事，做完交给
  下一个，不能越界。工具执行负责实际调用（查 SQL、
  读 RAG、查实时指标），回答生成负责把结构化结果转成
  业务语言，trace/eval 负责记录链路供回归评测。
### 3. 为什么 SQL 安全和权限校验不应该只依赖模型？
  重要程度：5/5
  我的回答：
模型只能负责意图识别，问题改写，结果解释等
非确定性的内容，安全和权限校验需要确定性的
代码来控制边界，模型容易猜测幻读，上下文等
容易忽略安全和权限校验

  评分：7/10。核心理解正确。但"模型容易猜测幻读"用词
  不准，应该说：模型可能编造不存在的字段名、漏掉分区
  条件、忽略危险关键字或越权查询。Prompt 是软约束，
  模型在复杂问题下容易遗漏安全规则。

  参考答案：
  不能只靠 prompt 要求模型生成安全 SQL，因为 prompt
  是软约束，模型可能理解错、遗漏规则，在复杂问题下
  生成越权字段、危险 SQL 或高成本查询。生产安全必须
  用代码层硬约束实现：敏感字段要有字段级权限标签，
  表字段要来自 Schema Catalog，危险关键字要静态拦截，
  大表必须有时间范围，明细查询必须有 limit。prompt
  可以帮助模型少犯错，但不能替代 SQL 校验器、权限系统、
  审计日志和查询网关。
### 4. 为什么综合项目要保留本地 fallback？
  重要程度：5/5
  我的回答：
ai 项目的不确定性远大于传统项目，fallback
就是必须选项，当模型不确定或者出问题时候
给于一个降级备用方案，不完美但是可控的方案

  评分：7/10。核心逻辑说对了，但只讲了一个维度。缺少
  三个关键点：①演示稳定性——模型 API 挂了也能跑；
  ②回归可重复——规则结果不会漂移，评测结果一致；
  ③关键逻辑不该依赖模型——SQL 安全、权限拦截、延迟
  判断这些本来就应该用代码兜底。

  参考答案：
  这个项目既是学习仓库也是作品集。保留本地 fallback
  有三个核心价值：第一，保证演示稳定——如果所有能力
  都强依赖外部模型和在线环境，演示时网络抖动或模型
  不可用就会翻车。第二，保证回归可重复——确定性规则
  的结果不会因为模型版本变化而漂移，评测集每次跑出来
  的结果一致。第三，把确定性规则和模型能力分开讲清楚
  ——对于 SQL 安全、权限拦截、延迟判断这类关键逻辑，
  本来就不应该完全依赖模型，它们应该走确定性代码，
  模型只负责理解和表达。

### 6. 模块边界不清晰会带来哪些工程问题？
  重要程度：5/5
  我的回答：
每个模块都有各自的任务，如果不清晰模块干了不属于他的
工作，导致出错，无法排查问题，无法复用，安全问题，

  评分：8/10。4 点都说到位了，简洁有力。可以补充一个
  反例：比如 SQL 生成器"顺便做了权限校验"，那校验层

  参考答案：
  模块边界不清晰会带来四个问题。第一，无法排查——

  以后实时工具想复用还得再写一遍。第三，安全形同
  虚设——SQL 生成器说"我顺便校验了权限"，但模型
  生成的校验不可靠，等于没校验。第四，评测无从下手

  职责应该是"只做一件事，做完交给下一个，不能越界"。

### 8. [Day 58] 为什么审计字段必须结构化？
  重要程度：4/5
  我的回答：
结构化可以进行更加精准得找打具体字段，
trace id ，request Id 等，方便审计追踪，

  评分：6/10。偏薄且有错别字（"找打"→"找到"）。缺少
  关键点：结构化字段可以被程序稳定解析、校验、路由；
  status 和 next_action 必须用固定枚举，不能是自然
  语言；可以触发告警（如 final_status=safely_blocked
  超阈值）；回归测试可按结构化字段精准匹配预期。

  参考答案：
  审计字段必须结构化，因为程序需要稳定解析、校验和
  路由。每条审计记录至少包含 request_id、trace_id、
  user_id、scenario、intent、tool_route、model_id、
  status、latency_ms 和 error_code。其中 status 和
  error_code 必须使用固定枚举，不能是自然语言。这样
  做的价值是：可以自动定位问题（哪个请求出了问题、
  出在哪个节点）、可以触发告警（safely_blocked 比例
  超过阈值）、可以做回归测试的精确匹配。
### 9. [Day 57] 数据质量状态为什么会影响 Agent 是否回答？
  重要程度：4/5
  我的回答：
质量差可能导致回答不准确拒答，数据有延迟，涉及边界
问题，都会触发 agent 拒答或者降级

  评分：6/10。太抽象，缺少具体例子体现生产理解。
  只看回答分不清你是真懂还是泛泛而谈。

  参考答案：
  Agent 的回答依赖离线分区、实时窗口、补偿任务和上游
  事件质量。具体来说：离线分区为空或未产出时，Agent
  查不到数据可能误报为"指标为零"；实时窗口因 Kafka
  积压数据不完整时，Agent 给出的指标偏低；补偿任务
  未完成时，逾期率等关键指标在短时间内是偏的；上游
  事件重复写入会导致实时指标虚高。所以 Agent 必须在
  回答前检查数据质量状态：质量 ok 才正常回答，有风险
  就降级说明，严重异常就阻断并给排查建议。质量状态要
  结构化输出（quality_warning / quality_blocked），
  方便编排层决策。
### 10. [Day 56] 告警误报和口径冲突为什么要进入回归测试集？
  重要程度：4/5
  我的回答：
回归测试集要包含尽可能多的报错信息，都是真实的生产
环境中需要面对的，告警误报和口径冲突也是很有可能
发生的错误

  评分：6/10。方向对但太宽泛。核心逻辑没点到——回归
  测试不是为了"包含所有错误"，而是为了防止修复一个
  问题时破坏已有的正确逻辑。你的回答换成任何 bug 类型
  都成立，缺乏针对性。

  参考答案：
  回归测试的核心目的不是"包含尽可能多的错误"，而是
  保证每次改动后已有正确逻辑不退化。告警误报和口径
  冲突是典型的生产风险，容易在修复其他问题时再次出现。
  具体来说：修了告警误报的逻辑后，必须确保正常告警
  解释不受影响；修了口径冲突后，必须确保原来正确的
  口径问答不退化。如果这些 bad case 不在回归测试集里，
  下次代码变更可能又引入同样的问题。每次修复后跑全量
  回归，才能从"人工排查"变成"工程闭环"。
