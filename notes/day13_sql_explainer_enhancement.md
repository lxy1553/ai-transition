## 生产实际

SQL 风险规则在生产里是底线能力。金融信贷数据里包含客户身份、手机号、授信额度、审批状态、逾期和催收信息，错误 SQL 可能带来隐私泄露、
成本浪费或错误经营判断。

因此风险检查不能只靠 LLM。高危关键字、`select *`、缺少时间范围、缺少 limit、扫描明细表、查询敏感字段，都应该先由规则确定性拦截。

## 工程取舍

这一天优先采用最小可运行方案，是为了先建立稳定基线。真实项目里不能一开始就追求复杂架构，应该先把输入、处理逻辑、输出和失败场景跑通，再逐步增加模型调用、 检索、
数据库、权限和评测。

对“项目 1 强化：风险规则与输出优化”这类能力，学习阶段更看重可解释和可复盘；生产阶段再根据准确率、延迟、成本、安全和维护成本决定是否引入更复杂的方案。

## 面试沉淀

Q023：为什么 SQL 风险检查不能完全依赖 LLM？
重要程度：4/5

### 回答

SQL 风险检查不能完全依赖 LLM，因为风险判断里有很多确定性规则。比如是否缺少分区条件、是否使用 DELETE、是否 SELECT *、是否包含高成本
JOIN， 这些更适合用规则或 SQL 解析器检查。LLM 适合解释业务含义、补充建议和总结风险，但不适合独自承担强约束判断。
生产方案通常是规则检查先兜底，LLM 再做解释和表达，这样稳定性更高。
