## 真实工作场景

假设你在做一个 **SQL 上线前风险检查助手**。

背景：

- 数据开发同学提交一段指标 SQL
- 系统需要检查 SQL 是否有明显风险
- 风险结果要展示在页面上，也可能进入审批流
- 如果风险等级是 `high`，系统要阻止直接上线
- 如果缺少表结构或指标口径，系统要提示补充上下文

这时模型如果只回答：

```text
这段 SQL 可能有一些性能问题，建议注意 group by 和分区。
```

对人能看，但对系统不好用。

系统更需要这样的结果：

```json
{
  "summary": "统计指定日期每个城市的活跃用户数",
  "risk_level": "medium",
  "can_publish": false,
  "risks": [
    "需要确认 active_users 的业务口径",
    "group by city 可能带来聚合开销"
  ],
  "missing_context": [
    "user_events 表结构",
    "活跃用户定义"
  ],
  "suggestions": [
    "补充 user_events 字段说明",
    "确认 dt 是否为分区字段"
  ]
}
```

这就是结构化输出的价值：**让 AI 的回答变成系统可以继续处理的数据**。

**自由文本的问题：**

- [ ] 字段不固定，程序不好解析

**SQL 风险检查建议字段：**

## 注意事项

⚠️ **今天的重点：**

- 结构化输出不是为了好看，是为了让系统能继续处理
- 字段名要稳定，枚举值要固定
- 不能完全相信模型输出，必须做校验
- 校验失败要能返回明确错误

⚠️ **避免的坑：**

- 不要让模型自由发挥字段名
- 不要把风险等级写成随意中文描述
- 不要只校验是不是合法 JSON，还要校验业务规则
- 不要把“能生成”误认为“能上线”

## 今日核心知识点详解

### 1. 什么是结构化输出

大白话：

> 让模型不要随便写一段话，而是按你规定的字段返回结果。

比如真实系统里，你可能要根据风险等级决定：

- `low`：允许上线
- `medium`：需要人工确认
- `high`：禁止上线

如果模型只写“风险比较高”，程序不知道该怎么判断。

### 2. 为什么数据项目特别需要结构化输出

数据项目里很多结果要进入下游流程：

- SQL 审核
- 指标口径确认
- 报表异常解释
- 数据质量告警
- NL2SQL 生成结果校验
- RAG 回答可信度判断

这些都不能只靠自然语言。

### 3. 结构化输出和 Prompt 的关系

Day 9 学的是怎么写 Prompt。

Day 10 要进一步要求模型：

```text
必须返回 JSON。
字段必须包含 summary、risk_level、risks、suggestions。
risk_level 只能是 low、medium、high。
不能输出 JSON 以外的解释文字。
```

这就是从“会提问”到“能工程化落地”的一步。

### 4. 面试表达

> 在真实 AI 应用里，我不会只让模型返回自然语言。
我会要求模型输出固定 JSON，并在服务端做字段、类型、枚举和业务规则校验。
这样后续系统才能根据风险等级、是否缺少上下文等字段继续执行流程。
比如拦截高风险 SQL、提示补充表结构或进入人工审核。

## 今日练习

### 练习 1：判断坏输出

下面这个输出有什么问题？

```json
{
  "desc": "这个 SQL 还行，但可能有点风险",
  "level": "比较高",
  "advice": "建议优化一下"
}
```

你要指出：

- 字段名不符合约定
- 风险等级不是固定枚举
- 缺少 `can_publish`
- 缺少 `risks` 数组
- 缺少 `missing_context`

### 练习 2：改成好输出

把它改成：

```json
{
  "summary": "统计用户事件表中的活跃用户数",
  "risk_level": "medium",
  "can_publish": false,
  "risks": ["缺少活跃用户口径说明"],
  "missing_context": ["user_events 表结构", "active_users 统计口径"],
  "suggestions": ["补充字段含义和指标口径后再上线"]
}
```
