## 真实工作场景

假设你在做一个 **SQL 上线前风险检查助手**。

背景：

- 数据开发同学提交一段指标 SQL
- 系统需要检查 SQL 是否有明显风险
- 风险结果要展示在页面上，也可能进入审批流
- 如果风险等级是 `high`，系统要阻止直接上线
- 如果缺少表结构或指标口径，系统要提示补充上下文

这时模型如果只回答：

```text
这段 SQL 可能有一些性能问题，建议注意 group by 和分区。
```

对人能看，但对系统不好用。

系统更需要这样的结果：

```json
{
  "summary": "统计指定日期每个城市的活跃用户数",
  "risk_level": "medium",
  "can_publish": false,
  "risks": [
    "需要确认 active_users 的业务口径",
    "group by city 可能带来聚合开销"
  ],
  "missing_context": [
    "user_events 表结构",
    "活跃用户定义"
  ],
  "suggestions": [
    "补充 user_events 字段说明",
    "确认 dt 是否为分区字段"
  ]
}
```

这就是结构化输出的价值：**让 AI 的回答变成系统可以继续处理的数据**。

## 今日核心知识点详解

### 1. 什么是结构化输出

大白话：

> 让模型不要随便写一段话，而是按你规定的字段返回结果。

比如真实系统里，你可能要根据风险等级决定：

- `low`：允许上线
- `medium`：需要人工确认
- `high`：禁止上线

如果模型只写“风险比较高”，程序不知道该怎么判断。

### 2. 为什么数据项目特别需要结构化输出

数据项目里很多结果要进入下游流程：

- SQL 审核
- 指标口径确认
- 报表异常解释
- 数据质量告警
- NL2SQL 生成结果校验
- RAG 回答可信度判断

这些都不能只靠自然语言。

### 3. 结构化输出和 Prompt 的关系

Day 9 学的是怎么写 Prompt。

Day 10 要进一步要求模型：

```text
必须返回 JSON。
字段必须包含 summary、risk_level、risks、suggestions。
risk_level 只能是 low、medium、high。
不能输出 JSON 以外的解释文字。
```

这就是从“会提问”到“能工程化落地”的一步。

### 4. 面试表达

> 在真实 AI 应用里，我不会只让模型返回自然语言。
我会要求模型输出固定 JSON，并在服务端做字段、类型、枚举和业务规则校验。这样后续系统才能根据风险等级、是否缺少上下文等字段继续执行流程。
比如拦截高风险 SQL、提示补充表结构或进入人工审核。

## 生产实际

结构化输出是 AI 应用能接入业务流程的关键。金融信贷系统不能只拿一段自然语言答案继续处理，更需要固定字段，例如 `risk_level`、
`can_execute`、`reject_reason`、 `citations`、`sql` 和 `audit_id`。

生产里 JSON 合法只是第一步，还要做业务校验。比如 SQL 风险等级只能是固定枚举，是否允许执行必须是布尔值，拒绝原因不能缺失。
否则下游风控、审批或数据查询流程无法稳定消费。

## 面试沉淀

Q014：结构化输出为什么是 AI 应用里的关键能力？
重要程度：5/5

### 回答

结构化输出能把模型返回的自然语言变成程序可继续处理的数据。比如 SQL 解释助手需要稳定拿到 summary、tables、fields、risks 和
suggestions； RAG API 需要稳定返回 answer、citations、confidence 和 request_id。
没有结构化输出，后端很难做校验、展示、存储、评测和自动化流转。

Q015：JSON 合法和业务校验通过有什么区别？
重要程度：5/5

### 回答

JSON 合法只说明字符串能被解析，不代表内容满足业务要求。比如字段存在但类型错误、风险等级不在枚举里、citations 为空、SQL 表名没解析出来，
这些都属于业务校验失败。生产里要先做语法解析，再做字段、类型、枚举、必填项和业务约束校验。这样才能避免“格式看起来对，但系统实际不可用”的问题。
