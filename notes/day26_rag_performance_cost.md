## 大白话解释

RAG 成本主要来自两块：

```text
检索和重排的计算成本 + 大模型输入/输出 token 成本
```

初学时容易觉得“多塞资料更稳”。
但生产里不是这样。
如果每次都把 top 10、top 20 个 chunk 全部塞进 prompt，模型输入会变长，响应会变慢，费用会变高，
而且无关资料还可能干扰答案。

所以生产 RAG 要做上下文控制：

```text
先召回一批候选；
再过滤、重排、去重；
最后只把最有价值的少量 chunk 放进 prompt。
```

对于重复问题、高频 FAQ、固定口径问题，还可以加缓存。
命中缓存后，不必重复走完整的检索、重排和模型生成链路。

## 生产实际

真实公司里，RAG 成本优化通常分成几层：

| 层级 | 优化点 | 说明 |
|------|--------|------|
| 检索层 | 限制候选数量、metadata filter | 不把明显无关或无权限资料送进后续链路 |
| 重排层 | rerank、去重、低分过滤 | 减少进入 prompt 的噪声 |
| 上下文层 | token budget、chunk 合并 | 控制模型输入长度 |
| 模型层 | 大小模型分流、max tokens | 简单问题不用最贵模型 |
| 缓存层 | 问题缓存、召回缓存、答案缓存 | 高频问题复用结果 |
| 监控层 | token、延迟、缓存命中率 | 知道钱花在哪、慢在哪 |

生产里不会只看“回答对不对”。
还要看：

- 单次请求平均 token；
- 平均响应时间；
- P95 / P99 延迟；
- 缓存命中率；
- 单日模型费用；
- 无效召回比例；
- 改策略后正确率有没有下降。

## 常见坑

| 问题 | 表现 | 处理方式 |
|------|------|----------|
| top-k 过大 | prompt 很长、费用高、答案被噪声干扰 | 召回多一点，最终入 prompt 少一点 |
| chunk 重复 | 同一资料反复出现，浪费 token | 按 source 和内容 hash 去重 |
| 低相关 chunk 进入上下文 | 模型引用无关资料 | 设置最低相关分，必要时拒答 |
| 缓存过期 | 用户看到旧口径或旧政策 | 缓存 key 绑定知识库版本 |
| 只缓存答案 | 权限变化后可能越权复用 | 缓存 key 要包含用户角色和权限范围 |
| 不记录成本 | 不知道优化有没有效果 | 每次请求记录 input/output token、延迟和缓存状态 |

## 工程取舍

第一版 RAG 成本优化不要一上来做复杂平台。
先把四件事做清楚：

1. 记录每次请求的 input token、output token 和估算成本；
2. 限制最终进入 prompt 的 chunk 数量和 token 上限；
3. 对 source、chunk_id 或内容 hash 做去重；
4. 对高频、低风险、权限稳定的问题加缓存。

缓存不是越多越好。
如果问题和权限强相关，缓存 key 里就必须包含用户角色、业务域、知识库版本和 prompt 版本。
否则今天省下的成本，明天可能变成越权或旧答案事故。

## 面试沉淀

Q052：生产 RAG 怎么控制 token 成本和响应延迟？

### 回答

生产 RAG 控制成本和延迟，核心是减少无效上下文和重复计算。
检索阶段先用 metadata filter 缩小范围，避免无权限、无关文档进入候选。
重排阶段对候选 chunk 做 rerank、低分过滤和去重。
上下文组装阶段设置 token budget，只保留最相关、最有依据的少量 chunk。
模型调用阶段控制 max tokens，并对简单问题使用更便宜的模型。
对高频问题、固定口径和低风险 FAQ，可以缓存召回结果或最终答案。
同时每次请求都要记录 input token、output token、延迟、缓存命中率和 request_id，
这样才能判断优化有没有真实效果。

Q053：RAG 系统里缓存应该怎么设计，为什么不能只按问题文本缓存？

### 回答

RAG 缓存不能只按问题文本缓存，因为同一个问题在不同用户、权限、业务域和知识库版本下，答案可能不同。
缓存 key 至少要考虑规范化后的问题、用户角色、业务域、知识库版本、prompt 版本和模型版本。
高风险或权限敏感问题可以只缓存召回结果，不直接缓存最终答案。
缓存还要有 TTL 和失效机制，文档更新、权限变更或 prompt 变更后必须失效。
否则缓存虽然降低成本，但可能返回旧答案、错答案或越权答案。

Q054：为什么 RAG 不是 top-k 越大越好？

### 回答

top-k 越大，召回覆盖面可能更高，但进入上下文的噪声、token 成本和延迟也会增加。
无关 chunk 进入 prompt 后，模型可能被干扰，引用错误资料，甚至生成看似有依据但实际不准确的答案。
生产里通常会“召回多一点，最终使用少一点”：
先召回一批候选，再用权限过滤、rerank、去重和 token budget 选出少量高质量资料。
top-k 的选择要通过固定评测集验证，而不是凭感觉调大。

### 2. 为什么上下文不是越长越好？

### 3. RAG 缓存 key 为什么不能只用用户问题文本？
  我的回答：

### 5. 如何在准确率、延迟和成本之间做取舍？
  我的回答：

回答评价：

“多召回，少使用”是很实用的 RAG 成本策略。

低风险高频问题，比如固定 FAQ，可以使用缓存、较小模型和更短上下文。
工程上常用策略是先多召回候选，再通过权限过滤、rerank、去重和 token budget 少量使用。
每次调整 top-k、缓存、模型和上下文长度，都要用固定测试集验证质量没有明显下降。

### 7. [Day 24] 为什么“没有依据时拒答”比“尽量猜一个答案”更适合生产系统？
  我的回答：

拒答是可控制的，是 rag 设计重要一环，猜测是不可控的属于 rag 设计缺陷

回答评价：
回答正确。
你说清了拒答是可控安全行为，猜测是不可控风险。
可以再补充：生产系统更重视可信和可追溯，而不是表面上“什么都能答”。

参考答案：
没有依据时拒答比猜答案更适合生产系统，因为拒答是可解释、可审计、可回归测试的安全行为。
猜测答案看起来有帮助，但本质上没有资料支撑，可能误导用户、扩大错误决策，甚至触发合规风险。
生产 RAG 的目标是基于可信资料回答；
如果没有 citations 或检索依据，就应该说明无法判断，而不是用模型常识补全。

### 8. [Day 25] 为什么 RAG 不能只靠 prompt 防止敏感信息泄露？
  我的回答：

 prompt不能代替 rag 进行权限的设置，在元数据的控制时候就进行权限控制，
引用上下文， prompt本身也可能暴露用户不该访问的资料

回答评价：
回答正确。
你已经覆盖 prompt 不能替代权限控制、metadata 需要提前控制、上下文和引用都可能泄露。
这也是 Day 25 的核心安全点。

参考答案：
不能只靠 prompt 防止敏感信息泄露，因为 prompt 只是对模型的行为约束，不是权限系统。
如果敏感 chunk 已经进入上下文，模型可能直接引用、总结或推断出敏感内容。
用户也可能通过 prompt 注入诱导模型忽略规则。
生产里要在模型之前完成数据分级、入库脱敏、权限标签、检索过滤和上下文过滤。
prompt 只能作为最后一层提醒，不能作为唯一边界。

  我的回答：

回答评价：

你提到了 citations 会泄露资料是否存在、名称和路径。
