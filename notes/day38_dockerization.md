## 大白话解释

Docker 的价值是把运行环境打包起来。

本地运行时，你的机器上可能已经装了 Python、依赖包、环境变量和项目文件。
换一台机器后，这些东西不一定存在。
Dockerfile 把这些环境要求写成可重复执行的步骤，让别人能按同样方式构建和启动服务。

```text
-> 健康检查和日志监控
```

Day 38 先完成本地 Dockerfile。
它不是最终生产部署，但能证明服务已经具备标准化运行入口。

## 面试沉淀

Q088：Docker 化解决了 AI 应用交付里的什么问题？

### 回答

Docker 化解决的是运行环境可复现和交付一致性问题。
AI 应用常依赖 Python 版本、系统库、模型 SDK、环境变量和项目文件结构。
如果只说“我本机能跑”，换机器、换环境或部署到服务器时很容易失败。
Dockerfile 把依赖安装、代码复制、启动命令和端口暴露写成标准步骤，
让开发、测试和部署使用同一套运行方式。
但 Docker 化不等于生产完备，生产还需要配置管理、密钥管理、日志、监控、健康检查和发布流程。

### 2. 为什么不能只说“本地能跑”就算可部署？
  我的回答：
本地部署不能保证换一个环境就可以跑到通，需要相同的依赖，环境，配置等

回答评价：
通过。你准确说明了本地可运行不代表其他环境具备相同依赖、配置和运行条件。
可以再补充：可部署还要求启动方式、健康检查、日志、配置和故障排查都清楚。

参考答案：
不能只说“本地能跑”就算可部署，因为本地机器上可能已经安装了依赖、配置了环境变量、存在演示文件或使用了特定路径。
换到测试、生产或其他同事机器后，这些条件不一定存在。
真正可部署需要明确依赖来源、启动命令、端口、配置项、健康检查、日志输出和错误排查方式。
Dockerfile 能把运行环境和启动入口标准化，但生产还需要配置管理、密钥管理、监控和发布流程。

### 5. 镜像里为什么不能写死密钥？
  我的回答：
密钥需要单独配置，切换不同环境还需要重新修改，上传到 git 也照成内部环境泄露，

回答评价：
通过。你指出了密钥应单独配置，以及写死后会带来环境切换和泄露风险。
表达上注意“造成”不要写成“照成”。

参考答案：
镜像里不能写死密钥，因为镜像会被构建、分发、缓存和推送到镜像仓库。
如果把数据库密码、模型 API key 或内部服务 token 写进镜像，镜像一旦泄露，密钥也会泄露。
同时，不同环境的密钥不同，写死密钥会导致镜像无法在 dev、test、prod 之间安全复用。
生产里应该通过环境变量、密钥系统或配置中心在运行时注入密钥。

### 6. [Day 36] 后端服务为什么需要清晰目录结构？
  我的回答：
不同层级结构负责不同的功能，接口，响应，业务编排，审计存储，错误解决等

回答评价：
通过。回答说明了清晰目录结构用于拆分接口、业务编排、审计存储和错误处理等职责。
可以再补充：目录清晰也有利于测试、扩展和后续替换实现。

参考答案：
后端服务需要清晰目录结构，是为了把不同职责拆开。
API 层负责路由和请求响应，schema 层负责数据结构，service 层负责编排业务，storage 层负责审计存储，
config 层负责配置，errors 层负责统一错误处理。
如果所有逻辑都堆在一个文件里，后续改接口、换存储、接真实 NL2SQL 链路或写测试都会更困难。

### 7. [Day 37] 为什么安全阻断不应该简单返回 500？
  我的回答：
安全阻断是本身策略和 500是服务器错误可能是系统错误有 bug

回答评价：
通过。你正确区分了安全阻断是策略结果，500 是服务器内部错误。
可以再补充：安全阻断应该用业务状态表达，例如 `final_status=safely_blocked`。

参考答案：
安全阻断不应该简单返回 500，因为安全阻断是系统安全策略按预期生效，不是服务故障。
例如敏感字段、危险 SQL、无权限查询或高成本查询被拦截，都应该返回清晰的业务状态和阻断原因。
HTTP 500 应该用于代码 bug、依赖故障、配置错误或数据库连接失败。
如果把安全阻断返回 500，调用方会误以为系统故障，也无法向业务用户解释合规边界。

### 8. [Day 37] 为什么接口要返回 request_id？
  我的回答：
 根据request_id来回溯整个链路，如果有错误可以进行追溯

回答评价：
通过。回答抓住了 `request_id` 的核心作用：回溯链路和定位错误。
可以再补充：`request_id` 还用于审计、日志关联、trace 查询和 bad case 复盘。

参考答案：
接口要返回 `request_id`，是为了让每一次请求都能被追踪和回放。
NL2SQL 链路包含问题解析、SQL 生成、SQL 校验、查询执行、结果解释和审计写入。
当用户反馈答案错误、安全阻断有争议或接口异常时，研发可以用 `request_id` 查日志和 trace，
定位问题发生在哪一层。
在金融信贷场景里，`request_id` 也用于合规审计和敏感查询追踪。
