## 今日核心知识点

### 1. RAG 是什么

RAG 全称是 Retrieval-Augmented Generation，中文叫“检索增强生成”。

大白话：

> 先从知识库里找相关资料，再让模型基于资料回答。

它解决的问题是：模型本身不知道你的项目文档、公司口径、表结构和最新规则，所以需要先检索外部资料。

### 2. 为什么先整理资料

如果知识库资料质量差，后续再好的模型也容易答错。

例如：

- 文档过期，模型会基于旧口径回答
- 文档冲突，模型不知道信哪一个
- 文档太长，检索不到重点
- 文档没有来源，回答无法追溯

所以今天先做资料清单。

### 3. 第一批 RAG 项目范围

推荐范围：

```text
SQL 解释助手知识库
```

它包含：

- SQL 解释助手项目说明
- SQL 风险规则
- Tool Use 说明
- 结构化输出说明
- 项目讲解提纲
- 术语表

这个范围小、清晰、和你的数据仓库背景强相关。

## 生产实际

RAG 准备阶段最重要的是资料治理。金融信贷公司不会把所有文档直接丢进向量库，而是要先筛选授信政策、风控规则、还款说明、贷后流程、表结构、指标口径和 FAQ，
并标记来源、版本、权限和适用范围。

资料清单做得好，后续切分、检索、引用和拒答才有基础。资料清单做得差，模型会基于过期、重复或无权限资料回答，生产风险很高。

## 常见坑

- 表格行 1
  - 类型：目标
  - 可能的问题：只完成 Demo，不知道对应真实项目环节
  - 生产处理方式：把“RAG 准备：文档资料整理与知识库清单”映射到 RAG、NL2SQL、SQL 解释助手或服务化链路
- 表格行 2
  - 类型：数据
  - 可能的问题：输入样例过干净，真实数据有缺失、重复、格式不统一
  - 生产处理方式：增加校验、异常分支和边界样例
- 表格行 3
  - 类型：工程
  - 可能的问题：代码能跑但缺少日志、配置、错误提示和复盘材料
  - 生产处理方式：保留 README、输出文件、运行命令和关键结果
- 表格行 4
  - 类型：安全
  - 可能的问题：忽略密钥、权限、敏感字段或外部调用风险
  - 生产处理方式：使用环境变量、权限控制、脱敏和审计记录
- 表格行 5
  - 类型：维护
  - 可能的问题：只写临时代码，后续无法扩展或讲清楚
  - 生产处理方式：按模块拆分，并补充大白话注释说明设计原因

## 工程取舍

这一天优先采用最小可运行方案，是为了先建立稳定基线。真实项目里不能一开始就追求复杂架构，应该先把输入、处理逻辑、输出和失败场景跑通，再逐步增加模型调用、 检索、
数据库、权限和评测。

对“RAG 准备：文档资料整理”这类能力，学习阶段更看重可解释和可复盘；生产阶段再根据准确率、延迟、成本、安全和维护成本决定是否引入更复杂的方案。

## 面试沉淀

Q001：生产环境里的 RAG 系统一般怎么设计？

### 回答

生产 RAG 通常不是简单的向量检索加模型生成。它会分成文档接入、解析清洗、切分、embedding、索引存储、在线检索、权限过滤、重排、上下文构造、LLM
生成、引用返回、 日志监控和评测优化。核心目标是让模型基于可信资料回答，并且答案可追溯、可控制、可排查。面试里要强调 RAG 是工程系统，
不只是一个模型调用技巧。
