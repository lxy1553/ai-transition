## 大白话解释

Day 35 的重点不是再学一个新概念，而是把前面分散的能力串起来。

单独看，问题解析、SQL 生成、SQL 校验、查询执行、结果解释都只是一个小模块。
但业务用户真正需要的是：

```text
我问一句业务问题，系统能安全查数，并给我一个能看懂的答案。
```

所以 NL2SQL 助手必须像一条流水线：

- 先听懂用户问什么；
- 再找到该查的表和字段；
- 再生成只读 SQL；
- 再检查 SQL 是否安全；
- 再执行安全 SQL；
- 最后把结果解释成业务语言。

如果中间任何一步有风险，比如导出手机号、缺少时间范围、危险 SQL，就应该安全阻断，
并告诉用户怎么改问，而不是硬查数据库。

## 生产实际

金融信贷场景里的 NL2SQL 助手通常服务于运营、风控、产品、贷后和管理层。

他们可能会问：

- 昨天授信申请量是多少？
- 上周每个渠道的授信通过率是多少？
- 最近 7 天放款金额趋势怎么样？
- 本周逾期率比上周变化多少？
- 查询某个申请编号的审批状态；
- 导出客户手机号列表。

这些问题不能全部一视同仁。
汇总指标可以自动查，客户敏感信息必须拦截或审批，明细查询必须限制权限和返回行数，
大表查询必须带时间范围。

生产版本通常会拆成这些模块：

| 模块 | 职责 |
|------|------|
| Schema Catalog | 管理可用表、字段、指标、维度、权限和口径 |
| Question Parser | 抽取指标、维度、时间、过滤条件和风险标记 |
| SQL Generator | 基于 Schema 和问题类型生成只读 SQL |
| SQL Validator | 校验只读、表字段白名单、敏感字段、时间范围、limit 和成本 |
| Query Executor | 使用只读连接执行安全 SQL，并控制超时和返回行数 |
| Result Interpreter | 把结果解释成业务语言，补充风险提示和建议追问 |
| Audit Logger | 记录用户、问题、SQL、状态、耗时、行数和 request_id |

Day 35 的本地版本先不做真正 API，而是把这些阶段的 JSON 产物串成一个可演示报告。
这能证明项目链路完整，也为 Day 36 后端重构做准备。

## 常见坑

| 类型 | 可能的问题 | 生产处理方式 |
|------|------------|--------------|
| 只演示成功样例 | 看起来能查数，但看不出安全能力 | 同时展示成功查询和安全阻断 |
| 所有逻辑堆一起 | 后续难测试、难替换、难排查 | 保持解析、生成、校验、执行、解释分层 |
| 没有链路状态 | 不知道失败发生在哪一步 | 每一步都记录状态、原因和产物 |
| 忽略 Day 34 解释层 | 只返回 SQL 或表格，业务不可读 | 输出业务回答、关键发现、风险提示 |
| 不记录 bad case | 项目讲不出边界和改进方向 | 保留阻断样例、执行失败和低置信度样例 |
| 没有演示入口 | 面试时只能零散讲代码 | 提供一键报告和单问题演示命令 |

原因：

- 整合日的重点是检查模块契约，而不是重写每个模块；

- 后续服务化时，可以把这些 JSON 契约改造成 API response schema；

```text
-> interpreter service

```

## 面试沉淀

Q083：一个生产级 NL2SQL 助手应该包含哪些模块？
重要程度：5/5

### 回答

一个生产级 NL2SQL 助手不应该只有“自然语言生成 SQL”这一步。
它至少应该包含 Schema Catalog、问题解析、SQL 生成、SQL 校验、权限控制、查询执行、
结果解释、审计日志和评测回归。
Schema Catalog 提供真实表字段、指标口径和权限标签；
问题解析负责抽取指标、维度、时间和风险；
SQL 生成负责在 Schema 约束下生成只读 SQL；
SQL 校验负责拦截危险、越权和高成本查询；
查询执行层使用只读连接和超时限制查数；
结果解释层把数据转成业务语言；
审计和评测则保证线上可追溯、可持续优化。

Q084：你会怎么演示一个 NL2SQL 项目？
重要程度：5/5

### 回答

演示 NL2SQL 项目时，不能只展示一个成功生成 SQL 的例子。
我会准备三类样例：正常指标查询、复杂对比或趋势查询、安全阻断查询。
演示顺序是先输入业务问题，再展示问题解析结果，包括指标、维度、时间和风险标记；
然后展示生成的只读 SQL，说明它如何受到 Schema Catalog 约束；
接着展示 SQL Validator 如何检查字段白名单、敏感字段、时间范围和 limit；
然后展示查询执行结果和业务解释；
最后展示被阻断样例，比如导出手机号或缺少时间范围的大表查询，说明系统如何保护权限、隐私和成本。

Q085：NL2SQL 项目整合时，为什么要保留每一层的中间结果？
重要程度：4/5

### 回答

NL2SQL 项目整合时必须保留每一层的中间结果，因为问题可能出在任何阶段。
如果最终答案错了，需要知道是问题解析漏了时间，Schema Router 选错表，SQL Generator 生成错字段，
SQL Validator 没拦住风险，Executor 执行方言不兼容，还是 Interpreter 解释过度。
保留中间结果可以支持调试、审计、回放和评测。
生产里还可以用 request_id 把用户问题、解析结果、SQL、校验结果、执行状态、返回行数和业务解释串起来，
这样 bad case 才能被定位和修复。
