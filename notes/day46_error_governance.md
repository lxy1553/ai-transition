## 大白话解释

错误治理就是把系统失败当成工程问题来处理。

不要只说：

```text
这个问题答错了，改一下 Prompt。
```

而是要问：

```text
错在哪一层？
是问题没识别清楚？
是工具路线选错了？
是缺少条件却继续查库？
是 RAG 没有引用还在回答？
是 SQL 没校验就执行？
是数据库失败后模型还在编？
```

只有把失败归因清楚，修复才会有效。
否则很容易出现一种情况：今天改 Prompt 修好了一个问题，明天另一个问题又被改坏。

## 生产实际

真实公司里，AI 应用上线后一定会持续产生 bad case。
这些 bad case 可能来自评测集、线上日志、业务反馈、客服工单、安全审计或面试演示前自测。

金融信贷 Agent 常见错误治理方式如下：

| 失败类型 | 例子 | 应该修哪里 |
|----------|------|------------|
| 工具路由错误 | 缺少时间范围却继续执行 SQL | 路由规则和澄清策略 |
| 引用缺失 | 风控规则没有资料来源却回答 | RAG 检索、citation 校验和无依据拒答 |
| 前置条件缺失 | query_executor 跳过 SQL Validator | 工具注册表和编排层硬约束 |
| 异常误判 | 数据库超时被包装成正常结果 | 异常分类、最终状态和解释层边界 |
| Prompt 漂移 | 改完 Prompt 后老问题退化 | Prompt 版本管理和回归评测 |

生产里更稳的做法是把修复分层。
Prompt 可以改表达和任务约束，但执行顺序、权限、安全、SQL 校验和系统异常必须用代码规则兜住。

## 常见坑

| 类型 | 可能的问题 | 生产处理方式 |
|------|------------|--------------|
| 归因粗糙 | 所有失败都归为模型不好 | 按责任层拆分失败原因 |
| 只修单例 | 修好当前问题，不看其他样例 | 每次修复后跑全量评测 |
| 过度改 Prompt | 用 Prompt 约束 SQL 执行顺序 | 执行类前置条件必须用代码控制 |
| 忽略异常 | 系统超时后让模型继续解释 | 异常状态直接阻断解释层 |
| 无版本记录 | 改了什么没人知道 | 记录修复计划、责任层和回归结果 |
| 忽略安全 | 敏感查询靠模型自觉拒答 | 权限、敏感字段和审计由规则兜底 |

## 工程取舍

### 取舍一：什么时候改 Prompt？

当问题出在表达、任务约束、输出格式、解释口径或回答风格时，可以改 Prompt。
例如结果解释层过度推断原因，可以在 Prompt 中明确“只能基于 rows、columns 和 citations 解释”。

但如果问题是跳过 SQL 校验、越权查库、缺少时间范围仍执行，就不应该只改 Prompt。
这些是编排层和工具前置条件的责任。

### 取舍二：为什么修复后要跑全量评测？

因为 AI 应用很容易局部变好、整体变差。
你修了一个无引用拒答问题，可能让正常规则解释也拒答太多；
你加强了时间范围限制，可能让原本合法的“近 7 天”查询被误判。

全量回归能发现这种退化。
Day 46 的本地练习就是把 Day 45 的 24 条样例重新跑一遍，验证修复是否影响旧能力。

### 取舍三：为什么要把修复写成计划？

修复计划能让团队知道：

- 哪个 case 失败；
- 失败属于哪一层；
- 修复方式是什么；
- 修复后用什么评测保护；
- 后续如果退化，应该回看哪条规则。

这比“我调了一下 Prompt”更接近生产工程实践。

## 面试沉淀

Q097：AI Agent 出现 bad case 后，你会怎么做错误治理？

### 回答

AI Agent 出现 bad case 后，我不会直接只改 Prompt，而是先做失败归因。
第一步是保留原始问题、工具路线、中间结果、最终状态、引用、SQL、校验结果和 request_id。
然后判断错误发生在哪一层：意图识别、工具路由、RAG 召回、SQL 生成、SQL 校验、查询执行、结果解释还是异常处理。

如果是缺少时间范围却继续查询，就修工具路由和澄清规则；
如果是 RAG 没有引用仍回答，就修 citation 校验和无依据拒答；
如果是实时问题误走离线 SQL，就修意图识别和工具路由；
如果是实时链路延迟被包装成正常答案，就修实时状态校验和异常分类。
Prompt 只适合修任务表达、输出格式和解释约束，不能替代权限、安全和执行前置条件。

修复后必须把 bad case 加入固定评测集，并跑全量回归。
通过率、失败类型、拒答率和安全阻断结果都要对比修复前后。
在金融信贷场景里，这一步尤其重要，因为授信、风控、逾期和客户明细都涉及权限、合规和审计。
错误治理的目标不是让某一条样例看起来答对，而是让系统在同类问题上稳定、可控、可追溯。

Q102：为什么 AI Agent bad case 必须按责任层归因？

### 回答

AI Agent bad case 必须按责任层归因，因为不同层的问题修法完全不同。
如果实时问题误走离线 SQL，核心问题在意图识别和工具路由；
如果缺少时间范围还继续查询，问题在澄清策略和执行前置条件；
如果口径解释没有引用仍回答，问题在 RAG 召回、citation 校验和无依据拒答；
如果实时链路延迟还输出正常结论，问题在实时状态校验和异常分类。

如果不做责任层归因，团队很容易把所有问题都说成“模型不好”或“改 Prompt”。
Prompt 可以修表达、任务约束和输出格式，但不能替代权限控制、SQL 校验、工具前置条件、
实时延迟检测和审计记录。
这些生产边界必须由确定性工程规则兜住。

按责任层归因还能让修复可回归。
每个 bad case 都要记录原始问题、工具路线、中间结果、最终状态、失败类型、责任层、修复动作和回归样例。
修复后再跑固定评测集，确认该类问题稳定解决，同时没有让正常查询、安全阻断或实时告警解释退化。

  我的回答：

  回答评价：

通过。你已经说清楚“实时意图识别”和“实时路线”。可以再补充禁止路线：实时问题不能进入 offline_sql_generator 和 offline_query_executor。

  参考答案：

修复时要把实时信号识别出来，强制路由到 realtime_metric_tool 或 alert_query_tool，并禁止离线 SQL 链路。

### 4. 实时链路延迟却给出正常结论，为什么属于实时状态校验和异常分类问题？
  我的回答：

  回答评价：

通过，但偏短。关键是实时链路延迟时，数据本身不可作为当前事实，最终状态应该是 execution_failed 或明确延迟，而不是正常回答。

  参考答案：

### 7. 为什么实时延迟问题不能只靠结果解释层兜底？
  我的回答：

结果解释层是模型识别的，容易模糊曲解，涉及到安全边界权限问题，需要代码来兜底，结构化输入输出，
  经过结构化校验，最大调用次数、失败回退和审计记录。

  回答评价：

通过。你提到了模型解释不稳定和代码兜底。这里重点是实时延迟属于数据可用性状态，应该在实时工具或状态校验层确定，不应该交给解释层自由发挥。

  参考答案：

实时延迟不是“怎么解释”的问题，而是“数据是否可用”的问题。
如果实时链路延迟，结果解释层不能把过期数据解释成当前状态。
应该由实时指标工具或状态校验层返回结构化状态，例如 `realtime_delay` 或 `execution_failed`，再由解释层按这个状态说明原因并记录审计。
