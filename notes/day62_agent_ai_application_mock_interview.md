## 常见坑

| 类型 | 可能的问题 | 生产处理方式 |
|------|------------|--------------|
| 回答过长 | 先把重点说丢了 | 先给一句结论，再展开 |
| 只有框架名 | 听起来空泛 | 回到你的项目和数据链路 |
| 只说做了什么 | 没说为什么这样做 | 补工程取舍和风险 |
| 没有数字 | 说服力弱 | 引用评测覆盖、工具数、规则数 |
| 不敢承认边界 | 显得不真实 | 说明限制和后续扩展方向 |

## 工程取舍

### 取舍一：为什么回答里不要先讲框架名？

因为面试官更关心你解决了什么问题。
框架是手段，不是项目价值本身。

### 取舍二：为什么模拟面试要保留 bad case 问题？

因为大多数深挖都发生在失败和风险场景。
成功路径大家都能讲，bad case 才能区分工程成熟度。

## 面试沉淀

Q141：为什么金融信贷 Agent 项目不能只强调用了 LangChain / LangGraph / LangSmith？
重要程度：5/5

### 回答

因为框架只是实现手段，业务价值来自数据边界、工具路由、安全控制、审计和评测。
如果只强调框架名，面试官会认为你是在堆技术词，而不是解决真实场景问题。

Q142：模拟面试里为什么一定要准备 bad case 和安全问题？
重要程度：5/5

### 回答

因为生产系统最容易在异常和风险场景暴露问题。
面试官也常用这类问题区分“会搭 Demo”和“理解生产约束”的候选人。

## 术语更新

- 模拟面试：按真实面试节奏对项目进行结构化问答练习。
- 追问题：面试官在第一层回答后继续下钻的深入问题。
- 短答案：30 秒内说清核心判断的回答形式。
- 展开答案：在短答案之后补充链路、取舍和风险的完整回答。
