## 大白话解释

查询执行层就是 NL2SQL 系统里真正“查数据库”的那一层。

前面几天做的是理解问题、选表字段、生成 SQL 和检查风险。这些都通过以后，执行层才可以拿 SQL 去数据库里查数。

但是执行层不是简单把 SQL 丢给数据库。它还要确认：

- 这条 SQL 是不是已经通过校验；
- 连接的是不是只读库；
- 返回行数会不会太多；
- 查询有没有超时；
- 结果字段和类型能不能被前端展示；
- 失败时能不能返回清楚的错误，而不是直接把数据库异常暴露给用户。

如果 Day 32 已经拦截了某条 SQL，Day 33 就必须跳过它。执行层不能因为“想试试看”再去执行高风险 SQL。

## 生产实际

在金融信贷数据问答里，查询执行层通常接在 SQL Validator 和权限系统后面。

典型链路是：

```text
用户问题
-> 问题解析
-> Schema Router
-> SQL 生成
-> SQL 校验
-> 权限校验
-> 查询执行
-> 结果格式化
-> 结果解释
-> 审计日志
```

真实公司里，执行层一般不会直接连生产主库。更常见的是连接数仓、只读副本、查询网关或内部 BI 查询服务。

原因很直接：

- 主库不能被临时分析查询拖慢；
- 明细和敏感数据必须受权限控制；
- 大查询要有超时、扫描量和返回行数限制；
- 每次查询要留下 request_id、用户、SQL、耗时和返回行数；
- 出问题后要能回放链路，判断是问题解析错、SQL 生成错、校验漏了，还是执行环境问题。

在信贷场景里，正常问题可能是“上周每个渠道的授信通过率是多少”，执行层可以返回按渠道分组的表格。
如果问题是“导出客户手机号列表”，前面应该已经阻断，执行层不能再访问客户维表。

## 常见坑

- 表格行 1
  - 类型：绕过校验
  - 可能的问题：执行层直接接收用户 SQL
  - 生产处理方式：只接受 SQL Validator 放行的任务 ID 或签名结果
- 表格行 2
  - 类型：连接错库
  - 可能的问题：临时查询打到生产主库
  - 生产处理方式：强制使用只读账号、只读副本和查询网关
- 表格行 3
  - 类型：无行数限制
  - 可能的问题：明细查询返回几十万行
  - 生产处理方式：设置 `limit`、分页、导出审批和最大结果大小
- 表格行 4
  - 类型：超时不可控
  - 可能的问题：大范围聚合长时间运行
  - 生产处理方式：设置查询超时、扫描量限制和异步任务
- 表格行 5
  - 类型：错误暴露
  - 可能的问题：把数据库异常原样返回给用户
  - 生产处理方式：对外返回业务错误码，内部日志保留详细错误
- 表格行 6
  - 类型：结果不可读
  - 可能的问题：只返回裸 JSON，业务看不懂
  - 生产处理方式：按查询类型格式化成指标卡、趋势、TopN 或明细表

## 工程取舍

Day 33 使用 SQLite 和固定样例数据，不直接接真实数据库。

这样做的原因：

- 本地练习可以稳定复现，不依赖外部服务；
- 能先练清楚执行层的输入输出契约；
- 可以验证 Day 32 通过的 SQL 确实能被执行；
- 可以明确哪些 SQL 被跳过，避免把风险样例误执行；
- 后续接 FastAPI 或真实数仓时，只需要替换执行适配层。

生产里更完整的执行层会包含：

```text
任务接收：只接收已校验通过的 SQL
权限确认：确认当前用户能查对应表、字段和数据范围
执行控制：只读连接、超时、最大行数、扫描量限制
结果格式化：指标卡、表格、趋势、对比、明细
审计日志：记录用户、SQL、耗时、行数、状态和 request_id
错误处理：对外返回安全错误，对内保留排查信息
```

## 面试沉淀

Q077：NL2SQL 查询执行层为什么必须放在 SQL 校验之后？

### 回答

查询执行层必须放在 SQL 校验之后，因为执行层一旦访问数据库，风险就从“生成错误”变成了真实的数据泄露、性能冲击或误操作风险。
SQL 生成层可能产生字段越权、缺少时间范围、全表扫描、敏感字段查询或危险关键字。这些问题必须先由 SQL Validator、权限系统和成本控制拦住。
执行层只应该接收已经通过校验的 SQL 或查询任务，使用只读连接执行，并记录执行状态、耗时、返回行数和审计信息。

Q078：NL2SQL 查询执行层要控制哪些生产风险？

### 回答

查询执行层要控制安全、性能、稳定性和可追溯四类风险。安全上，要确保只读账号、权限校验和敏感字段拦截已经生效。
性能上，要限制超时、扫描量、返回行数、分页和大结果导出。稳定性上，要处理数据库异常、连接失败、SQL 方言不兼容和空结果。
可追溯上，要记录 request_id、用户、SQL、参数、执行耗时、返回行数和失败原因。
在金融信贷场景里，这些控制能避免客户隐私泄露、 生产库被拖慢和错误指标影响业务判断。

Q079：NL2SQL 返回结果为什么不能只返回数据库原始行？

### 回答

不能只返回数据库原始行，因为业务用户通常需要的是可理解、可展示、可解释的答案。
同样是 SQL 结果，指标查询适合返回指标卡，趋势查询适合返回按日期排列的序列， TopN 适合返回排序表，对比查询适合返回当前值、上期值和差值，
明细查询要控制字段和行数。如果只返回原始行，前端不知道如何展示，业务用户也不一定能理解口径。
生产里执行层应该返回结构化结果，包括 `response_type`、`columns`、`rows`、`row_count`、`summary_text`
和必要的审计信息，再交给结果解释层转成业务语言。

### 8. [Day 32] SQL 校验层通常要检查哪些内容？
我的回答：
静态校验，schema 表结构 验证，字段白名单；
安全校验，权限等问题；
性能校验，有没有 limit 限制，大表扫描，时间是否加了条件

回答评价：
通过。回答结构清楚，覆盖静态校验、Schema、字段白名单、安全权限、limit、大表扫描和时间条件。

参考答案：
SQL 校验层通常检查三类内容。
第一是静态和 Schema 校验，包括是否只读、是否有危险关键字、表是否在白名单里、字段是否存在。

第三是性能和成本校验，包括大表是否带时间范围或分区条件，明细查询是否有精确过滤和 limit，
