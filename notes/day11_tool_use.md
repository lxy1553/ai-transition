## 大白话解释

“函数调用概念：Tool Use”这一天要解决的是一个很具体的工程问题：先把基础能力、工具或项目环节讲清楚，再把它变成能运行、能检查、能复盘的产物。
学习时不要只看代码是否跑通，还要能说清楚它为什么需要、输入输出是什么、出错后怎么定位。

## 真实工作场景

继续沿用 Day 10 的场景：**SQL 上线前风险检查助手**。

用户提交一段 SQL：

```sql
select city, count(distinct user_id) as active_users
from user_events
where dt between '2026-05-01' and '2026-05-07'
  and event_name = 'app_open'
group by city;
```

如果只让模型直接回答，它可能会凭经验解释，但它不一定知道：

- `user_events` 表结构是什么
- `dt` 是不是分区字段
- `event_name = 'app_open'` 是否就是活跃口径
- 这段 SQL 有没有命中公司内部规范

所以真实系统里，模型不应该自己硬猜。更合理的流程是：

1. 模型判断需要哪些工具
2. 系统调用工具
3. 工具返回结构化结果
4. 模型基于工具结果生成最终回答

### 任务 2：定义一个工具

一个工具至少要说明：

- 表格行 1
  - 字段：`name`
  - 说明：工具名称
- 表格行 2
  - 字段：`description`
  - 说明：工具能做什么
- 表格行 3
  - 字段：`parameters`
  - 说明：需要哪些参数
- 表格行 4
  - 字段：`result`
  - 说明：返回什么结果

示例：

```json
{
  "name": "check_sql_risk",
  "description": "检查 SQL 是否存在常见数仓风险",
  "parameters": {
    "sql": "用户提交的 SQL",
    "dialect": "SQL 方言，例如 hive/mysql"
  },
  "result": {
    "risk_level": "low / medium / high",
    "risks": ["风险列表"],
    "suggestions": ["修改建议"]
  }
}
```

## 今日核心知识点详解

### 1. Tool Use 是什么

大白话：

> 模型像一个会判断问题的调度员，工具像真正干活的专业工种。

比如用户问：

```text
这段 SQL 能上线吗？
```

模型不应该直接拍脑袋说“能”或“不能”。

它应该判断：

```text
我需要调用 SQL 风险检查工具。
```

工具检查完后返回：

```json
{
  "risk_level": "medium",
  "can_publish": false,
  "risks": ["缺少指标口径", "group by 可能有聚合开销"]
}
```

模型再把这个结果解释成人能看懂的话。

### 2. 为什么 Tool Use 很重要

因为真实项目里很多事情模型不能瞎猜：

- 数据库里有没有这张表
- 字段是不是存在
- 指标口径是什么
- SQL 是否真的能执行
- 用户有没有权限
- 当前数据质量是否正常

这些都应该由工具或系统查询出来。

### 3. Tool Use 和 Day 10 的关系

Day 10 学的是：工具或模型结果要返回稳定 JSON。

Day 11 学的是：模型什么时候该调用工具，以及工具怎么接入流程。

两者连起来就是：

```text
模型判断需要工具
-> 工具返回结构化 JSON
-> 系统校验 JSON
-> 模型生成最终解释
```

### 4. 面试表达

> 我理解的 Tool Use 是让模型负责意图理解和流程调度，把查数据库、查表结构、SQL 风险校验这类确定性任务交给工具。
工具返回结构化结果后，再由模型生成面向用户的解释。这样可以减少模型幻觉，也方便系统做权限、校验和审计。

## 生产实际

Tool Use 在生产里是让模型做调度，工具做确定性工作。金融信贷场景中，模型可以决定是否调用“查授信政策”“查 Schema Catalog”“检查 SQL
风险”“权限判断”等工具，但真正的查询、校验和拦截应该由工具完成。

这样做可以减少模型幻觉，也方便审计。例如用户问客户手机号时，工具层可以根据权限直接拒绝，不把敏感数据交给模型自由处理。

这一天优先采用最小可运行方案，是为了先建立稳定基线。真实项目里不能一开始就追求复杂架构，应该先把输入、处理逻辑、输出和失败场景跑通，再逐步增加模型调用、 检索、
数据库、权限和评测。

对“函数调用 / Tool Use：让模型学会用工具”这类能力，学习阶段更看重可解释和可复盘；
生产阶段再根据准确率、延迟、成本、安全和维护成本决定是否引入更复杂的方案。

## 面试沉淀

Q016：Tool Use 在生产 AI 应用里怎么设计？
重要程度：5/5

### 回答

Tool Use 的核心是让模型不要只靠生成文本，而是按规则调用外部工具或业务函数。
生产设计时要明确工具 schema、参数校验、权限控制、超时、错误返回和调用日志。
模型可以负责判断何时调用工具，但工具执行结果必须由系统校验和封装，不能完全相信模型。
在 SQL 检查、知识检索、数据查询和工单操作场景里，Tool Use 能把 LLM 连接到真实业务动作。
