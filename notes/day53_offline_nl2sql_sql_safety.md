## 大白话解释

NL2SQL 不是“把中文交给模型，让模型写 SQL”这么简单。

在金融信贷场景里，SQL 一旦执行，就可能带来这些风险：

- 查错表，导致指标口径不一致；
- 漏分区，扫描大表，拖慢数仓；
- 查 DWD 明细，暴露客户手机号和身份证号；
- 查实时问题时误走离线日报；
- 生成非只读 SQL，误删或误改数据；
- 没有时间范围，结果含义不清；
- 结果查出来了，但解释时编造业务原因。

所以生产级 NL2SQL 链路必须是：

```text
用户问题
-> 主题域和意图识别
-> 指标字典确认口径
-> Schema Catalog 找 ADS/DWS 表和字段
-> 生成候选 SQL
-> SQL Validator 校验
-> 查询执行
-> 结果解释
-> 审计记录
```

SQL 生成只是中间步骤，不是终点。

## 生产实际

用户问：

```text
近 7 天各渠道授信通过率是多少？
```

Agent 应该先判断：

- 主题域：授信申请；
- 意图：离线指标查询；
- 指标口径：授信通过率 = approved_cnt / apply_cnt；
- 表选择：渠道趋势优先 DWS；
- 分区字段：`dt`；
- 时间范围：近 7 天；
- 是否敏感：不涉及客户明细；
- 是否只读：只能 SELECT；
- 扫描成本：7 天分区，可接受。

然后才能生成并执行类似 SQL：

```sql
SELECT
  dt,
  channel,
  SUM(approved_cnt) * 1.0 / SUM(apply_cnt) AS approval_rate
FROM dws_credit_apply_channel_1d
WHERE dt BETWEEN '2026-05-28' AND '2026-06-03'
GROUP BY dt, channel;
```

如果用户问：

```text
导出昨天所有授信申请客户的手机号和身份证号。
```

即使模型能生成 SQL，也必须阻断。
因为这会访问 DWD 明细和敏感字段，不属于普通离线指标查询。

## Day 53 SQL Validator 设计

今天的 SQL Validator 先检查 6 类问题。

| 校验项 | 为什么重要 | 处理方式 |
|--------|------------|----------|
| 只读 SQL | 防止误删、误改数据 | 非 SELECT 阻断 |
| 表是否在 Catalog | 防止编造表或越权表 | 未登记表阻断 |
| ADS/DWS 优先 | 降低口径和敏感风险 | 普通指标不查 DWD/ODS |
| 时间范围和分区 | 控制扫描成本和结果范围 | 缺少则澄清或阻断 |
| 敏感字段 | 防止手机号、身份证号泄露 | 命中即阻断 |
| 扫描天数 | 防止大范围扫数仓 | 超过阈值阻断 |

生产里还会继续增加：

- SQL AST 解析；
- LIMIT 和 ORDER BY 约束；
- 聚合维度校验；
- join 白名单；
- 用户权限校验；
- 结果行数上限；
- 查询超时控制；
- 审计日志和 request_id。

## 常见坑

| 类型 | 可能的问题 | 生产处理方式 |
|------|------------|--------------|
| SQL 生成后直接执行 | 模型可能编造表字段或越权 | 必须经过 SQL Validator |
| 查 DWD 明细 | 暴露客户敏感信息 | 普通指标优先 ADS/DWS |
| 缺少时间范围 | 结果不清晰，扫描全表 | 先澄清，不执行 |
| 漏分区 | 成本高，拖慢集群 | 强制分区字段 |
| 大范围扫描 | 影响数仓稳定性 | 设置最大扫描天数 |
| 口径和 SQL 脱节 | SQL 能跑但业务错 | 先查指标字典确认口径 |
| 结果解释编造原因 | 用户误以为系统已查到原因 | 解释只能基于查询结果和口径 |

## 工程取舍

### 取舍一：为什么优先 ADS/DWS？

ADS 和 DWS 更接近业务指标，通常已经做过清洗、聚合和口径沉淀。
Agent 查询离线指标时优先 ADS/DWS，可以降低 SQL 复杂度、扫描成本和敏感字段风险。
DWD 是明细层，适合授权排查，不适合作为普通数据问答入口。

### 取舍二：为什么分区条件必须强制？

离线仓库表通常按 `dt` 或 `biz_date` 分区。
没有分区条件，SQL 可能扫描大量历史数据，成本不可控，结果范围也不清楚。
所以用户没给时间范围时应先澄清，SQL 里没带分区字段时应阻断。

### 取舍三：为什么不能只靠 Prompt 约束 SQL 安全？

Prompt 可以提醒模型生成安全 SQL，但不能保证模型一定遵守。
SQL 是否只读、字段是否敏感、表是否授权、扫描范围是否超限，都应该由确定性代码校验。
在金融信贷场景里，权限、安全和执行不能交给模型自由判断。

## 面试沉淀

Q119：生产级 NL2SQL 为什么不能生成 SQL 后直接执行？
重要程度：5/5

### 回答

生产级 NL2SQL 不能生成 SQL 后直接执行，因为模型生成的 SQL 可能存在表字段编造、口径错误、越权访问、
漏分区、扫描成本过高、访问敏感字段、非只读操作和结果范围不清等风险。
在金融信贷场景里，这些问题可能导致客户隐私泄露、数仓资源被打爆、报表口径错误，甚至影响风控和经营决策。

更稳的链路是：先识别主题域和意图，再用指标字典确认口径，用 Schema Catalog 找表和字段，
生成候选 SQL 后进入 SQL Validator。
Validator 要检查只读、表白名单、字段权限、敏感字段、时间范围、分区条件、扫描成本、limit 和用户权限。
只有通过校验的 SQL 才能执行，执行后还要写审计并基于查询结果解释。

一句面试总结是：模型可以生成候选 SQL，但 SQL 能不能执行必须由确定性校验层决定。

Q120：离线 NL2SQL 的 SQL Validator 通常要检查哪些内容？
重要程度：5/5

### 回答

离线 NL2SQL 的 SQL Validator 至少要检查：是否只读、表是否在白名单或 Catalog 中、字段是否存在、
是否命中敏感字段、用户是否有权限、是否带时间范围、是否带分区条件、扫描范围是否超限、
是否访问 ODS/DWD 明细、是否有危险函数或子查询、limit 是否合理、join 是否在允许关系内。

金融信贷场景里，重点是控制客户隐私、口径一致和扫描成本。
普通指标查询应优先走 ADS/DWS；DWD 明细只能在授权排查场景下访问；手机号、身份证号、银行卡号和客户名单必须阻断。
如果用户没给时间范围，Agent 应先澄清；如果 SQL 缺分区条件，应阻断执行。

Validator 的价值是把模型生成的不确定结果变成可控的工程动作。

Q121：为什么离线指标查询要强制时间范围和分区裁剪？
重要程度：5/5

### 回答

离线指标查询强制时间范围和分区裁剪，是为了保证业务含义清晰、扫描成本可控、结果范围可解释。
离线仓库表通常按 `dt` 或 `biz_date` 分区，如果 SQL 没有时间范围，就可能扫描全量历史分区，
不仅拖慢数仓，也会让“通过率是多少”这种问题缺少明确统计范围。

在金融信贷 Agent 中，用户问“授信通过率是多少”时，应先澄清是昨天、近 7 天、上月还是某个渠道。
用户问“近 7 天各渠道授信通过率”时，SQL 必须带 `dt BETWEEN ... AND ...` 这类分区条件。
如果扫描天数超过策略阈值，比如超过 31 天，系统应要求缩小范围或进入审批流程。

一句面试总结是：时间范围保证口径清楚，分区裁剪保证成本可控。

## 术语更新

今天新增或强化这些术语：

- SQL Validator：SQL 执行前的确定性校验层，检查只读、权限、分区、敏感字段和成本。
- 分区裁剪：通过分区字段限制扫描范围，减少离线仓库查询成本。
- 只读 SQL：只能查询数据，不能修改、删除或创建对象的 SQL。
- 扫描成本：SQL 需要扫描的数据分区、行数或资源规模。
- 字段白名单：允许查询的字段集合，用于防止编造字段和越权字段访问。
- SQL 安全阻断：SQL 不满足安全规则时拒绝执行。
