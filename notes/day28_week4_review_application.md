## 大白话解释

前 3 周的重点是把 RAG 链路搭起来。第 4 周的重点是回答一个更现实的问题：

```text
这个 RAG 项目拿去面试或试投时，能不能讲清楚生产问题？
```

生产项目不只看“能不能回答成功问题”。还要看：

- 问错时能不能拒答；
- 资料不准时能不能评测；
- 用户无权限时能不能过滤；
- 成本太高时能不能优化；
- 演示前能不能稳定复现。

Day 28 的任务就是把这些能力整理成投递材料。

## 生产实际

真实公司里，RAG 项目通常不是一次上线就结束。上线后会持续遇到这些问题：

- 表格行 1
  - 问题：用户问法太口语
  - 第 4 周对应能力：query rewrite
- 表格行 2
  - 问题：召回效果忽好忽坏
  - 第 4 周对应能力：固定测试集
- 表格行 3
  - 问题：模型没有依据也回答
  - 第 4 周对应能力：拒答策略
- 表格行 4
  - 问题：用户看到无权限资料
  - 第 4 周对应能力：权限标签和 metadata filter
- 表格行 5
  - 问题：答案或引用泄露敏感信息
  - 第 4 周对应能力：脱敏和 citation 过滤
- 表格行 6
  - 问题：成本和延迟太高
  - 第 4 周对应能力：token budget、缓存和上下文裁剪
- 表格行 7
  - 问题：演示或上线前不稳定
  - 第 4 周对应能力：smoke test 和固定回归问题

这就是第 4 周的核心价值： 你不只是做了一个 RAG Demo，而是开始补生产化能力。

## 工程取舍

这一天优先采用最小可运行方案，是为了先建立稳定基线。真实项目里不能一开始就追求复杂架构，应该先把输入、处理逻辑、输出和失败场景跑通，再逐步增加模型调用、 检索、
数据库、权限和评测。

对“第 4 周复盘与试投启动”这类能力，学习阶段更看重可解释和可复盘；生产阶段再根据准确率、延迟、成本、安全和维护成本决定是否引入更复杂的方案。

## 面试沉淀

Q058：如何把一个 RAG Demo 包装成可投递项目？

### 回答

可投递的 RAG 项目不能只说“我做了向量检索和问答”。要按生产链路包装： 离线侧讲文档接入、清洗、chunk 切分、embedding、索引和
metadata； 在线侧讲 query rewrite、检索、rerank、上下文构造、citations、拒答和 API；
质量侧讲测试集、命中率、bad case、引用准确性和回归评估； 安全侧讲权限过滤、敏感信息控制和审计；
工程侧讲日志、request_id、成本、缓存、参数校验和演示稳定性。这样项目才像一个可落地的 AI 应用，而不是一次性脚本。

Q059：试投前应该检查哪些材料？

### 回答

试投前至少检查简历、项目 README、演示脚本、面试题库和投递记录。简历里要把金融信贷开发经验和 RAG / NL2SQL 方向连接起来；
项目 README 要说明项目目标、运行方式、核心链路、边界处理和后续优化；
演示脚本要准备成功样例、bad case、citations 和 request_id； 面试题库要能支撑 RAG、SQL、FastAPI、评测、
安全和成本问题；
投递记录要记录公司、岗位、渠道、匹配度和反馈，方便复盘。

Q060：为什么试投要先小批量验证，而不是一开始海投？

### 回答

试投初期的目标不是立刻投很多，而是验证简历表达、项目包装和岗位匹配度。如果一开始海投，简历问题、项目讲法问题和岗位筛选问题会被放大，反馈也难分析。
先投 5-10 个高匹配岗位，可以观察是否有简历查看、沟通、约面或卡点反馈。根据反馈再调整简历关键词、项目描述、薪资预期和投递方向。
这比盲目扩大数量更容易提高命中率。

### 5. [Day 26] 为什么 RAG 不是 top-k 越大越好？
我的回答：

增加检索成本，引入噪声，延迟成本

回答评价：
回答正确。你已经覆盖了成本、噪声和延迟三个重点。还可以补充：top-k 过大还会占用 token budget，让模型更容易被无关上下文干扰。

参考答案：
RAG 不是 top-k 越大越好。top-k 变大会召回更多 chunk，但也会增加检索、rerank 和 LLM 上下文成本。
如果无关 chunk 进入 prompt，会占用 token budget，增加延迟，并干扰模型生成。
生产里要通过测试集、expected_sources、引用准确性和 bad case 来选择合适的 top-k， 而不是单纯追求召回数量。

### C. 周回顾问题

### 7. [Day 22] query rewrite 解决什么问题，风险是什么？
我的回答：

用户问题表达不清导致问题曲解，口语化，中英文结合，多重语义等，风险是重写后曲解用户问题，导致答案错误偏移

回答评价：
回答正确。你说到了口语化、中英文混用、多义表达和 rewrite drift。
这里可以再补充：query rewrite 的目标不是改写成好看的问题，而是让检索更容易命中正确资料。

参考答案：
query rewrite 解决的是用户原始问题不一定适合直接检索的问题。用户可能表达口语化、省略上下文、中英文混用、包含多个意图，或者和知识库里的术语不一致。
rewrite 可以补全上下文、扩展同义词、保留精确实体、拆分多意图问题，让检索更容易命中正确 chunk。
风险是 rewrite drift，也就是改写后偏离用户原意，导致召回和答案都偏到错误方向。
生产里要记录原始 query、改写 query、召回结果和 citations， 并用测试集评估改写是否真的提升效果。

### 9. [Day 24] RAG 系统什么时候应该拒答？
我的回答：

边界问题，权限问题，敏感问题，进行解释澄清建议说明，建议进行正常链路，权限申请，记录并且审计

回答评价：
基本正确，但表达可以更清楚。拒答场景包括无资料、低置信度、权限不足、敏感信息、问题模糊和资料冲突；
拒答时要给原因、澄清建议或下一步路径，并记录 request_id 便于审计。

参考答案：
RAG 系统应该在没有足够依据、检索结果低置信度、用户权限不足、问题涉及敏感信息、问题过于模糊、资料冲突或超出知识库范围时拒答。
拒答不是简单说“不能回答”，而是要说明原因，比如资料不足、权限不足或问题需要补充条件。如果可以继续推进，系统应该给出澄清建议、权限申请路径或人工处理建议。
同时要记录 request_id、用户、问题、召回结果、拒答原因和策略命中情况，方便后续审计和优化。

### 10. [Day 25] RAG 系统怎么做权限控制和敏感信息保护？
我的回答：

权限 tags，metadata filter 权限过滤，

回答评价：
方向正确，但答案偏短。权限 tag 和 metadata filter 是核心，但完整方案还要覆盖入库前脱敏、检索前后过滤、生成前上下文控制、答案和
citations 脱敏，以及日志审计。

参考答案：
RAG 权限控制要贯穿入库、检索、生成和返回全链路。入库前要做数据分级、脱敏和权限标签；
每个 chunk 要记录 security_level、allowed_roles、business_domain、source 和 version 等
metadata。 在线检索时要根据用户身份、角色和业务域过滤 chunk，不能让无权限资料进入 LLM 上下文。
生成时要求模型只基于授权 citations 回答， 返回前还要对答案和 citations 做敏感信息扫描。
最后记录 request_id、用户、召回结果、引用、答案和策略命中情况，便于审计。

### 11. [Day 26] 生产 RAG 怎么控制 token 成本和响应延迟？
我的回答：

多召回，少引用，只引用最精准少部分的 chunk

回答评价：
方向正确。“多召回、少引用”本质是先保证候选覆盖，再通过 rerank 和上下文压缩只把最有价值的 chunk 放进 prompt。
还可以补充缓存、top-k 控制、模型分层、超时和监控。

参考答案：
生产 RAG 控制 token 成本和响应延迟，核心是控制进入 LLM 的上下文。
可以先用相对宽的召回保证候选覆盖，再用 rerank、metadata filter 和去重选出少量高质量 chunk。
不要把所有召回结果都塞进 prompt，而是只放最相关、最新、授权且能支撑答案的片段。
同时可以使用缓存、合理 top-k、上下文压缩、小模型预处理、超时控制和成本监控。 高风险问题优先准确性，低风险高频问题可以更重视缓存和延迟。

### 12. [Day 27] RAG 项目演示前应该做哪些稳定性检查？
我的回答：

smoke test，前置条件，成功路径， 边界问题，bad case

回答评价：
回答正确。这几个检查项覆盖了演示稳定性的主干。可以再补充具体字段：索引是否存在、固定回归问题、answer、citations、request_id、
latency 和错误提示。

参考答案：
RAG 项目演示前要做三类稳定性检查。第一是前置条件：索引文件是否存在，依赖是否可用，配置是否正确，固定回归问题是否能跑通。
第二是成功路径：典型问题能返回 answer、citations、retrieved_chunks、confidence、request_id 和
latency。 第三是边界路径：空问题、top_k 越界、无答案、权限不足和 bad case 是否有清楚错误提示或
cannot_answer_reason。
演示前最好生成一份固定检查报告，避免现场临时选问题导致结果不稳定。
