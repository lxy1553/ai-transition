## 大白话解释

数据质量问题不是“数仓同学自己的事”。
只要仓库数据不稳定，Agent 就会把错误结果包装成看起来合理的回答。

例如：

```text
实时事件重复 -> 实时申请量虚高
分区延迟到达 -> 日报指标偏低
补偿任务没跑完 -> 逾期率短时间异常
状态字段为空 -> 风控拒绝率解释失真
```

所以 Day 57 的任务不是重新讲数据质量概念，而是把“数据质量 -> Agent 风险 -> 处理规则 -> 回归保护”
这条链路打通。

## 生产实际

金融信贷仓库里最常见的数据质量问题包括：

| 类型 | 典型例子 | 对 Agent 的影响 |
|------|----------|----------------|
| 空值 | 审批结果字段为空 | 拒绝率、通过率失真 |
| 重复 | 同一申请事件重复落库 | 实时申请量和命中率虚高 |
| 乱序 | 事件先到结果后到明细 | 窗口指标短时间异常 |
| 延迟 | 分区晚到、流任务积压 | Agent 错把延迟当异常 |
| 补偿 | 回补任务未完成 | 昨日指标暂时不完整 |
| 幂等失败 | 同一流水重复写入 | 经营日报和实时快照不一致 |

生产处理要点：

- 先识别数据质量问题，再决定是否回答；
- 可以降级说明时，不强行给确定结论；
- 需要阻断时，明确说明原因、范围和建议排查项；
- 所有质量异常都要写入审计字段，便于回放和回归。

---

## 常见坑

| 类型 | 可能的问题 | 生产处理方式 |
|------|------------|--------------|
| 只看指标值 | 指标异常其实来自重复或延迟 | 联查质量规则和任务状态 |
| 质量问题只报警不治理 | 问答结果持续错误 | 将 bad case 接入回归测试 |
| 没有质量标签 | Agent 无法判断能否回答 | 输出质量状态和可信度字段 |
| 只修一次 | 下次补偿又复发 | 建规则、建审计、建回归集 |
| 实时和离线混看 | 链路差异被忽略 | 分别记录窗口状态和分区状态 |

## 工程取舍

### 取舍一：为什么质量问题不能只交给报表 owner 解释？

因为 Agent 在一线回答问题，它必须先知道“当前数据能不能被信任”。
如果等用户看到错误结果再人工解释，已经失去自动化价值。

### 取舍二：为什么质量状态要结构化输出？

因为后续编排要基于状态路由：

```text
quality_ok -> 正常回答
quality_warning -> 降级说明
quality_blocked -> 阻断并给排查建议
```

这类判断不能只靠自然语言描述。

## 面试沉淀

Q131：为什么金融信贷仓库 Agent 必须感知数据质量状态？
重要程度：5/5

### 回答

因为 Agent 的最终回答依赖离线分区、实时窗口、补偿任务和上游事件质量。
如果它不知道当前数据是否完整、是否重复、是否延迟，就可能把暂时错误的数据说成确定结论。
金融信贷场景里这会直接影响经营判断、风险研判和告警处置。

Q132：数据质量问题为什么要接入错误治理和回归测试？
重要程度：5/5

### 回答

因为这类问题不是一次性故障，而是会重复出现的生产风险。
如果只靠人工排查，下次空分区、重复写入或补偿延迟发生时，Agent 还会犯同样的错。
正确做法是把质量异常做成规则、审计字段和固定 bad case，并在每次修改后执行全量回归。

## 术语更新

- 数据质量规则：判断数据是否完整、唯一、及时、可信的规则集合。
- 幂等：同一事件重复执行多次，结果仍然保持一致。
- 补偿任务：对晚到、漏跑或修正数据进行回补的任务。
- 质量状态：供 Agent 判断是否回答、降级或阻断的结构化字段。

## 每日核心问题自测

### A. 今日核心问题

### 1. 为什么金融信贷仓库 Agent 必须感知数据质量状态？
  重要程度：5/5
  我的回答：金融信贷仓库 Agent 的回答直接依赖四个数据来源：离线分区、实时窗口、
    补偿任务和上游事件质量。如果 Agent 不知道当前数据是否完整、是否重复、是否延迟，
    它就可能把暂时错误的数据包装成确定结论。具体来说：

    1）离线分区可能为空或未产出，Agent 查不到数据却可能误报为"指标为零"；
    2）实时窗口可能因 Kafka 积压导致数据不完整，Agent 给出的指标偏低；
    3）补偿任务未完成时，逾期率等关键指标在短时间内是偏的；
    4）上游事件重复写入或幂等失败，会导致实时指标虚高。

    在金融信贷场景里，这些错误会直接影响经营判断（如放款决策）、
    风险研判（如逾期率趋势）和告警处置（如误报或漏报）。
    因此 Agent 必须在回答前检查数据质量状态：质量 ok 才正常回答，
    有风险就降级说明，严重异常就阻断并给排查建议。

  回答评价：从四个数据来源切入，说明了每种质量异常对 Agent 回答的具体影响，
    落到金融信贷的三个核心场景（经营、风险、告警），
    最后给出了三种质量状态的路由结论。结构完整，因果关系清晰，面试可用。

  评分：9/10

  参考答案要点：
    - Agent 依赖离线分区、实时窗口、补偿任务、上游事件；
    - 不知道数据是否完整 → 暂时错误 → 确定结论；
    - 金融场景影响：经营判断、风险研判、告警处置；
    - 三种路由：正常回答、降级说明、阻断排查。
    需补充：数据质量问题bushu还会影响 Agent 的可信度——
    如果用户多次得到错误回答，会丧失对 Agent 的信任，即使后续数据恢复了用户也不信了。

### 2. 空值、重复、乱序和补偿会如何影响 Agent 回答？
  重要程度：5/5
  我的回答：这四类问题分别从不同角度污染 Agent 的回答质量。

    1）空值：当审批结果、风控标记等核心字段为空时，Agent 计算的拒绝率、通过率会失真。
    例如渠道 B 的 risk_flag 空值率 12%，该渠道拒绝率被严重低估，
    Agent 如果把所有渠道排序对比就会误导用户。正确做法是标注空值率、不直接对比。

    2）重复：同一申请事件重复落库会让实时申请量虚高 15%+，
    Agent 可能误判为"业务增长"或触发假告警。
    正确做法是同时给出原始值和去重值两个口径。

    3）乱序：结果事件的时间戳早于明细事件，会让窗口内的审批→放款链路指标异常，
    Agent 可能误判链路有问题。正确做法是标注乱序事件数和比例，建议拉大窗口。

    4）补偿：补偿任务未完成时，昨天的逾期率快照值偏低，
    Agent 如果当作最终值回答，用户做经营判断后等补偿完成才发现逾期率其实更高。
    正确做法是给出快照值和补偿后预计范围。

  回答评价：四类问题逐一拆解，每类都给出了影响机制、错误后果和正确做法，
    结合了具体业务场景（拒绝率、申请量、审批链路、逾期率），接地气。

  评分：9/10

  参考答案要点：
    - 空值 → 指标失真 → 标注空值率；
    - 重复 → 指标虚高 → 双口径对比；
    - 乱序 → 窗口异常 → 拉大窗口；
    - 补偿 → 快照偏低 → 给预计范围。
    补充：乱序还可能导致 Agent 在追溯事件链路时给出错误的因果顺序，
    补偿还可能影响环比/同比计算。

### 3. 什么情况下应该降级回答，而不是直接阻断？
  重要程度：5/5
  我的回答：降级和阻断的核心判断标准是"数据是否仍然包含可参考的信息"。
    降级回答适用于数据有瑕疵但仍有参考价值的场景：

    1）数据部分不完整但可以给出范围——
    如分区还在产出中（当前完成 60%），可以给出已有数据和预计完成时间；
    2）数据存在偏差但可以标注——
    如某渠道空值率 8%，可以给出正常渠道的精确值并标注异常渠道的不确定性；
    3）数据有时效差异但可以做对比——
    如实时和离线状态不一致，可以同时给出两个来源让用户自己判断；
    4）窗口数据不完整但不影响方向判断——
    如 5 分钟窗口有 20% 晚到数据，可以给出当前值和可能的影响方向。

    阻断回答适用于数据完全不可信的场景：空分区（0 行）意味着任何指标值都是错的，
    或幂等失败导致离线/实时偏差超过可接受阈值，这时不给确定结论。

  回答评价：给出了清晰的判断标准（"是否仍包含可参考信息"），
    列举了四种降级场景并各举例，最后用空分区和幂等失败两个阻断场景对比，逻辑自洽。

  评分：9/10

  参考答案要点：
    - 核心标准 → 数据是否仍有可参考信息；
    - 降级：部分不完整给范围、存在偏差可标注、时效差异可对比、窗口不完整不影响方向；
    - 阻断：空分区（0行）、幂等失败超阈值（不可忽略偏差）。
    补充说明：还要考虑问题的重要程度——
    如果是监管报送类问题，即使数据有轻微瑕疵也应阻断。

### 4. 为什么质量状态要结构化输出？
  重要程度：5/5
  我的回答：质量状态的结构化输出不是技术偏好，而是后续编排的刚性需求。

    1）路由依赖：Agent 编排层需要根据 quality_ok / quality_warning / quality_blocked
    三个枚举值做分支路由，不能靠解析自然语言文本来判断"Agent 觉得数据好不好"。

    2）审计可检索：后续需要按质量状态筛选请求（如"过去 7 天有多少次 quality_blocked"），
    结构化字段可以直接 SQL 过滤，自然语言不行。

    3）回归可验证：Bad case 回归需要断言 quality_status == "quality_blocked"，
    结构化值可以精确匹配，自然语言需要 NLP 评估增加不确定性。

    4）告警可触发：监控系统可以根据结构化状态自动触发告警
    （如 quality_blocked 比例突增），不需要人工读每条回答。

    Day 58 的审计存储设计里，quality_status、triggered_rules、completeness_score
    等都是结构化字段，正是为了实现这些能力。

  回答评价：从路由、审计、回归、告警四个维度论证了结构化输出的必要性，
    每个维度都给出了"结构化能做，自然语言做不了或做不好"的具体理由，
    并关联了 Day 58 审计存储设计。

  评分：9/10

  参考答案要点：
    - 路由依赖 → 枚举值做分支判断；
    - 审计可检索 → SQL 直接过滤结构化字段；
    - 回归可验证 → 精确匹配断言；
    - 告警可触发 → 自动监控质量状态比例。
    补充：结构化输出还可以减少模型幻觉——
    模型被强制输出枚举值比自由发挥更不容易出错。

### 5. 数据质量 bad case 应该如何进入回归集？
  重要程度：5/5
  我的回答：数据质量 bad case 进入回归集需要五个步骤。

    1）发现与归类：从生产日志、用户反馈或评测失败中收集数据质量导致的错误回答，
    按空值/重复/乱序/延迟/补偿/幂等等类型归类，每条 bad case 关联一条具体的质量规则 ID。

    2）结构化描述：每个 bad case 包含 case_id、用户问题、质量异常描述、预期质量状态、
    预期 Agent 行为、错误行为示例和回归断言列表，
    格式与 Day 57 的 quality_bad_cases.json 一致。

    3）联动已有评测集：将 bad case 与 Day 56 端到端评测场景关联
    （如空分区→空分区查询场景、重复→告警误报场景），
    确保每次回归既覆盖功能正确性也覆盖质量容错性。

    4）自动化回归：在 CI 或定时任务中，每次代码或规则变更后跑全量 bad case 回归，
    验证 quality_status 和回答行为是否符合预期断言。

    5）持续更新：每次发现新的质量异常模式（如新渠道的数据倾斜、新上游系统的幂等问题），
    都沉淀为新的 bad case 加入回归集，防止同类问题复发。

  回答评价：五步流程清晰可执行，从发现问题到持续更新形成闭环。
    特别强调了与 Day 56 评测集的联动和结构化格式要求。

  评分：9/10

  参考答案要点：
    - 发现归类 → 关联质量规则 ID；
    - 结构化 → case_id/问题/异常/预期状态/断言；
    - 联动 Day 56 → 功能正确 + 质量容错双覆盖；
    - 自动化 → CI 全量回归；
    - 持续更新 → 新问题新 case 防复发。
    补充：还应记录 bad case 的发现来源（生产/评测/用户反馈）和严重等级，
    用于回归优先级排序。

### 6. 实时质量问题和离线质量问题的处理区别是什么？
  重要程度：5/5
  我的回答：实时和离线质量问题有三个核心区别。

    1）时效要求不同：实时质量问题的处理窗口是秒到分钟级——
    比如 5 分钟窗口内发现重复事件，Agent 需要立刻降级说明并给出原始/去重双口径，
    用户需要的是即时的正确判断。
    离线质量问题的处理窗口是小时到天级——
    比如发现昨天分区未产出，Agent 可以告诉用户"等待 ETL 完成，预计 30 分钟后可查"，
    用户通常可以等。

    2）判断策略不同：实时问题需要基于当前窗口内的数据特征判断
    （如窗口内去重率、乱序比例），无法参考"全量数据"。
    离线问题可以参考历史分区趋势判断（如前一天同分区行数、同期环比），
    有更多的参考基准。

    3）降级策略不同：实时问题通常建议拉大窗口（从 5 分钟到 15 分钟）
    来缓解延迟和乱序的影响，因为更大的窗口会让晚到数据和乱序事件的比例自然下降。
    离线问题通常建议使用上一个可用分区或等待补偿完成，
    因为离线分区的粒度是固定的（T+1），无法"拉大"。

    此外，实时问题更关注窗口内的数据完整度（completeness_score），
    离线问题更关注分区产出状态和补偿任务状态。

  回答评价：从时效、判断策略、降级策略三个维度区分了实时和离线质量问题的处理差异，
    每个维度都有具体例子和可操作建议，体现了对两种仓库链路本质差异的理解。

  评分：9/10

  参考答案要点：
    - 时效：实时秒-分钟级立刻降级，离线小时-天级可建议等待；
    - 判断：实时看窗口内特征，离线看历史趋势；
    - 降级：实时→拉大窗口，离线→用上一个分区或等补偿。
    补充：实时问题的排查方向通常是上游事件源和 Kafka 积压，
    离线问题的排查方向通常是 ETL 调度和补偿任务依赖。

### 7. 数据质量规则至少应该包含哪些字段？
  重要程度：5/5
  我的回答：数据质量规则至少应该包含 8 个核心字段：

    1）rule_id：规则唯一标识，如 DQ-001，便于引用和审计追踪；
    2）rule_name：规则名称，如"审批结果字段空值检查"，人类可读；
    3）category：质量分类，从标准维度选取——
    空值/重复/乱序/延迟/补偿/幂等/状态不一致/晚到/空分区/数据倾斜；
    4）risk_level：风险等级，critical > high > medium > low，决定阻断还是降级；
    5）warehouse_type：适用仓库类型，offline / realtime / both，
    因为同一类问题在两种仓库的处理方式不同；
    6）trigger_condition：触发条件，用可量化的阈值描述——
    如"NULL 比例超过 5%"、"同一 event_id 出现超过 1 次"、"分区行数偏差超 30%"；
    7）quality_status：本条规则触发后的质量状态，quality_warning 或 quality_blocked，
    决定路由分支；
    8）action：处理动作的具体描述，是给 Agent 的执行指令——
    如"降级回答：标注空值比例并说明指标可能偏低"。

    Day 57 的 quality_rules.json 还额外包含 affected_agent_scenario、
    example_agent_response_template 和 fix_suggestion 三个扩展字段，
    分别用于关联业务场景、指导回答模板和提供排查建议。

  回答评价：核心 8 字段覆盖了标识、分类、风险、适用性、触发条件、状态和动作，
    每个字段都有明确的用途说明和量化要求，
    最后补充了 3 个扩展字段体现工程实用性。

  评分：9/10

  参考答案要点：
    - 核心 8 字段：rule_id / rule_name / category / risk_level /

    - 扩展：affected_agent_scenario / example_agent_response_template / fix_suggestion。

    last_updated（最后更新时间）等运维字段。

### B. 前三天核心回顾

### 8. [Day 56] 为什么 Agent 端到端评测不能只看最终回答文字？
  重要程度：4/5
  我的回答：端到端评测只看最终回答文字会遗漏四个关键维度。

    1）工具调用路径：Agent 可能用错了工具但碰巧得到了正确答案——
    比如应该用离线 NL2SQL 却用了 RAG 检索，虽然这次回答正确，
    但切换到其他问题时就会出错。评测必须检查 Agent 是否选择了正确的工具路线。

    2）质量状态判断：Agent 可能在数据质量异常时仍然给出了"看似合理"的确定结论——
    比如空分区时回答"授信通过率为 0%"，文字通顺但完全错误。
    评测必须验证 Agent 是否在回答前检查了 quality_status。

    3）审计记录完整性：回答过程是否记录了 request_id、工具调用链、中间结果
    和最终决策原因，这些是回放和排错的依据，只看最终回答无法验证。

    4）安全阻断执行：对于敏感查询（如导出身份证号），
    Agent 是否在工具层就阻断了而不是在回答文字中"委婉拒绝"？
    如果只在文字中说"不能查"但实际已经查了，就是安全漏过。

    所以 Day 56 的评测集每条样例都包含了 expected_tool_route、
    expected_quality_status、expected_audit_fields 等结构化断言，不只是检查回答文字。

  回答评价：从工具路线、质量状态、审计记录、安全阻断四个维度说明为什么不能只看文字，
    每个维度都有"看似对实际错"的反例，且关联了 Day 56 评测集的设计。

  评分：9/10

  参考答案要点：
    - 工具路径 → 用错工具碰巧对；
    - 质量状态 → 空分区答"0%"文字通顺但结论错；
    - 审计记录 → 没有记录无法回放；
    - 安全阻断 → 文字拒答但实际已查询。
    补充：还应检查回答的可信度标注（confidence）、引用来源（citations）是否有据可查。

### 9. [Day 55] 多工具 Agent 为什么必须写审计日志？
  重要程度：4/5
  我的回答：多工具 Agent 写审计日志不是"锦上添花"而是"生产必选"。

    1）排错定位：当用户反馈"昨天的日报数据不对"，没有审计日志就无法知道
    是 Agent 选错了工具（用了实时而非离线）、工具返回了错误数据、
    还是 Agent 解释错了结果。审计日志记录了每一步的工具调用、输入输出和时间戳，
    可以精确复现问题。

    2）责任界定：在金融信贷场景，如果 Agent 给出的逾期率偏低导致放款决策失误，
    需要追溯是数据质量问题（补偿未完成）还是 Agent 逻辑问题（没检查质量状态）。
    审计日志的 quality_check 字段可以明确责任链。

    3）回归测试：审计日志中的 request_id → tool_calls → final_answer 链路
    可以直接转化为回归样例，比人工构造测试覆盖更真实。

    4）合规要求：金融行业对自动化决策有审计追溯要求，
    Agent 如果参与了经营判断和告警处置，其决策链路必须可审计。
    Day 55 的 daily_alert_agent 实现了 4 步审计记录
    （query → check → generate → respond），就是为保证可追溯。

  回答评价：从排错定位、责任界定、回归测试、合规要求四个角度论证，
    紧密结合金融行业特点和 Day 55 项目实现。

  评分：9/10

  参考答案要点：
    - 排错 → 知道哪一步出错；
    - 责任界定 → 是数据问题还是逻辑问题；
    - 回归 → 真实链路转测试样例；
    - 合规 → 金融自动化决策可追溯。
    补充：审计日志还需要考虑敏感信息脱敏，不能把用户身份证号等明文写入日志。

### 10. [Day 54] 实时链路延迟超阈值时，Agent 应该怎么回答？
  重要程度：4/5
  我的回答：实时链路延迟超阈值时，Agent 不能直接报出当前窗口的指标值，
    而应该执行四步降级回答。

    1）明确标注延迟状态：开头就说明"当前实时链路存在延迟
    （当前延迟 {delay}，阈值 {threshold}），以下数据可能不完整"。

    2）给出有界结论而非确定结论：不报"拒绝率是 5.2%"，而报
    "当前可见数据中拒绝率为 5.2%，但窗口内预计还有 {missing_pct}% 的数据未到达，
    实际值可能在 {range_min}%-{range_max}% 之间"。

    3）说明影响方向和量级：如果晚到的数据中拒绝案例占比通常更高
    （因为拒绝是异常事件、采集链路更长），要说明"实际拒绝率可能高于当前值"。

    4）给出操作建议：建议用户拉大查询窗口（从 5 分到 15 分钟）、稍后重查、
    或切换到 T+1 离线数据（如果时效允许）。

    Day 54 的有界解释设计正是为这个场景——Agent 不能因为延迟就拒绝回答，
    也不能因为回答了就让用户误以为数据完整，
    需要给出"有边界的、带不确定性范围的答案"。

  回答评价：四步降级回答流程清晰可执行，每个步骤都有具体的措辞示例（带占位符），
    从标注延迟到给范围到给建议形成完整链路。

  评分：9/10

  参考答案要点：
    - 标注延迟状态 → 开门见山说数据不完整；
    - 有界结论 → 报范围不报精确值；
    - 说明影响方向 → 晚到数据对指标的方向性影响；
    - 操作建议 → 拉大窗口 / 稍后重查 / 切离线。
    补充说明：如果延迟严重到窗口内数据完整度低于 50%，应该直接阻断而非降级。
