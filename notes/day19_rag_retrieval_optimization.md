## 大白话解释

召回就是“先把可能有用的资料找出来”。

召回优化就是：

**让系统更容易找到正确资料，少找无关资料，并把重要资料排到前面。**

比如用户问：

```text
SQL 解释助手为什么要检查 dt 分区？
```

好的召回应该找到：

```text
缺少 dt 分区会导致扫描范围过大，存在全表扫描风险。
```

差的召回可能找到：

```text
FastAPI 怎么写 health 接口。
```

这时就算后面的 LLM 很强，也只能基于错误资料生成一个看起来合理的错误答案。

## 生产实际

真实公司里的 RAG 效果问题，很多不是模型不会回答，而是检索阶段没有找到正确资料。

常见场景：

- 用户问法和文档写法不一致
- 文档 chunk 切得太碎，关键上下文丢失
- chunk 切得太大，相关内容被无关内容稀释
- top-k 太小，漏掉关键资料
- top-k 太大，带入噪声，增加 token 成本
- 表名、字段名、错误码这类精确词没有匹配上
- 权限、业务域、版本过滤做得太粗
- 召回结果没有 rerank，正确资料排得太靠后

生产里召回优化一般会围绕这些手段展开：

| 方法 | 作用 |
|------|------|
| 调整 chunk 策略 | 控制文档切分粒度，避免太碎或太大 |
| 调整 top-k | 平衡召回率、噪声和成本 |
| query rewrite | 把用户问题改写成更适合检索的表达 |
| 混合检索 | 同时使用向量检索和关键词检索 |
| metadata 过滤 | 按业务域、权限、版本、文档类型过滤 |
| rerank | 对候选结果重新排序，让最相关资料靠前 |
| bad case 分析 | 找出召回失败原因，反向优化知识库和检索策略 |

## 今日实验

今天本地脚本会对比三种策略：

| 策略 | 说明 |
|------|------|
| baseline_top3 | 原始问题，取 top 3 |
| expanded_top3 | 对问题做简单关键词改写，取 top 3 |
| expanded_top5 | 改写后取 top 5，观察是否提升命中 |

评估重点不是答案写得好不好，而是看：

- expected source 有没有被召回
- 正确资料排第几名
- top-k 变化是否带来更多噪声
- query rewrite 是否改善命中
- 哪些问题是 bad case

## 常见坑

| 问题 | 影响 | 处理方式 |
|------|------|----------|
| 只看最终答案 | 找不到真正问题来源 | 先看召回结果和引用 |
| 只调 prompt | 检索错了 prompt 也救不了 | 先优化知识库、chunk、召回 |
| top-k 越大越好 | 噪声和 token 成本升高 | 结合 rerank 和评估集调参 |
| 没有测试集 | 优化靠感觉 | 准备标准问题、期望来源和 bad case |
| 没有记录失败样例 | 问题反复出现 | 保存 query、召回结果、原因和修复动作 |

## 工程取舍

召回优化本质是在三件事之间取平衡：

```text
召回率：尽量别漏掉正确资料
精准率：尽量少带入无关资料
成本和延迟：不要召回太多，拖慢 rerank 和 LLM
```

生产里一般不会只靠一个策略。

更常见的是：

```text
query rewrite
-> 向量检索 + 关键词检索
-> metadata 过滤
-> rerank
-> 选少量高质量 chunk 进入上下文
```

今天的本地练习先不引入真实 rerank 模型，而是用可解释的规则对比 top-k 和 query rewrite。
这样能先理解召回优化的评估思路。

## 面试表达

> 我理解 RAG 召回优化不是只调一个 top-k 参数，而是要先有测试问题和期望来源，
> 然后观察正确资料有没有进入候选集、排在第几名，以及无关资料有多少。
> 生产里会结合 chunk 策略、query rewrite、混合检索、metadata 过滤和 rerank。
> 如果召回没命中，应该先修知识库、切分和检索策略，而不是只改 prompt。
