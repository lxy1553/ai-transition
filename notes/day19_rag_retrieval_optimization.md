## 大白话解释

召回就是“先把可能有用的资料找出来”。

召回优化就是：

**让系统更容易找到正确资料，少找无关资料，并把重要资料排到前面。**

比如用户问：

```text
SQL 解释助手为什么要检查 dt 分区？
```

好的召回应该找到：

```text
缺少 dt 分区会导致扫描范围过大，存在全表扫描风险。
```

差的召回可能找到：

```text
FastAPI 怎么写 health 接口。
```

这时就算后面的 LLM 很强，也只能基于错误资料生成一个看起来合理的错误答案。

## 生产实际

真实公司里的 RAG 效果问题，很多不是模型不会回答，而是检索阶段没有找到正确资料。

常见场景：

- 用户问法和文档写法不一致
- 文档 chunk 切得太碎，关键上下文丢失
- chunk 切得太大，相关内容被无关内容稀释
- top-k 太小，漏掉关键资料
- top-k 太大，带入噪声，增加 token 成本
- 表名、字段名、错误码这类精确词没有匹配上
- 权限、业务域、版本过滤做得太粗
- 召回结果没有 rerank，正确资料排得太靠后

生产里召回优化一般会围绕这些手段展开：

- 表格行 1
  - 方法：调整 chunk 策略
  - 作用：控制文档切分粒度，避免太碎或太大
- 表格行 2
  - 方法：调整 top-k
  - 作用：平衡召回率、噪声和成本
- 表格行 3
  - 方法：query rewrite
  - 作用：把用户问题改写成更适合检索的表达
- 表格行 4
  - 方法：混合检索
  - 作用：同时使用向量检索和关键词检索
- 表格行 5
  - 方法：metadata 过滤
  - 作用：按业务域、权限、版本、文档类型过滤
- 表格行 6
  - 方法：rerank
  - 作用：对候选结果重新排序，让最相关资料靠前
- 表格行 7
  - 方法：bad case 分析
  - 作用：找出召回失败原因，反向优化知识库和检索策略

## 今日实验

今天本地脚本会对比三种策略：

- 表格行 1
  - 策略：baseline_top3
  - 说明：原始问题，取 top 3
- 表格行 2
  - 策略：expanded_top3
  - 说明：对问题做简单关键词改写，取 top 3
- 表格行 3
  - 策略：expanded_top5
  - 说明：改写后取 top 5，观察是否提升命中

评估重点不是答案写得好不好，而是看：

- expected source 有没有被召回
- 正确资料排第几名
- top-k 变化是否带来更多噪声
- query rewrite 是否改善命中
- 哪些问题是 bad case

## 常见坑

- 表格行 1
  - 问题：只看最终答案
  - 影响：找不到真正问题来源
  - 处理方式：先看召回结果和引用
- 表格行 2
  - 问题：只调 prompt
  - 影响：检索错了 prompt 也救不了
  - 处理方式：先优化知识库、chunk、召回
- 表格行 3
  - 问题：top-k 越大越好
  - 影响：噪声和 token 成本升高
  - 处理方式：结合 rerank 和评估集调参
- 表格行 4
  - 问题：没有测试集
  - 影响：优化靠感觉
  - 处理方式：准备标准问题、期望来源和 bad case
- 表格行 5
  - 问题：没有记录失败样例
  - 影响：问题反复出现
  - 处理方式：保存 query、召回结果、原因和修复动作

## 工程取舍

召回优化本质是在三件事之间取平衡：

```text
召回率：尽量别漏掉正确资料
精准率：尽量少带入无关资料
成本和延迟：不要召回太多，拖慢 rerank 和 LLM
```

生产里一般不会只靠一个策略。

更常见的是：

```text
query rewrite
-> 向量检索 + 关键词检索
-> metadata 过滤
-> rerank
-> 选少量高质量 chunk 进入上下文
```

今天的本地练习先不引入真实 rerank 模型，而是用可解释的规则对比 top-k 和 query rewrite。这样能先理解召回优化的评估思路。

## 面试沉淀

Q004：向量检索、关键词检索和混合检索有什么区别？

### 回答

向量检索适合找语义相近的内容，关键词检索适合找表名、字段名、编号、错误码等精确词。混合检索会把两者结合起来，既能处理同义表达，也能保留精确匹配能力。
企业 RAG、数据字典和 NL2SQL 场景通常更适合混合检索，因为业务实体和字段名不能只靠语义相似度。

Q005：rerank 在 RAG 里解决什么问题？

### 回答

rerank 解决的是“召回结果不等于最终可用上下文”的问题。第一阶段检索会拿到一批候选 chunk，但里面可能有噪声、重复或弱相关内容。
rerank 会结合用户问题重新排序，优先选择真正能回答问题的片段进入上下文。这样可以减少无关内容干扰，也能在 top-k 较大时控制最终 prompt 成本。

Q029：RAG 召回质量差时，你会怎么排查和优化？

### 回答

我会先看检索结果，而不是先改 prompt。排查顺序包括：知识库是否有正确资料，chunk 是否切得合理，top-k 是否合适，metadata 过滤是否过严，
关键词和向量检索是否命中，rerank 是否把正确资料排到前面，以及用户问题是否需要 query rewrite。
优化时要用固定测试集和 bad case 复盘，不能只凭感觉调参数。
