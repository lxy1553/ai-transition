## 大白话解释

Day55 的重点是把前几天的能力串起来。

业务方可能不会分开问：

```text
昨天日报怎么样？
近 1 小时有没有实时告警？
通过率下降要不要关注？
还款失败是不是业务异常？
```

他们更可能直接问：

```text
给我一份今天上午的信贷经营摘要，顺便看下有没有实时风险。
```

这时 Agent 不能只调用一个 SQL，也不能只查一个实时指标。
它要按路线调用多个工具：

```text
离线日报工具
-> 实时告警工具
-> 指标口径工具
-> 结果解释
-> 审计记录
```

最后输出一份“有证据、有边界、有建议动作”的摘要。

## 生产实际

一个生产级日报 + 告警 Agent 至少要处理 4 类信息。

| 信息 | 来源 | 作用 |
|------|------|------|
| 离线日报指标 | ADS 日报表 | 说明昨天经营表现 |
| 实时告警 | 告警平台/实时指标 | 说明当前是否有实时风险 |
| 指标口径 | 指标字典/RAG | 防止解释错指标 |
| 审计记录 | Agent 执行日志 | 支撑追溯和复盘 |

例如摘要里可以说：

```text
昨日授信申请量 18600，授信通过率 0.638，较前一日下降 3.2 个百分点；
近 1 小时触发 P1 风控拒绝率突增告警，实时拒绝率 0.42，高于阈值 0.35；
还款失败链路延迟 720 秒，超过 120 秒阈值，因此当前还款失败数不能给确定异常结论。
```

但不能说：

```text
通过率下降是因为欺诈用户变多。
```

除非 Agent 已经查到了渠道流量、黑名单命中、策略版本或上游事件质量等证据。

## Day 55 Agent 编排设计

今天的 Agent 使用 4 个工具：

| 工具 | 作用 | 关键约束 |
|------|------|----------|
| `offline_daily_metric_tool` | 查询昨日 ADS 日报指标 | 必须带日期分区 |
| `realtime_alert_tool` | 查询近 1 小时实时告警 | 必须带窗口和延迟状态 |
| `metric_definition_rag` | 查询指标口径 | 必须返回 citation |
| `audit_logger` | 记录工具调用和结果 | 每一步都要写 request_id |

Agent 输出分成 4 块：

- 离线日报摘要；
- 近 1 小时实时告警；
- 风险提示和排查建议；
- 解释边界和证据来源。

## 常见坑

| 类型 | 可能的问题 | 生产处理方式 |
|------|------------|--------------|
| 只查离线日报 | 忽略当前实时风险 | 同时查近 1 小时告警 |
| 只查实时告警 | 忽略昨日经营背景 | 联合日报和告警解释 |
| 混用离线和实时口径 | 把昨天指标说成当前状态 | 明确离线日期和实时窗口 |
| 编造根因 | 把告警现象说成业务原因 | 只解释证据，原因作为排查建议 |
| 无审计 | 复盘时不知道调用了什么 | 每个工具步骤写审计 |
| 敏感明细混入摘要 | 泄露客户名单 | 只输出聚合指标和告警证据 |

## 工程取舍

### 取舍一：为什么日报和实时告警要分开查？

离线日报来自 ADS 分区表，通常是 T+1 或固定批次产出。
实时告警来自实时指标或告警平台，依赖窗口、事件时间和延迟状态。
两者的时间口径和可信度来源不同，必须分开查询，再在摘要层合并。

### 取舍二：为什么摘要不能直接给根因？

日报和告警摘要通常只能证明“发生了什么”，不一定证明“为什么发生”。
例如通过率下降和风控拒绝率升高可能有关，但也可能来自渠道流量、策略调整、客群变化或数据质量问题。
Agent 应给排查方向，而不是编造确定原因。

### 取舍三：为什么审计记录是 Agent 项目的一部分？

多工具 Agent 不是一次模型回答。
它会查日报、查告警、查口径、合成摘要。
如果没有审计日志，后续无法知道用了哪些数据、哪些工具、哪些证据，也无法复盘 bad case。

## 面试沉淀

Q125：如何设计一个金融信贷日报 + 实时告警 Agent？
重要程度：5/5

### 回答

我会把金融信贷日报 + 实时告警 Agent 设计成多工具编排系统，而不是一个单纯的 SQL 查询。
入口先识别用户是要“经营日报 + 当前风险摘要”，然后分别调用离线日报工具、实时告警工具、指标口径工具和审计工具。

离线日报工具查询 ADS 日报指标，例如授信申请量、授信通过率、放款金额、M1 逾期率，并强制日期分区。
实时告警工具查询近 1 小时实时告警，例如风控拒绝率突增、还款失败链路延迟，并返回窗口、阈值、延迟和证据。
指标口径工具负责解释关键指标，避免 Agent 把通过率、放款率、逾期率讲错。
最后由结果解释层生成摘要、风险提示和建议排查项，同时写审计日志。

这个 Agent 的关键边界是：可以解释查到的指标和告警证据，但不能编造根因；
可以返回聚合指标和告警证据，但不能返回客户手机号、身份证号或客户名单。

Q126：日报 + 实时告警摘要为什么必须区分离线日期和实时窗口？
重要程度：5/5

### 回答

日报 + 实时告警摘要必须区分离线日期和实时窗口，因为两类数据的时间口径完全不同。
离线日报通常来自 ADS 分区表，表示昨天或某个批次的经营结果，重点是 `biz_date`、分区产出和指标口径。
实时告警来自近 5 分钟、近 10 分钟或近 1 小时窗口，重点是事件时间、处理时间、延迟状态和告警规则。

如果把昨天日报指标当成当前状态，就会误导业务；如果把实时 10 分钟波动当成全天经营结果，也会夸大风险。
所以摘要里必须明确：哪些结论来自离线日报，统计日期是什么；哪些结论来自实时告警，窗口和延迟状态是什么。

一句面试总结是：离线日期决定历史经营口径，实时窗口决定当前风险口径，不能混在一起。

Q127：多工具 Agent 为什么必须写审计日志？
重要程度：5/5

### 回答

多工具 Agent 必须写审计日志，因为它的回答不是一次模型生成，而是由多个工具调用共同组成。
如果日报摘要出错，团队需要知道它查了哪个 ADS 分区、用了哪个实时告警窗口、引用了哪个指标口径、有没有触发安全阻断。
没有审计日志，就无法复盘、回归和追责。

审计日志至少应记录 request_id、用户问题、工具名称、输入参数、输出摘要、执行状态、失败原因、证据 ID、时间戳和最终回答。
在金融信贷场景里，审计尤其重要，因为指标口径、风控告警、客户隐私和经营报表都涉及合规和可追溯要求。

一句面试总结是：Agent 的可信不是只靠回答文本，而是靠可追溯的工具链路和审计记录。

## 术语更新

今天新增或强化这些术语：

- 日报 Agent：自动查询离线日报指标并生成经营摘要的 Agent。
- 告警摘要：把实时告警、指标值、阈值、窗口和证据整理成可读结论。
- 多工具编排：一个 Agent 按顺序或条件调用多个工具完成复杂任务。
- 审计日志：记录 request_id、工具调用、输入输出、状态和证据的日志。
- 解释边界：说明哪些结论来自证据，哪些只是建议排查方向。
- 聚合摘要：只返回指标、趋势、告警和证据，不返回客户级敏感明细。

  当前指标值、告警阈值、触发规则、事件时间、处理时间、链路延迟、来源事件、证据 ID 和建议排查方向。
  原因解释要有边界，不能把告警现象编造成未经验证的业务根因。
