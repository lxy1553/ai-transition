## 大白话解释

RAG 的风险不只是“答错”，还有“把不该给用户看的资料答出来”。

比如知识库里有薪酬表、客户名单、手机号、合同金额和内部策略。
如果系统只看语义相似度，不看用户权限和数据敏感等级，就可能把敏感资料召回给普通用户。

所以生产级 RAG 必须把安全控制前移：

```text
先判断资料能不能入库；
再判断用户能不能检索；
最后判断答案能不能返回。
```

## 生产实际

真实公司里，RAG 权限和敏感信息控制通常分成几层：

| 层级 | 控制点 | 说明 |
|------|--------|------|
| 数据层 | 数据分级、脱敏、权限标签 | 决定资料能不能入库 |
| 入库层 | metadata、版本、来源、owner | 决定 chunk 后续怎么过滤 |
| 检索层 | 用户身份、角色、业务域过滤 | 防止无权限 chunk 进入上下文 |
| 生成层 | 只基于引用回答、输出脱敏扫描 | 防止模型泄露或补全敏感信息 |
| 审计层 | request_id、日志、引用记录 | 出问题后能追踪和复盘 |
| 评测层 | 安全测试集、误放行率、误拒率 | 防止改动后破坏安全边界 |

这也是你前面问到的“RAG 是不是分层设计”的具体落地。
权限与敏感信息控制横跨规则层、权限层、检索层、引用层、生成层和评测层。

---

## 常见安全风险

| 风险 | 例子 | 处理方式 |
|------|------|----------|
| 敏感信息入库 | 手机号、身份证、密钥进入 chunk | 入库前脱敏或禁止入库 |
| 越权召回 | 普通用户召回薪酬或客户名单 | 检索前按用户权限过滤 |
| 引用泄露 | citation 暴露敏感文档标题或路径 | citation 也要做权限和脱敏处理 |
| 输出泄露 | 模型把手机号或 token 写进答案 | 输出前做敏感信息扫描 |
| Prompt 注入 | 用户要求忽略权限或输出内部资料 | 规则层拦截，权限层兜底 |
| 审计缺失 | 用户投诉后无法回放链路 | 记录 request_id、用户、召回和答案 |

## 工程取舍

第一版安全控制不要追求复杂，先把三件事做扎实：

1. 每个 chunk 都要有 `security_level` 和 `allowed_roles`；
2. 检索前或检索后必须按用户角色过滤；
3. 答案返回前要做敏感信息扫描。

不要只依赖 prompt。
prompt 可以告诉模型“不要泄露敏感信息”，但它不能替代权限系统、脱敏规则和审计日志。

## 面试沉淀

Q049：RAG 系统怎么做权限控制和敏感信息保护？

### 回答

生产 RAG 的权限和敏感信息保护要贯穿全链路。
入库前要做数据分级、脱敏和权限标签，避免高敏资料直接进入普通知识库。
每个 chunk 要记录 security_level、allowed_roles、业务域、来源和版本。
在线检索时要基于用户身份、角色和业务域过滤 chunk，不能让无权限资料进入 LLM 上下文。
生成时要要求模型只基于授权 citations 回答，答案返回前还要做敏感信息扫描。
最后要记录 request_id、用户、召回结果、引用和答案，方便审计和 bad case 复盘。

Q050：为什么 RAG 不能只靠 prompt 防止敏感信息泄露？

### 回答

prompt 只能约束模型行为，不能替代权限系统和数据治理。
如果敏感 chunk 已经进入上下文，模型就可能直接引用、改写或推断出敏感内容。
用户也可能通过 prompt 注入诱导模型忽略规则。
生产里应该把安全控制放在模型之前：
入库前脱敏，chunk 带权限标签，检索时按用户权限过滤，生成前只传授权上下文。
prompt 只是最后一层提醒，不应该成为唯一边界。

Q051：RAG 的 citations 也会有安全风险吗？

### 回答

citations 也会有安全风险。
引用里可能包含敏感文档标题、路径、chunk 内容、业务域、版本或内部系统地址。
即使答案正文没有泄露，citation 也可能暴露用户无权知道的资料存在。
生产里 citations 要和答案正文一样做权限过滤和脱敏处理。
普通用户只看到允许访问的来源信息；
研发和审计人员可以在后台通过 request_id 查看更完整的召回链路。
