## 大白话解释

RAG 的风险不只是“答错”，还有“把不该给用户看的资料答出来”。

比如知识库里有薪酬表、客户名单、手机号、合同金额和内部策略。如果系统只看语义相似度，不看用户权限和数据敏感等级，就可能把敏感资料召回给普通用户。

所以生产级 RAG 必须把安全控制前移：

```text
先判断资料能不能入库；
再判断用户能不能检索；
最后判断答案能不能返回。
```

## 生产实际

真实公司里，RAG 权限和敏感信息控制通常分成几层：

- 表格行 1
  - 层级：数据层
  - 控制点：数据分级、脱敏、权限标签
  - 说明：决定资料能不能入库
- 表格行 2
  - 层级：入库层
  - 控制点：metadata、版本、来源、owner
  - 说明：决定 chunk 后续怎么过滤
- 表格行 3
  - 层级：检索层
  - 控制点：用户身份、角色、业务域过滤
  - 说明：防止无权限 chunk 进入上下文
- 表格行 4
  - 层级：生成层
  - 控制点：只基于引用回答、输出脱敏扫描
  - 说明：防止模型泄露或补全敏感信息
- 表格行 5
  - 层级：审计层
  - 控制点：request_id、日志、引用记录
  - 说明：出问题后能追踪和复盘
- 表格行 6
  - 层级：评测层
  - 控制点：安全测试集、误放行率、误拒率
  - 说明：防止改动后破坏安全边界

这也是你前面问到的“RAG 是不是分层设计”的具体落地。权限与敏感信息控制横跨规则层、权限层、检索层、引用层、生成层和评测层。

## 常见坑

- 表格行 1
  - 风险：敏感信息入库
  - 例子：手机号、身份证、密钥进入 chunk
  - 处理方式：入库前脱敏或禁止入库
- 表格行 2
  - 风险：越权召回
  - 例子：普通用户召回薪酬或客户名单
  - 处理方式：检索前按用户权限过滤
- 表格行 3
  - 风险：引用泄露
  - 例子：citation 暴露敏感文档标题或路径
  - 处理方式：citation 也要做权限和脱敏处理
- 表格行 4
  - 风险：输出泄露
  - 例子：模型把手机号或 token 写进答案
  - 处理方式：输出前做敏感信息扫描
- 表格行 5
  - 风险：Prompt 注入
  - 例子：用户要求忽略权限或输出内部资料
  - 处理方式：规则层拦截，权限层兜底
- 表格行 6
  - 风险：审计缺失
  - 例子：用户投诉后无法回放链路
  - 处理方式：记录 request_id、用户、召回和答案

## 工程取舍

第一版安全控制不要追求复杂，先把三件事做扎实：

1. 每个 chunk 都要有 `security_level` 和 `allowed_roles`；
2. 检索前或检索后必须按用户角色过滤；
3. 答案返回前要做敏感信息扫描。

不要只依赖 prompt。prompt 可以告诉模型“不要泄露敏感信息”，但它不能替代权限系统、脱敏规则和审计日志。

## 面试沉淀

Q049：RAG 系统怎么做权限控制和敏感信息保护？

### 回答

生产 RAG 的权限和敏感信息保护要贯穿全链路。入库前要做数据分级、脱敏和权限标签，避免高敏资料直接进入普通知识库。
每个 chunk 要记录 security_level、allowed_roles、业务域、来源和版本。
在线检索时要基于用户身份、角色和业务域过滤 chunk，不能让无权限资料进入 LLM 上下文。
生成时要要求模型只基于授权 citations 回答，答案返回前还要做敏感信息扫描。
最后要记录 request_id、用户、召回结果、引用和答案，方便审计和 bad case 复盘。

Q050：为什么 RAG 不能只靠 prompt 防止敏感信息泄露？

### 回答

prompt 只能约束模型行为，不能替代权限系统和数据治理。如果敏感 chunk 已经进入上下文，模型就可能直接引用、改写或推断出敏感内容。
用户也可能通过 prompt 注入诱导模型忽略规则。生产里应该把安全控制放在模型之前： 入库前脱敏，chunk 带权限标签，检索时按用户权限过滤，
生成前只传授权上下文。 prompt 只是最后一层提醒，不应该成为唯一边界。

Q051：RAG 的 citations 也会有安全风险吗？

### 回答

citations 也会有安全风险。引用里可能包含敏感文档标题、路径、chunk 内容、业务域、版本或内部系统地址。
即使答案正文没有泄露，citation 也可能暴露用户无权知道的资料存在。生产里 citations 要和答案正文一样做权限过滤和脱敏处理。
普通用户只看到允许访问的来源信息； 研发和审计人员可以在后台通过 request_id 查看更完整的召回链路。

参考答案：

### 2. 敏感信息进入知识库前应该怎么处理？
我的回答：

资料入库前都有权限标签，问问题的时候会更加用户权限来判断是否有权限访问信息

回答评价：
方向部分正确。你说到了权限标签和按用户权限访问，但题目问的是“进入知识库前”。这里还需要补充数据分级、脱敏、过滤不应入库的高敏资料，以及记录来源和 owner。

参考答案：
敏感信息进入知识库前，应该先做数据分级，判断它是公开、内部、敏感还是机密资料。对手机号、身份证、邮箱、密钥、薪酬和客户名单等内容，要脱敏、摘要化或禁止入库。
允许入库的资料要带上 security_level、allowed_roles、业务域、来源、版本和 owner。
这样在线检索时才能按用户权限过滤，避免敏感 chunk 进入模型上下文。

### 3. chunk metadata 里为什么要有权限标签？
我的回答：

在线链路会根据用户权限信息判断是否有权访问，防止用户越权获得权限之外的信息，而且不仅返回问题需要权限，上下文和引用也需要权限，citations也可能泄露信息

回答评价：
回答正确。你已经说清楚了 metadata 权限标签用于在线过滤，并且指出上下文和 citations 也需要权限控制。这是 Day 25 最关键的点。

参考答案：
chunk metadata 里要有权限标签，是因为 RAG 检索的最小单位通常是 chunk。
如果 chunk 没有 allowed_roles、security_level、business_domain 等信息，
系统就无法判断当前用户是否能看这段资料。权限标签可以用于检索过滤、上下文构造、citations 展示和审计回放。
这样能防止无权限资料进入 LLM，也能防止引用来源泄露内部信息。

### 4. 为什么不能只靠 prompt 防止敏感信息泄露？
我的回答：

不能代替权限层是系统进行权限的限制

回答评价：
回答方向正确，但偏短。prompt 不能代替权限层，因为敏感内容一旦进入上下文，模型可能直接引用、改写或推断出来。

参考答案：
不能只靠 prompt 防止敏感信息泄露，因为 prompt 只是对模型的行为约束，不是权限系统。
如果敏感 chunk 已经被召回并放进上下文，模型可能直接引用、 总结或推断出敏感内容。用户也可能通过 prompt 注入要求模型忽略规则。
生产里要在模型之前完成数据分级、入库脱敏、权限标签、检索过滤和上下文过滤； prompt 只能作为最后一层提醒，不能作为唯一边界。

### 5. citations 为什么也可能泄露信息？
我的回答：

防止获取到用户并没有权限访问的文档 chunk，metadata

回答评价：
方向正确，但表述可以更完整。citations 不只是文档 chunk，也可能暴露文档标题、路径、业务域、版本、内部系统地址和资料是否存在。

参考答案：
citations 也可能泄露信息，因为引用来源里通常包含文档标题、路径、chunk_id、metadata、业务域、版本和原文位置。

### 6. 生产里如何审计一次 RAG 越权或敏感信息问题？

回答评价：
方向正确，但需要补充审计要记录什么。一次越权或敏感信息问题，不能只记录“没有权限”，还要能回放用户、问题、召回、引用、答案和策略命中情况。

参考答案：

审计结果要反哺规则、权限配置、测试集和 bad case，避免同类问题再次发生。
