## 1. LLM API 真实调用

### 它是什么

LLM API 真实调用，就是不再只停留在本地模拟，而是通过 HTTP/API 调用真实大模型服务。

大白话：

> 以前我们是在本地写规则模拟模型行为；真实调用就是把 prompt 发给模型服务，让模型真正返回结果。

一次真实调用通常包括：

- 表格行 1
  - 部分：API Key
  - 作用：身份凭证，证明你有权限调用
- 表格行 2
  - 部分：model
  - 作用：选择使用哪个模型
- 表格行 3
  - 部分：messages / input
  - 作用：发送给模型的内容
- 表格行 4
  - 部分：parameters
  - 作用：控制输出稳定性、长度、成本
- 表格行 5
  - 部分：response
  - 作用：模型返回的结果

### 为什么要补

因为后续 RAG / NL2SQL 项目最终一定要接真实模型。

如果只会写本地规则，就还停留在“模拟 AI 应用”；会调用 LLM API 后，才进入真正的 AI 应用开发。

### 和 SQL 解释助手的关系

当前 SQL 解释助手已经能输出规则结果：

```json
{
  "risk_level": "medium",
  "risks": ["group by 可能产生较大的聚合开销"],
  "missing_context": ["active_users 活跃用户业务口径"]
}
```

后续可以把这段 JSON 交给 LLM，让模型生成自然语言解释：

```text
这段 SQL 当前不建议直接上线，因为它包含 group by 聚合，并且 active_users 的业务口径尚未确认。
```

### 最小练习

先不要一上来做复杂项目，只做一个最小调用：

```text
输入：请用一句话解释 RAG 是什么
输出：模型返回一句解释
```

然后再接入 SQL 解释助手：

```text
输入：SQL 风险 JSON
输出：适合业务人员阅读的解释
```

### 面试表达

> 我理解 LLM API 调用不是简单把问题发给模型，而是要控制输入结构、模型参数、输出格式、异常处理和调用成本。
后续我会把 SQL 解释助手的规则分析结果作为上下文传给模型，让模型负责生成更自然的解释。

## 2. Prompt 模板版本管理

### 它是什么

Prompt 模板版本管理，就是把 prompt 当成项目资产来管理，而不是每次临时手写。

大白话：

> Prompt 也像代码一样，需要有版本、记录变化、能回滚、能比较效果。

例如：

```text
sql_explain_v1：只解释 SQL
sql_explain_v2：增加风险提示
sql_explain_v3：要求输出 JSON
sql_explain_v4：增加缺失上下文判断
```

### 为什么要补

真实项目里，prompt 经常会改。

如果不管理版本，会出现几个问题：

- 不知道线上现在用的是哪版 prompt
- 改完之后效果变差，不知道怎么回滚
- 不同人改 prompt，结果互相覆盖
- 没法比较 v1 和 v2 哪个效果更好

### 和 SQL 解释助手的关系

SQL 解释助手后续接 LLM 时，至少会有这些 prompt：

- 表格行 1
  - 模板：`sql_explain_v1`
  - 作用：解释 SQL 做什么
- 表格行 2
  - 模板：`sql_risk_v1`
  - 作用：解释风险和建议
- 表格行 3
  - 模板：`business_friendly_v1`
  - 作用：把技术解释改写给业务看
- 表格行 4
  - 模板：`json_output_v1`
  - 作用：强制输出结构化 JSON

### 推荐做法

可以先在项目里建一个目录：

```text
prompts/
├── sql_explain_v1.md
├── sql_risk_v1.md
└── business_friendly_v1.md
```

每个 prompt 里记录：

- 适用场景
- 输入变量
- 输出格式
- 修改记录

### 面试表达

> 我会把 Prompt 当成可维护资产来管理，而不是写在代码里的临时字符串。
对于 SQL 解释、风险判断、业务化表达这类不同任务，我会拆成不同模板，并通过版本记录和测试样例比较效果。

## 3. JSON 输出失败后的重试策略

### 它是什么

JSON 输出失败，就是模型没有按要求返回合法或可用 JSON。

常见失败：

```text
1. 返回了普通文字，不是 JSON
2. JSON 格式不合法，少了引号或逗号
3. JSON 合法，但缺字段
4. 字段类型不对
5. risk_level 不是 low / medium / high
```

重试策略就是：当模型输出不符合要求时，系统怎么补救。

### 为什么要补

真实系统不能假设模型每次都听话。

如果模型输出坏 JSON，后续系统直接解析就会报错；如果业务字段不合法，系统可能做出错误决策。

### 重试流程

推荐流程：

```text
第一次调用模型
-> 尝试解析 JSON
-> 字段和业务规则校验
-> 如果失败，带着错误信息重试一次
-> 如果仍失败，进入人工复核或返回明确错误
```

### 示例

模型第一次返回：

```json
{
  "risk_level": "比较高",
  "can_publish": true
}
```

系统校验失败：

```text
risk_level 必须是 low / medium / high
缺少 risks
缺少 suggestions
```

第二次 prompt 可以这样写：

```text
你上一次输出不符合 JSON Schema：
- risk_level 必须是 low / medium / high
- 必须包含 risks 和 suggestions

请只返回修正后的 JSON，不要输出解释文字。
```

### 面试表达

> 我不会默认模型一定返回正确 JSON。
服务端会先做 JSON 解析，再做字段、类型、枚举和业务规则校验。校验失败时，可以把错误信息反馈给模型重试；多次失败后进入人工复核或返回明确错误。

## 4. SQL 解析准确性

### 它是什么

SQL 解析准确性，就是系统能不能正确理解 SQL 里的表、字段、过滤条件、聚合、排序、子查询等结构。

当前 SQL 解释助手用的是简单正则：

```sql
from xxx
join xxx
select ... from
```

这种方式适合学习，但不适合复杂 SQL。

### 为什么要补

真实 SQL 可能很复杂：

- 多表 join
- 子查询
- CTE：`with tmp as (...)`
- 字段别名
- 函数嵌套
- 不同 SQL 方言：Hive、MySQL、Spark SQL

如果解析不准，后续解释和风险判断都会出错。

### 当前正则方式的局限

例如：

```sql
with t as (
  select user_id, city
  from user_events
  where dt = '2026-05-09'
)
select city, count(*) as cnt
from t
group by city;
```

简单正则可能只识别到 `t`，但真正的底层表是 `user_events`。

### 后续增强方向

学习阶段可以先继续用规则。

真实项目可以考虑：

- 引入 SQL 解析库
- 针对常用 SQL 方言做兼容
- 把 SQL 解析结果做成结构化对象
- 对无法解析的 SQL 返回“需要人工确认”

### 面试表达

> 当前项目用正则实现 SQL 解析，是为了快速验证工程流程。
真实项目中我会引入 SQL 解析库。它可以提升复杂 SQL、CTE、join、字段别名和不同方言的解析准确性。在无法确定时明确返回需要人工确认，而不是强行解释。

## 5. RAG 的文档切分、Embedding、检索和引用

### 它是什么

RAG 是“先检索资料，再让模型回答”。

完整流程：

```text
原始文档
-> 文档切分 chunk
-> 生成 embedding
-> 存入向量库
-> 用户提问
-> 检索 top-k 相关片段
-> 把片段交给 LLM
-> 生成带依据的回答
```

### 文档切分 Chunk

大白话：

> 把长文档切成小段，方便检索。

如果整篇文档太长，检索时不精准；切得太碎，又可能丢上下文。

常见切分方式：

- 按标题切
- 按段落切
- 按问答切
- 按表或指标切

### Embedding

大白话：

> 把文字变成一串数字，用来计算语义相似度。

例如：

```text
活跃用户怎么算
```

和：

```text
active_users 指标口径
```

虽然字面不同，但语义接近，向量距离应该比较近。

### 检索 Top-k

大白话：

> 用户提问后，从知识库里找最相关的前 k 个片段。

例如：

```text
top_k = 5
```

就是取最相关的 5 个文档片段。

### 引用 Citation

大白话：

> 回答时告诉用户依据来自哪段文档。

例如：

```text
active_users 指在指定时间范围内发生 app_open 事件的去重用户数。
依据：指标口径文档 第 2 节
```

### 和 SQL 解释助手的关系

当前 SQL 解释助手遇到：

```text
active_users 活跃用户业务口径
```

只能提示缺少上下文。

接入 RAG 后，它可以：

```text
检索 active_users 指标口径
-> 找到定义
-> 把定义交给 LLM
-> 生成解释
```

### 面试表达

> 我理解 RAG 的关键不是简单把文档塞给模型，而是先做好文档切分、Embedding、检索和引用。
对于数据问答场景，知识库质量尤其重要，因为字段口径、指标定义和表结构会直接影响回答准确性。

## 面试沉淀

Q018：一次 LLM 调用需要关注哪些参数和工程问题？
重要程度：5/5

### 回答

一次 LLM 调用要关注模型版本、Prompt、temperature、max_tokens、超时、重试、成本、日志和输出校验。
这些参数会影响稳定性、可控性和线上成本。

Q017：Prompt 在生产项目里应该怎么管理？
重要程度：5/5

### 回答

Prompt 要像配置和代码一样管理版本。每次改动都要记录原因，并用固定样例回归格式稳定性、准确率和 bad case。

Q015：JSON 合法和业务校验通过有什么区别？
重要程度：5/5

### 回答

JSON 合法只代表能解析，业务校验通过才代表字段、类型、枚举和业务约束都满足系统要求。生产里两层校验都要做，否则模型输出可能看起来合法但不能被系统使用。

Q023：为什么 SQL 风险检查不能完全依赖 LLM？
重要程度：4/5

### 回答

SQL 风险检查里有很多确定性规则，比如分区条件、危险操作、SELECT * 和 JOIN 风险。这些应该由规则或解析器兜底，LLM 更适合解释含义和补充建议。

Q003：RAG 文档切分时怎么设计 chunk？
重要程度：5/5

### 回答

chunk 要按标题、段落、语义边界或业务对象切分，并保留 source、版本、权限和位置等 metadata。目标是在召回精准和上下文完整之间取得平衡。
