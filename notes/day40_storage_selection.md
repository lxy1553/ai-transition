## 大白话解释

NL2SQL 服务不只是返回答案，还要能回放一次请求。

比如用户反馈“这个答案不对”，研发要能查到：

- 用户问了什么；
- 最终状态是什么；
- 有没有安全阻断；
- SQL 是否生成；
- 校验是否通过；
- 查询是否执行；
- 返回了多少行；
- 业务解释是什么。

这些都需要审计存储。

---

## 生产实际

金融信贷系统对审计要求更高。
因为查询可能涉及授信、风控、逾期、客户明细和敏感字段。

生产审计通常至少保存：

- `request_id`；
- `user_id`；
- `question`；
- parsed result；
- SQL；
- validation issues；
- execution status；
- row count；
- final status；
- created time；
- request source；
- error code；
- policy decision。

Day 40 使用 SQLite 是为了本地可运行。
生产多实例、高并发、权限控制和长期留存场景，应该使用 Postgres、MySQL 或公司审计平台。

## 存储链路图

```mermaid
flowchart TD
    Ask[POST /nl2sql/ask] --> Service[NL2SQL Service]
    Service --> Result[生成回答或安全阻断]
    Service --> Audit[AuditStore.save]
    Audit --> SQLite[(本地 SQLite)]
    SQLite --> Trace[GET /nl2sql/trace/{request_id}]
    Trace --> Debug[研发排查 / 审计回放]
```

这条链路说明：审计存储不是额外装饰，而是 trace 接口能工作的基础。

## SQLite 和 Postgres 取舍

| 维度 | SQLite | Postgres |
|------|--------|----------|
| 部署成本 | 低，一个本地文件即可 | 需要独立数据库服务 |
| 本地学习 | 很适合 | 偏重 |
| 并发写入 | 不适合高并发 | 更适合高并发和多连接 |
| 多实例共享 | 不适合 | 适合 |
| 权限控制 | 很弱 | 支持账号、角色、权限和审计 |
| 生产长期留存 | 不推荐 | 推荐 |
| 当前项目定位 | 本地演示和学习 | 生产替换方向 |

当前项目先用 SQLite 是合理的，因为它让学习版无需额外数据库就能跑通审计闭环。
但如果服务进入多人使用、多实例部署或长期留存，就应该升级到 Postgres 或公司统一审计平台。

## 常见坑

| 类型 | 可能的问题 | 生产处理方式 |
|------|------------|--------------|
| 数据 | 只保存最终答案，不保存中间结果 | 保存关键链路状态和阻断原因 |
| 安全 | 审计表里保存敏感明文 | 对敏感字段脱敏或加密，控制访问权限 |
| 留存 | 审计文件写在容器临时目录 | 使用外部数据库或持久化挂载 |
| 并发 | 多实例同时写 SQLite | 生产切换到 Postgres / MySQL |
| 查询 | trace 接口无权限控制 | trace 只给管理员、研发或审计角色 |
| 成本 | 保存完整大结果导致存储膨胀 | 大结果抽样、摘要或放对象存储 |

## 工程取舍

### 取舍一：为什么学习版先用 SQLite？

SQLite 不需要安装数据库服务，一个文件就能保存审计记录。
它非常适合本地学习、面试演示和小规模样例。
当前阶段重点是证明审计链路存在，而不是搭建复杂数据库基础设施。

### 取舍二：什么时候升级到 Postgres？

当服务需要多人使用、多实例部署、高并发写入、权限管理、长期留存、复杂查询或接公司审计平台时，
就应该从 SQLite 升级到 Postgres、MySQL 或公司统一数据服务。

### 取舍三：审计记录和普通日志有什么区别？

普通日志偏运行过程，比如接口启动、异常堆栈和调试信息。
审计记录偏业务追责和回放，重点回答谁在什么时间问了什么、系统做了什么判断、是否触发安全阻断、最终返回了什么。
金融信贷场景里，审计记录比普通日志更接近合规证据。

## 面试沉淀

Q090：NL2SQL 服务为什么需要审计存储？

### 回答

NL2SQL 服务需要审计存储，因为每次自然语言查询都可能涉及业务口径、权限、安全和成本风险。
当用户质疑结果或出现越权风险时，研发需要回放完整链路：
用户问了什么、解析出了什么、生成了什么 SQL、校验是否通过、执行状态是什么、返回了多少行、
最后如何解释给用户。

在金融信贷场景里，授信、风控、逾期和客户明细都有合规要求。
审计存储可以支持排查 bad case、追踪责任、证明敏感查询被阻断，并为后续评测优化提供数据。
如果没有审计存储，NL2SQL 服务一旦答错或被质疑，只能靠日志碎片和人工回忆排查，无法形成可追溯闭环。

## 术语更新

今天新增或强化这些术语：

- 审计存储：专门保存请求、状态、链路详情和风险判断的存储。
- SQLite：轻量级本地数据库，适合学习版和小规模演示。
- Postgres：生产常用关系型数据库，适合多用户、高并发和长期留存。
- Trace：一次请求的完整回放信息。
- 数据留存：按合规要求保存审计记录一段时间。

### 7. [Day 39] 为什么密钥不能写入代码仓库？
  我的回答：
容易暴露密钥，导致核心配置暴露，安全问题

回答评价：
通过。回答正确指出密钥写入代码仓库会导致密钥和核心配置泄露。
可以再补充：泄露后可能导致数据库、模型账号或内部服务被滥用。

参考答案：
密钥不能写入代码仓库，因为仓库可能被多人访问、复制、打包或泄露。
一旦数据库密码、模型 API key 或内部服务 token 被提交，可能导致业务数据泄露、模型账户被盗用、内部服务被非法调用，
还可能产生费用和合规风险。
真实密钥应该通过环境变量、密钥系统或配置中心注入，并且泄露后要及时轮换。
