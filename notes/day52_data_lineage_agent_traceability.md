## 大白话解释

数据血缘可以理解为“数据家谱”。

它回答这些问题：

- 这个指标来自哪些上游表；
- 中间经过了哪些清洗、汇总和发布任务；
- 哪些下游报表、指标或告警依赖它；
- 如果某张表或某个任务异常，会影响哪些结果；
- 一个实时告警是从哪个事件、哪个窗口任务、哪个告警规则来的。

指标字典回答“指标怎么算”。
Schema Catalog 回答“有哪些表字段能查”。
数据血缘回答“数据从哪里来、流向哪里、哪里出问题会影响谁”。

这三类信息都很重要，但不能混用。
用户问“授信通过率怎么算”，应该查指标字典。
用户问“近 7 天授信通过率是多少”，应该走离线 NL2SQL。
用户问“授信通过率来自哪些表，会影响哪些报表”，才应该走血缘工具。

## 生产实际

金融信贷场景里，血缘追溯通常用于三类问题。

第一类是来源追溯：

```text
授信通过率来自哪些上游表？
```

Agent 应该返回：

```text
ODS 授信申请订单表
-> DWD 授信申请明细表
-> DWS 授信渠道日汇总表
-> ADS 信贷经营日报指标表
-> 授信通过率指标
```

第二类是影响分析：

```text
如果 dws_credit_apply_channel_1d 异常，会影响哪些下游？
```

Agent 应该返回受影响的 ADS 表、指标和报表。
这在生产事故排查里很关键，因为数据团队需要快速判断影响范围、通知哪些业务方、是否暂停报表使用。

第三类是实时告警追溯：

```text
实时风控拒绝率告警来自哪些事件和任务？
```

Agent 应该说明：

```text
risk_decision_made 事件
-> 实时 10 分钟窗口计算任务
-> rt_risk_reject_rate_10m 实时指标
-> 风控拒绝率突增告警
```

同时要带上窗口、延迟、告警阈值和规则证据。
如果实时任务延迟或事件乱序严重，Agent 不能只解释告警，还要提示当前证据可信度下降。

## 常见坑

| 类型 | 可能的问题 | 生产处理方式 |
|------|------------|--------------|
| 只返回表名 | 用户不知道路径和任务 | 返回节点、边、任务和证据 |
| 血缘和口径混用 | 问来源却解释公式 | 先识别意图，血缘问题走血缘工具 |
| 只看上游不看下游 | 无法判断事故影响范围 | 同时支持 upstream 和 downstream |
| 实时血缘忽略延迟 | 告警证据可能不可信 | 返回事件时间、窗口、延迟和规则 |
| 明细血缘泄露敏感字段 | 可能越权暴露客户信息 | 血缘工具也要做权限和敏感字段阻断 |
| 没有版本信息 | 事故回放时对不上 | 生产里应记录任务版本、口径版本和运行批次 |

## 工程取舍

### 取舍一：为什么血缘不是普通 RAG 文档？

血缘是结构化图关系，不只是自然语言文档。
它包含节点、边、任务、依赖方向、版本和影响范围。
如果只用 RAG 文档回答，模型可能漏掉某个下游报表，或者把上游和下游方向说反。

更稳的做法是：血缘图用结构化数据存储，Agent 通过血缘工具查询路径，再把结果解释成业务语言。

### 取舍二：为什么血缘回答必须有证据？

血缘问题通常用于生产排查和审计。
如果 Agent 说“这个指标受 DWS 表影响”，但不给任务名、依赖边或版本证据，排查人员很难确认。
所以回答要带上来源节点、目标节点、依赖关系、加工任务、告警规则或报表 ID。

### 取舍三：为什么血缘工具也要做权限控制？

血缘工具虽然主要回答“来源和影响”，但它可能暴露敏感表、敏感字段或客户明细入口。
例如 `dwd_credit_apply_detail_di` 里有手机号和身份证号。
Agent 可以说明这张表是上游依赖，但不能继续展开敏感字段或导出客户明细。

## 面试沉淀

Q116：数据血缘在金融信贷 Agent 里解决什么问题？
重要程度：5/5

### 回答

数据血缘在金融信贷 Agent 里主要解决来源追溯、影响分析、异常排查和审计解释。
它回答的不是“指标怎么算”，也不是“指标当前是多少”，而是“这个数据从哪里来、经过哪些任务、会影响哪些下游”。

例如授信通过率的血缘可以从 ODS 授信申请订单表追到 DWD 明细表、DWS 渠道汇总表、ADS 日报指标表，
再追到授信通过率指标和信贷经营日报。
如果 DWS 汇总任务异常，Agent 可以基于血缘判断哪些 ADS 表、指标和报表可能受影响。
实时场景里，风控拒绝率告警可以追溯到风控决策事件、实时窗口计算任务、实时指标和告警规则。

生产级 Agent 回答血缘问题时，应该返回节点、路径、加工任务、下游影响和证据，而不是只用自然语言猜测。
同时血缘工具也要遵守权限和敏感字段控制，不能因为查询血缘就暴露客户手机号、身份证号或明细名单。

Q117：离线指标血缘和实时告警血缘有什么区别？
重要程度：5/5

### 回答

离线指标血缘关注批处理链路，通常是 ODS 到 DWD、DWS、ADS，再到指标、报表或下游应用。
它重点看分区日期、调度任务、依赖表、产出表、指标口径版本和报表影响范围。
比如授信通过率异常，要排查 ADS 日报指标表、DWS 渠道汇总、DWD 申请明细、ODS 原始申请单以及对应日批任务。

实时告警血缘关注流处理链路，通常是实时事件、消息流、实时计算任务、窗口指标、延迟状态、告警规则和通知渠道。
它重点看事件时间、处理时间、窗口大小、乱序、重复、延迟阈值和告警规则。
比如实时风控拒绝率突增，要追溯 `risk_decision_made` 事件、10 分钟窗口计算任务、实时拒绝率指标和告警规则。

一句面试总结是：离线血缘重在批次、表依赖和报表影响；实时血缘重在事件、窗口、延迟和告警证据。

Q118：Agent 做血缘追溯时，为什么必须返回证据和影响范围？
重要程度：5/5

### 回答

血缘追溯通常服务生产排查和审计，不能只回答“可能是某张表的问题”。
如果没有证据，排查人员无法确认 Agent 的结论来自真实血缘图还是模型猜测。
如果没有影响范围，数据团队也不知道要通知哪些业务方、暂停哪些报表、回补哪些任务。

生产级血缘回答至少要包含上游节点、下游节点、依赖关系、加工任务、指标或报表 ID、告警规则和必要的版本信息。
例如 `dws_credit_apply_channel_1d` 异常时，要说明它会影响 `ads_credit_daily_metrics`、
`metric_credit_approval_rate` 和 `report_credit_operation_daily`，并给出对应调度任务或依赖边。

在金融信贷场景里，错误报表可能影响授信经营分析、风控策略复盘、贷后逾期判断和管理层日报。
因此 Agent 的血缘回答必须可追溯、可复核、可审计。

## 术语更新

今天新增或强化这些术语：

- 数据血缘：描述数据从上游来源到下游报表、指标或告警的依赖关系。
- 上游追溯：从指标、报表或告警向前查来源表、事件和加工任务。
- 下游影响分析：从某张表、任务或事件向后查受影响指标、报表和告警。
- 血缘节点：血缘图中的表、事件、任务、指标、报表或告警。
- 血缘边：血缘图中两个节点之间的依赖关系和证据。
- 实时告警血缘：实时事件到窗口任务、实时指标和告警规则的链路。
