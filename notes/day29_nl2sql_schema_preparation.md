## 大白话解释

NL2SQL 是把用户的人话问题转成 SQL。比如用户问：

```text
上周每个渠道的授信通过率是多少？
```

系统最终可能要生成：

```sql
select channel, approval_rate
from dws_credit_application_daily
where dt between '2026-05-18' and '2026-05-24'
group by channel;
```

但真正困难的地方不是“写 SQL 语法”，而是理解业务问题应该查哪张表、用哪个字段、按什么维度分组、用哪个时间字段过滤，以及这个用户有没有权限查这类数据。

所以生产级 NL2SQL 一般不会直接把用户问题丢给模型生成 SQL。更稳的做法是先做 Schema 准备：

- 有哪些表；
- 每张表有哪些字段；
- 字段是什么意思；
- 哪些字段是指标；
- 哪些字段是维度；
- 哪些字段是时间或分区；
- 哪些表或字段有权限限制；
- 常见业务问题应该映射到哪些候选表。

## 生产实际

真实公司的 NL2SQL 链路通常至少包含这些步骤：

1. 用户输入自然语言问题。
2. 系统识别问题类型，比如指标查询、明细查询、趋势查询、TopN 查询或对比查询。
3. 系统根据业务域、关键词、指标名和字段别名找到候选表。
4. 系统抽取指标、维度、时间范围和过滤条件。
5. 系统生成 SQL 草稿。
6. SQL 校验模块检查表字段是否存在、权限是否允许、是否有分区过滤、是否有危险扫描。
7. 执行 SQL 或返回待确认的 SQL。
8. 把查询结果解释成业务语言。

Schema 准备是这条链路的地基。如果表结构和字段解释不清楚，模型就容易编造不存在的字段，或者把授信申请量、授信通过率、放款金额、逾期率这些口径混在一起。

对于金融信贷开发工程师，这是很强的迁移优势。因为 NL2SQL 需要的不是“会写一句 SQL”，而是理解授信、风控、放款、还款、逾期、贷后这些业务流程，
以及事实表、 维度表、指标口径、分区字段和权限边界。

## Schema Catalog 应该包含什么

今天先用一个小型 Schema Catalog 表示信贷数据问答需要的表结构信息。

- 表格行 1
  - 信息：table_name
  - 作用：让系统知道可用表名，禁止模型编造表
- 表格行 2
  - 信息：description
  - 作用：帮助模型理解表的业务用途
- 表格行 3
  - 信息：business_domain
  - 作用：按业务域筛选候选表
- 表格行 4
  - 信息：grain
  - 作用：表粒度，比如申请天、放款天、还款计划明细、客户维度
- 表格行 5
  - 信息：columns
  - 作用：字段名、类型、含义和角色
- 表格行 6
  - 信息：metrics
  - 作用：表里可直接统计或派生的指标
- 表格行 7
  - 信息：dimensions
  - 作用：可用于 group by 的维度
- 表格行 8
  - 信息：time_fields
  - 作用：时间过滤和分区字段
- 表格行 9
  - 信息：permission_level
  - 作用：判断当前用户是否能查询
- 表格行 10
  - 信息：sample_questions
  - 作用：训练系统做候选表匹配

生产里这些信息可能来自数据地图、元数据平台、指标平台、Hive Metastore、dbt 文档或数据治理系统。本地练习先用 JSON 文件模拟。

## 问题类型

NL2SQL 不是所有问题都按同一种方式生成 SQL。先分类能降低后续生成难度。

- 表格行 1
  - 问题类型：指标查询
  - 用户问题例子：昨天授信申请量是多少？
  - 典型 SQL 特征：聚合函数、时间过滤
- 表格行 2
  - 问题类型：维度分组
  - 用户问题例子：上周每个渠道的授信通过率是多少？
  - 典型 SQL 特征：group by 维度
- 表格行 3
  - 问题类型：趋势查询
  - 用户问题例子：最近 7 天放款金额趋势怎么样？
  - 典型 SQL 特征：按日期 group by
- 表格行 4
  - 问题类型：TopN 查询
  - 用户问题例子：上周放款金额最高的 10 个城市是什么？
  - 典型 SQL 特征：order by + limit
- 表格行 5
  - 问题类型：对比查询
  - 用户问题例子：本周逾期率比上周变化多少？
  - 典型 SQL 特征：两个时间窗口对比
- 表格行 6
  - 问题类型：明细查询
  - 用户问题例子：查询申请 A123 的审批状态
  - 典型 SQL 特征：where 精确过滤
- 表格行 7
  - 问题类型：不适合回答
  - 用户问题例子：导出客户手机号列表
  - 典型 SQL 特征：权限或敏感信息拦截

问题分类不是为了展示标签，而是为了后面生成 SQL 时控制结构。比如趋势查询必须按时间维度分组，TopN 必须限制 limit，明细查询要避免扫全表。

## 常见坑

- 表格行 1
  - 类型：Schema
  - 可能的问题：模型编造不存在的表或字段
  - 生产处理方式：只允许使用 Schema Catalog 中的表字段
- 表格行 2
  - 类型：指标
  - 可能的问题：授信申请量、授信通过率、放款金额、逾期率口径混淆
  - 生产处理方式：接入指标口径和字段别名
- 表格行 3
  - 类型：时间
  - 可能的问题：用户说“最近”，没有明确日期范围
  - 生产处理方式：做时间解析或澄清
- 表格行 4
  - 类型：粒度
  - 可能的问题：用申请明细表统计渠道日指标，导致重复
  - 生产处理方式：记录表粒度和可聚合方式
- 表格行 5
  - 类型：权限
  - 可能的问题：普通用户查询客户手机号、身份证号或授信明细
  - 生产处理方式：表字段权限过滤和审计
- 表格行 6
  - 类型：性能
  - 可能的问题：没有分区条件导致扫描大表
  - 生产处理方式：强制 dt 分区过滤或拒绝执行
- 表格行 7
  - 类型：质量
  - 可能的问题：候选表选错，SQL 语法再对也没用
  - 生产处理方式：先评估候选表命中率

## 工程取舍

Day 29 先做规则版 Schema Router，而不是直接接 LLM。原因很简单：

- Schema 匹配里有很多确定性信息，比如表名、字段名、指标别名和权限等级。
- 规则版容易解释和调试，适合先建立基线。
- 后续接 LLM 时，也应该把 Schema Catalog 作为上下文，而不是让模型自由发挥。
- 先做候选表和字段推荐，可以降低 SQL 生成阶段的搜索空间。

生产里更稳的设计通常是“规则 + 检索 + LLM”组合： 规则负责硬约束，检索负责找相关 schema 和指标口径，LLM 负责理解复杂表达并生成 SQL
草稿。

## 面试沉淀

Q065：NL2SQL 为什么不能直接把用户问题交给模型生成 SQL？
重要程度：5/5

### 回答

生产环境里不能直接把用户问题交给模型生成 SQL。因为模型可能编造不存在的表和字段，混淆指标口径，漏掉分区条件，生成高成本 SQL，或者绕过权限查询敏感数据。
更稳的做法是先提供 Schema Catalog、指标口径、权限规则和 SQL 约束，让模型只能在可用表字段和业务规则范围内生成 SQL。
生成后还要做 SQL 校验、权限检查和执行风险控制。

Q066：NL2SQL 的 Schema Catalog 应该包含哪些信息？
重要程度：5/5

### 回答

Schema Catalog 至少要包含表名、表描述、业务域、表粒度、字段名、字段类型、字段含义、指标字段、维度字段、时间字段、分区字段、
权限等级和常见问题示例。
这些信息能帮助系统判断用户问题应该查哪张表、用哪些字段、按什么维度聚合，也能限制模型不要编造表字段。
生产里 Schema Catalog 通常来自元数据平台、指标平台、数据地图、Hive Metastore 或 dbt 文档。

Q067：NL2SQL 问题分类有什么价值？
重要程度：4/5

### 回答

NL2SQL 问题分类能降低 SQL 生成难度。指标查询、维度分组、趋势查询、TopN、对比查询和明细查询，对应的 SQL 结构不一样。
先识别问题类型，可以决定是否需要聚合、group by、order by、limit、时间窗口或精确过滤。
如果问题涉及敏感信息或权限不足，还可以在生成 SQL 前拦截。 这比直接生成 SQL 更容易控制质量、成本和安全风险。

### 4. 为什么 NL2SQL 必须关注表粒度和指标口径？
  重要程度：5/5
我的回答：
方便筛选，分组，减少计算成本

回答评价：
评分：8/10
方向部分正确。筛选、分组和成本确实相关，但这题最核心的是“算出来的数是否业务正确”。表粒度选错会导致重复计算或漏算；
指标口径不清楚会导致 SQL 能跑但结果不可信。

参考答案：
NL2SQL 必须关注表粒度和指标口径，因为 SQL 结果是否正确，不只取决于语法是否能执行。
表粒度决定一行数据代表什么，比如授信申请明细表是一行一笔申请， 渠道日汇总表是一行一个渠道一天。如果粒度选错，可能重复计数、漏算或错误 join。
指标口径决定一个指标到底怎么算，比如授信通过率分母是否包含撤销申请、 放款金额是否按放款成功时间统计、逾期率按客户数还是本金余额计算。
生产里必须把表粒度和指标口径放进 Schema Catalog， 否则模型生成的 SQL 可能看起来正确但业务结果错误。
