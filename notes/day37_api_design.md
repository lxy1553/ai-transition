## 大白话解释

API 契约就是前端、机器人、业务系统和后端服务之间的“约定书”。

对于 NL2SQL 服务来说，调用方不能只知道“发一个问题，返回一个答案”。它还必须知道：

- 请求字段有哪些，哪些必填，长度限制是什么；
- 成功时返回哪些字段；
- 安全阻断算不算系统故障；
- 参数错误、业务错误和内部错误分别是什么格式；
- `request_id` 怎么拿，后续怎么查 trace；
- SQL、pipeline、中间结果是否对外暴露。

没有稳定契约，前端和业务系统就只能猜字段。今天返回 `answer`，明天改成 `result`，下游就会出错。
在生产系统里，接口字段不是随手写的 JSON，而是跨系统协作的边界。

## 生产实际

金融信贷系统里的 NL2SQL API 通常会接到数据门户、运营后台、业务机器人、风控分析工具或内部 BI 平台。
这些调用方关心的不只是答案，还关心本次查询是否合规、是否可追踪、是否可以给业务用户展示。

例如用户问：

```text
导出客户手机号列表
```

系统安全阻断后，不应该返回 500。
因为这不是服务故障，而是安全策略按预期生效。

更合理的处理是：

- `final_status=answered`：正常回答；
- `final_status=safely_blocked`：命中敏感字段、危险 SQL、权限边界或高成本规则，被安全阻断；
- HTTP 422：请求参数不合法，例如 `question` 为空或超过长度；
- HTTP 404：演示版不支持该问题，或查不到指定 `request_id`；
- HTTP 500：服务内部异常，例如依赖文件损坏、数据库连接失败或代码 bug。

这样前端可以根据 `final_status` 展示业务提示，根据 HTTP 状态码处理系统错误。
审计系统也能通过 `request_id` 回放用户问题、SQL、校验结果、执行状态和最终解释。

## 接口链路图

```mermaid
flowchart TD
    U[用户 / 前端 / 机器人] --> API[POST /nl2sql/ask]
    API --> Pydantic[Pydantic 请求校验]
    Pydantic -->|参数非法| E422[HTTP 422 参数错误]
    Pydantic --> Service[NL2SQL Service 编排]
    Service --> Demo[读取 Day35 演示结果]
    Demo --> Judge{最终状态}
    Judge -->|answered| OK[HTTP 200 + final_status=answered]
    Judge -->|safely_blocked| Block[HTTP 200 + final_status=safely_blocked]
    Service --> Audit[写入审计记录 request_id]
    OK --> Caller[调用方展示答案]
    Block --> Caller
    Audit --> Trace[GET /nl2sql/trace/{request_id}]
    Service -->|业务未命中| E404[统一错误响应 HTTP 404]
    Service -->|内部异常| E500[统一错误响应 HTTP 500]
```

这条链路里最关键的取舍是：安全阻断不是异常，而是可预期的业务结果。
真正的异常是参数不合法、资源不存在或服务内部失败。

## 接口契约拆解

### 1. 请求模型

`AskRequest` 定义调用方能传什么：

```text
question: 用户自然语言问题，必填，长度 1-300
user_id: 用户标识，可选，用于审计
include_trace: 是否返回链路信息，默认 true
```

请求模型的价值是把基础校验前置。
如果 `question` 为空，应该在进入业务逻辑前被拦住，而不是让 service 层猜怎么处理。

### 2. 响应模型

`AskResponse` 定义服务稳定返回什么：

```text
request_id
question
final_status
answer
key_findings
risk_notes
follow_up_questions
pipeline
sql
```

成功回答和安全阻断共用同一个响应模型。
区别不靠换结构，而靠 `final_status`、`risk_notes` 和 `pipeline` 表达。
这样前端解析逻辑更稳定，测试也更容易写。

### 3. 错误响应

统一错误响应使用类似结构：

```json
{
  "success": false,
  "error": {
    "code": "question_not_supported",
    "message": "演示版暂不支持该问题"
  }
}
```

`code` 给机器判断，`message` 给人阅读。
生产里还会补充 `request_id`、可重试标记、文档链接和错误级别。

### 4. Trace 接口

`/nl2sql/trace/{request_id}` 用来查看审计轨迹。
它的价值不是给普通用户看热闹，而是给研发、测试、风控和合规排查问题。

一次 NL2SQL 请求可能错在问题解析、Schema 选择、SQL 生成、SQL 校验、查询执行或结果解释任意一层。
没有 trace，排查只能靠猜；有 trace，才能通过 `request_id` 定位每一步发生了什么。

## 常见坑

| 类型 | 可能的问题 | 生产处理方式 |
|------|------------|--------------|
| 契约 | 字段名频繁变化，下游解析失败 | 用 Pydantic schema 固化请求响应，改字段要兼容或升版本 |
| 状态 | 把安全阻断返回 500 | 安全阻断返回业务状态，系统故障才返回 500 |
| 错误 | 直接暴露 Python 异常、SQL、文件路径 | 对外返回统一错误码，详细异常写日志 |
| 审计 | 没有 request_id，无法排查用户反馈 | 每次请求生成 request_id，并保存审计记录 |
| 权限 | trace 接口暴露 SQL 和中间结果 | trace 接口只给管理员或研发，必要时脱敏 |
| 测试 | 只测成功回答，不测阻断和错误 | 同时覆盖成功、安全阻断、参数错误和 trace |

## 工程取舍

### 取舍一：安全阻断返回 200，还是返回 4xx？

当前项目把安全阻断放在 `AskResponse` 里返回 200，并通过 `final_status=safely_blocked` 表达。
原因是安全阻断是 NL2SQL 链路的正常业务结果，不是 HTTP 请求失败。

如果是用户无权限访问某个独立资源，也可以返回 403。
但在数据问答场景里，用户问题本身被安全策略拒绝执行，前端通常更需要展示“不能查这个问题”的业务解释。

### 取舍二：成功和阻断是否共用响应模型？

共用响应模型更适合本项目。
前端永远读取 `request_id`、`question`、`final_status`、`answer`、`risk_notes` 和 `pipeline`。
如果每种状态返回完全不同结构，下游会写很多分支，后续维护成本更高。

### 取舍三：内部错误要不要返回真实异常？

不能直接返回。
真实异常可能包含服务器路径、SQL、表名、字段名、密钥位置或依赖细节。
这些信息对业务用户没有帮助，还可能造成安全风险。
正确做法是对外返回统一错误码和简短提示，对内通过日志和 `request_id` 排查。

## 面试沉淀

Q087：NL2SQL API 为什么要区分安全阻断和系统错误？
重要程度：5/5

### 回答

NL2SQL API 必须区分安全阻断和系统错误，因为两者含义完全不同。

安全阻断是系统按预期拦截了敏感字段、危险 SQL、缺少时间范围或高成本查询。
它说明安全策略生效了，应该返回清晰的业务状态，例如 `final_status=safely_blocked`，
并告诉调用方为什么不能执行。

系统错误才是服务异常、依赖缺失、数据库连接失败、配置错误或代码 bug。
这类问题应该返回统一的错误响应，并在服务端日志中记录详细异常。

如果把安全阻断都返回 500，调用方会误以为系统故障，监控会产生误报，
前端也无法向业务用户解释权限和合规边界。
如果把系统错误包装成正常回答，研发又无法及时发现线上故障。

生产里一般会同时使用两层状态：

- HTTP 状态码表达接口调用是否成功；
- 业务状态字段表达 NL2SQL 链路最终结果。

在金融信贷场景里，例如用户请求导出客户手机号、查询无权限字段、执行全表扫描或缺少时间范围的大表查询，
这些都应该进入安全阻断，而不是伪装成服务宕机。

## 术语更新

今天新增或强化这些术语：

- 接口契约：前后端或服务之间约定好的请求字段、响应字段、错误格式和状态语义。
- 统一错误响应：所有可预期错误和内部错误都按固定 JSON 结构返回，方便调用方处理。
- 业务状态：描述业务链路结果的字段，例如 `answered`、`safely_blocked`。
- HTTP 状态码：描述 HTTP 请求层结果的状态，例如 200、404、422、500。
- Trace API：按 `request_id` 查询一次请求完整审计轨迹的接口。

API 层和 service 层要分开，是因为它们负责的问题不同。
API 层负责 HTTP 入口、路由、请求模型、响应模型和状态码；
service 层负责业务编排，例如查找演示结果、组织回答、判断是否返回 trace、写入审计记录。
如果把所有逻辑都写在 API 层，接口代码会变得很厚，后续改业务规则、换数据库、接真实 NL2SQL 链路或写测试都更困难。
分层后，API、service、storage、config 和 errors 各自负责一类问题，维护成本更低，也更接近生产后端结构。
