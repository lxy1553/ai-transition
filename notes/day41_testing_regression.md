## 大白话解释

测试不是为了证明代码永远没问题，而是为了每次改动后快速知道主链路有没有坏。

NL2SQL 服务最少要测：

- `/health` 是否正常；
- 正常问题是否返回 `answered`；
- 敏感问题是否返回 `safely_blocked`；
- `request_id` 是否能查到审计记录。

如果只测成功查询，不测安全阻断，就无法证明系统真的有权限和安全边界。

## 生产实际

生产 NL2SQL 项目每次改 prompt、Schema、校验规则、执行逻辑或解释模板，都可能影响结果。

固定回归集可以帮助判断：

- 成功样例是否还成功；
- 敏感样例是否仍被阻断；
- 大表无时间范围是否仍被拦截；
- 接口响应字段是否兼容前端；
- 审计记录是否还能写入；
- trace 接口是否还能按 `request_id` 回查。

Day 41 先做最小测试集。
后续可以扩展成更完整的端到端评测集，例如 50 条问题、覆盖多表、多指标、多时间范围、多阻断原因和多种错误类型。

## 测试链路图

```mermaid
flowchart TD
    Test[unittest 测试] --> Client[FastAPI TestClient]
    Client --> Health[GET /health]
    Client --> AskOK[POST /nl2sql/ask 正常问题]
    Client --> AskBlock[POST /nl2sql/ask 敏感问题]
    Client --> Trace[GET /nl2sql/trace/{request_id}]
    Health --> Assert[断言状态和字段]
    AskOK --> Assert
    AskBlock --> Assert
    Trace --> Assert
```

测试链路的重点是验证外部可见行为，而不是只验证某个函数是否返回固定值。

## 当前最小测试集

当前 `tests/test_api.py` 覆盖 4 个场景：

| 测试 | 验证内容 | 价值 |
|------|----------|------|
| `test_health` | `/health` 返回 200，并包含 `demo_ready` | 确认服务和演示产物可用 |
| `test_ask_success_case` | 正常问题返回 `final_status=answered` | 确认主链路可回答 |
| `test_ask_safe_block_case` | 敏感问题返回 `final_status=safely_blocked` | 确认安全阻断有效 |
| `test_trace_after_ask` | 问答后可通过 `request_id` 查 trace | 确认审计链路有效 |

这组测试不追求覆盖所有边界，但覆盖了 Week 6 项目的最小交付闭环。

## 常见坑

| 类型 | 可能的问题 | 生产处理方式 |
|------|------------|--------------|
| 测试范围 | 只测成功，不测阻断 | 成功、阻断、错误、trace 都要覆盖 |
| 断言 | 只看状态码，不看业务字段 | 同时断言 `final_status`、`pipeline`、关键字段 |
| 数据 | 测试依赖不稳定外部服务 | 本地测试使用固定样例或 mock |
| 回归 | 改 prompt 后不跑测试 | 改解析、生成、校验、解释逻辑后都跑回归 |
| 安全 | 敏感样例未进入测试集 | 手机号、身份证、全表扫描、危险 SQL 都要有样例 |
| 维护 | 测试数据和业务口径脱节 | 测试集跟随 Schema 和指标口径更新 |

## 工程取舍

### 取舍一：为什么先用 `unittest`？

当前项目优先保证本地可运行。
`unittest` 是 Python 标准库，不需要额外安装 `pytest`。
对于 Week 6 的最小回归测试，标准库已经足够。
后续如果测试规模变大，可以再引入 `pytest`、fixture、覆盖率和 CI。

### 取舍二：接口测试和单元测试怎么分工？

接口测试从 HTTP 层验证完整行为，能覆盖路由、schema、service、storage 和错误处理。
单元测试更适合验证单个函数或规则，例如 SQL 校验器是否拦截危险关键字。
NL2SQL 项目两类测试都需要：接口测试保交付链路，单元测试保核心规则。

### 取舍三：为什么安全阻断样例必须测试？

因为 NL2SQL 的生产风险不只在“能不能回答”，还在“该拒绝的时候能不能拒绝”。
如果安全阻断样例没有进入回归测试，后续改代码可能让敏感查询绕过校验。
金融信贷场景里，这类问题比普通回答错误更严重。

## 面试沉淀

Q091：NL2SQL 服务最小回归测试应该覆盖哪些场景？

### 回答

NL2SQL 服务最小回归测试至少要覆盖健康检查、正常查询、安全阻断和审计追踪。
健康检查确认服务和关键产物可用；正常查询确认主链路能返回业务答案；
安全阻断确认敏感字段、危险 SQL 或缺少条件的问题不会绕过校验；
审计追踪确认 `request_id` 能回查完整记录。

如果只测试成功样例，无法证明系统具备生产需要的权限、安全和可追溯能力。
每次修改 Prompt、Schema、SQL 校验规则或解释逻辑后，都应该跑这组回归。
更成熟的 NL2SQL 项目还会把真实 bad case 沉淀进评测集，持续验证解析准确率、SQL 安全性和结果解释稳定性。

## 术语更新

今天新增或强化这些术语：

- 回归测试：每次改动后重复执行的固定测试，确认已有能力没坏。
- 冒烟测试：用少量关键用例快速判断主链路是否可用。
- 接口测试：从 HTTP 接口层验证请求、响应和业务行为。
- TestClient：FastAPI 提供的本地接口测试客户端。
- 测试集：固定问题和期望结果集合，用于判断系统改动是否影响质量。

### 8. [Day 40] SQLite 和 Postgres 怎么取舍？
  我的回答：
 sqlite 适合小规模演示学习，pg 库适合大规模存储，和生产

回答评价：
通过。回答抓住了核心取舍：SQLite 适合小规模学习演示，Postgres 适合生产和更大规模存储。
可以再补充：Postgres 还适合高并发、多实例、权限控制和长期留存。

参考答案：
SQLite 适合本地学习版和小规模演示，因为它不需要单独部署数据库服务，一个文件就能保存审计记录。
Postgres 更适合生产场景，因为它支持多用户、高并发、多实例共享、权限控制、索引、备份恢复和长期留存。
当前 Week 6 项目用 SQLite 是为了降低本地运行成本；如果进入真实生产，就应该迁移到 Postgres、MySQL 或公司统一审计平台。
