## 大白话解释

离线仓库可以理解成公司每天加工好的“业务数据底账”。
实时仓库可以理解成公司正在流动的“业务现场监控台”。

业务系统里有授信申请、设备行为、风控审批、放款、还款、逾期、催收等数据。
这些数据有些适合按天加工成日报和指标，有些必须在几秒或几分钟内被发现。

用户问的问题也分两类：

```text
昨天消费贷审批通过率是多少？
近 30 天 M1+ 逾期余额趋势怎么样？
近 5 分钟申请量为什么突然升高？
刚才黑名单命中告警来自哪个渠道？
审批通过率的离线口径和实时口径有什么区别？
```

离线仓库解决“历史指标、日报、口径、回溯”的问题。
实时仓库解决“当前状态、风险告警、实时特征、分钟级监控”的问题。
Agent 不应该随便查所有原始表或事件流，而应该基于可信的指标表、实时指标、指标字典、
Schema Catalog、血缘和权限规则工作。

## 生产实际

金融信贷公司里，离线/实时仓库通常会支撑这些场景：

| 场景 | 业务问题 | Agent 需要的数据底座 |
|------|----------|----------------------|
| 经营日报 | 昨天申请数、通过率、放款金额、逾期率有什么变化 | ADS 日报指标表 |
| 风控分析 | 哪些产品线风控拒绝率升高 | DWS 风控汇总表和风控规则口径 |
| 贷后监控 | M1+ 逾期余额、逾期率、回收率趋势 | DWS / ADS 贷后指标表 |
| 实时风控 | 近 5 分钟申请量、拒绝率、黑名单命中是否异常 | 实时指标和告警表 |
| 反欺诈 | 同设备、同 IP、同渠道短时间内是否聚集申请 | 实时事件和实时特征 |
| 口径问答 | 审批通过率怎么算，离线和实时口径有什么差异 | 指标字典和口径知识库 |
| 血缘追溯 | 某个指标来自哪些来源表、事件流和加工任务 | 血缘元数据和调度/流任务记录 |
| 安全审计 | 谁问了什么、查了哪些指标、是否越权 | 查询审计、告警审计和权限标签 |

如果没有这些仓库基础，Agent 看起来能回答问题，但生产风险很高：

- 可能把明细表当指标表查，导致口径错误；
- 可能漏掉日期分区，扫描成本失控；
- 可能把实时窗口指标和离线日指标混用；
- 可能忽略实时链路延迟、乱序、重复和补偿；
- 可能访问手机号、身份证号、银行卡号等敏感字段；
- 可能在离线任务失败、实时任务延迟或分区缺失时仍然给出确定结论。

## 离线仓库四层

### ODS：原始数据层

ODS 保存从业务系统同步过来的原始数据。
它尽量保留原始字段、原始状态和原始时间，方便追溯。

金融信贷例子：

- 授信申请原始表；
- 风控审批流水表；
- 放款交易表；
- 还款计划表；
- 实际还款流水表；
- 催收跟进记录表。

Agent 通常不应该直接查 ODS。
ODS 字段多、口径粗、明细敏感，适合追溯和数据开发，不适合作为业务问答的第一入口。

### DWD：明细事实层

DWD 会把 ODS 清洗成标准明细事实表。
它解决字段命名、状态标准化、主键对齐、脏数据过滤和业务事件统一的问题。

金融信贷例子：

- 授信申请明细事实表；
- 风控审批明细事实表；
- 放款明细事实表；
- 还款明细事实表；
- 逾期明细事实表。

DWD 的关键是粒度清楚。
例如一行是“一次授信申请”，还是“一次风控策略命中”，还是“一个用户一天的逾期状态”，必须写清楚。
如果粒度不清，NL2SQL 很容易重复计数或选错表。

### DWS：汇总主题层

DWS 会按用户、产品、渠道、日期、城市、风险等级等维度做轻度汇总。
它比 DWD 更适合分析，但还没有完全变成报表。

金融信贷例子：

- 按日、产品线汇总的授信申请和审批结果；
- 按日、渠道汇总的放款金额；
- 按用户风险等级汇总的逾期表现；
- 按催收阶段汇总的回收情况。

Agent 做离线指标查询时，经常优先查 DWS 或 ADS。
因为 DWS 已经减少明细扫描成本，也更接近业务问题的粒度。

### ADS：应用指标层

ADS 面向报表、看板、日报和业务应用。
它通常已经把指标算好，比如申请数、通过率、放款金额、逾期率、M1+余额。

金融信贷例子：

- 信贷经营日报表；
- 风控效果监控表；
- 贷后逾期监控表；
- 产品线经营看板表。

信贷日报 Agent 最适合优先接 ADS。
因为日报 Agent 的目标不是重新做复杂加工，而是基于已经沉淀好的指标表做解释、对比和风险提示。

## 实时仓库核心概念

实时仓库不是把离线仓库简单跑快一点。
它要处理持续进入的事件流，并在短时间窗口内产出指标、特征和告警。

### 实时事件

实时事件是业务系统持续产生的动作记录。

金融信贷例子：

- 用户提交授信申请；
- 设备指纹识别完成；
- 风控规则命中；
- 黑名单命中；
- 放款成功或失败；
- 还款成功或失败；
- 催收动作记录。

实时事件的关键是事件时间、处理时间、事件唯一 ID 和业务主键。
没有这些字段，就很难处理乱序、重复和延迟。

### 实时指标

实时指标是在分钟级或秒级窗口内计算出的业务状态。

金融信贷例子：

- 近 5 分钟授信申请量；
- 近 5 分钟风控拒绝率；
- 近 10 分钟黑名单命中数；
- 近 1 小时还款失败数；
- 某渠道当前设备聚集风险。

实时指标必须说明窗口大小、统计口径、延迟阈值和是否允许补偿。
同一个“通过率”，离线按天统计和实时按 5 分钟窗口统计，含义并不一样。

### 实时特征

实时特征通常给风控策略、模型或 Agent 使用。

金融信贷例子：

- 用户近 1 小时申请次数；
- 同设备近 10 分钟申请用户数；
- 同 IP 近 5 分钟申请次数；
- 渠道近 30 分钟拒绝率；
- 用户近 7 天还款失败次数。

实时特征要关注延迟、幂等和一致性。
如果重复事件没有去重，风控模型或 Agent 解释可能会误判风险升高。

### 实时告警

实时告警是实时指标或实时特征触发的异常信号。

金融信贷例子：

- 某渠道近 5 分钟申请量超过阈值；
- 黑名单命中数突然升高；
- 风控拒绝率突增；
- 还款失败数异常升高；
- 实时链路延迟超过阈值。

Agent 解释告警时，只能说明告警事实、关联指标、可能需要排查的方向和证据来源。
不能把“申请量升高”直接编造成“某活动导致”，除非有活动数据或运营记录作为证据。

## Agent 应该怎么接离线/实时仓库

Agent 不是直接拿到数据库账号和事件流就自由查询。
更稳的做法是把仓库信息整理成几个工具和知识源：

| 能力 | 作用 | 典型输入 | 典型输出 |
|------|------|----------|----------|
| Schema Catalog | 告诉 Agent 有哪些离线表、实时指标、字段、粒度、权限和分区 | 用户问题、业务域 | 候选表字段或实时指标 |
| 指标字典 | 解释指标口径、公式、分母、分子和适用范围 | 指标名 | 指标定义和口径 |
| 实时指标字典 | 解释窗口、事件时间、处理时间、延迟阈值和告警规则 | 实时指标名 | 实时口径说明 |
| 血缘工具 | 追溯指标来自哪些离线表、事件流、任务和分区 | 指标名或表名 | 来源链路 |
| 离线 SQL 工具 | 基于 Catalog 生成只读 SQL | 结构化问题 | 候选 SQL |
| 实时状态工具 | 查询实时指标、延迟状态和告警列表 | 业务域、窗口 | 实时状态 |
| SQL 校验工具 | 检查权限、敏感字段、分区、limit 和成本 | 候选 SQL | 通过或阻断 |
| 结果解释工具 | 把离线指标、实时状态和告警转成业务语言 | 查询结果 | 摘要和风险提示 |

金融信贷 Agent 至少要区分五类问题：

- 查离线指标：走 Schema Catalog、SQL 生成、SQL 校验、查询执行和结果解释；
- 查实时状态：走实时指标字典、实时状态工具和结果解释；
- 问口径：走指标字典或 RAG，不应该直接查表；
- 问血缘：走血缘工具，返回来源表、事件流、加工任务和下游应用；
- 敏感明细：安全阻断或要求更高权限，不能靠模型自由决定。

## 常见坑

| 类型 | 可能的问题 | 生产处理方式 |
|------|------------|--------------|
| 表层级混乱 | 用户问日报指标，Agent 去查 ODS 明细 | Catalog 标清 ODS/DWD/DWS/ADS 和推荐查询层 |
| 粒度不清 | 申请数重复计算，用户粒度和申请粒度混用 | 每张表写清主键、业务粒度和去重规则 |
| 离线实时混用 | 把实时 5 分钟通过率当成昨日通过率解释 | 指标字典区分离线口径和实时窗口口径 |
| 分区缺失 | SQL 没有日期条件，扫描全表 | SQL 校验强制时间范围和分区裁剪 |
| 实时延迟 | 实时任务延迟，Agent 仍说当前状态正常 | 查询实时链路延迟和任务状态后再回答 |
| 乱序重复 | 事件重复导致实时指标虚高 | 事件 ID 去重、watermark、幂等写入 |
| 权限越界 | 查询手机号、身份证号、银行卡号 | 字段级权限标签和敏感字段阻断 |
| 告警编造 | 只有指标异常，却编造业务原因 | 只解释事实和证据，原因必须有引用或关联数据 |
| 血缘不可追 | 指标错了不知道影响哪些报表或告警 | 建立来源表、事件流、加工任务和下游应用血缘 |

## 工程取舍

### 取舍一：为什么离线指标优先查 ADS / DWS，而不是 ODS？

ODS 是原始数据层，适合追溯和重算，不适合直接做业务问答。
Agent 直接查 ODS 会遇到明细量大、字段敏感、状态复杂、口径不稳定和成本高的问题。

ADS / DWS 已经按业务口径汇总，更适合日报、看板和指标查询。
生产里通常让 Agent 优先查 ADS 或 DWS，只有排查数据问题或追溯血缘时才进入 DWD / ODS。

### 取舍二：为什么实时指标不能直接替代离线指标？

实时指标适合发现当前异常，但它可能受延迟、乱序、补偿和窗口大小影响。
离线指标更适合做最终口径、历史回溯、经营复盘和对外汇报。

例如“审批通过率”如果按实时 5 分钟窗口计算，适合监控突增突降；
如果按天计算，适合进入日报和经营分析。
Agent 必须先判断用户问的是实时状态还是离线经营指标。

### 取舍三：为什么指标口径要进知识库？

指标口径不是 SQL 字段名那么简单。
例如“审批通过率”至少要说明分子、分母、时间口径、产品范围、是否排除撤销申请、是否按申请粒度计算。

这些内容适合放入指标字典和 RAG 知识库。
用户问“通过率怎么算”时，Agent 应该引用口径文档回答；
用户问“昨天通过率是多少”时，Agent 才进入 NL2SQL 离线指标查询；
用户问“刚才通过率为什么异常”时，Agent 才进入实时指标和告警链路。

### 取舍四：为什么血缘对 Agent 很重要？

Agent 给出的指标结论如果出错，研发必须能追溯：

- 指标来自哪个 ADS 表或实时指标表；
- ADS 表由哪些 DWS / DWD 表加工；
- 实时指标由哪些事件流和流任务加工；
- 哪个离线任务或实时任务产出；
- 哪个分区或窗口是否成功；
- 哪些报表、告警或 Agent 回答会受影响。

没有血缘，Agent 项目只能演示，不能生产化排查。

### 练习一：信贷离线日报 + 实时告警 Agent

准备一个本地 ADS 指标表，字段可以包括：

```text
biz_date
product_line
apply_cnt
approve_cnt
approve_rate
loan_amount
overdue_m1_balance
risk_reject_rate
```

再准备一个实时告警 JSONL，字段可以包括：

```text
event_time
alert_type
product_line
metric_name
window_size
current_value
threshold
severity
evidence
```

Agent 输入：

```text
生成 2026-06-01 消费贷业务日报，并汇总最近 1 小时实时风险告警
```

Agent 输出：

- 今日关键离线指标；
- 和前一日或近 7 日均值对比；
- 最近 1 小时实时告警摘要；
- 异常指标提示；
- 不能确认原因时明确说明“需要进一步结合业务、规则日志和实时链路排查”。

### 练习二：仓库口径、实时指标与血缘问答 Agent

准备一个指标字典、实时指标字典和血缘清单：

```text
指标：审批通过率
离线口径：按 biz_date 统计进入终审申请中的审批通过占比
实时口径：按 5 分钟窗口统计当前进入终审申请中的审批通过占比
离线来源：ads_credit_daily_metrics
实时来源：rt_credit_approval_metrics
上游：dws_credit_approval_daily, dwd_credit_apply_detail, credit_apply_event_stream
分区：biz_date
窗口：5 minutes
敏感等级：低
```

用户问“审批通过率来自哪些表和事件流”时，Agent 走血缘工具；
用户问“审批通过率怎么算”时，Agent 走 RAG 口径问答；
用户问“昨天审批通过率是多少”时，Agent 走 NL2SQL 离线查询；
用户问“近 5 分钟审批通过率是否异常”时，Agent 走实时状态查询。

## 面试沉淀

Q098：离线/实时仓库如何支撑金融信贷 Agent 或 NL2SQL 数据问答？

### 回答

离线/实时仓库是金融信贷 Agent 和 NL2SQL 数据问答的数据底座。
如果没有仓库治理，Agent 只能看到一堆业务原始表和事件流，很容易选错表、混淆指标口径、
漏掉分区、忽略实时延迟、访问敏感字段，或者在离线任务失败、实时任务延迟时仍然给出错误结论。

生产里通常会把信贷离线数据按 ODS、DWD、DWS、ADS 分层。
ODS 保留原始业务数据，DWD 清洗成标准明细事实表，DWS 按主题和维度做轻度汇总，
ADS 面向日报、看板和经营分析沉淀应用指标。
Agent 做离线指标查询时，应该优先使用 DWS 或 ADS，而不是直接查 ODS 明细表。

实时仓库则面向实时风控、反欺诈、实时特征、告警和分钟级监控。
它需要管理实时事件、实时指标、窗口口径、事件时间、处理时间、延迟、乱序、重复和幂等。
例如近 5 分钟申请量、黑名单命中数、实时拒绝率突增和还款失败告警，都应该走实时状态查询工具，
不能和离线日报指标混在一起解释。

同时，还要把仓库元数据整理成 Schema Catalog、指标字典、实时指标字典、权限标签和血缘信息。
Schema Catalog 告诉 Agent 有哪些离线表、实时指标、业务粒度、分区字段和权限边界；
指标字典解释指标的分子、分母、过滤条件和适用范围；
实时指标字典解释窗口、延迟阈值和告警规则；
血缘信息说明指标来自哪些上游表、事件流和加工任务；
权限标签控制敏感字段和明细访问。

在工具编排上，查离线指标走 NL2SQL 链路，查实时状态走实时指标工具，
问口径走 RAG 链路，问来源走血缘工具，敏感明细查询则走安全阻断或权限校验。
这样 Agent 才能把“能回答”升级成“回答可信、可审计、可追溯、可控成本、能处理实时风险”。

## 术语更新

今天新增或强化这些术语：

- 离线数据仓库：把业务系统数据按天或按批次加工成可分析数据底座。
- 实时数据仓库：把持续进入的业务事件加工成实时指标、实时特征和告警的数据底座。
- ODS：原始数据层，保留业务系统同步来的原始记录。
- DWD：明细事实层，对原始数据做清洗、标准化和粒度统一。
- DWS：汇总主题层，围绕主题域和常用维度做轻度汇总。
- ADS：应用指标层，面向日报、看板、数据问答和业务应用。
- 实时事件：业务系统持续产生的行为或状态变化记录。
- 实时指标：按秒级或分钟级窗口计算出的当前业务状态。
- 实时特征：给风控策略、模型或 Agent 使用的实时聚合特征。
- 实时告警：实时指标或特征触发的异常信号。
- 指标字典：记录指标含义、公式、分子、分母、粒度、负责人和适用范围。
- 数据血缘：记录数据从来源表或事件流到加工任务、指标表和下游应用的链路。
- 分区裁剪：通过日期等分区条件减少扫描范围，控制查询成本。
- 事件时间：业务事件真实发生的时间。
- 处理时间：系统实际处理这条事件的时间。
