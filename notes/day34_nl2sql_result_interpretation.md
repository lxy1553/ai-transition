## 大白话解释

结果解释层就是把“数据库查出来的结果”翻译成“业务能听懂的话”。

数据库返回的是：

```json
{
  "channel": "APP",
  "approval_rate": 0.6487
}
```

业务更需要看到的是：

```text
上周 APP 渠道授信通过率最高，为 64.87%。
这个结果只说明当前查询周期内的渠道表现，后续可以继续按产品或风险等级拆分。
```

这一步不是让模型自由发挥。
结果解释层必须忠于查询结果，只能基于 `rows`、`columns`、`summary_text` 和查询上下文解释。
如果数据里没有原因，就不能说“因为某个渠道质量变差”；只能提示“需要继续按产品、风险等级或日期拆分排查”。

## 生产实际

金融信贷数据问答里，业务用户通常不想看裸 SQL 或裸表格，而是想知道：

- 昨天申请量是多少，是否异常；
- 哪个渠道通过率最高，哪个最低；
- 最近 7 天放款金额是上升还是下降；
- 本周逾期率比上周改善还是变差；
- 某个申请编号当前审批状态是什么；
- 为什么手机号导出请求不能执行。

所以结果解释层通常接在查询执行之后：

```text
SQL 校验
-> 查询执行
-> 结果格式化
-> 结果解释
-> 前端展示 / 业务追问 / 审计日志
```

生产里的解释结果一般至少包含：

| 字段 | 作用 |
|------|------|
| `business_answer` | 直接给业务看的解释 |
| `key_findings` | 关键发现，适合卡片或报告展示 |
| `risk_notes` | 口径、权限、样本量、时间范围等风险提示 |
| `follow_up_questions` | 建议下一步追问或拆分方向 |
| `source_result` | 原始查询结果或结果引用，便于追溯 |

---

## 常见坑

| 类型 | 可能的问题 | 生产处理方式 |
|------|------------|--------------|
| 编造原因 | 只看到下降，就解释成“渠道质量变差” | 没有维度拆分和证据时，只能提示继续分析 |
| 口径不清 | 不说明时间范围、分子分母和数据来源 | 解释里补充查询周期、指标口径和限制 |
| 忽略空结果 | 空结果被解释成业务不存在 | 区分无数据、权限拦截、SQL 被阻断和执行失败 |
| 过度简化 | 趋势、TopN、对比都只写一句话 | 根据 response_type 使用不同解释模板 |
| 泄露风险 | 对被拦截的敏感查询给出猜测性信息 | 只说明拦截原因和可走的审批或替代查询 |
| 不可追溯 | 解释没有保留原始结果和 SQL 链路 | 关联 request_id、SQL、rows、columns 和审计日志 |

## 工程取舍

Day 34 先做规则版结果解释器，不直接接 LLM。

原因：

- 规则解释稳定，适合建立 baseline；
- 不会编造数据库里没有的信息；
- 能按 `response_type` 明确区分指标、趋势、TopN、对比和明细；
- 后续接 LLM 时，可以把规则解释作为结构化上下文，让 LLM 只负责润色。

生产里更稳的路线是：

```text
结构化结果
-> 规则生成关键发现和风险提示
-> LLM 可选润色
-> 引用原始数据和查询链路
-> 前端展示和审计记录
```

如果直接让 LLM 看原始行自由总结，容易出现两个问题：
一是编造原因，二是忽略权限和口径边界。
所以结果解释层仍然要有硬规则和可追溯数据来源。

## 面试沉淀

Q080：NL2SQL 为什么需要结果解释层？

### 回答

NL2SQL 需要结果解释层，是因为业务用户通常不只需要 SQL 返回的原始行列结果，
还需要理解这个结果代表什么、是否有风险、下一步该怎么追问。
查询执行层返回的是结构化数据，比如字段、行数、指标值、趋势序列和明细记录。
结果解释层要把这些内容转换成业务语言，例如说明哪个渠道通过率最高、逾期率比上周下降了多少、
某个申请编号当前审批状态是什么。
同时，解释层必须补充时间范围、指标口径、权限边界和样本限制，避免业务误读。
它不能编造 SQL 没查出来的原因，只能基于查询结果和元数据解释。

Q081：结果解释层怎么避免模型编造业务原因？

### 回答

结果解释层要避免编造，核心是把解释建立在结构化查询结果上。
系统应该先由规则层生成事实性结论，比如最高值、最低值、环比差值、趋势方向和返回行数，
再把这些事实交给 LLM 做可选润色。
Prompt 里要明确要求模型只能引用提供的 rows、columns、指标口径和时间范围，不能自行猜测原因。
如果需要原因分析，应该输出“建议继续按渠道、产品、风险等级或日期拆分”，而不是直接说原因已经确定。
生产里还要保留原始查询结果、SQL、request_id 和审计日志，方便回溯解释是否忠于数据。

Q082：被 SQL 校验拦截的问题，结果解释层应该怎么回答？

### 回答

被 SQL 校验拦截的问题，结果解释层不能回答成“没有数据”，也不能猜测敏感信息。
它应该明确说明该问题没有执行数据库查询，原因是权限、敏感字段、缺少时间范围、危险 SQL 或高成本风险。
同时要给出安全替代路径，例如补充时间范围、改查汇总指标、走敏感数据审批流程，或联系数据 owner。
这样业务用户能理解系统是在做安全控制，而不是查询失败或数据不存在。
在金融信贷场景里，这一点尤其重要，因为手机号、身份证号、客户明细和贷后敏感数据都必须有权限和审计边界。

静态校验，schema 验证，字段是否在白名单，权限安全校验，成本性能校验

参考答案：

第一是静态和 Schema 校验，包括是否只读、是否有危险关键字、表是否在白名单里、字段是否存在。
第二是安全和权限校验，包括是否访问敏感字段，当前用户是否能查对应表、字段和数据范围。
第三是性能和成本校验，包括大表是否带时间范围或分区条件，明细查询是否有精确过滤和 limit，
是否存在无限制排序、高成本 join 或大范围扫描。

NL2SQL 查询执行层必须放在 SQL 校验之后，因为执行层一旦访问数据库，
风险就从生成阶段的错误变成真实的数据泄露、性能冲击或审计问题。
SQL 生成层可能产生字段越权、缺少时间范围、大表扫描、敏感字段查询或危险关键字。
这些问题必须先由 SQL Validator、权限系统和成本规则拦住。
执行层只应该接收已经通过校验的 SQL 或查询任务，并使用只读连接执行。
