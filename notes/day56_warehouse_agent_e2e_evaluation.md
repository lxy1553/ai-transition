## 大白话解释

端到端评测不是问“答案看起来对不对”。

它要验证整条链路：

```text
用户问题
-> 意图识别
-> 工具路由
-> 前置条件检查
-> 工具执行
-> 安全校验
-> 证据返回
-> 最终回答
-> 审计记录
```

如果最终回答文字看起来正常，但工具路线错了，也应该判失败。
例如实时问题误走离线 SQL，可能也能生成一段像样的回答，但业务上是错的。

如果敏感导出问题最终没有返回手机号，但工具中间已经查了 DWD 明细，也应该判失败。
因为安全边界已经被突破。

## 生产实际

金融信贷仓库 Agent 的端到端评测至少要覆盖这些场景：

| 场景 | 应该验证什么 |
|------|--------------|
| 成功日报摘要 | 是否同时调用离线日报、实时告警和口径工具 |
| 空分区 | 是否返回无数据，不编造指标 |
| 实时延迟 | 是否返回不可判断，而不是强行结论 |
| 告警误报 | 是否基于指标值、阈值和规则判断 |
| 口径冲突 | 是否澄清版本或 owner，而不是统一编造 |
| 敏感导出 | 是否安全阻断并审计 |
| SQL 缺时间范围 | 是否澄清，而不是扫描全表 |
| 工具异常 | 是否降级并说明失败原因 |
| 有界解释 | 是否只解释证据，不编造业务根因 |

这些样例是后续改 Prompt、改规则、接 LangGraph 或接真实仓库时的回归保护。

## 常见坑

| 类型 | 可能的问题 | 生产处理方式 |
|------|------------|--------------|
| 只看最终答案 | 路由错也可能回答像样 | 检查 route、status、evidence、audit |
| 只测成功样例 | 上线后安全和异常场景崩掉 | 成功、拒答、澄清、降级、异常都要测 |
| bad case 修完不回归 | 新规则破坏旧能力 | 每次修复跑全量测试集 |
| 空分区编造结果 | 用户误以为当天有数据 | 返回 no_data 和分区证据 |
| 延迟仍给结论 | 实时判断误导业务 | 返回 degraded 或不可判断 |
| 敏感请求只看最终文本 | 中间工具可能已越权 | 检查工具调用路线和审计 |

## 工程取舍

### 取舍一：为什么端到端评测要检查工具路线？

Agent 的回答是否正确，不只取决于最终文本。
金融信贷数据问答里，路线本身就是安全和口径的一部分。
口径问题应该走 RAG，离线指标应该走 NL2SQL，实时状态应该走实时工具，血缘问题应该走血缘工具，敏感导出应该走安全阻断。
路线错了，即使回答像样，也不能算通过。

### 取舍二：为什么要专门测空分区和实时延迟？

空分区和实时延迟是数据系统常见生产问题。
空分区时 Agent 容易编造昨日指标；实时延迟时 Agent 容易强行给“当前正常/异常”结论。
这两类问题不会靠更好的措辞解决，必须有固定评测样例约束系统行为。

### 取舍三：为什么告警误报和口径冲突也要进评测？

告警可能因为规则版本、快照时间或延迟状态导致误报。
口径文档可能存在版本冲突或 owner 未确认。
Agent 如果在这些场景里强行给确定答案，会制造新的业务风险。
正确行为是基于证据说明不确定性，并要求澄清、复核或升级处理。

## 面试沉淀

Q128：金融信贷仓库 Agent 的端到端评测应该覆盖哪些场景？
重要程度：5/5

### 回答

金融信贷仓库 Agent 的端到端评测不能只覆盖成功查询。
它应该同时覆盖离线指标查询、实时状态查询、口径解释、血缘追溯、日报告警摘要、权限阻断、空分区、实时延迟、工具异常、告警误报和口径冲突。

每条样例要明确用户问题、预期意图、预期工具路线、预期状态、必须包含的证据和不允许出现的行为。
例如近 7 天授信通过率应该走离线 NL2SQL；实时风控拒绝率应该走实时工具；授信通过率怎么算应该走指标字典 RAG；
要求导出手机号和身份证号应该走安全阻断。

生产级评测还要检查审计记录、引用来源、SQL 安全、延迟状态和拒答类型。
只有最终回答、工具路线、证据和安全状态都符合预期，才算真正通过。

Q129：为什么 Agent 端到端评测不能只看最终回答文字？
重要程度：5/5

### 回答

Agent 端到端评测不能只看最终回答文字，因为最终回答可能看起来合理，但内部路线是错的。
例如实时问题误走离线日报，口径问题误走 SQL，敏感导出问题中间已经查了 DWD 明细，最终文本即使没有泄露数据，也说明安全边界已经被突破。

金融信贷数据 Agent 的正确性包括多层：意图是否识别正确、工具是否路由正确、前置条件是否满足、SQL 是否安全、实时延迟是否检查、证据是否返回、
审计是否记录。 这些都不是只看最终自然语言能判断的。

所以端到端评测要检查结构化 trace，包括 route、tool_calls、status、evidence、audit、refusal_reason 和 final_answer。
一句面试总结是：生产 Agent 评测的是链路正确，不只是回答像样。

Q130：如何设计空分区、实时延迟、告警误报和口径冲突这类 bad case？
重要程度：5/5

### 回答

这类 bad case 要按生产风险设计，而不是随便写几个失败问题。
空分区样例要验证 Agent 是否返回 no_data 和分区证据，而不是编造指标。
实时延迟样例要验证 Agent 是否检查延迟阈值，并在延迟超阈值时返回 degraded 或不可判断。
告警误报样例要验证 Agent 是否对比指标值、阈值、规则版本和快照时间，而不是只要有告警就当真。
口径冲突样例要验证 Agent 是否识别文档版本冲突、owner 冲突或定义冲突，并要求澄清，而不是强行合并口径。

每个 bad case 都要写清 expected_status、expected_route、must_contain 和 forbidden_behavior。
修复后必须跑全量回归，确认新规则没有破坏成功查询、安全阻断和正常告警解释。

## 术语更新

今天新增或强化这些术语：

- 端到端评测：从用户问题到最终回答的整条链路评测。
- 回归测试集：用于每次修复后重复验证旧能力没有被破坏的固定样例集。
- 空分区：目标日期或范围没有数据的仓库分区。
- 告警误报：告警触发但证据显示不满足规则或数据不可用的情况。
- 口径冲突：多个指标文档、版本或 owner 对同一指标定义不一致。
- 评测 Trace：记录意图、工具路线、状态、证据和审计的结构化执行轨迹。
