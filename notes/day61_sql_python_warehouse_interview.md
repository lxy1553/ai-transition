## 大白话解释

面试官不会因为你做了项目，就自动认为你掌握了 SQL、Python 和数仓。
你需要把项目里的具体实现翻译成他们熟悉的问题语言。

例如：

```text
为什么离线指标查询必须带分区条件

为什么要做幂等、超时和异常处理

```

## 常见坑

| 类型 | 可能的问题 | 生产处理方式 |
|------|------------|--------------|
| 只背概念 | 一问项目细节就空 | 每题都映射到本项目 |
| 只讲成功案例 | 工程深度不足 | 补异常、越权和延迟场景 |
| SQL 只讲语法 | 没有生产语境 | 结合分区、成本和权限 |
| Python 只讲语法糖 | 难体现工程能力 | 讲异常、结构化输出和审计 |
| 仓库只讲分层 | 不会讲业务价值 | 连接授信、风控、贷后场景 |

## 工程取舍

### 取舍一：为什么面试准备要围绕项目，而不是另背一套题库？

因为项目是你最能说细的材料。
如果项目和题库是两张皮，面试官一追问实现细节就会暴露。

### 取舍二：为什么 SQL 和仓库问题要一起准备？

因为在这个项目里，SQL 从来不是脱离业务和数仓单独存在的。
你写的 SQL 是否安全、是否带分区、是否走对层级，本身就是仓库设计问题。

## 面试沉淀

Q139：为什么生产级 NL2SQL 必须带分区、权限和扫描成本控制？
重要程度：5/5

### 回答

因为生产仓库查询不仅追求能跑出结果，还要保证口径正确、成本可控和权限合规。
如果没有分区条件，容易扫全表；如果没有权限控制，可能访问敏感明细；如果没有成本约束，Agent 可能生成高开销 SQL 影响仓库稳定性。

Q140：为什么实时窗口和离线日报指标不能混着解释？
重要程度：5/5

### 回答

因为两者的数据来源、时间边界、延迟特征和业务用途都不同。
实时窗口更适合监控和告警，离线日指标更适合日报和历史分析。
如果混着解释，会把实时波动当成日级结论，或者把延迟问题误解为业务异常。

## 术语更新

- 分区裁剪：通过时间或业务分区限制查询扫描范围。
- 扫描成本：一次查询对仓库资源的消耗估算。
- 项目深挖题：围绕你做过的系统，追问实现细节和取舍的问题。
- 工程语言：不只讲概念，还讲边界、风险、约束和取舍的表达方式。
