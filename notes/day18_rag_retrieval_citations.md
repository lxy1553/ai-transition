## 生产实际

真实公司里的 RAG 在线问答一般会接在一个 API 服务后面。

用户从网页、企业微信、飞书机器人或内部系统提问，后端服务会做：

1. 校验用户身份和权限
2. 判断问题类型和业务域
3. 从向量库、关键词索引或混合检索里召回候选 chunk
4. 做权限过滤、版本过滤和 rerank
5. 构造上下文 prompt
6. 调用 LLM 生成答案
7. 返回答案、引用、耗时、trace_id
8. 记录日志和用户反馈

今天本地练习只做最小闭环：

```text
SQLite 索引
-> 本地向量相似度检索
-> top-k 结果
-> 答案草稿
-> citations 引用列表
```

这个版本还不是完整生产系统，但已经具备 RAG 问答最核心的可追溯能力。

## 常见坑

| 问题 | 影响 | 处理方式 |
|------|------|----------|
| 只返回答案不返回引用 | 用户无法判断答案依据 | 每个答案必须带 source、chunk_id、position |
| top-k 太小 | 漏掉关键资料 | 根据问题复杂度调整 top-k |
| top-k 太大 | 无关内容干扰模型，token 成本升高 | 检索后 rerank，再选少量高质量 chunk |
| 检索结果重复 | 上下文浪费，答案啰嗦 | 按 doc_id、标题、内容 hash 去重 |
| 引用资料过期 | 答案基于旧口径 | metadata 里保留 version、status、updated_at |
| 没有权限过滤 | 可能泄露内部资料 | 检索前后都要按用户权限过滤 |

## 工程取舍

今天为什么不直接接 LLM？

因为 RAG 的第一层能力不是“模型会不会写答案”，而是“系统有没有找到正确资料”。

如果检索错了，模型写得再自然也只是把错误资料包装得更像真的。
所以今天先用本地规则生成答案草稿，重点检查：

- top-k 是否命中相关 chunk
- citation 是否能追溯到原始文件
- 问题、检索结果、答案之间是否能解释清楚

后续 Day 19 再做召回质量优化，Day 20 再把它接口化。

## 今日核心理解

RAG 问答不是“模型自由回答”，而是“先找依据，再基于依据回答”。

今天要重点记住：

1. 检索结果决定答案上限。
2. 引用来源决定答案是否可信。
3. 没有引用的企业 RAG 很难排查问题，也很难让用户信任。

## 面试沉淀

Q007：RAG 为什么要做引用来源和可追溯？

### 回答

RAG 返回引用来源，是为了让用户和研发都能确认答案依据。
用户可以查看答案来自哪个文档、章节、版本和片段，
研发可以据此判断问题是资料过期、检索不准还是生成错误。
没有 citations 的 RAG 很难在生产里落地，因为错误答案无法排查，也难以建立信任。
在财务、人事、合同、指标口径等场景里，引用来源不是展示细节，而是上线门槛。

Q028：RAG 在线问答为什么必须返回引用来源？

### 回答

在线问答返回 citations，可以把 answer 和 retrieved chunks 连接起来。
当答案正确时，citations 提供可信依据；当答案错误时，citations 能帮助定位是召回错、
排序错、资料错还是生成错。
生产 API 里通常会返回 source、chunk_id、标题、位置、更新时间和相关性分数。
这样 RAG 系统才具备可解释、可审计和可持续优化的能力。
