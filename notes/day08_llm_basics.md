## 1. LLM 调用是什么

LLM 调用可以理解为：把一段带有角色、任务、上下文和约束的文本发送给模型，模型根据这些信息生成回答。

一个最小调用通常包括：

- 表格行 1
  - 组成：system prompt
  - 作用：规定模型的角色、边界和全局行为
- 表格行 2
  - 组成：user prompt
  - 作用：用户当前提出的问题或任务
- 表格行 3
  - 组成：model
  - 作用：选择使用哪个模型
- 表格行 4
  - 组成：parameters
  - 作用：控制输出风格、长度和随机性
- 表格行 5
  - 组成：response
  - 作用：模型返回的结果

面试表达：

工程上不能只关注回答内容，还要关注稳定性、成本、可复现性和异常处理。

## 2. System Prompt 与 User Prompt

### System Prompt

system prompt 是对模型的全局指令，通常用于定义角色、任务边界、输出规范和禁止事项。

示例：

```text
你是一个面向数据仓库工程师的 SQL 解释助手。
请用准确、简洁的语言解释 SQL，不编造不存在的表结构。
如果信息不足，请明确说明缺少哪些上下文。
```

### User Prompt

user prompt 是本轮具体任务。

示例：

```sql
请解释下面 SQL 的业务含义，并指出可能的性能风险：

select user_id, count(*) as order_cnt
from orders
where dt = '2026-05-08'
group by user_id;
```

### 为什么要拆开

- system prompt 保持稳定，适合放通用规则。
- user prompt 每轮变化，适合放具体问题和上下文。
- 拆开后更容易复用、测试和排查问题。

## 3. 常见参数

### temperature

控制输出随机性。

- 表格行 1
  - 值：0-0.2
  - 特点：更稳定、更保守
  - 适合场景：SQL 解释、结构化输出、事实问答
- 表格行 2
  - 值：0.3-0.7
  - 特点：稍有变化
  - 适合场景：总结、改写、普通问答
- 表格行 3
  - 值：0.8+
  - 特点：更发散
  - 适合场景：创意文案、头脑风暴

对于 RAG / NL2SQL / 数据问答，通常优先低 temperature，因为这些场景更重视准确和稳定。

### top_p

控制模型从多大概率范围内选择下一个 token。

实践中一般不要同时大幅调整 `temperature` 和 `top_p`。初学阶段先固定 `top_p=1`，主要观察 `temperature` 的影响。

### max_tokens

限制模型最多生成多少 token。

设置过小会导致回答截断；设置过大会增加成本，也可能让回答变啰嗦。

### token

token 是模型计费和上下文长度的基本单位，可以粗略理解为文字片段。一次调用成本通常与输入 token 和输出 token 都有关。

## 4. 成本意识

AI 应用工程不是只要“能回答”，还要控制成本。

需要关注：

- Prompt 是否过长。
- 是否重复传入不必要的上下文。
- 输出长度是否被约束。
- 是否可以缓存重复问题。
- 是否需要更小模型完成简单任务。

面试表达：

> 我会把 LLM 调用看成有成本的外部服务。
设计时会控制上下文长度、设置合理的 max_tokens，并对重复请求、失败重试和输出校验做工程化处理。

## 生产实际

LLM 基础参数在生产里直接影响成本、稳定性和回答风格。金融信贷场景通常不适合高随机性回答，尤其是授信政策、风控规则、逾期口径和 SQL 解释，更需要低温度、
可追溯、 格式稳定和边界清楚。

生产里还要记录模型版本、输入 token、输出 token、延迟、错误率和 request_id。这不是为了记流水账，而是为了评估成本、排查坏答案和支撑审计。
