## 大白话解释

SQL 校验层就是 NL2SQL 系统里的“执行前安检”。

Day 31 生成 SQL 草稿，像是系统写好了一张“查询申请单”。Day 32 不能直接把申请单交给数据库执行，而是要先问几个问题：

- 这是不是只读 SQL？
- 有没有 `delete`、`update`、`drop` 这类危险操作？
- 表和字段是不是都在白名单里？
- 有没有查手机号、身份证号这类敏感字段？
- 大表有没有带时间范围？
- 明细查询有没有精确过滤和 `limit`？
- 这个 SQL 会不会扫太多数据、成本太高？

如果这些问题过不了，系统应该拒绝执行、要求用户补充条件，或者进入人工审批。
这一步的价值不是让 SQL 更好看，而是防止 NL2SQL 把错误、 越权或高成本查询推到生产数据库。

## 生产实际

在金融信贷场景里，SQL 校验层非常关键。业务同学可能问的是“上周每个渠道的授信通过率”，也可能问“导出客户手机号列表”。
前者是正常数据分析，后者涉及个人敏感信息，必须被拦截或走审批。

生产 NL2SQL 链路通常是：

```text
用户问题
-> 问题解析
-> Schema Router
-> SQL 生成
-> SQL 校验
-> 权限校验
-> 成本预估
-> 查询执行
-> 结果解释
-> 审计日志
```

SQL 校验层接在 SQL 生成之后、查询执行之前。它要把模型生成的不确定性挡在数据库外面。

常见校验项：

- 表格行 1
  - 校验项：只读校验
  - 作用：只允许 `select` 或安全的 `with ... select`
- 表格行 2
  - 校验项：危险关键字校验
  - 作用：拦截 `insert`、`update`、`delete`、`drop`、`alter`
- 表格行 3
  - 校验项：表白名单校验
  - 作用：禁止查询 Schema Catalog 外的表
- 表格行 4
  - 校验项：字段白名单校验
  - 作用：禁止查询不存在或未授权字段
- 表格行 5
  - 校验项：敏感字段校验
  - 作用：拦截手机号、身份证号、银行卡号等字段
- 表格行 6
  - 校验项：时间范围校验
  - 作用：大表必须带分区或时间条件
- 表格行 7
  - 校验项：limit 校验
  - 作用：明细查询必须限制返回行数
- 表格行 8
  - 校验项：成本风险校验
  - 作用：识别全表扫描、大范围排序和 join 放大

## 常见坑

- 表格行 1
  - 类型：只看语法
  - 可能的问题：SQL 能跑，但业务口径错或越权
  - 生产处理方式：校验 Schema、指标口径、权限和审计
- 表格行 2
  - 类型：只靠 prompt
  - 可能的问题：prompt 要求不要查敏感字段，但模型仍可能生成
  - 生产处理方式：在代码层做敏感字段和权限拦截
- 表格行 3
  - 类型：时间缺失
  - 可能的问题：大表没有 `dt` 或时间范围，导致全表扫描
  - 生产处理方式：强制时间范围或要求用户补充
- 表格行 4
  - 类型：明细失控
  - 可能的问题：明细查询没有精确过滤和 limit
  - 生产处理方式：要求申请编号、客户 ID 等精确条件
- 表格行 5
  - 类型：字段编造
  - 可能的问题：模型生成不存在字段
  - 生产处理方式：只允许使用 Schema Catalog 白名单
- 表格行 6
  - 类型：成本失控
  - 可能的问题：无限制排序、join 放大、扫描过多分区
  - 生产处理方式：增加成本预估和执行前审批

## 工程取舍

Day 32 先做规则版 SQL 校验器，不直接接数据库执行。

原因：

- 规则校验稳定、可解释，适合作为执行前硬闸门；
- SQL 生成层可能来自模板，也可能来自 LLM，但校验层必须统一；
- 在真正执行 SQL 前，先把只读、权限、字段、敏感信息和成本风险控制住；
- 本地阶段没有真实数据库，也可以通过静态检查建立生产级安全意识。

生产里更完整的 SQL 校验可以分三层：

```text
静态校验：只读、关键字、表字段白名单、敏感字段、limit、时间范围
权限校验：当前用户角色是否能查对应表、字段和数据范围
成本校验：预估扫描分区、扫描行数、join 放大、排序成本和超时时间
```

静态规则不是万能的。复杂 SQL 最好使用 SQL parser，而不是只靠正则。真正上线时还要结合数据库执行计划、权限系统、审计系统和查询网关。

## 面试沉淀

Q074：NL2SQL 为什么必须有 SQL 校验层？
重要程度：5/5

### 回答

NL2SQL 必须有 SQL 校验层，因为生成 SQL 正确不代表可以安全执行。
模型或模板可能生成语法正确但业务错误、权限越界、缺少时间范围、扫描大表、 查询敏感字段或包含危险关键字的 SQL。
在金融信贷场景里，SQL 可能涉及客户隐私、授信审批、风控评级、逾期和贷后数据，如果直接执行，可能造成数据泄露、 错误决策或生产数据库压力。
SQL 校验层应该放在 SQL 生成之后、查询执行之前，统一检查只读、表字段白名单、敏感字段、时间范围、limit、权限和成本风险。
只有校验通过的 SQL 才能进入执行层。

Q075：SQL 校验层通常要检查哪些内容？
重要程度：5/5

### 回答

SQL 校验层至少要检查三类内容。
第一是安全约束，包括只读 SQL、危险关键字、敏感字段、权限边界和审计要求。
第二是业务和 Schema 约束，包括表是否在白名单里、字段是否存在、指标口径是否匹配、明细查询是否有精确过滤。
第三是性能和成本约束，包括大表是否带时间范围或分区条件、是否有无限制 `order by`、是否缺少 `limit`、是否存在高成本 join 或大范围扫描。
这些检查应该用代码层规则、SQL parser、权限系统和查询网关共同完成，不能只依赖 prompt。

Q076：为什么不能只靠 prompt 要求模型生成安全 SQL？
重要程度：5/5

### 回答
不能只靠 prompt 要求模型生成安全 SQL，因为 prompt 是软约束，模型可能理解错、遗漏规则，也可能在复杂问题下生成越权字段、危险 SQL
或高成本查询。 生产安全必须用代码层硬约束实现。比如敏感字段要有字段级权限标签，表字段要来自 Schema Catalog，危险关键字要静态拦截，
大表必须有时间范围， 明细查询必须有 `limit` 和精确过滤。prompt 可以帮助模型少犯错，但不能替代 SQL 校验器、权限系统、审计日志和查询网关。
