## 大白话解释

项目收口就是把散落的代码、笔记、命令和结论整理成一个可展示作品。

面试官或同事不会替你翻每个脚本。他们需要快速看懂：

- 这个项目解决什么问题
- 怎么启动
- 怎么输入问题
- 输出长什么样
- 为什么这样设计
- 当前还有哪些限制

---

## 生产实际

真实公司交付 AI 项目时，README 和演示材料不是形式主义。

它们解决的是协作问题：

- 表格行 1
  - 列 1：使用者
  - 列 2：关心点
--------|--------|
- 表格行 1
  - 列 1：业务方
  - 列 2：这个系统能回答什么、不能回答什么
- 表格行 2
  - 列 1：后端同事
  - 列 2：API 怎么调、错误怎么处理
- 表格行 3
  - 列 1：数据同事
  - 列 2：知识库资料来自哪里、口径是否可信
- 表格行 4
  - 列 1：测试同事
  - 列 2：哪些问题要回归、成功标准是什么
- 表格行 5
  - 列 1：面试官
  - 列 2：你是否理解完整工程链路

RAG 项目尤其要写清楚失败场景，比如知识库没有资料、召回不相关、引用过期、问题超出范围、权限不足、回答无法判断。

---

## 常见坑

- 表格行 1
  - 问题：README 只写启动命令
  - 影响：看不出项目价值
  - 处理方式：补背景、架构、产出和限制
- 表格行 2
  - 问题：只展示成功样例
  - 影响：面试追问时容易露怯
  - 处理方式：主动列 bad case 和改进方向
- 表格行 3
  - 问题：没有接口样例
  - 影响：别人无法快速调用
  - 处理方式：写请求、响应和错误示例
- 表格行 4
  - 问题：没有测试问题
  - 影响：改动后无法回归
  - 处理方式：保留固定问题集
- 表格行 5
  - 问题：不说明边界
  - 影响：用户以为什么都能答
  - 处理方式：写清知识库范围和拒答规则

## 工程取舍

项目展示不是把所有细节堆上去，而是按主线组织。

建议顺序：

```text
问题背景
-> 系统目标
-> 离线入库
-> 在线问答
-> 引用来源
-> 召回优化
-> 失败场景
-> 后续改进
```

这个顺序能体现你理解的不只是模型调用，而是一套可落地的知识问答系统。

---

## 本地练习

补全项目 README 时至少包含：

- 项目简介
- 架构流程图或文字流程
- 启动命令
- 请求示例
- 响应示例
- 测试问题
- 当前限制
- 后续优化方向

演示时控制在 5 分钟内：

1. 说明项目目标
2. 展示入库和问答流程
3. 跑一个正常问题
4. 展示 citations
5. 讲一个 bad case 和改进方向

## 面试沉淀

Q031：一个 RAG 项目怎样包装成可展示、可复盘的作品？
重要程度：4/5

### 回答

RAG 项目展示不能只说“我调了一个模型”。我会按工程链路包装：先说明业务问题，比如企业文档、指标口径或 SQL 规范查询；
再说明离线入库如何做文档接入、清洗、切分、embedding、索引和 metadata；然后说明在线问答如何做问题处理、检索、引用返回和错误处理。
README 里要有启动命令、接口示例、响应示例、测试问题和 bad case。
演示时要展示一个成功问题，再展示一个失败或边界问题，说明怎么排查是知识库缺失、 召回不准还是生成错误。
这样项目看起来不是 Demo，而是有工程闭环、质量意识和迭代方向的作品。

## 今日复盘

今天要记住三句话：

1. 项目收口的目标是让别人能快速理解、启动和评价。
2. RAG 展示必须包含引用、bad case 和边界。
3. README 是工程交付的一部分，不是附属文档。

我的回答：

回答评价：
评分：8/10
方向正确，抓住了边界和后续处理；可以再补“体现工程排查能力”和“不是只挑成功样例”。

参考答案：

### 4. Day 17 到 Day 20 的 RAG 项目完整链路是什么？
  重要程度：5/5
我的回答：

分为两条，离线入库和在线问答，离线入库包含数据文档等资料的入库，然后资料的清洗，chunk 文档的切分，，向量化 embedding 在线问答包含 召回，
问题id，上下文， quarry rewrite ，引用来源，错误处理

回答评价：
评分：8/10
主链路方向正确，能区分离线入库和在线问答；需要补 Day 18、Day 19、Day 20 的分工，并把 request_id、confidence、API
化等字段说清楚。

参考答案：
Day 17 到 Day 20 串起来是一个最小 RAG 工程闭环。Day 17 是离线入库：读取学习笔记和项目资料，清洗正文，切分 chunk，生成学习版
embedding，把 chunk、向量、来源、 位置和 metadata 写入 SQLite 索引。
Day 18 是在线问答：接收用户问题，从索引中召回 top-k chunk，并返回答案草稿和 citations。
Day 19 是召回优化：用固定测试问题对比 top-k、query rewrite 等策略，记录 bad case。
Day 20 是 API 化： 用 FastAPI 提供 `/health` 和 `/rag/ask`， 响应中返回 answer、citations、
request_id、confidence、cannot_answer_reason 和 latency_ms。
