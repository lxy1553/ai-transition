## 大白话解释

RAG 的目标不是让模型“看起来很会答”，而是让它基于可靠资料回答。

如果知识库里没有答案，正确做法不是让模型猜；
而是明确告诉用户：

```text
当前知识库没有足够依据，不能给出可靠答案。
```

如果问题涉及权限、隐私、敏感信息或执行高风险操作，也不能只靠模型自由发挥。
系统应该先用规则、权限和检索结果做边界判断，再决定是否调用模型生成答案。

---

## 生产实际

真实公司里的 RAG 系统通常会把回答决策拆成三类：

| 决策 | 含义 | 典型场景 |
|------|------|----------|
| `answer` | 可以回答 | 有足够依据，用户有权限，问题在业务范围内 |
| `refuse` | 必须拒答 | 无资料依据、无权限、敏感信息、危险操作 |
| `clarify` | 需要澄清 | 问题太模糊、缺少指标、缺少时间范围或业务域 |

拒答不是失败，而是生产系统的安全能力。
一个成熟的 RAG 系统应该能说清楚：

- 拒答原因是什么；
- 缺少什么依据；
- 用户可以怎么补充问题；
- 是否可以给出安全替代回答。

例如用户问：

```text
把所有用户手机号导出来。
```

系统不应该生成导出 SQL。
它应该拒答，并说明涉及敏感个人信息，需要权限审批和脱敏流程。

---

## 常见拒答边界

| 边界类型 | 触发条件 | 系统动作 |
|----------|----------|----------|
| 无依据 | 检索不到可靠 chunk | 拒答，并提示知识库没有足够资料 |
| 低置信度 | top-k 分数低或来源不一致 | 拒答或澄清，不生成确定答案 |
| 权限不足 | 用户无权访问资料 | 拒答，并记录审计日志 |
| 敏感信息 | 涉及手机号、身份证、密钥、工资等 | 拒答或要求脱敏审批 |
| 危险操作 | 删除数据、绕过权限、批量导出 | 拒答，并给出安全流程 |
| 问题模糊 | 缺少指标、时间、口径或对象 | 先澄清，再检索 |

## 工程取舍

第一版拒答策略不要完全交给 LLM 判断。

原因很简单：

```text
LLM 本身也可能误判边界。
```

生产里更稳的做法是分层控制：

1. 规则层先拦截明显敏感、越权和危险问题；
2. 检索层判断是否有足够依据；
3. 权限层过滤用户不可见资料；
4. 生成层要求模型只基于 citations 回答；
5. 评测层用固定测试集检查拒答准确率。

LLM 可以参与解释拒答原因，但不应该是唯一的安全边界。

## 面试沉淀

Q046：RAG 系统什么时候应该拒答？

### 回答

RAG 系统在没有可靠依据、用户无权限、问题涉及敏感信息、请求危险操作、
检索置信度过低或问题表达过于模糊时，应该拒答或先澄清。
拒答不是系统没能力，而是为了防止模型在没有资料支撑时编造答案。
生产里通常会结合规则拦截、权限过滤、检索分数、引用来源和测试集来判断是否可以回答。
如果拒答，也要返回清楚的原因，比如知识库没有足够依据、资料权限不足或问题需要补充时间范围。

Q047：生产环境里怎么设计 RAG 拒答策略？

### 回答

生产 RAG 的拒答策略通常分层设计。
第一层用规则识别敏感信息、危险操作和越权请求；
第二层用权限系统过滤用户不可见资料；
第三层用检索结果判断 expected source 是否命中、top-k 分数是否足够、来源是否一致；
第四层在 prompt 中要求模型只基于 citations 回答，不能基于常识补全；
第五层用测试集评估拒答准确率、误拒率和幻觉率。
这样可以把拒答从“模型自己决定”变成可解释、可测试、可审计的工程能力。

Q048：RAG 幻觉通常怎么产生，怎么降低？

### 回答

RAG 幻觉常见原因包括知识库资料缺失、chunk 切分不合理、检索没有命中正确资料、
query rewrite 改写偏移、无关上下文被塞进 prompt、模型没有被限制只基于引用回答。
降低幻觉要从链路上处理，而不是只改 prompt。
离线侧要保证资料可信、切分合理、版本清楚；
检索侧要提高 expected source 命中率，必要时使用混合检索和 rerank；
生成侧要要求答案必须有 citations 支撑；
边界侧要在无依据、低置信度、越权和敏感场景下拒答；
评测侧要长期维护 bad case 和拒答样本。

### 2. 拒答和回答错误有什么区别？

我的回答：
拒答是rag 系统设计的必要一环，是保证系统安全，可信度高的重要环节；
错误是 rag 设计本身缺陷，可能是引用错误，曲解改写问题，边界不明，幻觉等重大错误

回答评价：
回答正确。
已经区分了拒答是主动边界控制，错误回答是系统链路缺陷。
可以再补一句：拒答是可解释、可审计的安全行为；
错误回答则会误导用户，降低系统可信度。

参考答案：
拒答是系统在判断依据不足、权限不足、问题敏感或风险过高时，主动选择不生成确定答案。
它是生产 RAG 的安全能力。
回答错误则是系统本该回答正确，但因为检索、引用、改写、生成或边界判断出错，
给出了不可靠甚至误导性的答案。
前者是受控行为，后者是质量问题或安全风险。

### 3. 为什么拒答策略不能完全交给 LLM 判断？

我的回答：
 llm 也有可能曲解问题，需要通过多个链路策略来判断是否拒答

回答评价：
回答正确，但偏简短。
核心点说到了：LLM 也可能误判或曲解问题。
生产里还要强调拒答需要结合规则、权限、检索结果、引用来源和测试集，而不是只靠模型自由判断。

参考答案：
拒答策略不能完全交给 LLM，因为 LLM 本身可能误解用户意图、忽略权限边界、
低估敏感风险，或在没有资料时仍然生成看似合理的答案。
生产里应该用多层策略控制：
规则层拦截敏感和危险请求，权限层过滤不可见资料，检索层判断是否有足够依据，
生成层要求只基于 citations 回答，评测层用固定测试集检查拒答准确率。

### 4. 生产里怎么判断一个 RAG 回答是否有足够依据？

我的回答：
通过固定测试集测试和 bad case 长期维护，返回的答案的期望引用率，置信度等

回答评价：
方向正确，但需要补完整。
固定测试集和 bad case 适合做长期评估；
单次线上回答是否有依据，还要看 top-k 是否命中可靠资料、citations 是否正确、
检索分数是否足够，以及答案是否严格基于引用上下文。

参考答案：
生产里判断一个 RAG 回答是否有足够依据，不能只看模型说得是否流畅。
需要先看检索结果是否命中可靠资料，top-k 中是否包含 expected source；
再看 citations 是否指向正确文档、chunk 和位置；
然后看检索分数、来源一致性和权限过滤是否通过。
如果没有可靠引用、引用和答案不一致、分数过低或来源冲突，就应该拒答或澄清。
固定测试集和 bad case 用来长期验证这些规则是否稳定有效。

### 5. 敏感信息、权限不足和无答案问题分别应该怎么处理？

我的回答：
都应该拒绝回答，给出解释说明敏感信息或者包含敏感信息，无答案的说明相关资料不足

回答评价：
回答基本正确。
三类问题都不能直接给确定答案，但处理方式可以更细：
敏感信息要拒答或要求脱敏审批；
权限不足要拒答并记录审计；
无答案要说明知识库没有足够依据，并引导用户补充资料或问题范围。

参考答案：
敏感信息问题不能直接回答，应该拒答、提示涉及隐私或敏感数据，
必要时引导走脱敏、审批或合规流程。
权限不足问题应该拒答，不能暴露用户无权查看的资料，同时记录审计日志。
无答案问题应该说明当前知识库没有足够依据，不能基于猜测回答；
可以提示用户补充资料、明确业务域或改问更具体的问题。

### 6. 怎么用测试集评估拒答策略是否有效？

我的回答：
通过修改规则查看拒答率是否改变

回答评价：
方向对，但不完整。
拒答率变化只能说明策略变严或变松，不能说明策略一定更好。
评估时还要看 expected_decision、拒答准确率、误拒率、误放行率和 bad case 是否减少。

参考答案：
评估拒答策略要准备固定测试集，每条样本标注 expected_decision，
比如 answer、refuse 或 clarify。
测试集要覆盖无答案、低置信度、权限不足、敏感信息、危险操作、问题模糊和正常可回答问题。
运行策略后对比 actual_decision 和 expected_decision；
重点看拒答准确率、误拒率、误放行率、澄清准确率和 bad case 数量。
如果规则调整后只是拒答率升高，但正常问题也被大量误拒，说明策略并没有真正变好。
