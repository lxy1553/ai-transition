## 大白话解释

Schema Catalog 可以理解为“数据资产说明书”。

它告诉 Agent：

- 有哪些表和实时指标；
- 每张表属于 ODS、DWD、DWS、ADS 还是实时层；
- 表的粒度是什么；
- 时间字段和分区字段是什么；
- 哪些字段是敏感字段；
- 普通数据问答能不能查；
- 查这张表必须满足什么条件。

工具注册表可以理解为“Agent 工具使用说明书”。

它告诉 Agent：

- 有哪些工具；
- 每个工具解决什么意图；
- 调用工具需要哪些输入；
- 调用前必须满足哪些前置条件；
- 工具风险等级是什么；
- 失败时应该澄清、降级、拒答还是返回资料不足；
- 是否必须写审计。

如果没有 Schema Catalog，Agent 可能编造字段、选错表、漏分区或查敏感明细。
如果没有工具注册表，Agent 可能把口径问题拿去查 SQL，把实时告警问题拿去查离线日报，或者把敏感导出请求当成普通查询执行。

## 生产实际

金融信贷公司里，数据问答 Agent 不应该直接看到所有物理表。
它应该看到经过治理的 Catalog。

例如用户问：

```text
近 7 天各渠道授信通过率是多少？
```

Agent 不应该直接去猜 SQL。
正确路线是：

```text
识别主题域：授信申请
识别意图：离线指标查询
读取指标字典：确认授信通过率口径
读取 Schema Catalog：找到 DWS/ADS 表和字段
读取工具注册表：确认 offline_nl2sql_query 的前置条件
检查时间范围、分区、权限和敏感字段
生成只读 SQL
SQL 校验通过后执行
结果解释并写审计
```

再例如用户问：

```text
看一下当前实时风控拒绝率是不是异常，按近 10 分钟。
```

这类问题应该走实时指标工具。
工具调用前必须确认：

- 实时指标是否存在；
- 窗口是否明确；
- 延迟阈值是多少；
- 当前链路是否延迟；
- 返回结果是否需要带窗口、事件时间和延迟状态。

如果用户问：

```text
导出昨天所有授信申请客户的手机号和身份证号。
```

即使 Catalog 里存在 DWD 明细表，也不能把它当普通查询执行。
因为手机号、身份证号和客户名单属于敏感明细，工具注册表应该把这类请求路由到 `safe_block`，并写审计。

## Day 51 Catalog 设计

今天先登记 5 类资产：

| 资产 | 类型 | 作用 | 关键约束 |
|------|------|------|----------|
| `ads_credit_daily_metrics` | 离线 ADS 表 | 经营日报总览 | 必须带 `biz_date` |
| `dws_credit_apply_channel_1d` | 离线 DWS 表 | 渠道/产品/风险等级趋势分析 | 必须带 `dt` |
| `dwd_credit_apply_detail_di` | 离线 DWD 明细表 | 授权排查和明细核对 | restricted，含敏感字段 |
| `rt_risk_reject_rate_10m` | 实时指标 | 近 10 分钟风控拒绝率 | 必须检查窗口和延迟 |
| `lineage_credit_approval_rate` | 血缘入口 | 授信通过率来源追溯 | 返回上游、任务、下游 |

这些资产共同说明一件事：Agent 不是“知道表名就能查”。
它必须理解表层级、业务粒度、时间字段、分区字段、权限等级和风险边界。

## Day 51 工具注册表设计

今天登记 5 个工具：

| 工具 | 适用意图 | 关键前置条件 | 风险等级 |
|------|----------|--------------|----------|
| `metric_definition_rag` | 口径解释 | 必须有引用来源 | medium |
| `offline_nl2sql_query` | 离线指标查询 | 必须有时间范围和分区条件 | high |
| `realtime_metric_query` | 实时状态查询 | 必须有窗口并检查延迟 | high |
| `lineage_lookup` | 血缘追溯 | 问题关注来源或影响范围 | medium |
| `safe_block` | 敏感导出/越权 | 命中敏感字段或越权访问 | critical |

工具注册表不是给模型看的“提示词装饰”，而是执行前的工程约束。
模型可以提出候选工具，但系统要按注册表检查前置条件。

例如：

- 缺少时间范围：离线 SQL 工具不能执行，应先澄清；
- 缺少实时窗口：实时指标工具不能执行，应先澄清；
- 实时链路延迟超阈值：不能给出确定异常结论；
- 命中手机号、身份证号、客户名单：不能进入 SQL 查询，应安全阻断；
- 没有可靠口径资料：RAG 不能编造，应返回资料不足。

## 常见坑

| 类型 | 可能的问题 | 生产处理方式 |
|------|------------|--------------|
| Catalog 只有字段名 | Agent 不知道粒度、时间字段和权限 | Catalog 必须补粒度、分区、时间、权限和 owner |
| DWD 明细随便查 | 可能导出客户敏感信息 | 普通问答优先 ADS/DWS，DWD 需要授权和审计 |
| 工具描述太空 | Agent 不知道什么时候不能调用 | 工具注册表必须写前置条件和 fallback |
| 实时指标无窗口 | 查询结果没有业务含义 | 实时工具缺窗口必须澄清 |
| 漏分区 | SQL 扫描成本高、结果范围失控 | SQL 校验强制时间范围和分区条件 |
| 只靠 prompt 控制工具 | 模型仍可能选错工具或越权 | 用 Catalog 和注册表做确定性校验 |

## 工程取舍

### 取舍一：为什么 Catalog 不只是字段列表？

字段列表只能告诉 Agent “有哪些字段”，不能告诉它“这张表能不能查、按什么粒度查、时间字段是什么、是否含敏感字段”。
生产里 NL2SQL 最常见的问题不是 SQL 语法错，而是选错表、漏分区、口径不一致和越权查明细。
所以 Catalog 必须包含业务说明、层级、粒度、分区、时间字段、权限标签、敏感字段、owner 和查询约束。

### 取舍二：为什么工具注册表要写风险等级？

不同工具的风险不一样。
口径 RAG 工具主要风险是资料不足或引用错误；
离线 NL2SQL 工具风险包括扫描成本、越权字段和错误口径；
实时指标工具风险包括窗口缺失、延迟和误判；
安全阻断工具则处理 critical 级请求。

风险等级能决定是否必须审计、是否需要二次确认、是否允许自动执行，以及失败后如何回退。

### 取舍三：为什么 Agent 不能自己决定所有工具调用？

Agent 可以理解问题和提出候选路线，但工具调用涉及权限、成本、实时状态和合规。
这些规则不能完全交给模型判断。
正确做法是：模型负责候选意图和候选工具，工程层用 Catalog、工具注册表、权限系统和校验器决定是否执行。

## 面试沉淀

Q113：金融信贷 Agent 的 Schema Catalog 应该包含哪些信息？
重要程度：5/5

### 回答

生产级 Schema Catalog 不能只记录表名和字段名。
它应该同时记录表描述、业务主题域、仓库层级、表粒度、主键、分区字段、时间字段、字段类型、字段含义、
可用维度、指标字段、敏感字段、权限等级、数据更新频率、owner、查询约束和下游使用场景。

在金融信贷 Agent 里，Catalog 的作用是约束 NL2SQL 和工具路由。
例如 `ads_credit_daily_metrics` 适合日报总览，`dws_credit_apply_channel_1d` 适合渠道趋势，
`dwd_credit_apply_detail_di` 含客户手机号和身份证号，只能在授权排查时访问。
Agent 生成 SQL 前要根据 Catalog 判断应该选 ADS、DWS 还是 DWD，必须带时间范围和分区条件，
不得编造字段，不得访问无权限的敏感字段。

一句面试总结是：Schema Catalog 保证 Agent “查得对、查得安全、查得可控成本”。

Q114：Agent 工具注册表应该如何设计？
重要程度：5/5

### 回答

Agent 工具注册表应该把每个工具当成一个有边界的生产能力，而不是只写一个工具名。
至少要包含工具 ID、工具名称、适用意图、功能描述、允许访问的实体类型、必填输入参数、前置条件、
权限要求、风险等级、超时和失败处理、是否需要审计、返回结构和典型不可调用场景。

例如离线 NL2SQL 工具要写清：只适合离线指标查询，优先使用 ADS/DWS，必须有时间范围和分区条件，
不能访问 restricted 敏感字段；实时指标工具要写清：必须有窗口，必须检查延迟，延迟超阈值时不能给确定结论；
安全阻断工具要处理手机号、身份证号、银行卡号和客户名单这类敏感导出。

生产里可以让模型选择候选工具，但系统必须按注册表做确定性校验。
这样才能避免模型自由调用工具导致误查表、越权、漏分区或把实时问题错误路由到离线 SQL。

Q115：为什么工具调用前必须检查前置条件和风险等级？
重要程度：5/5

### 回答

工具调用前检查前置条件和风险等级，是为了把“模型建议”变成“可执行动作”之前先过一层工程闸门。
金融信贷数据问答涉及经营指标、实时风控、客户明细和合规审计，如果工具调用没有前置检查，
Agent 可能在缺少时间范围时扫描大表，在缺少实时窗口时给出无意义结果，在链路延迟时误判正常，
或者把客户手机号、身份证号导出给无权限用户。

前置条件通常包括：意图是否匹配、必填参数是否齐全、时间范围是否明确、分区条件是否存在、
是否命中敏感字段、用户权限是否足够、实时延迟是否可接受、是否有可靠引用来源。
风险等级用于决定是否自动执行、是否需要二次确认、是否必须审计、失败后是澄清、降级还是阻断。

一句面试总结是：Agent 可以负责理解和编排，但工具执行必须由权限、Catalog、注册表和校验器共同控制。

## 术语更新

今天新增或强化这些术语：

- Schema Catalog：记录表、字段、粒度、分区、权限和查询约束的数据资产目录。
- Tool Registry：记录 Agent 可调用工具、输入参数、前置条件、风险等级和失败处理的工具注册表。
- 前置条件：工具调用前必须满足的条件，例如时间范围、窗口、权限和引用来源。
- 风险等级：用于区分工具调用风险高低，决定是否审计、确认、降级或阻断。
- 敏感字段标签：Catalog 中标记手机号、身份证号、银行卡号、客户 ID 等敏感字段的权限标签。
