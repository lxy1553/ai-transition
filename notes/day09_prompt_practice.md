## 大白话解释

“Prompt 实战”这一天要解决的是一个很具体的工程问题：先把基础能力、工具或项目环节讲清楚，再把它变成能运行、能检查、能复盘的产物。
学习时不要只看代码是否跑通，还要能说清楚它为什么需要、输入输出是什么、出错后怎么定位。

## Prompt 五要素

- 表格行 1
  - 要素：Role
  - 说明：模型扮演什么角色
  - 示例：你是数据仓库 SQL 解释助手
- 表格行 2
  - 要素：Task
  - 说明：要完成什么任务
  - 示例：解释 SQL 的业务含义和风险
- 表格行 3
  - 要素：Context
  - 说明：提供哪些背景信息
  - 示例：表结构、字段口径、业务场景
- 表格行 4
  - 要素：Constraints
  - 说明：有哪些限制
  - 示例：不编造表结构；信息不足要说明
- 表格行 5
  - 要素：Format
  - 说明：按什么格式输出
  - 示例：JSON、Markdown、固定小节

一个工程化 Prompt 应该能被复制、测试和迭代，而不是每次临时手写。

## 好 Prompt 和坏 Prompt 的差别

### 不够好的写法

```text
帮我解释这个 SQL。
```

问题：

- 没有角色，模型不知道要按什么专业程度回答。
- 没有上下文，模型容易猜字段含义。
- 没有约束，可能编造业务口径。
- 没有输出格式，结果不稳定，不方便接入程序。

### 更好的写法

```text
你是一个数据仓库 SQL 解释助手。

请解释下面 SQL 的业务含义、关键字段、统计口径和潜在风险。

约束：
- 只基于已给 SQL 和表结构解释
- 不确定的字段含义必须标记为“待确认”
- 不编造不存在的表或字段

输出格式：
1. 业务含义
2. 字段与口径
3. 风险提示
4. 需要补充的上下文
```

核心变化：从“提问”变成“任务说明书”。

---

## Prompt 改写原则

1. 先定角色：让模型知道回答视角。
2. 再定任务：明确要做什么，不要只说“帮我看看”。
3. 补上下文：表结构、字段口径、业务背景、目标读者。
4. 加约束：哪些不能做，哪些不确定要明说。
5. 固定格式：让输出稳定，后续才能自动解析或复用。

面试表达：

> 我理解的 Prompt 工程不是写漂亮句子，而是把业务任务拆成角色、上下文、约束和输出格式。
尤其在数据问答和 NL2SQL 场景里，Prompt 必须限制模型不要编造表结构和业务口径。

## 模板 1：SQL 解释

```text
你是一个数据仓库 SQL 解释助手。

任务：
解释用户提供的 SQL，包括：
1. 这段 SQL 的业务含义
2. 每个关键字段的作用
3. 可能的性能风险
4. 是否缺少必要上下文

约束：
- 不要编造不存在的表结构
- 如果字段含义不明确，请直接说明
- 回答要适合数据开发面试讲解

输出格式：
## 业务含义
## 字段解释
## 风险提示
## 需要补充的上下文
```

---

## 模板 2：数据问答

```text
你是一个数据分析助手。

任务：
根据给定的数据表说明回答业务问题。

约束：
- 只基于已给出的表结构和字段说明回答
- 如果无法确认口径，请说明不能直接判断
- 优先指出口径、时间范围、过滤条件

输出格式：
1. 结论
2. 推理依据
3. 口径风险
4. 建议 SQL 思路
```

## 模板 3：简历项目包装

```text
你是一个 AI 应用工程师简历优化助手。

任务：
把我的学习项目改写成简历项目描述。

约束：
- 不夸大真实经历
- 强调工程能力和数据背景迁移
- 每条 bullet 使用“动作 + 技术 + 结果/价值”

输出格式：
- 项目名称
- 技术栈
- 3 条项目描述
- 面试讲解亮点
```

## 今日练习

### 练习 1：SQL 解释 Prompt

把下面 SQL 放进 SQL 解释模板里：

```sql
select
    city,
    count(distinct user_id) as active_users
from user_events
where dt between '2026-05-01' and '2026-05-07'
  and event_name = 'app_open'
group by city;
```

你要重点约束模型：

- 不要猜 `user_events` 的完整表结构。
- 说明 `active_users` 是去重用户数。
- 提醒 `dt` 和 `event_name` 的口径需要确认。

### 练习 2：数据问答 Prompt

业务问题：

```text
上周哪个城市的活跃用户最多？
```

你要给模型的上下文至少包括：

- 可用表：`user_events`
- 关键字段：`city`、`user_id`、`event_name`、`dt`
- 活跃定义：发生过 `app_open` 事件的去重用户
- 时间范围：上周

## 生产实际

Prompt 在生产里应该像代码一样管理，而不是临时写一句话。金融信贷场景里的 Prompt 要明确角色、任务、上下文、边界和输出格式，
例如“只基于授信政策回答”、 “不要编造风控规则”、“敏感字段必须拒绝”。

生产中还要做 Prompt 版本管理和回归测试。同一个 RAG 问答或 SQL 解释 Prompt 改动后，必须用固定测试集验证有没有引入新的幻觉、
越权或格式错误。

## 常见坑

- 表格行 1
  - 类型：目标
  - 可能的问题：只完成 Demo，不知道对应真实项目环节
  - 生产处理方式：把“Prompt 实战”映射到 RAG、NL2SQL、SQL 解释助手或服务化链路
- 表格行 2
  - 类型：数据
  - 可能的问题：输入样例过干净，真实数据有缺失、重复、格式不统一
  - 生产处理方式：增加校验、异常分支和边界样例
- 表格行 3
  - 类型：工程
  - 可能的问题：代码能跑但缺少日志、配置、错误提示和复盘材料
  - 生产处理方式：保留 README、输出文件、运行命令和关键结果
- 表格行 4
  - 类型：安全
  - 可能的问题：忽略密钥、权限、敏感字段或外部调用风险
  - 生产处理方式：使用环境变量、权限控制、脱敏和审计记录
- 表格行 5
  - 类型：维护
  - 可能的问题：只写临时代码，后续无法扩展或讲清楚
  - 生产处理方式：按模块拆分，并补充大白话注释说明设计原因

## 工程取舍

这一天优先采用最小可运行方案，是为了先建立稳定基线。真实项目里不能一开始就追求复杂架构，应该先把输入、处理逻辑、输出和失败场景跑通，再逐步增加模型调用、 检索、
数据库、权限和评测。

对“Prompt 实战：约束输出与角色设定”这类能力，学习阶段更看重可解释和可复盘；生产阶段再根据准确率、延迟、成本、安全和维护成本决定是否引入更复杂的方案。

## 面试沉淀

Q017：Prompt 在生产项目里应该怎么管理？

### 回答

生产里的 Prompt 不应该散落在代码里随手修改。它应该有清晰的任务目标、输入变量、输出格式、边界约束和版本记录。
每次修改 Prompt 都要用固定测试集回归，观察准确率、格式稳定性、拒答效果和 bad case。
对于 SQL 解释、数据问答和 RAG 场景， Prompt 管理本质上是在管理模型行为的可控性和可追溯性。
