## Prompt 五要素

| 要素 | 说明 | 示例 |
|------|------|------|
| Role | 模型扮演什么角色 | 你是数据仓库 SQL 解释助手 |
| Task | 要完成什么任务 | 解释 SQL 的业务含义和风险 |
| Context | 提供哪些背景信息 | 表结构、字段口径、业务场景 |
| Constraints | 有哪些限制 | 不编造表结构；信息不足要说明 |
| Format | 按什么格式输出 | JSON、Markdown、固定小节 |

一个工程化 Prompt 应该能被复制、测试和迭代，而不是每次临时手写。

## 好 Prompt 和坏 Prompt 的差别

### 不够好的写法

```text
帮我解释这个 SQL。
```

问题：

- 没有角色，模型不知道要按什么专业程度回答。
- 没有上下文，模型容易猜字段含义。
- 没有约束，可能编造业务口径。
- 没有输出格式，结果不稳定，不方便接入程序。

### 更好的写法

```text
你是一个数据仓库 SQL 解释助手。

请解释下面 SQL 的业务含义、关键字段、统计口径和潜在风险。

约束：
- 只基于已给 SQL 和表结构解释
- 不确定的字段含义必须标记为“待确认”
- 不编造不存在的表或字段

输出格式：
1. 业务含义
2. 字段与口径
3. 风险提示
4. 需要补充的上下文
```

核心变化：从“提问”变成“任务说明书”。

---

## Prompt 改写原则

1. 先定角色：让模型知道回答视角。
2. 再定任务：明确要做什么，不要只说“帮我看看”。
3. 补上下文：表结构、字段口径、业务背景、目标读者。
4. 加约束：哪些不能做，哪些不确定要明说。
5. 固定格式：让输出稳定，后续才能自动解析或复用。

面试表达：

> 我理解的 Prompt 工程不是写漂亮句子，而是把业务任务拆成角色、上下文、约束和输出格式。
尤其在数据问答和 NL2SQL 场景里，Prompt 必须限制模型不要编造表结构和业务口径。

## 模板 1：SQL 解释

```text
你是一个数据仓库 SQL 解释助手。

任务：
解释用户提供的 SQL，包括：
1. 这段 SQL 的业务含义
2. 每个关键字段的作用
3. 可能的性能风险
4. 是否缺少必要上下文

约束：
- 不要编造不存在的表结构
- 如果字段含义不明确，请直接说明
- 回答要适合数据开发面试讲解

输出格式：
## 业务含义
## 字段解释
## 风险提示
## 需要补充的上下文
```

---

## 模板 2：数据问答

```text
你是一个数据分析助手。

任务：
根据给定的数据表说明回答业务问题。

约束：
- 只基于已给出的表结构和字段说明回答
- 如果无法确认口径，请说明不能直接判断
- 优先指出口径、时间范围、过滤条件

输出格式：
1. 结论
2. 推理依据
3. 口径风险
4. 建议 SQL 思路
```

## 模板 3：简历项目包装

```text
你是一个 AI 应用工程师简历优化助手。

任务：
把我的学习项目改写成简历项目描述。

约束：
- 不夸大真实经历
- 强调工程能力和数据背景迁移
- 每条 bullet 使用“动作 + 技术 + 结果/价值”

输出格式：
- 项目名称
- 技术栈
- 3 条项目描述
- 面试讲解亮点
```

## 今日练习

### 练习 1：SQL 解释 Prompt

把下面 SQL 放进 SQL 解释模板里：

```sql
select
    city,
    count(distinct user_id) as active_users
from user_events
where dt between '2026-05-01' and '2026-05-07'
  and event_name = 'app_open'
group by city;
```

你要重点约束模型：

- 不要猜 `user_events` 的完整表结构。
- 说明 `active_users` 是去重用户数。
- 提醒 `dt` 和 `event_name` 的口径需要确认。

### 练习 2：数据问答 Prompt

业务问题：

```text
上周哪个城市的活跃用户最多？
```

你要给模型的上下文至少包括：

- 可用表：`user_events`
- 关键字段：`city`、`user_id`、`event_name`、`dt`
- 活跃定义：发生过 `app_open` 事件的去重用户
- 时间范围：上周
