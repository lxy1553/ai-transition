## 大白话解释

前面做的日报、告警、NL2SQL、血缘和评测，如果都只是本地脚本，就很难真正接到前端、调试平台或演示接口里。

服务化的核心不是“包一层 FastAPI”。
核心是把这些东西固定下来：

```text
谁发起了请求
请求要查什么
走了哪些节点和工具
用了哪个模型和配置
结果是什么
为什么拒答或降级
后续怎么回放
```

---

## 生产实际

金融信贷 Agent 服务化至少要有四类审计：

| 类型 | 记录重点 |
|------|----------|
| 查询审计 | 请求内容、用户、场景、工具路线、结果状态 |
| 告警审计 | 告警时间、规则版本、证据、解释结果 |
| 任务审计 | 定时日报、批量评测、补偿重跑 |
| 权限审计 | 用户角色、敏感字段访问、拒答原因 |

最小字段建议：

```text
request_id
trace_id
user_id
scenario
intent
tool_route
model_id
status
latency_ms
error_code
audit_created_at
```

## 常见坑

| 类型 | 可能的问题 | 生产处理方式 |
|------|------------|--------------|
| 日志无主键 | 无法串起一次请求 | 强制使用 request_id 和 trace_id |
| 审计只写文本 | 难以检索和回放 | 保存结构化字段 |
| 日志过多敏感信息 | 合规风险高 | 统一脱敏和字段白名单 |
| 契约频繁漂移 | 前后端联调困难 | 固定错误码和状态字段 |
| trace 不可回放 | 无法定位 bad case | 记录节点、工具和版本信息 |

## 工程取舍

### 取舍一：为什么先定审计字段，再扩展接口？

因为接口随时能加，但一旦没有稳定审计字段，后面很难补全历史行为。
Day 58 更像给后续集成定“骨架”。

### 取舍二：为什么权限审计要单独看？

因为金融场景里“有没有回答”只是结果层。
更重要的是中间有没有查过不该查的数据、有没有走到不该走的工具。

## 面试沉淀

Q133：为什么金融信贷 Agent 服务化必须先定 request_id 和审计结构？
重要程度：5/5

### 回答

因为 Agent 的问题定位、权限追溯、bad case 回放和版本回归都依赖稳定主键和结构化审计字段。
如果每次请求没有 request_id、trace_id 和统一状态，后续即使发现错误答案，也很难知道问题出在模型、工具还是数据链路。

Q134：为什么不能把完整用户问题和原始查询结果直接写进日志？
重要程度：5/5

### 回答

因为金融信贷问题里可能包含手机号、身份证号、客户名单、订单号和贷款明细。
日志系统通常比业务库更广泛地被访问和复制，所以必须采用字段白名单、脱敏和最小留存策略。

## 术语更新

- request_id：一次业务请求的唯一标识。
- trace_id：串联多个节点、工具和下游调用的链路标识。
- 审计回放：基于历史结构化记录复原一次请求执行过程的能力。
- 错误码契约：稳定表达拒答、降级、越权和工具失败等状态的编码规范。
