## 大白话解释

前面做的日报、告警、NL2SQL、血缘和评测，如果都只是本地脚本，就很难真正接到前端、调试平台或演示接口里。

服务化的核心不是“包一层 FastAPI”。
核心是把这些东西固定下来：

```text
谁发起了请求
请求要查什么
走了哪些节点和工具
用了哪个模型和配置
结果是什么
为什么拒答或降级
后续怎么回放
```

---

## 生产实际

金融信贷 Agent 服务化至少要有四类审计：

| 类型 | 记录重点 |
|------|----------|
| 查询审计 | 请求内容、用户、场景、工具路线、结果状态 |
| 告警审计 | 告警时间、规则版本、证据、解释结果 |
| 任务审计 | 定时日报、批量评测、补偿重跑 |
| 权限审计 | 用户角色、敏感字段访问、拒答原因 |

最小字段建议：

```text
request_id
trace_id
user_id
scenario
intent
tool_route
model_id
status
latency_ms
error_code
audit_created_at
```

## 常见坑

| 类型 | 可能的问题 | 生产处理方式 |
|------|------------|--------------|
| 日志无主键 | 无法串起一次请求 | 强制使用 request_id 和 trace_id |
| 审计只写文本 | 难以检索和回放 | 保存结构化字段 |
| 日志过多敏感信息 | 合规风险高 | 统一脱敏和字段白名单 |
| 契约频繁漂移 | 前后端联调困难 | 固定错误码和状态字段 |
| trace 不可回放 | 无法定位 bad case | 记录节点、工具和版本信息 |

## 工程取舍

### 取舍一：为什么先定审计字段，再扩展接口？

因为接口随时能加，但一旦没有稳定审计字段，后面很难补全历史行为。
Day 58 更像给后续集成定“骨架”。

### 取舍二：为什么权限审计要单独看？

因为金融场景里“有没有回答”只是结果层。
更重要的是中间有没有查过不该查的数据、有没有走到不该走的工具。

## 面试沉淀

Q133：为什么金融信贷 Agent 服务化必须先定 request_id 和审计结构？
重要程度：5/5

### 回答

因为 Agent 的问题定位、权限追溯、bad case 回放和版本回归都依赖稳定主键和结构化审计字段。
如果每次请求没有 request_id、trace_id 和统一状态，后续即使发现错误答案，也很难知道问题出在模型、工具还是数据链路。

Q134：为什么不能把完整用户问题和原始查询结果直接写进日志？
重要程度：5/5

### 回答

因为金融信贷问题里可能包含手机号、身份证号、客户名单、订单号和贷款明细。
日志系统通常比业务库更广泛地被访问和复制，所以必须采用字段白名单、脱敏和最小留存策略。

## 术语更新

- request_id：一次业务请求的唯一标识。
- trace_id：串联多个节点、工具和下游调用的链路标识。
- 审计回放：基于历史结构化记录复原一次请求执行过程的能力。
- 错误码契约：稳定表达拒答、降级、越权和工具失败等状态的编码规范。

### A. 今日核心问题

### 1. 为什么 Agent 不应该只停留在本地脚本？
  重要程度：5/5
  我的回答：本地脚本无法接入前端、调试平台或演示接口，
    没有 request_id 和 trace_id，bad case 无法定位和回放，
    服务化才能把工具路线、质量状态、权限决策固定为可复现的链路。

  回答评价：三点层层递进。第一点"无法接入前端/调试/演示"讲的是服务可达性——
    本地脚本只有开发者自己能跑，业务方、产品、前端都接不上。
    第二点"没有 request_id 和 trace_id，bad case 无法定位回放"讲的是可运维性——
    出了问题只能看终端输出，没有结构化链路可以复盘。
    第三点"服务化把工具路线、质量状态、权限决策固定为可复现链路"讲的是行为确定性——
    这是面试中能加分的表述，说明你理解的不是"包一层 HTTP"而是"把 Agent 的决策路径固化下来"。
    可以再补一句：金融合规要求自动化决策可审计，本地脚本做不到。

  评分：8/10

  参考答案要点：
    - 本地脚本无法接入前端、调试平台或演示接口，只能开发者自用；
    - 没有 request_id 和 trace_id，bad case 无法定位和回放；
    - 没有统一的错误码和状态，前后端联调困难；
    - 金融场景必须可审计，本地脚本没有审计存储能力；
    - 服务化才能把工具路线、质量状态、权限决策固定为可复现的链路。

### 2. request_id 和 trace_id 分别解决什么问题？
  重要程度：5/5
  我的回答：request_id是记录一次请求的，只能调用日志不包含上下文，具体操作，用于请求定位，
    trace_id是记录完整链路中每个请求的具体操作，包含上下文调用什么函数节点工具版本，用于请求完整回放 定位问题
    request_id + trace_id相互配合来调试 agent

  回答评价：抓住了核心区别——request_id 定位单次请求，trace_id 串联全链路上下文。
    提到"函数节点工具版本"是关键点，trace_id 不只是串联节点，还承载版本信息供回放。
    表述上"只能调用日志不包含上下文"不够准确，实际是 request_id 关注请求边界，
    trace_id 关注跨节点传播。

  评分：7/10

  参考答案要点：
    - request_id：一次业务请求的唯一标识，对外暴露给调用方，用于请求定位和状态查询；
    - trace_id：分布式链路追踪 ID，在 Agent、工具、数据源之间传播，用于串联所有节点；
    - 配合关系：一个 request_id 下可以有多个 span，trace_id 负责把它们串成完整调用链；
    - 生产价值：request_id 定位"哪个请求出了问题"，trace_id 定位"问题出在哪个节点"。
### 3. 查询审计、告警审计、任务审计和权限审计分别记录什么？
  重要程度：5/5
  我的回答：查询审计、用户请求，状态
    告警审计、告警是否正常，是否延迟，版本
    任务审计，证据，定时日报
    权限审计，查询敏感信息的操作，拒答

  回答评价：四个审计方向都对，区分了各自的关注重点。但每题只有一个关键词级别，
    缺少具体记录内容。面试时会被追问"查询审计除了用户请求和状态还记录什么"。
    需补充：查询审计还要记录工具路线和结果状态；任务审计要记录执行时间和重跑次数。

  评分：5/10

  参考答案要点：
    - 查询审计：request_id、user_id、问题原文、意图类型、工具路线、模型 ID、质量状态、延迟、最终状态；
    - 告警审计：告警触发时间、规则 ID 和版本、当前窗口数据、阈值、是否延迟、解释结果、是否误报；
    - 任务审计：任务类型（日报/评测/补偿）、执行时间窗口、成功/失败状态、重试次数、输出摘要；
    - 权限审计：用户角色、请求资源、敏感字段访问记录、是否有越权尝试、阻断原因、审批记录。
### 4. 为什么审计字段必须结构化，而不是只存自然语言日志？
  重要程度：5/5
  我的回答：必须统一主键，结构化字段request_id + trace_id，保证请求能够精准定位，和记录，
    路由依赖，可触发告警，方便审计追踪，回归测试可精准匹配

  回答评价：点到了路由、告警、审计追踪和回归匹配四个应用场景，方向很对。
    但缺少最关键的论据——SQL 过滤 vs 自然语言检索的对比。
    自然语言日志需要全文搜索才能找到"哪些请求被阻断了"，结构化字段一个 WHERE quality_status='blocked' 即可。
    面试时补上这个对比会让回答更有说服力。

  评分：7/10

  参考答案要点：
    - SQL 可直接过滤：结构化字段支持 WHERE、GROUP BY 精准检索，自然语言只能模糊搜索；
    - 路由可编程：编排层根据 status 枚举值做 if/else 分支，不能解析自然语言文本做判断；
    - 告警可自动化：监控系统直接统计 quality_blocked 比例，不需要人工读每条日志；
    - 回归可精确匹配：断言 quality_status == "quality_blocked"，自然语言需要 NLP 增加不确定性；
    - 成本：结构化存储 + 索引的查询成本远低于对大量自然语言文本做全文检索。
### 5. 为什么权限审计不能只看最终回答？
  重要程度：5/5
  我的回答：回答看似合理，不代表中间链路操作合规合法，要完整记录用户 crud，
    权限敏感等关键操作，并且记录证据

  回答评价：一句话抓住了核心——"回答看似合理不代表中间链路合规"。
    这是面试中的高价值论点。提到的"记录证据"也很关键。
    可以补充一个具体反例增强说服力：Agent 文字说"不能查"但实际已经执行了 SQL 查询，
    这种"文字拒答实际已查"只有审计中间链路才能发现。

  评分：7/10

  参考答案要点：
    - 回答层和工具层是两个层面：Agent 可以在回答里说"无法查询"，但工具层可能已经执行了敏感查询；
    - 需要记录：用户请求了什么资源、Agent 调用了哪个工具、工具是否实际查询了敏感表/字段；
    - 金融合规要求：即使最终没有返回敏感数据，中间访问行为本身也需要留痕；
    - 证据链：权限决策（为什么允许/拒绝）+ 工具执行记录 + 最终回答，三者构成完整证据。
### 6. 服务化接口至少应该固定哪些状态和错误码？
  重要程度：5/5
  我的回答：正确错误拒答

  回答评价：三个词抓住了最基础的三种结果状态（成功/失败/拒答），方向对但粒度过粗。
    面试场景只回答五个字基本等于没答。需要展开每种状态下的子类型，
    至少覆盖：成功（正常回答/降级回答）、失败（工具异常/超时/模型错误）、
    拒答（权限阻断/质量阻断/资料不足）。生产接口还需要定义对应的 error_code 枚举。

  评分：3/10

  参考答案要点：
    - 成功状态：ok（正常回答）、degraded（降级回答，数据有瑕疵但可参考）；
    - 阻断状态：blocked_permission（权限不足）、blocked_quality（数据质量不可用）、
      blocked_insufficient（资料不足无法回答）；
    - 异常状态：error_tool（工具执行失败）、error_timeout（超时）、
      error_model（模型调用异常）；
    - 每个状态对应一个稳定枚举值（如 "ok" / "degraded" / "blocked_xxx" / "error_xxx"），
      不能每次返回不同的自然语言描述。
### 7. 如何避免审计日志泄露金融敏感数据？
  重要程度：5/5
  我的回答：在权限层过滤敏感字段，返回上下文，输入输出都不可以记录敏感字段

  回答评价：方向正确——权限层过滤 + 输入输出不记录敏感字段。这是最小权限原则的体现。
    但缺少具体策略：哪些字段算敏感？怎么过滤？日志保留多久？
    面试时需补充字段白名单、脱敏算法和最小留存三个具体策略。

  评分：6/10

  参考答案要点：
    - 字段白名单：明确列出可以写入日志的字段，不在白名单的一律不记录；
    - 脱敏写入：必须记录的标识（如 user_id）使用脱敏 hash，不存明文手机号/身份证；
    - 原始内容不入日志：用户原始问题、查询结果在确认不含敏感字段前不直接写日志，
      或只记录脱敏后的摘要；
    - 最小留存：审计日志设保留期限，过期自动清理，降低泄露面；
    - 访问控制：审计日志本身也要有权限控制，不能所有人可查。
### B. 前三天核心回顾

### 8. [Day 57] 数据质量问题为什么要接入错误治理和回归测试？
  重要程度：4/5
  我的回答：数据质量高低会直接影响 agent回答准确率，甚至错误，
    空值、重复、乱序和补偿，会导致指标失真异常

  回答评价：抓住了直接影响关系和四类质量问题的例子。但 Day 57 的核心论点是
    "质量问题不是一次性故障，会重复出现"——当前回答缺少这个关键点。
    面试时需补上：只靠人工排查，下次空分区或重复写入发生时还会犯同样错，
    所以必须规则化 + 审计化 + 回归化。

  评分：6/10

  参考答案要点：
    - 数据质量问题会重复出现（空分区、补偿延迟不是偶发），不是一次性故障；
    - 只靠人工排查每次都要重新定位，效率低且容易遗漏；
    - 正确做法：质量异常 → 规则 → 审计字段 → 固定 bad case → 全量回归；
    - 每次代码或规则变更后自动跑回归，防止同类问题复发。
### 9. [Day 56] 端到端评测里为什么要检查工具路线？
  重要程度：4/5
  我的回答：因为路线本身就是口径、安全和治理的一部分。
    口径问题应该走 RAG，离线指标应该走 NL2SQL，实时状态应该走实时工具，敏感导出应该直接阻断。
    路线错了，即使回答像样，也不能算通过。

  回答评价：这是 10 题中最好的回答。三句话逻辑递进：先定性（路线=口径+安全+治理），
    再举例（四种场景对应四种路线），最后点出评测标准（路线错了不算通过）。
    "路线错了，即使回答像样，也不能算通过"——面试金句级别，可以直接用。

  评分：8/10

  参考答案要点：
    - 工具路线是 Agent 决策的核心，选错工具 = 回答来源不可信；
    - 四种典型错路：口径问题走 NL2SQL（方向错）、实时问题走离线（时效错）、
      敏感问题不阻断（安全漏过）、异常问题不走异常处理（容错失效）；
    - 路线检查让评测从"只看结果"升级到"看过程和来源"。
### 10. [Day 55] 日报 + 实时告警 Agent 通常要串哪些工具？
  重要程度：4/5
  我的回答：离线日报工具，实时告警工具，口径审核工具，日志审计工具

  回答评价：四个工具都列对了，覆盖了离线、实时、口径和审计四个维度。
    但只是一份清单，缺少每个工具的具体职责和编排关系。
    面试时如果被追问"这四个工具的执行顺序和依赖关系是什么"，
    需要能说出：先并行调用日报和告警工具获取数据，
    再调用口径工具校验指标口径一致性，最后统一写入审计日志。

  评分：6/10

  参考答案要点：
    - 离线日报工具：查询 ADS/DWS 层离线指标（申请量、通过率、放款金额、逾期率）；
    - 实时告警工具：查询近 1 小时实时窗口指标和告警状态，标注延迟和阈值；
    - 口径审核工具：校验日报指标和实时指标的口径是否一致，不一致时标注原因；
    - 日志审计工具：记录完整的 request_id、工具调用链、质量状态和最终摘要；
    - 编排顺序：日报 + 告警并行 → 口径审核（依赖前两者结果）→ 生成摘要 → 写入审计。
