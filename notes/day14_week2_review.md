———

我这个项目叫 SQL 解释助手 CLI，它的目标是把我的金融信贷开发经验迁移到 AI 应用方向。

这个项目的背景是：在真实的数据开发、数据问答或者 NL2SQL 场景里，SQL 不是生成出来就结束了。
系统还需要知道这段 SQL 在查什么、涉及哪些表和字段、有没有性能风险、缺不缺业务口径，以及能不能直接上线。

比如业务同学问一句：

上周每个渠道的授信通过率是多少？

未来 NL2SQL 系统可能会生成一段 SQL：

```sql
select
  channel,
  sum(approval_count) / nullif(sum(application_count), 0) as approval_rate
from dws_credit_application_daily
where dt between '2026-05-01' and '2026-05-07'
group by channel;
```

但生成 SQL 以后，还需要继续检查几个问题：

第一，这段 SQL 查的是不是正确的表。第二，字段含义是否清楚，比如 approval_rate 到底是不是授信通过率。第三，有没有分区过滤，比如 dt。
第四，有没有性能风险，比如 group by、order by、select *。第五，缺少上下文时能不能明确提示，而不是让模型瞎猜。

所以我先做了一个本地命令行版本，不依赖真实大模型。它现在的作用是：输入一段 SQL，输出结构化 JSON，里面包括 SQL 摘要、表名、字段、风险等级、
是否建议上线、 风险列表、优化建议和缺失上下文。

运行方式是：

```bash
python3 projects/day12_sql_explainer_cli/main.py --example
```

或者传入自定义 SQL：

```bash
python3 projects/day12_sql_explainer_cli/main.py --sql "select * from \
dwd_credit_application_detail limit 10"
```

这个项目目前主要做了三件事。

第一，提取 SQL 的基本信息。

它会从 SQL 里提取涉及的表，比如：

```json
{"tables": ["dws_credit_application_daily"]}
```

也会提取 select 字段，比如：

```json
{"fields": ["channel",
"sum(approval_count) / nullif(sum(application_count),
0) as approval_rate"]}
```

这些信息后续可以和表结构、字段口径、元数据系统结合起来。

第二，识别常见数仓风险。

目前内置了几类规则：

- `select *`：可能读取不必要字段；
- 缺少 `where`：可能全表扫描；
- 缺少 `dt`：可能缺少分区过滤；
- `group by`：可能有聚合开销；
- `order by`：可能有全局排序成本；
- `limit` 但无过滤：不代表扫描成本低。

比如输入：

```sql
select * from dwd_credit_application_detail limit 10
```

它会识别出高风险，因为这里既用了 select *，又没有 where，也没有 dt 分区条件。
虽然有 limit 10，但在大数据场景下，limit 不一定能降低扫描成本， 因为底层可能还是要先扫大量授信申请明细。

第三，输出稳定 JSON。

这是我前几天学的结构化输出能力。项目不会只返回一段自然语言，而是返回固定字段，例如：

```json
{
  "summary": "这段 SQL 用于查询授信申请数据，并可能包含过滤、聚合、排序或限制返回条数。",
  "tables": ["dws_credit_application_daily"],
  "fields": ["channel", "approval_rate"],
  "risk_level": "medium",
  "can_publish": false,
  "risks": [],
  "suggestions": [],
  "missing_context": ["approval_rate 授信通过率业务口径"]
}
```

这样做的好处是，后续系统可以根据字段继续处理。

比如：

- `risk_level = high` -> 拦截上线；
- `risk_level = medium` -> 进入人工复核；
- `missing_context` 不为空 -> 提示补充表结构或指标口径；
- `can_publish = false` -> 不建议直接上线。

这就不是简单的“解释 SQL”，而是能接入真实审批、审核、数据开发流程的结构化结果。

这个项目也串起了我第 2 周学的几个核心概念。

Day 9 学的是 Prompt 思路。也就是不要随便问模型，而是要明确角色、任务、上下文、约束和输出格式。
放到这个项目里，就是 SQL 解释助手要明确：不能编造表结构，缺少字段口径要提示，输出必须稳定。

Day 10 学的是结构化输出。也就是模型或工具不能只返回自然语言，而要返回系统能处理的 JSON。
这个项目里的 risk_level、can_publish、risks、missing_context 都是为了让系统能继续判断流程。

Day 11 学的是 Tool Use。也就是模型不应该什么都自己猜，确定性的事情交给工具。
这个 SQL 解释助手目前就是一个工具层：它用规则检查 SQL 风险，后续可以被 LLM 调用。

所以这个项目的定位是：

> 它不是最终完整的 AI 产品，而是 RAG / NL2SQL / 数据问答系统里的一个基础模块。

后续我会从三个方向增强它。

第一，接入真实 LLM。现在的解释比较规则化，后续可以让大模型基于结构化结果生成更自然、更适合业务人员看的解释。

第二，接入表结构和指标口径。比如 active_users 不能只靠字段名猜，它应该去查指标口径文档，确认活跃用户到底怎么定义。这个部分可以和 RAG 结合。

第三，增强 SQL 解析准确性。现在用的是简单正则，适合学习和 Demo。
真实项目里可以引入 SQL 解析库，支持更复杂的 join、子查询、CTE 和不同 SQL 方言。

总结一下：

> 我做这个 SQL 解释助手，是为了把数据仓库里的 SQL 经验，转换成 AI 应用里的工具能力。
它现在能输入 SQL、识别表字段和常见风险、输出稳定 JSON。后续可以接入 LLM 和 RAG，变成 NL2SQL 系
> 统里的 SQL 校验、解释和风险控制模块。

## 工程取舍

这一天优先采用最小可运行方案，是为了先建立稳定基线。真实项目里不能一开始就追求复杂架构，应该先把输入、处理逻辑、输出和失败场景跑通，再逐步增加模型调用、 检索、
数据库、权限和评测。

对“第 2 周复盘与项目讲解”这类能力，学习阶段更看重可解释和可复盘；生产阶段再根据准确率、延迟、成本、安全和维护成本决定是否引入更复杂的方案。
