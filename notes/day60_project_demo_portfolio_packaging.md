## 生产实际

一个有效的演示顺序通常是：

1. 业务问题：为什么金融信贷场景同时需要离线仓库和实时仓库。
2. 系统架构：RAG、NL2SQL、实时工具、血缘和审计各自负责什么。
3. 成功路径：给一个正常查询或日报摘要样例。
4. 风险路径：给一个延迟、越权或口径冲突样例。
5. 工程价值：为什么这不是普通聊天机器人，而是有数据边界和审计能力的 Agent。

## 常见坑

| 类型 | 可能的问题 | 生产处理方式 |
|------|------------|--------------|
| 只讲功能点 | 听众抓不住主线 | 先讲业务问题和架构边界 |
| 演示过长 | 关键价值被稀释 | 控制在 3 分钟核心链路内 |
| 只展示成功样例 | 显得像普通 Demo | 补一条 bad case 证明治理能力 |
| README 太碎 | 难以形成作品集 | 统一目录和入口说明 |
| 截图无上下文 | 面试官看不懂 | 每张图配一句业务含义 |

## 工程取舍

### 取舍一：为什么要展示 bad case？

因为金融信贷 Agent 的价值不只是回答问题，还包括知道什么时候不能回答、什么时候要降级、什么时候要阻断。

### 取舍二：为什么演示脚本不能按代码目录讲？

因为代码目录是实现视角，演示应该是业务视角。
先讲用户问题和系统响应，再回头解释模块分工。

## 面试沉淀

Q137：为什么金融信贷 Agent 项目演示必须同时展示成功路径和风险路径？
重要程度：5/5

### 回答

因为成功路径只能说明系统能工作，风险路径才能说明系统在错误、越权、延迟和口径冲突场景下仍然可控。
这类治理能力比单纯答对一个问题更接近生产价值。

Q138：3 分钟项目演示最应该讲什么？
重要程度：5/5

### 回答

最应该讲业务问题、系统边界、核心链路和工程取舍。
不要花大量时间介绍学习过程或细枝末节代码，而要讲清楚为什么需要离线/实时仓库、为什么要用 RAG 和 NL2SQL 分工、为什么要做审计和安全阻断。
