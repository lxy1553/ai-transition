## 生产实际

一个有效的演示顺序通常是：

1. 业务问题：为什么金融信贷场景同时需要离线仓库和实时仓库。
2. 系统架构：RAG、NL2SQL、实时工具、血缘和审计各自负责什么。
3. 成功路径：给一个正常查询或日报摘要样例。
4. 风险路径：给一个延迟、越权或口径冲突样例。
5. 工程价值：为什么这不是普通聊天机器人，而是有数据边界和审计能力的 Agent。

## 常见坑

| 类型 | 可能的问题 | 生产处理方式 |
|------|------------|--------------|
| 只讲功能点 | 听众抓不住主线 | 先讲业务问题和架构边界 |
| 演示过长 | 关键价值被稀释 | 控制在 3 分钟核心链路内 |
| 只展示成功样例 | 显得像普通 Demo | 补一条 bad case 证明治理能力 |
| README 太碎 | 难以形成作品集 | 统一目录和入口说明 |
| 截图无上下文 | 面试官看不懂 | 每张图配一句业务含义 |

### B. 前三天核心回顾

### 8. [Day 59] 为什么综合项目必须提供统一入口？
  重要程度：4/5
  我的回答：
统一整合方便开发，也方便各个部分相互协调追踪，审计，提高效率，比如统一的 chunk id ，统一的 request id ，trace id 等，
统一入口负责把这些能力按"意图识别→
  权限校验→工具路由→工具执行→结果校验→回答生成→
  审计落库"的流程编排串联起来。

  评分：7/10。提到了统一 ID 和主流程，细节到位。但"方便开发、提高
  效率"是结果，不是根本原因。根本原因是：没有统一入口，多个脚本就是
  零散能力清单，面试官无法理解它们如何协同工作。

  参考答案：
  因为面试和生产都看重系统边界，而不是零散能力清单。多个独立脚本的
  问题在于：无统一入口和复用，配置散落难以维护，工具重复实现导致审计
  和错误码不一致，用户（或面试官）也无法直观理解这些能力如何协同工作。
  统一入口意味着意图识别、权限校验、工具路由、结果校验和审计逻辑被
  真正整合成一个主流程，这才是"一个系统"而不是"一堆脚本拼成的故事"。
### 9. [Day 58] 为什么日志里不能直接写完整原始查询结果？
  重要程度：4/5
  我的回答：
权限和边界问题，结果很可能包含核心业务数据或者敏感字段，日志容易暴露，对结果要进行审查，白名单，最小留存，访问控制和脱敏

  评分：7/10。理解正确，列出了风险点和防范措施。建议补充一个关键点：
  日志系统的访问范围通常比业务库更广，泄露风险更大。

  参考答案：
  因为金融信贷问题里可能包含手机号、身份证号、客户名单、订单号和贷款
  明细。日志系统通常比业务库更广泛地被访问和复制，如果明文写入这些
  敏感数据，泄露面会被极大放大。生产里必须采用多层防护：字段白名单
  （明确列出可以写入日志的字段）、脱敏写入（user_id 使用 hash）、
  原始内容不入日志、最小留存（设保留期限）和访问控制（审计日志本身
  也要有权限控制）。
### 10. [Day 57] 数据质量问题为什么会直接影响 Agent 回答质量？
  重要程度：4/5
  我的回答：
Agent 回答质量直接来源于数据质量，在对数据进行切片同统一 chunk id，进行 bm25监所和向量检索后采用 rrf 排序，最后 rerank 取 topk，
每一步都会影响数据质量

  评分：4/10。偏题。题目问的是"数据质量"（空分区、实时延迟、补偿失败、
  重复写入），但你回答成了"检索质量"（chunk、BM25、RRF、rerank）。
  两者是不同的概念。数据质量出问题，再好的检索链路也无法挽回。

  参考答案：
  Agent 的回答直接依赖数据质量，具体来说：离线分区为空或未产出时，
  Agent 查不到数据可能误报为"指标为零"；实时窗口因 Kafka 积压数据
  不完整时，Agent 给出的指标偏低；补偿任务未完成时，逾期率等关键指标
  在短时间内是偏的；上游事件重复写入会导致实时指标虚高。所以 Agent
  必须在回答前检查数据质量状态，质量 ok 才正常回答，有风险就降级说明，
  严重异常就阻断并给排查建议。
