## 大白话解释

指标字典可以理解为“公司统一承认的指标说明书”。

业务同学问：

```text
授信通过率怎么算？
M1 逾期率是不是所有逾期都算？
实时拒绝率窗口是多少？
放款金额按申请时间还是放款成功时间？
```

这些问题不是让 Agent 去查数据库，而是让 Agent 查口径资料。
因为用户问的是“这个指标是什么意思”，不是“这个指标当前值是多少”。

如果 Agent 对口径问题直接生成 SQL，就容易出现两个问题：

- SQL 能跑，但口径解释可能是错的；
- 模型可能按常识编一个定义，和公司指标平台不一致。

所以 Day 50 的重点是：先把指标口径写成结构化字典，再让 RAG 基于字典回答。

## 生产实际

金融信贷公司里的指标口径通常来自指标平台、数据地图、数仓文档、风控规则文档或报表说明。
Agent 要回答口径问题，至少要知道：

| 字段 | 为什么重要 | 信贷例子 |
|------|------------|----------|
| 指标名 | 确认用户问的是哪个指标 | 授信通过率、M1 逾期率 |
| 主题域 | 防止同名指标混淆 | 风控拒绝率和放款失败率不是一回事 |
| 分子 | 说明统计对象 | approved_cnt、reject_event_cnt |
| 分母 | 说明比例口径 | apply_cnt、risk_decision_event_cnt |
| 时间口径 | 决定按哪个时间统计 | 申请时间、放款成功时间、还款到期日 |
| 窗口 | 实时指标必须写清 | 近 5 分钟、近 10 分钟 |
| 来源表/事件 | 支撑可追溯 | DWS 表、ADS 表、实时事件 |
| 适用粒度 | 防止乱按维度拆 | dt + channel + product |
| 权限等级 | 控制谁能看 | internal、confidential |
| 使用限制 | 防止误用 | M1 不等于所有逾期 |

生产里，指标字典一般不是给人看的静态表，而是 Agent、NL2SQL、RAG 和评测系统共同使用的语义层。

## Day 50 指标字典设计

今天先建立 6 个指标：

| 指标 | 类型 | 主题域 | 重点口径 |
|------|------|--------|----------|
| 授信申请量 | 离线 | 授信申请 | 计数型，按 dt 统计 |
| 授信通过率 | 离线 | 授信申请 | approved_cnt / apply_cnt |
| 放款金额 | 离线 | 放款 | 按放款成功日期统计 |
| M1 逾期率 | 离线 | 逾期贷后 | overdue_days >= 30 / due_loan_cnt |
| 实时风控拒绝率 | 实时 | 风控决策 | 10 分钟滚动窗口，延迟阈值 3 分钟 |
| 实时还款失败数 | 实时 | 还款 | 5 分钟滚动窗口，延迟阈值 2 分钟 |

离线指标和实时指标最大的区别：

- 离线指标强调分区日期、来源表、日批口径和历史稳定性；
- 实时指标强调窗口、事件时间、处理时间、延迟阈值和告警解释。

如果 Agent 把实时指标当离线指标解释，可能会忽略延迟和窗口。
如果 Agent 把离线指标当实时指标解释，可能会把昨天日报误说成当前状态。

## 常见坑

| 类型 | 可能的问题 | 生产处理方式 |
|------|------------|--------------|
| 口径缺失 | 只写指标名，不写分子分母 | 指标字典必须结构化 |
| 时间混淆 | 放款金额按申请时间统计 | 写清时间字段和业务含义 |
| 实时窗口缺失 | 实时拒绝率没写窗口 | 写清窗口、延迟阈值和事件时间 |
| 口径问题误走 SQL | 用户问“怎么算”却查表 | 意图识别为 metric_definition，走 RAG |
| 无引用回答 | 模型凭常识解释指标 | 必须返回 citation |
| 敏感明细混入 | 口径问答返回客户名单 | 敏感导出直接阻断 |

## 工程取舍

### 取舍一：为什么不用 SQL 解释口径？

SQL 可以查指标值，但不一定能解释口径。
例如 `approved_cnt / apply_cnt` 能算通过率，但用户还需要知道分母是否包含拒绝、待处理、撤销申请，
时间按申请时间还是审批时间，来源表是 DWS 还是 ADS。

这些信息通常不在 SQL 结果里，而在指标字典和口径文档里。

### 取舍二：为什么 RAG 回答必须有 citation？

口径解释属于高风险回答。
如果模型编错一个指标定义，业务可能按错误口径做分析、报表或决策。

返回 citation 能让用户知道答案来自哪个指标字典 chunk，也方便后续审计和 bad case 修复。

### 取舍三：为什么实时指标字典要比离线指标多写窗口和延迟？

实时指标的“当前值”依赖窗口和链路状态。
近 5 分钟、近 10 分钟和近 1 小时得到的结论可能完全不同。
如果实时链路延迟超过阈值，Agent 不能说“当前正常”，只能说明数据不可用或延迟。

## 面试沉淀

Q110：金融信贷 Agent ，指标字典应该包含哪些字段？
重要程度：5/5

### 回答

金融信贷 Agent 需要指标字典，是因为数据问答不能只知道表名和字段名，还必须知道指标怎么算、按什么时间统计、能按哪些维度拆分、谁有权限查看。
如果没有指标字典，Agent 很容易混淆授信通过率、放款成功率、风控拒绝率、M1 逾期率这类指标，
生成的 SQL 可能能跑，但业务口径是错的。

一个生产级指标字典至少应该包含指标 ID、指标名称、主题域、指标类型、定义、分子、分母、时间口径、统计粒度、来源表或来源事件、
可用维度、权限等级、owner、适用范围和使用限制。
如果是实时指标，还要额外写清窗口大小、事件时间、处理时间、延迟阈值和告警规则。

在 Agent 链路里，用户问“这个指标怎么算”时，应该走指标字典或 RAG 口径问答；
用户问“这个指标当前是多少”时，才进入离线 SQL 或实时指标工具。
这样才能把口径解释、指标查询和敏感明细访问分开控制，保证回答可追溯、可审计、可回归。

Q111：为什么口径解释问题应该走 RAG，而不是直接生成 SQL？
重要程度：5/5

### 回答

口径解释问题问的是“指标是什么意思、分子分母是什么、窗口和限制是什么”，不是问“当前指标值是多少”。
SQL 只能查数据，不能完整表达业务定义、排除条件、时间字段、适用范围和版本说明。
如果用户问“授信通过率怎么算”，Agent 直接生成 SQL，可能只返回一个数值，无法解释分母是否包含拒绝和待处理申请。

更稳的做法是把指标字典、口径文档和实时指标说明作为 RAG 知识库。
Agent 先识别意图为 `metric_definition`，再检索对应指标 chunk，返回定义、分子、分母、时间口径、来源表、窗口和使用限制，并附带 citation。
如果没有可靠引用，就应该返回资料不足，而不是让模型凭常识编造。

在金融信贷场景里，口径错误会直接影响授信、风控、放款、逾期和催收分析。
所以口径解释必须基于可信资料，SQL 查询必须基于口径约束，两者不能混在一起。

## 术语更新

今天新增或强化这些术语：

- 指标字典：集中记录指标定义、分子分母、时间口径、来源和权限的清单。
- 指标口径：一个指标到底怎么算、按什么时间和范围统计。
- 实时指标口径：包含窗口、事件时间、处理时间和延迟阈值的实时指标定义。
- 口径问答：用户询问指标定义、公式、分子分母、窗口和限制的问答。
- 指标 Citation：口径回答引用的指标字典来源。

  我的回答：
 ads 和 dws 都经过汇总，分类指标，查询难度低成本低，准确率高，口径一致，ADS 适合总览，DWS 适合维度趋势。

  回答评价：
  评分：8/10
  回答正确，已经说到了汇总、成本、口径和 ADS/DWS 的分工。
  可以再补充：优先 ADS/DWS 还能减少访问 DWD/ODS 敏感明细和原始表的风险。

  参考答案：
  Agent 查询离线指标时优先选择 ADS 或 DWS，是因为这两层更接近业务指标。
  ADS 通常服务日报、看板和经营总览；DWS 通常服务主题汇总、维度趋势和对比分析。
  优先使用 ADS/DWS 可以降低 SQL 复杂度和扫描成本，减少口径不一致，避免直接访问 DWD/ODS 明细里的敏感字段。
  只有授权明细查询或上游排查时，才考虑进入 DWD；ODS 默认不作为普通数据问答入口。
