# Day 20 - 项目 2 接口化：RAG 问答 API

## 今日目标

今天把前几天的 RAG 检索能力封装成 API。

目标不是只让脚本能跑，而是让外部系统可以通过 HTTP 调用 RAG 能力：

- 输入用户问题
- 返回答案草稿
- 返回引用来源
- 返回必要的排查信息
- 对错误输入给出稳定响应

今天产出：

- 一份 RAG API 设计说明
- 一个接口请求和响应字段清单
- 5 个真实问题回归测试样例

---

## 大白话解释

脚本是自己在终端里跑，API 是让别人也能调用。

RAG API 就像给知识库问答系统开一个服务窗口。业务系统、前端页面、内部工具都不需要知道
底层怎么切 chunk、怎么检索、怎么拼 prompt，只要按约定把问题传进来，就能拿到答案和引用。

---

## 生产实际

真实公司里 RAG 很少只以脚本形式存在。它通常会被包装成服务，接到：

- 内部知识库问答页面
- 数据平台指标问答入口
- BI 或报表平台助手
- 客服系统和工单系统
- 企业微信、飞书、钉钉机器人

一个可用的 RAG API 至少要考虑这些字段：

| 类型 | 字段 | 作用 |
|------|------|------|
| 请求 | `question` | 用户原始问题 |
| 请求 | `top_k` | 控制召回数量 |
| 请求 | `user_id` | 记录和权限判断 |
| 请求 | `business_domain` | 限定业务域 |
| 响应 | `answer` | 最终回答 |
| 响应 | `citations` | 引用来源 |
| 响应 | `request_id` | 排查链路 |
| 响应 | `confidence` | 粗略可信度 |
| 响应 | `cannot_answer_reason` | 无法回答原因 |

生产里的关键点是：API 返回的不只是答案，还要返回能排查问题、做审计和持续优化的信息。

---

## 常见坑

| 问题 | 影响 | 处理方式 |
|------|------|----------|
| 只返回答案 | 无法核对依据 | 必须返回 citations |
| 请求字段随意 | 前后端难对齐 | 用 Pydantic 或 schema 固定字段 |
| 错误直接抛栈 | 暴露内部实现 | 统一错误码和错误消息 |
| 没有 request_id | 线上问题难追踪 | 每次请求生成唯一 ID |
| 不记录召回结果 | 无法优化效果 | 保存 query、top-k、引用和耗时 |

---

## 工程取舍

接口化时先做最小闭环，不急着上复杂权限和多模型调度。

第一版重点是：

```text
POST /rag/ask
输入 question
输出 answer + citations + request_id
```

等接口稳定后，再逐步加入用户身份、权限过滤、缓存、query rewrite、rerank 和模型分层。
这样每一层能力都能单独测试，问题也更容易定位。

## 面试沉淀


Q030：RAG 问答 API 在生产环境里应该怎么设计？

### 回答

RAG API 不能只设计成一个返回字符串的接口。生产里至少要包括请求校验、用户身份、
权限上下文、问题文本、召回参数、答案、引用来源、request_id、错误码和日志记录。
最小接口可以是 `POST /rag/ask`，请求里包含 `question`、`user_id`、`top_k`、
`business_domain` 等字段，响应里包含 `answer`、`citations`、`cannot_answer_reason`
和 `request_id`。引用来源要能追溯到文档、chunk、位置和版本，方便用户核对，也方便研发排查。
工程上还要有 timeout、异常处理、统一响应格式和回归测试。这样 RAG 才能从本地脚本变成可接入业务系统的服务。

### 1. RAG 知识入库怎么设计？

我的回答：
设计时候至少包含请求和响应俩个大模块，
请求包含请求验证，用户身份，问题上下文，问题，请求 id
返回包含答案，引用来源，
另外生产中还要包含日志记录，timeout，错误码

参考答案：
RAG 知识入库是离线链路，不是 API 请求响应设计。生产里通常先接入文档，
比如 Markdown、PDF、指标口径、数据字典、SQL 规范等；然后做解析、清洗、去重、
脱敏和版本判断；再按标题、段落、业务对象或语义边界切成 chunk；之后用同一个
embedding 模型把 chunk 转成向量，并把原文、向量、source_path、doc_id、chunk_id、
position、业务域、权限标签、版本、状态等 metadata 写入索引。这样在线问答时才能
快速检索、按权限过滤、返回引用来源，并能排查资料是否过期或切分是否有问题。

### 2. RAG 为什么要返回引用来源？

我的回答：
方便后续问题的追踪排查，维护和审计，

参考答案：
RAG 返回引用来源是为了让答案可核对、可追溯、可排查。企业场景里用户不能只相信
一个自然语言答案，还要知道它来自哪份文档、哪个 chunk、哪个位置和哪个版本。
如果答案错了，研发可以根据 citations 判断是资料本身错误、检索召回错误、rerank
排序错误，还是模型生成错误。引用来源也方便做审计和持续优化，是生产级 RAG 和普通
聊天机器人的关键区别。

### 3. SQL 解释助手能检查哪些风险？

我的回答：
sql 语法错误，sql 优化，比如没有 加where，使用 order by进行全局排序， 没有 dt 的限制 等导致资源token的消耗

参考答案：
SQL 解释助手可以检查语法和结构风险，也可以检查生产执行风险。常见风险包括：
没有 `where` 条件、没有 `dt` 或分区限制、`select *` 读取过多字段、全局
`order by` 成本高、缺少 `limit`、大表 join 条件不清晰、字段或表名不明确、
可能扫描过大数据量、以及存在 `delete`、`update`、`drop` 等危险语句。
这里关注的是数据库扫描资源、查询成本和数据安全风险，不是 token 消耗。

### 4. query rewrite 解决什么问题？

我的回答：
核心作用是把用户提出的问题进行重写，变成更方便理解、向量检索和调用的 query。
具体方法包括问题拆分、基于上下文的补全、同义词替换。
从而提高召回率，当然也要控制改写偏移，防止曲解用户提问的意思

参考答案：
Query rewrite 解决的是用户原始问题不适合直接检索的问题。用户可能表达很口语、
省略上下文、使用同义词、指代不清，或者一个问题里混着多个意图。rewrite 可以把
问题改写成更适合向量检索和关键词检索的表达，比如补全业务对象、替换同义词、
拆分多意图、补充表名字段名或指标名。它能提高召回率，但生产里必须控制改写偏移，
不能把用户原意改错，所以要保留原始 query、记录 rewritten query，并通过评测集观察效果。

### 5. RAG 召回质量差怎么排查？

我的回答：
查看错误码，引用上下文，引用来源，检查 topk，k是否太大照成内容稀释，太小照成引用被截取

参考答案：
排查召回质量要先看测试问题、期望来源和实际 top-k 结果。第一步看 expected source
有没有进入 top-k，如果没有进入，说明召回失败；如果进入但排序靠后，说明排序或
rerank 有问题；如果进入且排序靠前但答案仍错，可能是 prompt 或生成问题。然后继续
排查 query 是否表达不清、是否需要 query rewrite，chunk 是否切得太碎或太大，
embedding 词表或模型是否覆盖关键语义，关键词、表名、字段名是否能精确匹配，
metadata 过滤是否误过滤，top-k 是否过小漏召回或过大带入噪声。生产里要把 bad case
保存下来，用固定评测集反复验证 chunk、top-k、混合检索、rerank 和改写策略是否变好。

### 6. RAG 问答 API 在生产环境里应该怎么设计？

我的回答：
api 的设计第一性原则原则是保证外部能够稳定的调用接口，
基于这个原则所以不能简单的的设计成返回字符串，和答案，还要有错误信息方便后续的维护和审计，问题追踪
设计时候至少包含请求和响应俩个大模块，
请求包含请求验证，用户身份，问题上下文，问题，请求 id
返回包含答案，引用来源，
另外生产中还要包含日志记录，timeout，错误码

参考答案：
RAG 问答 API 不能只设计成一个返回字符串的接口。生产里最小可以是
`POST /rag/ask`，请求包含 `question`、`top_k`、`user_id`、`business_domain`
等字段，用于问题输入、召回数量控制、权限过滤和审计。响应至少包含 `answer`、
`citations`、`request_id`、`confidence`、`cannot_answer_reason` 和耗时等字段。
`citations` 要能追溯到文档、chunk、位置和版本，`request_id` 用来串起日志和用户反馈。
工程上还要有请求校验、统一错误码、timeout、异常处理、日志记录、回归测试和必要的
权限控制。这样 RAG 才能从本地脚本变成可接入前端、机器人或业务系统的稳定服务。
