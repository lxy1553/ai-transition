## 大白话解释

Agent 不能看到一个问题就直接选工具。
它要先问两件事：

```text
这个问题属于哪个业务域？
这个问题到底想做什么？
```

比如：

```text
近 7 天各渠道授信通过率趋势。
```

主题域是“授信申请”，意图是“离线指标查询”，应该走 ADS/DWS + NL2SQL。

```text
近 5 分钟 STR_BLACKLIST 策略拒绝率是否异常？
```

主题域是“风控决策”，意图是“实时状态查询”，应该走实时指标工具和延迟检查。

```text
审批通过率的分子和分母分别是什么？
```

主题域还是“授信申请”，但意图不是查表，而是“指标口径解释”，应该走 RAG 口径问答。

```text
导出所有逾期客户的手机号和身份证号。
```

主题域是“逾期贷后”，意图是“敏感导出”，应该安全阻断和审计。

## 生产实际

金融信贷主题域可以先按业务链路拆：

| 主题域 | 典型问题 | 常见工具路线 |
|--------|----------|--------------|
| 授信申请 | 申请量、通过率、渠道趋势 | 离线指标 / 实时申请量 / 口径问答 |
| 额度管理 | 额度审批、额度使用率、冻结额度 | 离线指标 / 明细权限查询 |
| 风控决策 | 策略命中、拒绝率、风险等级 | 离线指标 / 实时指标 / 告警解释 |
| 放款 | 放款金额、放款成功率、放款失败 | 离线指标 / 实时告警 |
| 还款 | 还款成功率、失败原因、渠道异常 | 离线指标 / 实时失败数 |
| 逾期贷后 | 逾期率、M1/M2、迁徙率 | 离线指标 / 血缘追溯 / 敏感阻断 |
| 催收 | 触达率、承诺还款率、催回金额 | 离线指标 / 明细权限查询 |

Agent 识别主题域后，才能知道要查哪些指标、表、实时事件、口径资料和权限标签。
Agent 识别意图后，才能决定走哪条工具路线。

## 意图分类标准

Day 49 先把用户问题分成 6 类：

| 意图 | 典型问题 | 工具路线 |
|------|----------|----------|
| 离线指标 | 昨天、近 7 天、上月、趋势、按渠道/产品 | 离线分层路由 + SQL |
| 实时状态 | 近 5 分钟、当前、实时、是否异常 | 实时路由 + 实时指标工具 |
| 口径解释 | 指标怎么算、分子分母、定义 | RAG 口径问答 |
| 血缘追溯 | 来自哪些表、上游、下游 | 血缘工具 |
| 敏感导出 | 手机号、身份证、客户名单、全部明细 | 安全阻断 |
| 缺参澄清 | 看一下是否异常、没有窗口/时间/维度 | 澄清 |

结构化输出应该长这样：

```json
{
  "domain": "风控决策",
  "intent": "realtime_metric",
  "required_context": ["window", "strategy_id", "product_code"],
  "route": ["realtime_router", "realtime_metric_tool", "delay_checker"],
  "risk_level": "medium",
  "fallback": "clarification"
}
```

这个 JSON 不是为了好看，而是为了让后续工具路线可控。

## 工程取舍

### 取舍一：为什么先识别主题域？

同一个词在不同主题域里含义可能不同。
“失败率”在放款域可能是放款失败率，在还款域可能是扣款失败率，在风控域可能是策略拒绝率。
先识别主题域，可以减少选错指标和选错表的概率。

### 取舍二：为什么再识别意图？

同一个主题域的问题可能走完全不同的工具。
授信申请既可能问离线通过率，也可能问近 5 分钟申请量，还可能问通过率口径或上游表来源。
如果只识别主题域，不识别意图，Agent 仍然会选错工具。

### 取舍三：为什么要结构化输出？

结构化输出可以让后续系统检查字段。
例如没有 `window` 就不能查实时指标，没有 `time_range` 就不能查离线指标，`risk_level=high` 就要进权限或安全阻断。
这比让模型自然语言说“我觉得应该查实时指标”更可控。

## 面试沉淀

Q108：金融信贷 Agent 为什么要先做主题域识别，再做意图识别？
重要程度：5/5

### 回答

金融信贷 Agent 需要先做主题域识别，是因为信贷业务链路长，授信、额度、风控、放款、还款、逾期、催收对应的指标、
表、实时事件和权限边界都不同。
同样是“失败率”，在放款域可能指放款失败率，在还款域可能指扣款失败率，在风控域可能指策略拒绝率。
如果不先识别主题域，Agent 很容易选错指标或表。

识别主题域之后，还必须识别意图。
同一个主题域的问题可能是离线指标查询、实时状态查询、口径解释、血缘追溯、敏感导出或缺参澄清。
例如授信申请域里，“近 7 天各渠道通过率”走离线指标；
“近 5 分钟申请量是否异常”走实时指标；
“通过率怎么算”走 RAG 口径问答；
“通过率来自哪些表”走血缘工具。

所以生产级 Agent 的路由应该是：先识别主题域，再识别意图，再选择工具路线，并做权限、前置条件和审计。
这样才能避免把所有问题都粗暴交给 SQL 或模型自由选择工具。

Q109：如何区分离线指标、实时状态、口径解释、血缘追溯和敏感导出问题？
重要程度：5/5

### 回答

可以先看用户问题里的触发信号。
出现昨天、近 7 天、上月、趋势、按渠道、按产品，通常是离线指标；
出现近 5 分钟、当前、实时、是否异常、突增，通常是实时状态；
出现怎么算、定义、口径、分子、分母，通常是口径解释；
出现来自哪些表、上游、下游、血缘，通常是血缘追溯；
出现导出、手机号、身份证、客户名单、全部明细，通常是敏感导出。

不同意图的处理路线完全不同。
离线指标走 ADS/DWS、Schema Catalog、SQL 生成、SQL 校验和离线执行；
实时状态走实时指标工具、延迟检查和告警工具；
口径解释走 RAG 并要求引用；
血缘追溯走 lineage_tool；
敏感导出走权限校验和安全阻断。

如果缺少必要上下文，例如“看一下是否异常”没有窗口、业务线或指标名，就应该先澄清。
意图分类的目标不是让模型显得聪明，而是让后续工具调用可控、可审计、可回归。

### 8. [Day 48] 实时仓库和离线仓库在 Agent 数据问答里的最大区别是什么？
  重要程度：4/5
