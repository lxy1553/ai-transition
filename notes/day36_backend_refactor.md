## 大白话解释

Day 35 的项目像一个“演示脚本”，适合命令行跑通。
Day 36 要把它变成“后端服务”，让其他系统可以通过 HTTP 调用。

服务化不是把脚本套一层 FastAPI 就结束。
更重要的是把职责拆清楚：

- `main.py` 负责接口入口；
- `schemas.py` 负责请求和响应格式；
- `services.py` 负责业务编排；
- `storage.py` 负责审计存储；
- `config.py` 负责配置；
- `errors.py` 负责统一错误处理。

这样后续改 SQL 生成、换数据库、加权限或接前端时，不会牵一发动全身。

---

## 生产实际

在金融信贷公司里，NL2SQL 助手不会长期以脚本形式存在。
业务系统、数据门户、运营后台或机器人都需要通过 API 调用它。

生产后端通常要回答这些问题：

- 请求进来后由哪个接口处理？
- 参数在哪里校验？
- 业务逻辑在哪里编排？
- 查询和审计记录写到哪里？
- 错误怎么统一返回？
- 日志和 request_id 怎么串起来？

Day 36 的重构重点是建立服务边界，而不是追求复杂架构。
本地项目先把 Day 35 的静态演示 JSON 封装成 API，
后续再逐步替换为真实 parser、generator、validator、executor 和 interpreter。

## 面试沉淀

Q086：为什么 AI 应用要从脚本重构成后端服务？

### 回答

AI 应用从脚本重构成后端服务，是为了让能力能被前端、业务系统、机器人或其他服务稳定调用。
脚本适合本地验证，但不适合权限控制、接口复用、审计追踪、错误处理和多用户访问。
服务化后，需要把 API 层、service 层、storage 层、config 层和 errors 层拆清楚。
这样后续替换模型、数据库、SQL 生成器或权限系统时，不会影响接口契约。
在 NL2SQL 场景里，服务化还能统一处理 request_id、审计记录、安全阻断和响应格式。

### 1. 为什么 NL2SQL 演示脚本要重构成后端服务？
  我的回答：
变成可调用接口的后端服务

回答评价：
通过。回答抓住了核心：脚本只能本地运行，服务化后才能变成可被前端、机器人或其他业务系统调用的接口能力。
可以再补充权限、审计、错误处理和多用户访问这些生产要求。

参考答案：
NL2SQL 演示脚本要重构成后端服务，是因为脚本适合本地验证和面试演示，
但不适合长期被业务系统调用。
服务化后，前端、数据门户、运营后台或企业 IM 可以通过 HTTP API 调用 NL2SQL 能力。
同时，后端服务可以统一处理参数校验、权限边界、request_id、审计记录、错误响应、配置管理和测试回归。
这能让 NL2SQL 从“我本地能跑”变成“别人可以稳定调用、可以排查、可以交付”的工程能力。
### 2. API 层、service 层、storage 层、config 层分别负责什么？
  我的回答：
服务业务层，审计储存层，参数配置层，

回答评价：
通过，但回答少了 API 层的职责。
完整表达应该是：API 层负责接口请求响应，service 层负责编排业务流程，storage 层负责审计存储，
config 层负责环境变量和运行配置。

参考答案：
API 层负责 HTTP 接口入口，包括接收请求、校验请求体、返回响应和绑定路由。
service 层负责业务编排，比如根据用户问题查找 NL2SQL 链路结果、组织回答、决定是否返回 trace。
storage 层负责数据存储，比如保存 request_id、问题、用户、最终状态和完整链路详情。
config 层负责配置管理，比如服务名、环境、演示结果路径、审计库路径、最大问题长度和是否暴露 SQL。
分层的价值是让接口、业务、存储和配置各自独立，后续替换数据库或接真实 NL2SQL 链路时风险更小。
### 3. 后端重构时为什么不能把所有逻辑都写在 `main.py`？
  我的回答：
分开不同的层级不同的工作

回答评价：
通过。回答说明了要按职责拆分。
可以再补充：所有逻辑放在 `main.py` 会导致测试困难、复用困难、修改风险高。

参考答案：
后端重构时不能把所有逻辑都写在 `main.py`，因为 `main.py` 应该主要负责 FastAPI 应用入口和路由注册。
如果把参数校验、业务编排、文件读取、审计写入、错误处理和配置都堆在一个文件里，
后续改任何一个环节都容易影响其他逻辑，也很难写测试。
更好的方式是拆成 `schemas.py`、`services.py`、`storage.py`、`config.py` 和 `errors.py`。
这样每层职责清楚，后续服务化、测试、部署和排查都会更容易。

Day 36 的服务化项目则把这个能力封装成 FastAPI 后端服务，提供 `/health`、`/nl2sql/ask`、

### 5. 后端服务为什么需要 request_id 或审计记录？
  我的回答：
链路某一步出错，通过request_id 或审计记录回溯追查错误

回答评价：
通过。回答抓住了 request_id 和审计记录的核心用途：定位错误链路、回溯问题。

参考答案：
后端服务需要 request_id 和审计记录，是为了让每一次请求都能被追踪和回放。
NL2SQL 链路很长，错误可能发生在问题解析、Schema 路由、SQL 生成、SQL 校验、查询执行或结果解释任意一层。
当用户反馈答案不对或出现安全阻断争议时，研发可以通过 request_id 查到用户问题、
中间结果、SQL、校验结果、执行状态、返回行数和最终解释。
在金融信贷场景里，审计记录还用于权限合规、敏感查询追踪和 bad case 复盘。
### B. 前两天核心回顾

### 6. [Day 34] 结果解释层怎么避免模型编造业务原因？
  我的回答：
静态验证后，对比 schema，基于事实数据进行业务解释

回答评价：
通过。你提到了基于事实数据解释，这是关键。
更准确地说，结果解释层要基于查询结果、字段含义和口径说明，而不是自由猜业务原因。

参考答案：
结果解释层要避免编造业务原因，核心是把解释建立在结构化查询结果上。
系统应该先由规则层生成事实性结论，比如最高值、最低值、趋势方向、环比差值和返回行数。
如果接 LLM，也只能把这些事实、字段含义、时间范围和口径说明作为上下文交给它润色，
不能让它自由猜测原因。
如果数据里没有原因维度，就只能提示继续按产品、渠道、风险等级或日期拆分分析。
### 7. [Day 35] 一个生产级 NL2SQL 助手应该包含哪些模块？
  我的回答：
 schema 校验，问题解析，验证层，sql 执行生成，问题解释，审计，测试回归

回答评价：
通过。回答覆盖了 Schema、问题解析、校验、SQL 执行、解释、审计和测试。
可以把“SQL 执行生成”拆成 SQL 生成和查询执行两个模块。

参考答案：
一个生产级 NL2SQL 助手至少应该包含 Schema Catalog、问题解析、SQL 生成、SQL 校验、
权限控制、查询执行、结果解释、审计日志和评测回归。
Schema Catalog 管理表字段、指标口径和权限标签；问题解析抽取指标、维度、时间和过滤条件；
SQL 生成在 Schema 约束下生成只读 SQL；SQL 校验拦截危险、越权和高成本查询；
查询执行层使用只读连接查数；结果解释层把结果转成业务语言；
审计和测试保证系统可追溯、可回归。
### 8. [Day 35] 为什么要保留每一层的中间结果？
  我的回答：
某个链路出错，通过到中间层回溯错误

回答评价：
通过。回答说明了中间结果用于定位链路错误。

参考答案：
NL2SQL 要保留每一层的中间结果，因为问题可能出在任何阶段。
如果最终答案错了，需要知道是问题解析漏了时间，Schema Router 选错表，SQL Generator 生成错字段，
SQL Validator 没拦住风险，Executor 执行失败，还是 Interpreter 解释过度。
保留中间结果可以支持调试、审计、回放和评测。
生产里通常用 request_id 把用户问题、解析结果、SQL、校验结果、执行状态、返回行数和业务解释串起来。
