### Q001
**分类：** Agent 与框架
**题目：** MCP 与 Function Calling 的区别是什么？
**参考答案：** MCP 和 Function Calling 看似都在让模型“调⽤功能”，但它们的定位和机制完全不同。
MCP 其实是⼀种协议，更准确地说是 Model Context Protocol，它定义了外部系统如何向⼤模型提供上下⽂、⼯具
描述以及如何接收模型的结构化输出。它不绑定具体实现，强调的是模型与环境之间的标准化交互⽅式。你可以把它
理解成⼀个通⽤的“插件通信规范”，像 GitHub Copilot Extensions 就基于这类思想构建扩展能⼒。
⽽ Function Calling 是具体的能⼒，指的是⼤模型能根据输⼊决定是否调⽤某个预定义函数，并⽣成符合 schema 的
参数。主流闭源模型如 GPT-4 都⽀持这个特性，它是实现 Agent ⾏为的基础组件之⼀。⽐如你⽤ LangChain 写个天⽓
查询，模型会判断⽤户意图后返回 {"name": "get_weather", "arguments": {"city": "Beijing"}}  这样
的结构。
关键区别在于：Function Calling 是单⼀模型对外暴露的能⼒接⼝，MCP 则是多个系统间协作的协议框架，⽀持更复
杂的双向交互和资源共享。

### Q002
**分类：** Agent 与框架
**题目：** 什么是 LangChain？
**参考答案：** LangChain 是个帮助开发者快速搭建应⽤的框架，重点是让⼤模型和外部系统打通。你不⽤从零写 prompt 调⽤、上
下⽂管理、⼯具调⽤这些重复代码，它都给你封装好了。
它的核⼼能⼒之⼀是链式调⽤，你可以把多个步骤串起来，⽐如先查数据库，再让⼤模型**总结**，最后发个通知。每个
环节它都抽象成组件，像积⽊⼀样拼接。
另⼀个**关键点**是跟外部数据对接。⽐如你要做个客服机器⼈，不能光靠模型内置知识，得让它查你的产品⽂档。
LangChain 提供了和向量数据库集成的能⼒，像和 Pinecone、Weaviate 对接，做检索增强⽣成（RAG），这样回答就
有据可依。
它还⽀持 Agent 模式，就是让模型⾃⼰决定下⼀步⼲什么。⽐如⽤户问“昨天销售额最⾼的订单是谁”，模型可以⾃⼰
判断要先查订单表，再查客户信息，然后组织语⾔回复。背后其实是模型输出特定格式指令，框架负责解析并执⾏。
代码上它很简洁，⼏⾏就能定义⼀个链：
from langchain.chains import LLMChain
chain = LLMChain(llm=llm, prompt=prompt)
response = chain.run(" 中国的⾸都是哪⾥？ ")
适合做需要动态输⼊、多步推理、结合私有数据的场景，⽐如智能客服、数据分析助⼿。但如果你只是简单问答，可
能显得重了。

### Q003
**分类：** Agent 与框架
**题目：** 什么是 LangGraph ？
**参考答案：** LangGraph 是基于 LangChain 构建的有状态的多智能体协作框架。它把多个 AI Agent 的交互过程建模
成“图”（Graph），每个节点是⼀个执⾏单元，边代表控制流或数据流向。
这种图结构让你能清晰定义复杂的 agent 协作逻辑。⽐如⼀个节点是⽤户输⼊处理，另⼀个是⼯具调⽤，第三个是决
策判断，通过边连接它们的执⾏顺序。整个流程的状态被显式维护，后续节点可以访问前⾯所有步骤的数据。
它特别适合做需要反复来回交互的任务。像客服系统⾥，⼀个 agent 处理意图识别，发现信息不全就转给追问
agent，补全后再交给业务办理 agent。这种循环、条件跳转、状态保持的场景，传统流⽔线搞不定，⽽ LangGraph
能扛住。
代码上其实就是定义⼀堆节点函数，然后⽤ add_node  和 add_edge  把它们串起来，最后编译成⼀个可执⾏的
graph。调试时还能看到每⼀步的状态快照，排查问题⽅便多了。
举个实际场景：⽤ LangGraph 搭⼀个多轮审批流，结合 Human-in-the-loop，在关键决策点暂停等待⼈⼯确认，确认
后继续往下⾛，整个过程可追溯。
这个模型让复杂 agent ⼯作流变得结构化，⽽不是⼀堆混乱的回调和全局变量。

### Q004
**分类：** Agent 与框架
**题目：** LangGraph 编排的原理是什么？
**参考答案：** LangGraph 的本质是把⼤模型的执⾏过程变成⼀个有向图，每个节点是⼀个动作或决策点，边代表执⾏路径。你给它
⼀个输⼊，它就从起点开始，按图的结构⼀步步⾛，⾛到哪、下⼀步⼲啥，由当前节点的逻辑决定。
1）节点可以是调⽤⼤模型、执⾏⼯具、做条件判断，甚⾄是⼈⼯审核这种阻塞操作。⽐如在客服机器⼈⾥，⼀个节点
可能是“识别⽤户意图”，下⼀个节点根据意图跳转到“查订单”或者“投诉处理”。
2）每⼀步的状态都通过⼀个共享的 state 对象传递，这个 state 是可变的，后续节点能读也能改。这就让整个流程有
了上下⽂记忆，不像普通函数调⽤那样孤⽴。
3）控制流靠的是 conditional edges，也就是条件边。你可以写个函数判断 state ⾥的字段，返回下⼀个节点的名字。
⽐如判断“是否需要⼈⼯介⼊”，返回 "human_in_the_loop" 或 "continue_automated"。
代码上其实就是定义节点函数和边规则：
graph.add_node("generate", generate)
graph.add_conditional_edges("generate", route_generate_result)
整个执⾏过程是同步推进的，但每步之间可以暂停、恢复，适合⻓周期任务。像 AutoGen、CrewAI 这些框架底层也⽤
了类似思路，只是 LangGraph 更灵活，⽀持复杂 DAG。

### Q005
**分类：** Agent 与框架
**题目：** LangChain 和 LangGraph 有什么区别？
**参考答案：** LangChain 是个⼯具箱，帮你快速搭起和⼤模型交互的流⽔线。你拿它做点简单的链式调⽤，⽐如⽂档加载、切分、
检索再喂给 LLM ⽣成答案，整个流程像⼀条直线，串⼏个组件就跑起来。适合做些原型或者轻量级应⽤，⽐如基于⽂
档问答的⼩⼯具。
LangGraph 就不⼀样了，它是专⻔搞有状态、多步骤、带循环的复杂 Agent 流程的。你可以把它看成是 LangChain
的“升级控制器”。它⽤有向图来建模流程，节点是动作，边是转移逻辑，能表达“失败重试”、“条件跳转”、“⼈类介
⼊”这种复杂控制流。⽐如你做个多智能体协作系统，A ⼲完活交给 B，B 觉得不⾏⼜回滚给 A，这种环状依赖，
LangChain 原⽣链式结构压根不经过，但 LangGraph 天⽣⽀持。
打个⽐⽅：LangChain 是条流⽔线，⼯件从头⾛到尾；LangGraph 是个调度⽹络，能根据情况动态派单、回流、并⾏
处理。
代码上，LangGraph 核⼼是 StateGraph ，你定义⼀个共享状态，每个节点函数读写这个状态，然后通过条件边决
定下⼀步去哪：
from langgraph.graph import StateGraph
graph = StateGraph(MyState)
graph.add_node("draft", draft_node)
graph.add_node("review", review_node)
graph.add_conditional_edges("review", should_rework)
graph.add_edge("draft", "review")
这种模式在做 AutoGen、MetaGPT 那类复杂 Agent 编排时特别扛住，脏活累活都让它揽了。
什么是 Manus？说说你对它的了解
Manus 不是 Java 或中间件领域的技术组件，它其实是⼀个由上海 AI 实验室推出的开源⼤模型智能体框架。你可以把
它理解成⼀个能让⼤语⾔模型“动起来”的⼯具链，让模型不仅能回答问题，还能执⾏任务。
它最⼤的特点是⽀持⾃主规划与⼯具调⽤。⽐如给它⼀个复杂⽬标，像“分析某⽀股票的历史⾛势并⽣成报告”，
Manus 能⾃⼰拆解任务：先搜索实时股价，再调⽤ Python 分析数据，最后⽣成可视化图表和⽂字**总结**。背后依赖的
是强化学习和思维链（CoT）的结合。
这类框架⼀般会集成常⻅⼯具接⼝，⽐如数据库连接、API 调⽤、代码解释器等。你在⽤的时候只需要定义⽬标，剩下
的步骤由 agent ⾃主决策。类似项⽬还有 LangChain + AutoGPT 的组合，但 Manus 更强调端到端的⾃治能⼒和中⽂
场景优化。
典型架构包含⼏个核⼼模块：任务分解器、⼯具路由、记忆存储和反馈循环。它不像传统微服务那样靠 RPC 通信，⽽
是通过语义驱动动作选择。
⽬前适⽤于⾃动化数据分析、智能运维助⼿等场景。如果你在做 AIGC 相关系统集成，可以考虑⽤它快速搭建 agent 流
程，但⽣产环境还得注意结果可解释性和执⾏超时控制。
ReAct 是什么？说说它的原理
ReAct 是⼀种让⼤模型在做推理时，既能思考（Reasoning）⼜能⾏动（Action）的框架。它的核⼼不是单纯输出答
案，⽽是通过交错的“想⼀步、做⼀步”来完成复杂任务。
模型在处理问题时，会先⽣成⼀段思考内容，⽐如分析当前情况、设定下⼀步⽬标，然后决定⼀个具体操作，⽐如调
⽤⼯具、查数据库或搜索⽹⻚。这个动作执⾏后拿到结果，再喂回模型继续下⼀轮思考，直到得出最终答案。
这种模式特别适合需要外部交互的任务，像客服机器⼈查订单状态，或者智能助⼿执⾏多步操作。相⽐纯推理，它能
获取实时数据；相⽐纯指令驱动，它多了规划和反思能⼒。
举个例⼦，你要查天⽓并决定是否带伞，ReAct 会先想“我需要知道当前位置的天⽓”，然后触发“调⽤天⽓API”动
作，拿到结果后再判断“预报有⾬，建议带伞”。
整个过程就像⼈在解决问题时的习惯：思考 → ⾏动 → 观察 → 再思考。语⾔模型不再只是被动回答，⽽是主动推进任
务。

### Q008
**分类：** Agent 与框架
**题目：** LLM Agent 的基本架构有哪些组成部分？
**参考答案：** LLM Agent 不是简单地调⽤⼤模型，⽽是⼀个能感知环境、做出决策并执⾏动作的智能体。它的结构更像是⼀个闭环
控制系统。
1）感知模块（Perception）
负责接收外部输⼊，⽐如⽤户指令、环境状态或历史交互记录。这部分会把原始信息转换成模型可理解的格式，通常
就是拼接到 prompt ⾥。
2）规划模块（Planning）
这是 Agent 的“⼤脑”，⼜可以拆成两块：
推理（Reasoning）：让模型对当前问题进⾏多步思考，⽐如使⽤思维链（Chain-of-Thought），先想“我该
怎么做”，再决定下⼀步。
任务分解（Task Decomposition）：复杂任务会被拆成多个⼦任务，按序或并⾏处理，像 LangChain 中的
Plan-and-Execute 就是典型实现。
3）记忆模块（Memory）
维持短期和⻓期上下⽂。短期记忆⼀般是当前对话的 token 上下⽂窗⼝，受限于模型⻓度；⻓期记忆则依赖向量数据
库，⽐如⽤ Milvus 或 Chroma 存储历史经验，需要时检索召回。
4）⼯具调⽤与执⾏（Tool Use & Action）
Agent 能主动调⽤外部⼯具，⽐如查天⽓、执⾏代码、操作数据库。主流做法是通过函数调⽤（Function Calling）机
制，让模型输出结构化 JSON 去触发 API，例如在 OpenAI 的接⼝中定义 tools 列表。
整个流程⾛下来，其实就是⼀个“观察 → 思考 → 决策 → ⾏动 → 观察”的循环。

### Q009
**分类：** Agent 与框架
**题目：** LLM Agent 常⻅功能有哪些？
**参考答案：** LLM Agent 不是简单的问答机器⼈，它得能感知环境、做规划、调⼯具、持续执⾏直到⽬标完成。你可以把它看成⼀
个“AI 执⾏官”，⾃⼰拆任务、调接⼝、纠错重试。
1）规划与⽬标分解
Agent 接到“分析竞品并写报告”这种⼤任务，不会⼀⼝吞，⽽是拆成查资料、读 PDF、对⽐数据、⽣成初稿、润⾊
⼏步。像 LangChain 的 PlanAndExecute  就是⼲这个的典型模式。
2）⼯具调⽤（Function Calling）
它得知道什么时候该让别⼈⼲活。⽐如查实时股价，直接调 Alpha Vantage API；订会议室就⾛钉钉或⻜书开放平台。
模型输出结构化指令，框架负责转成真实调⽤。
3）记忆能⼒
短时记忆靠上下⽂窗⼝存对话历史，⻓时记忆就得接外部存储。⽐如⽤ Chroma 或 Milvus 存过往决策记录，下次遇到
类似任务能参考。
4）⾃主执⾏与反思
有些 Agent 能⾃⼰决定下⼀步。AutoGPT 是个例⼦，但它容易陷⼊循环。更稳的做法是 ReAct 模式：每步“思考-⾏
动-观察”，像⼈⼀样边做边看结果。
5）多 Agent 协作（进阶）
复杂场景下会分⼯。⽐如⼀个当产品经理提需求，⼀个当⼯程师写代码，⼀个当测试员验证，通过内部通信达成⽬
标。MetaGPT 就模拟了这种⻆⾊分⼯。
别指望它百分百靠谱，超⻓上下⽂推理还是会乱，⼯具调⽤也可能失败。关键是设计好 fallback 机制和⼈⼯介⼊点。

### Q017
**分类：** Agent 与框架
**题目：** LangChain 的核⼼组件有哪些？
**参考答案：** LangChain 的设计是为了解耦⼤模型应⽤的各个关键环节，让开发者能灵活组装。它不是单⼀⼯具，⽽是⼀套拼图。
1）Model I/O
负责与⼤模型交互，封装了输⼊输出逻辑。⽀持多种模型类型，⽐如 OpenAI、Anthropic、Hugging Face 等。你不⽤
⾃⼰写 HTTP 请求，它帮你统⼀调⽤⽅式。
2）Prompts
提⽰词管理模块，包括提⽰词模板（PromptTemplate）和⽰例选择器（ExampleSelector）。你可以定义变量化的
prompt，⽐如把⽤户问题动态填⼊模板，还能做 few-shot ⽰例注⼊。
3）Chains
把多个步骤串起来执⾏。⽐如先调⽤模型⽣成摘要，再⽤另⼀个 prompt 做情感分析，整个流程封装成⼀个 chain。最
常⽤的是 LLMChain ，也可以⾃定义复杂链式逻辑。
4）Agents
允许模型“决策”下⼀步动作。⽐如模型判断当前需要查天⽓，就会调⽤指定⼯具（Tool），执⾏完再继续推理。背后

5）Memory
维持对话状态，常⻅有 ConversationBufferMemory （保存全部历史）和 ConversationSummaryMemory
（只存摘要）。适合聊天机器⼈这类需要上下⽂记忆的应⽤。
6）Retrieval
结合向量数据库做检索增强。⽐如⽤ FAISS 或 Chroma 存⽂档嵌⼊，查询时先检索相关⽚段，再喂给模型⽣成答案，
有效缓解幻觉。
这些组件可以单独使⽤，也能组合出复杂应⽤，关键是按需选取，别⼀上来就全堆上去。
LangChain核⼼架构是什么样的
LangChain 的设计其实围绕⼀个核⼼⽬标：让⼤模型能像搭积⽊⼀样灵活接⼊各种外部能⼒。它不追求把所有功能塞
进引擎，⽽是通过标准化接⼝降低扩展成本。
链（Chain） 是整个架构的组织单元，你可以把多个处理步骤串成⼀条流⽔线。⽐如先⽤ PromptTemplate ⽣成⽂
本，再交给 LLM 处理，最后调⽤ API 写⼊数据库，每个环节都是可替换的组件。
关键抽象有四个： 1）Model I/O 封装了⼤模型的输⼊输出，⽀持 OpenAI、Anthropic、HuggingFace 等主流服务 2）
Retriever 负责从外部获取数据，典型的是⽤ FAISS 或 Chroma 做向量检索 3）Tool 允许模型调⽤外部函数，⽐如搜索
天⽓、执⾏ Python 代码 4）Agent 根据 prompt 决定调⽤哪些 Tool，实现动态决策流
实际项⽬⾥，你常会看到 Agent 结合 ReAct 模式⼯作。模型输出带特殊标记的⽂本，框架解析后触发⼯具调⽤，结果
再拼回去继续推理。这种设计让逻辑闭环可以在⼀次会话中反复迭代。
和直接调 API 的最⼤区别是，LangChain 把“感知-决策-⾏动”循环变成了可配置的⼯作流。但要注意，链路越⻓，
延迟叠加越明显，调试也越困难。简单任务直接⽤ LLMClient 反⽽更稳。

### Q018
**分类：** Agent 与框架
**题目：** Agent 死循环问题有遇到过吗？如何解决？
**参考答案：** Agent 死循环通常发⽣在字节码增强逻辑⾥，⽐如⽤ Java Agent 做监控或 APM 时，不⼩⼼把⾃⼰ instrument 的类⼜
触发了新的 transform，结果⽆限加载 class，CPU 直接打满。
最常⻅的场景是，你在 ClassFileTransformer  ⾥处理某个类的时候，执⾏了某些代码，这些代码⼜动态⽣成类
或者触发类加载，⽽你的 transformer 没做过滤，就会重新进⼊ transform ⽅法，形成递归调⽤。特别是⽤到 ASM、
CGLIB、Javassist 这些框架时很容易中招。
解决办法其实就两条：
1）加 类名过滤，直接排除不必要的类，尤其是你 agent ⾃⾝的包路径，⽐如 com.mycompany.agent.**  全部
return null 不做处理。
2）⽤ 本地线程标记（ThreadLocal）标记当前是否已经在 transform 流程中，如果是，直接跳过。这个特别关键，因
为很多间接类加载是你控制不了的。
static final ThreadLocal<Boolean> transforming = new ThreadLocal<>();
public byte[] transform(ClassLoader loader, String className, Class<?>
classBeingRedefined,
ProtectionDomain protectionDomain, byte[] classfileBuffer) {
if (transforming.get() != null) return null;
try {
transforming.set(true);
// 执⾏实际的字节码修改逻辑
return doTransform(loader, className, classfileBuffer);
} finally {
transforming.remove();
}
}
另外，像 SkyWalking、Pinpoint 这些 APM ⼯具都遇到过这类问题，它们的源码⾥都有对应的 guard 机制防⽌重⼊。
参考它们的实现能少踩很多坑。

### Q019

### Q025
**分类：** MCP 与协议
**题目：** MCP 协议⽀持哪两种模式？
**参考答案：** MCP（Model Context Protocol）⽬前主要⽀持两种交互模式，对应不同的上下⽂处理⽅式。
1）请求-响应模式
这是最常⽤的模式，客户端发送⼀个请求，携带特定上下⽂，服务端完全基于该上下⽂⽣成响应，交互结束。整个过

2）会话模式
系统会维护⼀个持续的上下⽂会话状态，客户端可以连续发送多条消息，每条都基于之前的对话历史进⾏理解与回

户前⼏轮提到的信息要能在后续对话中被引⽤。
会话模式的关键在于服务端需要管理 Session 状态，可能结合 UUID 或 token 来标识不同⽤户的上下⽂隔离。⽽请求-
响应模式⽆状态，更容易⽔平扩展。
要不要⽤会话模式，得看业务是否需要“记住”前⾯的内容。如果只是做离散任务，⽐如内容审核、关键词提取，压
根不需要维持状态，⽤第⼀种就够了。

### Q026

### Q033
**分类：** Prompt 与结构化输出
**题目：** 什么是护栏技术？
**参考答案：** 护栏技术本质是⼀种在系统运⾏时动态控制流量、防⽌雪崩的保护机制。它不追求彻底解决问题，⽽是快速⽌损，让
系统能继续对外提供部分服务。
1）核⼼思路是“宁可限流，不可拖垮”。⽐如⼀个电商系统的下单接⼝突然变慢，⼤量请求堆积，数据库连接被打
满，这时候如果不加⼲预，整个应⽤可能直接宕机。护栏技术就是在这个临界点介⼊，把超出处理能⼒的请求直接拒
绝或降级。
2）常⻅的实现⽅式有熔断、限流、降级。熔断像电路保险丝，当错误率超过 50% 就直接切断请求，避免连锁故障；
限流控制每秒最多放⾏ 1000 个请求，多余的就地拦截；降级则是返回兜底数据，⽐如商品详情⻚的推荐模块挂了，就
直接返回空列表，保证主流程可⽤。
3）实际项⽬⾥，Sentinel 和 Hystrix 都是典型的护栏⼯具。拿 Sentinel 来说，你可以给某个接⼝配置 QPS 超过 50
就⾃动限流，规则可以实时推送，不⽤重启服务。
// 定义资源
SphU.entry("getOrder")
// 业务逻辑
这种机制特别适合⽤在⾼并发场景，⽐如秒杀系统。你不可能靠堆机器扛住所有流量，得靠这些“安全策略”来兜
底。真正的稳定性不是不出问题，⽽是出问题时系统还能活着。

### Q034
**分类：** RAG 检索增强
**题目：** RAG 的完整流程是怎么样的？
**参考答案：** RAG 的流程其实就三步：准备数据、查资料、⽣成答案。整个过程让⼤模型在回答前，先去“翻书”找依据。
1）数据准备阶段，先把外部知识库的⽂档切⽚，⽐如把 PDF 或数据库表结构说明拆成⼀段段⽂本块。接着⽤
embedding 模型给每⼀块转成向量，存进向量数据库，像 Milvus 或 Pinecone 这类⼯具⼲的就是这事。
2）⽤户提问时，系统不会直接扔给⼤模型。⽽是先把问题也转成向量，去向量库⾥做相似度搜索，找出最相关的⼏个
⽂本块。这⼀步很关键，决定了模型能“看到”哪些背景知识。
3）最后，把这些检索到的⽂本块拼接成提⽰词，连同原问题⼀起喂给⼤模型。模型基于这些上下⽂⽣成答案，⽽不是
靠记忆瞎编。这样出来的回答有据可查，幻觉少。
这个流程⾥，embedding 模型和向量检索的质量直接决定效果。如果检索错了，后⾯再强的模型也搞不定。常⻅坑是
⽂本分块不合理，⽐如按固定字符切，把⼀个完整概念切碎了，导致查不准。

### Q035
**分类：** RAG 检索增强
**题目：** 什么⾃查询？为什么在 RAG 中需要⾃查询？
**参考答案：** ⾃查询其实是让模型⾃⼰⽣成查询条件，去检索它需要的信息。⽐如你问“苹果最新财报怎么样”，模型不会直接回
答，⽽是先拆解问题，⽣成像“苹果 2024 年 Q2 营收、利润、同⽐增⻓”这样的检索关键词，再拿这些词去查⽂档。
这在 RAG ⾥特别关键，因为原始问题往往太笼统或包含多个维度。直接拿原问题去搜，召回的内容可能不精准。⾃查
询相当于加了⼀层智能路由，把复杂问题翻译成适合检索的结构化查询语句。
1）能处理时间范围过滤，⽐如“去年销售额最⾼的产品”会被转成带时间约束的查询
2）⽀持属性筛选，“便宜的安卓⼿机”会拆成“价格低于 2000 元、操作系统为 Android”
3）适配向量库的元数据检索，⽐如⽤ Chroma 或 Pinecone 的 where 条件过滤
代码上⼀般⽤ LLM 接⼀个提⽰模板：
String prompt = " 根据⽤户问题⽣成检索查询语句 : " + userQuestion;
List<String> queries = llm.generate(prompt);
List<Document> results = vectorStore.search(queries);
整个流程像是给 RAG 装了个“理解⼤脑”，先把⼈话转化成机器能⾼效检索的指令，避免压根不经过语义解析就硬
搜。

### Q036
**分类：** RAG 检索增强
**题目：** 什么提⽰压缩？为什么在 RAG 中需要提⽰压缩？
**参考答案：** 提⽰压缩其实就是把 RAG 中过⻓的上下⽂输⼊给“瘦⾝”。你想想，⼤模型的上下⽂⻓度是有限的，⽐如 32K 或
128K，但检索回来的⽂档⽚段⼀多，加上原始问题和模板，很容易就超了。
1）最直接的问题就是 token 数爆掉，请求直接被拒，压根不经过模型推理。
2）就算没超，⻓上下⽂推理成本⾼，响应慢，延迟上去⽤户体验就差了。
3）⽽且很多检索出的内容其实跟问题关系不⼤，纯属噪⾳，反⽽⼲扰⽣成质量。
所以得做提⽰压缩，把那些冗余信息、重复段落、⽆关句⼦给剔除或简化。不是简单删减，⽽是保留关键事实和语
义，让输⼊更紧凑⾼效。
常⻅的做法有⼏种：
⽤⼩模型先做⼀轮重排序或摘要，⽐如⽤ BGE 或 Cohere reranker 挑最相关的 top-k。
在 prompt 组装时加逻辑，只保留⾼分 chunk，甚⾄对每个 chunk 做句⼦级裁剪。
也可以引⼊类似 HyDE 的思路，先⽣成假设答案再反向匹配，过滤低相关度内容。
代码上没啥复杂语法，核⼼是控制组装后的总 token 数：
# 伪代码⽰意
context = "\n".join([chunk["text"] for chunk in relevant_chunks[:3]])  # 限制数量
prompt = f" 基于以下内容回答： \n{context}\n\n 问题：{query}"
要不要压缩，其实看场景。如果检索精度⾼、返回少，可能不需要。但⾯对海量⽂档库，⽐如⽤在企业知识库搜寻，
就得靠这招扛住⻓上下⽂压⼒。

### Q037
**分类：** RAG 检索增强
**题目：** 什么是 RAG 中的分块？为什么需要分块？
**参考答案：** RAG 的上下⽂⻓度是有限的，⽐如主流模型⼀般⽀持 32k 或 128k 的 token 上下⽂。你塞进去的 prompt 越⻓，能留
给检索内容的空间就越少。分块其实就是把原始⽂档切成适合模型处理的⼩段落，让每次检索和⽣成都能在上下⽂窗
⼝内完成。
1）分块不是简单按段落切。如果按固定字符数切，可能把⼀句话从中间劈开，语义就断了。好的分块策略会尽量保持
语义完整，⽐如⽤ 递归分割，先按标题切，再按段落，最后才是固定⻓度。LangChain ⾥的
RecursiveCharacterTextSplitter  就是这么⼲的。
2）块太⼩，信息不全，模型容易“只⻅树⽊不⻅森林”；块太⼤，⼜挤占上下⽂，影响其他 prompt 内容。⼀般来
说，512 到 1024 个 token 是⽐较常⻅的块⼤⼩，具体得看你的数据和模型。
3）有些场景还得考虑跨块关联。⽐如⼀个技术⽅案分布在三个块⾥，单靠⼀个块检索出来，模型也拼不全逻辑。这时
候就得靠后续的查询扩展或多步检索来补救。
代码上⼤概⻓这样：
from langchain.text_splitter import RecursiveCharacterTextSplitter
splitter = RecursiveCharacterTextSplitter(
chunk_size=512,
chunk_overlap=50,
separators=["\n\n", "\n", " 。 ", " ", ""]
)
docs = splitter.split_text(large_document)
要不要分块，其实取决于你的数据形态和检索⽬标。像 PDF、⻓⽹⻚这种肯定要分，但如果是短问答对，可能直接喂
进去就⾏。

### Q038
**分类：** RAG 检索增强
**题目：** 在 RAG 中，常⻅的分块策略有哪些？分别有什么区别？
**参考答案：** 分块策略直接影响检索效果，选不好后⾯全⽩搭。关键得看⽂本的语义连贯性和上下⽂依赖。
1）固定⼤⼩分块
按字符或 token 数硬切，⽐如每 512 个 token 切⼀块。简单粗暴，适合结构规整的⽂档。但容易把⼀句话从中间劈
开，语义被撕裂，检索回来的⽚段可能压根不经过原意。像 PDF 或⽹⻚正⽂这种连贯性不强的还⾏，但技术⽂档、论
⽂就搞不定。
2）基于段落或标题分割
利⽤⾃然断点，⽐如⼀级/⼆级标题之间、空⾏处分割。能保留完整语义单元，适合 Markdown、Word 这类有结构的
⽂档。问题在于块⼤⼩不均匀，有的标题下内容贼⻓，检索时信息密度太低。
3）滑动窗⼝重叠分块
在固定⼤⼩基础上，让相邻块有部分重叠，⽐如每次滑动 512，重叠 128。这样能缓解语义断裂，关键信息⼤概率出现
在多个块⾥。但会增加索引量和检索耗时，重复内容多，存储成本翻倍也不稀奇。
4）语义感知分块（如 RecursiveCharacterTextSplitter）
先按标点切，再按⻓度合并，尽量保证句⼦完整。LangChain ⾥默认⽤这个，实际项⽬中表现最稳。更进⼀步可以⽤
嵌⼊模型做句⼦相似度判断，在语义边界处分割，不过延迟⾼，⼀般⽤在对精度要求极⾼的场景。
选哪种？短平快任务⽤滑动窗⼝+固定⼤⼩，追求效果上语义分块，有明确结构就优先标题分割。

### Q039
**分类：** RAG 检索增强
**题目：** 在 RAG 中的 Embedding 嵌⼊是什么？
**参考答案：** Embedding 在 RAG ⾥⼲的事，就是把⽂本转成机器能算的“数字向量”。你给它⼀句话、⼀段⽂档，它就输出⼀个⻓
⻓的浮点数数组，这个数组得能保留原⽂的语义信息。
⽐如你搜“怎么修⾃⾏⻋”，系统其实不是在关键词匹配，⽽是先把这句话做个 Embedding 变成向量，再去向量空间
⾥找和它最接近的那些⽂档⽚段。像⽤百度搜索那种关键词匹配早就跟不上了，现在都靠这种语义相似度来找答案。
1）这活⼀般由预训练模型来⼲，⽐如 BERT、Sentence-BERT（SBERT），或者专⻔做检索的 bge 模型。它们会把不
同⻓度的⽂本压到固定维度的向量⾥，常⻅的是 768 或 1024 维。
2）RAG 流程中，Embedding 主要⽤于两个地⽅：⼀是离线把知识库⽂档切⽚后全部向量化存进向量数据库，⽐如⽤
FAISS、Milvus 或 Pinecone；⼆是线上⽤户⼀提问，⽴刻转成向量去库⾥查 top-k 最相似的 chunk。
3）效果好不好，关键看 Embedding 是否真能把相近意思的⽂本“拍”到向量空间⾥挨得近。如果“如何重启路由
器”和“拔电源再插上”这两个句⼦向量距离很近，那说明嵌⼊质量⾼。
代码上基本就是调个 model.encode(text) 就完事，但背后模型选型、是否微调、向量索引构建⽅式都会影响最终召回
率。

### Q041
**分类：** RAG 检索增强
**题目：** 什么是 GPTCache？
**参考答案：** GPTCache 是⼀个专为⼤模型应⽤设计的语义缓存中间件，⽬标是把重复或语义相近的请求直接拦截下来，减少对⼤
模型 API 的调⽤次数，从⽽降成本、提响应速度。
它的核⼼思路不是简单地做字符串匹配，⽽是通过向量相似度来判断新来的请求和历史请求是否“意思差不多”。⽐如
⽤户问“今天天⽓怎么样”和“现在外⾯晴吗”，⽂字不同但意图接近，GPTCache 就可能命中缓存。
整个流程⼀般是这样：
1）⽤户请求进来，先被转换成 embedding 向量
2）在向量数据库⾥搜最近似的⼏个结果
3）如果相似度超过阈值，就返回缓存⾥的答案；否则⾛正常链路调⼤模型，并把新的 question-answer 存进缓存
它适合⽤在⾼频问答、客服机器⼈、RAG 系统这类场景，尤其是请求重复率⾼、对延迟敏感的应⽤。像 LangChain、
LlamaIndex 这些框架都能集成 GPTCache。
不过要注意，embedding 模型的选择和距离计算⽅式会直接影响命中精度，太松容易误中，太严⼜起不到缓存作⽤。
⼀般建议搭配 Milvus、Pinecone 或 Chroma 做向量存储，性能和扩展性更稳。

### Q042
**分类：** RAG 检索增强
**题目：** 使⽤LangChain时，如何实现多路召回结果的动态权重分配？
**参考答案：** 多路召回的动态权重分配，关键在于根据查询实时计算各路结果的置信度或相关性得分，⽽不是⽤固定权重拍脑袋。
1）最直接的做法是引⼊⼀个重排序模型（Reranker），⽐如 BGE-Reranker 或 Cohere Rerank。把不同来源召回的候
选集合并后，统⼀喂给 Reranker 打分，它会输出基于 query 的相关性排序，⾃然就实现了“动态加权”。LangChain
⾥可以⽤ RunnableLambda  包装 reranker 调⽤。
2）另⼀种思路是基于查询特征做路由加权。⽐如⽤户问的是时间敏感问题，就提⾼来⾃新闻索引或时序数据库那⼀路
的权重；如果是专业术语，就抬⾼知识图谱或维基数据源的分值。这在 LangChain 可以通过 RouterRetriever  实
现，配合 LLM 判断 query 类型，动态选择或加权⼦检索器。
3）还可以让 LLM ⾃⼰评估各路结果的相关性。把每路召回的 top-k 和 query ⼀起输⼊ LLM，让它打分或排序。虽然
贵点慢点，但灵活性最⾼，适合对效果要求极⾼的场景。
代码上，核⼼是组合多个 BaseRetriever  并在外层做融合：
from langchain.retrievers import EnsembleRetriever
from langchain_community.retrievers import BM25Retriever
ensemble_retriever = EnsembleRetriever(
retrievers=[vector_retriever, BM25Retriever.from_texts(...)],
weights=[0.6, 0.4]  # 这⾥可以动态算出来再传
)
重点是 weights  别写死。你可以先跑个轻量模型预估各路匹配质量，再填进去。像 Jina Reranker、M3E + BGE 组
合在 C-MTEB 上能涨 5~8 个点，⽐静态融合强不少。

**参考答案：** RAG 系统召回结果偏离⽤户意图，问题通常出在检索和⽣成两个环节的衔接上。先定位是“没找到”还是“找到了但

有时候检索是对的，但 prompt 写得太松，模型⾃⼰脑补。要明确约束“基于以下上下⽂回答”，并在输⼊⾥只放⾼相

线上可以接反馈闭环，⽐如⽤户点赞/点踩驱动 embedding 或 reranker 微调，持续迭代。

### Q045
**分类：** RAG 检索增强
**题目：** 什么是 RAG 中的 Rerank？具体需要怎么做？
**参考答案：** RAG ⾥的 Rerank 其实就是对检索阶段召回的多个⽂档⽚段，重新打分排序，挑出最相关的结果喂给⼤模型⽣成答
案。
⼀般流程是先⽤向量数据库做相似度搜索，⽐如从知识库⾥找出 top-5 的候选⽂档。但向量检索有个问题，它只看语
义相似，不⼀定和⽤户问题真正匹配。这时候就需要 Rerank 模型再筛⼀遍。
Rerank 模型通常是交叉编码器（Cross-Encoder），⽐如 BGE-Reranker 或 Cohere Rerank API。它会把⽤户问题和每
个召回的⽂档拼在⼀起输⼊模型，输出⼀个相关性分数。这个过程⽐向量检索慢，但精度⾼得多。
具体做法很简单，假设你有 5 个候选段落：
1）把 query 和每个段落组合成⼀对⽂本 2）送进 Rerank 模型得到相关性得分 3）按得分从⾼到低排序，取前 2~3 个
⾼质量段落输⼊ LLM
代码上类似这样：
# 伪代码⽰意
pairs = [(query, doc) for doc in retrieved_docs]
scores = reranker.predict(pairs)
ranked_docs = [doc for _, doc in sorted(zip(scores, retrieved_docs), reverse=True)]
要不要加 Rerank 得看场景。如果你的系统能接受⼀定噪⾳，直接⽤向量检索也⾏。但要是追求准确率，尤其是客服、
医疗这类容错低的场景，这⼀步的提升⾮常明显，基本能提 10%~20% 的最终回答准确率。

### Q046

**题目：** 什么混合检索？在基于⼤模型的应⽤开发中，混合检索主要解决什么问题？
**参考答案：** 混合检索不是简单把关键词和向量拼在⼀起。它要解决的是单⼀检索⽅式的短板，⽐如纯关键词匹配会因为字⾯不⼀

这招在问答系统、知识库检索⾥特别实⽤。你既要语义理解能⼒，⼜要⽀持字段过滤和精确匹配，光靠⼤模型⽣成不
⾏，得靠混合检索把底座打好。

### Q047
**分类：** RAG 检索增强
**题目：** 在 RAG 应⽤中为了优化检索精度，其中的数据清洗和预处理怎么做？
**参考答案：** 数据清洗和预处理在 RAG 中直接决定检索质量，很多效果差的问题其实都出在这⼀步。
1）⽂本切分要合理，不能简单按固定⻓度硬切。⽐如⽤ LangChain 的 RecursiveCharacterTextSplitter

5）可选做实体增强。⽤ spaCy 或 HanLP 抽关键词、⼈名、地名，附加到 chunk metadata ⾥，在检索时⽀持基于实
体的扩展查询。
整个流程可以串成 pipeline，⽤ Apache Beam 或 Airﬂow 调度，确保数据进 Embedding 模型前是⼲净、结构化、带
上下⽂的。

### Q048
**分类：** RAG 检索增强
**题目：** 什么查询扩展？为什么在 RAG 应⽤中需要查询扩展？
**参考答案：** 查询扩展本质是把⽤户那句简单的原始问题，变形成⼀个或多个更完整、信息更丰富的查询语句。你别⼩看这⼀步，
它直接决定了后续检索能不能捞出真正相关的内容。
1） ⽤户搜“苹果⼿机续航差”，其实他关⼼的是 iPhone 15 Pro 的电池表现和省电技巧。原始 query 太短，关键词稀
疏，直接去向量库搜索容易匹配到⽆关的“苹果⽔果”或者泛泛的“⼿机耗电”。这时候就得靠查询扩展来补全意图。
2） 常⻅做法有⼏种。⼀种是同义词/近义词扩写，“续航”换成“电池寿命”“使⽤时间”；⼀种是基于⼤模型做多跳推
理，让 LLM 把原始问题拆成⼏个⼦问题，⽐如“iPhone 15 Pro 满电能⽤多久”“哪些设置能延⻓待机”；还有⼀种是
查询重写，把⼝语化表达转成更标准的检索语⾔。
3） 不做查询扩展的 RAG 系统，召回率通常很低。特别是⾯对缩写、俚语、模糊表达时，向量检索压根不经过语义深
化，直接就去算相似度，结果就是 top-k ⽂档⾥没有命中关键段落。像 LangChain、LlamaIndex 都内置了 query
transformation 模块，就是为了解决这个断层。
这步看着是脏活累活，但它是连接“⼈话”和“机器可检索语义”的关键桥梁。搞不定查询扩展，RAG 效果就很难稳
定。
如何进⾏ RAG 调优后的效果评估？请给出真实应⽤场景中采⽤的效果评估标准与⽅法
RAG 系统调优后不能只看⽣成结果“好不好”，得⽤量化指标和真实场景反馈来判断是否真的提升了效果。重点是结合
⾃动评估指标和⼈⼯评估，覆盖相关性、准确性和可⽤性。
1）先看召回阶段，⽤传统信息检索指标。⽐如在企业知识库问答场景中，我们拿 1000 条历史⽤户问题做测试集，计
算 top-5 ⽂档的 Recall@5 和 MRR。如果 Recall@5 从 0.6 提升到 0.78，说明检索更准了，这是硬指标。
2）再看整个 RAG 流程的输出质量。可以⽤ Rouge-L 衡量⽣成答案与标准答案的⽂本相似度，但别太依赖它。更关键
的是 BERTScore 或 BARTScore，它们能更好捕捉语义匹配程度。⽐如在客服机器⼈中，我们发现 Rouge-L 变化不
⼤，但 BERTScore 提⾼了 12%，实际回答更贴切了。
3）引⼊事实⼀致性检测。⽤ Faithfulness 指标判断⽣成内容是否基于检索到的⽂档。⽐如模型说“报销周期是7
天”，但⽂档写的是“5个⼯作⽇”，这就是幻觉。可以⽤专⻔打分模型如 RAGAS 来量化。
4）最终绕不开⼈⼯评估。设计评分卡，让业务⼈员从相关性、准确性、完整性、可读性四个维度打分。⽐如在⾦融投
研场景，分析师对调优前后的回答盲评，发现新版本在“数据来源清晰”这⼀项得分提升明显。
真实项⽬⾥，像⽤ LangChain + Pinecone 做法律咨询系统时，就是靠这套组合拳验证调优有效，上线后⼀次解决率提
⾼了 23%。

上重型⽅案。中等以上规模，就得考虑分布式能⼒，Milvus 和 Qdrant 都⽀持分⽚和副本，能扛住⾼并发。然后看⽣

少踩坑。

### Q052
**分类：** RAG 检索增强
**题目：** 向量数据库中的 HNSW、LSH、PQ 分别是什么意思？
**参考答案：** HNSW、LSH、PQ 是向量数据库⾥常⽤的 近似最近邻搜索（ANN）技术，⽤来在⾼维空间快速找相似向量，毕竟暴⼒
遍历 O(n) 在亿级数据上根本扛不住。
1）HNSW（Hierarchical Navigable Small World）本质是个多层图结构。每⼀层都是个近邻图，⾼层稀疏⽤
来“跳”得快，底层密集做精细搜索。查询时从顶层开始往下⾛，像导航⼀样逐步逼近⽬标点。Milvus、Weaviate 都
⽤它，召回率⾼，延迟也稳，但建索引内存开销⼤。
2）LSH（Locality-Sensitive Hashing）思路更粗暴：把相似的向量通过特定哈希函数尽可能甩到同⼀个桶⾥。查的时
候只搜对应桶，⼤幅缩⼩范围。优点是实现简单、存储省，但召回率不如 HNSW，尤其分布不均时容易漏。早期
Annoy ⽤过类似思想。
3）PQ（Product Quantization）⼲的是降维压缩的脏活。它把⾼维向量切成⼏段，每段独⽴聚类，⽤聚类中⼼代替
原始向量分量，存个码本就⾏。原来 128 维 ﬂoat 能压到 16 字节内，检索时⽤对称距离近似算相似度。Faiss 就拿它和
IVF 搭配，扛住了⼗亿级向量库。
这仨经常混着⽤，⽐如 Faiss ⾥ IVFPQ 就是先⽤聚类定位候选集，再 PQ 压缩算距。选哪个看场景：要精度选
HNSW，要省资源可以试 LSH + PQ 组合。

### Q053
**分类：** RAG 检索增强
**题目：** 向量数据库中的 ANN 是什么？为什么需要⽤它？
**参考答案：** 近似最近邻（ANN）是向量数据库⾥⽤来加速相似性搜索的核⼼技术。原始的暴⼒搜索要算查询向量和所有向量的距
离，数据量⼀⼤，⽐如上亿条，根本没法实时响应。ANN 的思路就是牺牲⼀点点精度，换来⼏倍甚⾄上百倍的速度提
升。
它通过构建特殊的索引结构来缩⼩搜索范围。常⻅的⽅法有：
1）基于图的 HNSW，把向量连成近邻图，从⼀个⼊⼝点开始“爬⼭”，逐步逼近最近的邻居。Faiss 和 Milvus 都⽀持
这种，效果好，吞吐⾼。
2）基于聚类的 IVF，先把向量聚成若⼲簇，搜索时只查离查询点最近的⼏个簇，避免全量扫描。适合⼤数据集，配合
PQ 编码还能⼤幅降内存。
3）基于 LSH 的哈希映射，把相似向量尽量散列到同⼀个桶⾥，但召回率在⾼维下容易掉得厉害，现在⽤得少了。
为什么⾮要⽤它？因为纯 brute-force 在千万级以上向量场景下，延迟动辄⼏百毫秒甚⾄秒级，压根不经过。⽽ ANN
能把响应控制在⼏毫秒内，才能⽀撑推荐、图⽂搜、语义匹配这些实时业务。像推荐系统⾥⽤户⼀刷就触发⼀次向量
化检索，不⽤ ANN 根本扛不住。
向量数据库的⼯作流程有哪些？请简述下
向量数据库不是简单存个数组就完事了，它得解决“相似性检索”这个核⼼问题。整个流程从数据进来开始，到最后
返回结果，每⼀步都围绕⾼效近似最近邻搜索展开。
1） 数据预处理阶段，原始数据⽐如⽂本、图⽚先过嵌⼊模型，转成⾼维向量。这步通常在外部做，⽐如⽤ Sentence-
BERT 或 CLIP 模型⽣成向量，维度常⻅ 384 到 1024。
2） 向量写⼊时，数据库会做索引构建。直接暴⼒计算所有距离肯定搞不定，所以要⽤专⻔的索引结构，⽐如 HNSW
构建图索引，IVF 做聚类分组，或者 PQ 做量化压缩。这些技术能让查询时跳过⼤部分不相关的向量。
3） 查询阶段，输⼊⼀个⽬标向量，系统先定位可能相近的候选集。⽐如 HNSW 从⼊⼝点出发沿图遍历，IVF 先算落在
哪个聚类中⼼附近，再只搜那个分区⾥的向量。最后对候选集做精确距离计算，返回 Top-K 最相似的结果。
整个过程的关键在于平衡精度和性能。索引构建会影响写⼊速度和内存占⽤，⽽搜索策略决定延迟和召回率。典型产
品像 Milvus、Pinecone、Weaviate 都是基于这类流程设计的。

### Q054
**分类：** RAG 检索增强
**题目：** 什么是 Advanced RAG？
**参考答案：** Advanced RAG 是在基础 RAG（Retrieval-Augmented Generation）之上，通过优化检索和⽣成两个环节的协同效
率，提升回答质量的⼀套⽅法论。它不是某个具体⼯具，⽽是⼀系列增强技术的组合。
1）传统 RAG 通常直接⽤⽤户问题去向量库检索，但⾃然语⾔存在歧义或表述不清的问题，导致召回不准。Advanced
RAG 会先做查询重写，⽐如⽤ LLM 把原始问题拆解成多个⼦问题，或者改写成更适合检索的关键词形式，像 DPR
（Dense Passage Retrieval）这类模型就常被⽤来做语义对⻬。
2）在检索阶段，除了单纯找 top-k 最相似的⽂档块，还会引⼊重排序（Re-Ranking）。⽐如⽤ Cross-Encoder 对初步
召回的结果做精细打分，把真正相关的排前⾯。这块可以结合 BM25 等稀疏检索做混合召回，提升覆盖度。
3）⽣成环节也不再是“⼀把喂给 LLM”。有些⽅案会做上下⽂压缩，只提取出与问题最相关的句⼦⽚段，减少噪声和
token 消耗。还有像 Self-RAG 这样的思路，让模型⾃⼰判断是否需要检索、要不要采纳检索结果，增加推理可控性。
代码上其实变化不⼤，核⼼是流程增强：
# 伪代码： Advanced RAG 流程
rewritten_query = rewrite_prompt(user_query)
docs = vector_store.search(rewritten_query, k=10)
reranked_docs = cross_encoder_rerank(rewritten_query, docs)
context = compress_context(rewritten_query, reranked_docs)
answer = llm.generate(context, user_query)
整个链路更像是⼀个可调教的系统⼯程，典型应⽤有 LangChain + FAISS/BGE + Cohere Rerank 的组合，或者是⽤
LlamaIndex 做节点后处理。关键是要能根据业务场景动态调整各模块权重，⽽不是堆模块。

### Q055
**分类：** RAG 检索增强
**题目：** 什么是 Modular RAG？
**参考答案：** Modular RAG 不是某个固定框架，⽽是指⼀种将 RAG（检索增强⽣成）系统拆解成独⽴、可替换模块的设计思路。你
完全可以把它看作 RAG 的“微服务化”⸺每个环节单独优化、灵活组合。
整个流程通常包括⼏个关键阶段：⾸先从外部知识库中检索相关⽂档，然后对检索结果做重排序（Rerank），最后把处
理好的上下⽂喂给⼤模型⽣成答案。传统 RAG 把这些步骤耦合在⼀起，⽽ Modular RAG 强调每个部分都能独⽴迭
代。
⽐如你在⽤ LangChain 或 LlamaIndex 时，可以⾃由替换不同的检索器（如从 BM25 切到向量检索）、换上 Cross-
Encoder 做精排、再接⼊不同的 prompt ⼯程策略。这种结构下，⼀个团队可以专注优化检索质量，另⼀个团队负责
提升⽣成流畅度，互不⼲扰。
1）检索模块：⽀持多路召回，⽐如同时⾛关键词和语义向量 2）重排序模块：⽤更精细的模型对 Top-K 结果打分，提
升相关性 3）上下⽂融合：对多个⽚段做去重、摘要或拼接，减少噪声 4）⽣成控制：动态决定是否启⽤检索，避
免“硬套答案”
这种⽅式特别适合复杂场景，像在企业知识库中查政策，可能需要结合时间过滤、权限判断等额外逻辑，直接塞进端
到端 RAG 很难维护。拆成模块后，脏活累活交给对应组件，主链路反⽽更清晰。
Computer Use 是什么？说说它的原理
⽬前没有明确的技术标准或⼴泛认知的系统叫做 "Computer Use"。这个术语在主流计算机科学、软件⼯程或IT基础设
施领域并不指代某个具体的技术、协议或架构。
如果你指的是“计算机的使⽤”（the use of computers），那涵盖范围⾮常⼴，包括操作系统调度、程序执⾏、内存管
理、⽹络通信等基础⾏为。但作为⼀个专有名词，“Computer Use”并不是像 Redis、Kafka 或 JVM 这样的具体技术
组件。
有可能是提问时表述有误，⽐如本意想问的是：
Compute Unit：计算单元，常⻅于分布式计算框架（如 Spark 中的 executor）或硬件架构中。
Computer Vision：计算机视觉，AI 的⼀个分⽀。
Computer Cluster：计算机集群，⽤于⾼性能计算或服务扩容。
或者某个特定产品/项⽬中的内部命名，例如某系统的“ComputerUseService”类。
建议确认题⽬是否准确。如果是⾯试中遇到这个问题，可以先和⾯试官澄清概念，避免答偏。很多时候，这种模糊题
是在考察你如何应对需求不明确的问题⸺这时候沟通能⼒和问题拆解能⼒⽐技术本⾝更重要。
与其纠结术语定义，不如展⽰分析思路： 1）询问上下⽂场景
2）列举可能的相近概念进⾏排除
3）基于合理假设给出解答路径
这样反⽽更容易拿到⾼分。

### Q056
**分类：** RAG 检索增强
**题目：** 你有多个知识库，做 RAG 的时候，怎么保证查询效率和准确性兼容，并尽可能减少幻觉？
**参考答案：** 多个知识库做 RAG，关键是在检索阶段就控制好数据的“来源质量”和“语义相关性”，否则后⾯⽣成再怎么优化也压
根不经过。
1） 先做知识库的预分类或打标。⽐如⽤ Milvus 或 Elasticsearch 存向量时，给每个⽂档加 metadata 标记来源类型
（如产品⽂档、客服记录、API ⼿册）。查询时先根据 query 的意图路由到最可能相关的库，避免全量检索拖慢速度。
2） ⽤ 混合检索 策略。单纯靠向量可能召回语义相近但事实错误的内容，容易引发幻觉。结合关键词匹配（BM25）和
向量相似度，像 Weaviate 或 Vespa 都⽀持这种 hybrid search，能同时抓住字⾯匹配和深层语义，准确率提升明显。
3） 检索后做重排序（rerank）。别拿到 top-k 就直接喂给 LLM，⽤⼀个轻量级模型⽐如 BGE-reranker 对结果按相关
性重新打分，把真正靠谱的段落往前排。这⼀步能过滤掉不少“看起来像对其实不对”的内容。
4） 在 prompt 中显式标注信息来源。把检索到的⽂本块加上出处标签，⽐如 [来⾃：API ⼿册 v3.2]，让⼤模型知道依
据从哪来，⽣成时更倾向引⽤原⽂，减少脑补。
5） 最后加⼀层验证机制。对关键回答，可以⽤⼩模型做⼀致性校验，⽐如把⽣成结果和原始⽂档⽚段做⼀次
entailment 判断，不⼀致就触发⼈⼯审核或 fallback 回答。

### Q057
**分类：** 其他
**题目：** 当需要处理超⻓⼤模型上下⽂窗⼝限制时，有哪些可⾏的⼯程解决⽅案？
**参考答案：** 处理超⻓上下⽂，本质是绕过或优化模型的 token ⻓度硬限制。直接喂满动辄 32K、128K 上下⽂，成本⾼且效果不⼀
定好，得靠⼯程⼿段拆解。
1）分块与索引检索
把⻓⽂档切⽚存⼊向量数据库，⽐如⽤ LangChain + Chroma/Pinecone，查询时只召回相关 chunk。关键在分块策
略，按段落切可能割裂语义，结合滑动窗⼝重叠（overlap）能缓解。但太碎了会丢失全局逻辑。
2）摘要增强（Summary Augmentation）
对历史对话或⽂档提前⽣成摘要，新请求到来时，把摘要当上下⽂注⼊。像 LLM-powered summarizer 在对话系统⾥
很常⻅，能把前 10 轮压缩成 1-2 句，省下⼤量 token。
3）位置插值（Position Interpolation）
针对 Transformer 架构的位置编码做⼿脚。原⽣ RoPE ⽀持外推但衰减快，采⽤ NTK-by-parts 或 YaRN 等⽅法，在推
理时动态调整位置频率，让模型“脑补”超出训练⻓度的位置信息。实测能在 32K 模型上跑 64K 输⼊，效果接近原⽣
⻓上下⽂。
4）侧边缓存（Side Cache）
Google 提出的思路，把已计算的 KV 缓存复⽤起来。⽐如⽤户修改了 prompt 中⼀⼩段，不需要全量 recompute，命
中部分直接从 cache 读。适合交互式场景，降低延迟和算⼒消耗。
5）分层上下⽂路由
复杂系统如 Agent ⼯作流，⽤调度层决定哪些信息进⻓期记忆（vector store），哪些放短期上下⽂。AutoGPT 类项⽬
就这么⼲，核⼼状态放 context，细节查数据库。

### Q058
**分类：** 其他
**题目：** 提⽰词优化考虑哪些维度？你们提⽰词模板有哪些字段？
**参考答案：** 这问题其实是在问提⽰词⼯程的系统化设计思路，不只是随便写句话。
1）⾸先看维度。⼀个有效的提⽰词必须明确任务⽬标、输⼊约束和输出格式。任务类型决定结构，⽐如分类、⽣成、
抽取需要的模板完全不同。上下⽂信息也得给⾜，模型才能理解语义边界。还要考虑安全合规要求，像敏感词过滤、
价值观对⻬这些都得前置控制。
2）再说模板字段。我们常⽤的提⽰词模板包含这⼏个关键部分：⻆⾊定义，说明这个 prompt 是给谁⽤的；背景描
述，交代清楚业务场景；指令主体，这是核⼼，告诉模型具体要做什么；⽰例样本，few-shot 的例⼦能显著提升准确
率；输出约束，⽐如 JSON 格式、字数限制、枚举值范围；最后是异常处理机制，定义模糊输⼊时的降级策略。
代码⽰例⻓这样：
prompt = """
⻆⾊：电商客服助⼿
背景：⽤户在商品详情⻚发起咨询
指令：根据商品信息和⽤户问题，⽣成专业且友好的回复
输⼊：
- 商品标题： {title}
- ⽤户问题： {query}
输出格式： JSON ，包含  reply 字段，不超过  150 字
"""
这种结构能让⼤模型稳定输出符合预期的结果，⽽不是靠运⽓。

### Q059
**分类：** 工程与场景
**题目：** 当⼤模型API响应延迟超过1秒时，前端可以采取哪些优化策略保证⽤户体验？
**参考答案：** 前端能做的其实很有限，核⼼是“让⽤户感觉快”，⽽不是真的让模型变快。⼤模型API本⾝耗时主要在服务端推理，
前端更多是体验层的兜底。
1）请求发出去之后，⽴刻给⽤户反馈，别⼲等。⽐如⻢上展⽰⼀个“思考中...”的动画或者⻣架屏，告诉⽤户系统已
经在处理了，避免误操作重复提交。
2）如果接⼝超过800ms还没回来，可以主动提⽰“⽣成时间较⻓，请稍候”。这种预期管理很重要，⽤户知道要等，
就不会觉得卡死了。
3）优先考虑流式响应（streaming）。让后端以SSE或WebSocket⽅式分段返回结果，前端拿到⼀段就渲染⼀段。就像
ChatGPT那样逐字输出，视觉上⽐等1秒再刷出来舒服得多。
4）结合loading状态和取消按钮。允许⽤户在等待时⼿动中断请求，提升控制感。特别是移动端，⻓时间⽆响应容易

### Q065
**分类：** 微调与 PEFT
**题目：** 常⻅的微调⽅法有哪些？
**参考答案：** 微调本质上是拿预训练模型在特定任务上做⼆次训练，让模型更适配具体场景。常⻅的⽅法选择得看数据量、算⼒和
任务复杂度。
1）全量微调就是把整个模型的参数都放开训练。效果通常最好，但显存消耗⼤，8 张 A100 都不⼀定扛住 Llama 2 7B

# LoRA 核⼼思想： A 下降维度， B 升维，中间做低秩分解
lora_weights = (x @ A) @ B

### Q066
**分类：** 微调与 PEFT
**题目：** PEFT 是什么？为什么需要 PEFT？
**参考答案：** PEFT 全称是 Parameter-Eﬃcient Fine-Tuning，直⽩点说就是“只动⼀⼩部分参数来微调⼤模型”。现在⼤模型动不
动就上百亿、上千亿参数，⽐如 Llama、ChatGLM 这种，直接全量微调成本太⾼了，显存、算⼒、时间都吃不消。
其实我们发现，微调的时候没必要更新所有参数。PEFT 的思路是冻结原模型绝⼤部分权重，只训练少量额外引⼊或选
中的参数，就能达到接近全量微调的效果。这样显存能省下 70% 以上，训练速度也快很多，普通⼏张卡甚⾄单卡也能
搞。
常⻅的 PEFT ⽅法⾥，LoRA 是最⽕的。它在原始层旁边加个低秩矩阵做增量，训练时只更新这个⼩矩阵。⽐如⼀个
7B 模型，⽤ LoRA 可能只改 0.1% 的参数，效果却不差。HuggingFace 的 peft  库已经⽀持得⾮常好，配合
transformers 能轻松上⼿。
代码⻓这样：
from peft import LoraConfig, get_peft_model
lora_config = LoraConfig(
r=8,
lora_alpha=16,
target_modules=["q_proj", "v_proj"],
lora_dropout=0.1,
bias="none",
task_type="CAUSAL_LM"
)
model = get_peft_model(model, lora_config)
除了 LoRA，还有 Adapter、Preﬁx-tuning 等⽅法，但 LoRA 因为⽆推理延迟、实现简单，成了主流选择。特别是资源
有限⼜想定制⼤模型的场景，⽐如企业私有模型微调，PEFT ⼏乎是标配⽅案。
参数⾼效微调（PEFT）的核⼼思路是什么？列举 3 种典型⽅法
参数⾼效微调的核⼼思路是只微调模型中极⼩⼀部分参数，冻结绝⼤部分原始参数，从⽽在保持下游任务性能的同
时，⼤幅降低计算和存储开销。
⼤模型动辄上百亿参数，全量微调需要⼏⼗甚⾄上百 GB 显存，普通团队根本搞不定。PEFT 的做法是“以⼩博⼤”，
通过引⼊少量可训练参数来适配新任务，⽐如只调 0.1%~1% 的参数就能达到接近全量微调的效果。
1） LoRA（Low-Rank Adaptation）
在原始权重旁注⼊低秩矩阵，训练时只更新这些⼩矩阵。推理时可以合并到原权重中，完全不增加推理延迟。
HuggingFace Transformers 已深度集成，现已成为主流选择。
2） Adapter
在 Transformer 模块间插⼊⼩型前馈⽹络（通常两层 MLP），原模型冻结，只训练插⼊的模块。虽然效果好，但会增
加推理延迟，因为每次前向都要⾛额外⽹络。
3） Prompt Tuning / Preﬁx Tuning
通过可学习的“软提⽰”向量引导模型输出，本质是把任务建模成“填提⽰”的过程。Preﬁx Tuning 把这些向量拼接
在每⼀层输⼊前，Prompt Tuning 只加在第⼀层。节省参数，但对初始化敏感，⼩模型上容易不稳定。

### Q067
**分类：** 微调与 PEFT
**题目：** PEFT 和全量微调的区别？
**参考答案：** PEFT 本质上是解决⼤模型微调成本太⾼的问题。全量微调要更新整个模型所有参数，显存和计算资源消耗巨⼤，⽐如
⼀个 13B 的模型，光梯度存储就得上百 GB 显存。
⽽ PEFT 的思路是冻结原模型⼤部分参数，只训练⼀⼩部分新增或改造的模块。这样⼀来，显存占⽤能降到原来的
1/10 甚⾄更低，普通单卡也能跑。
1）全量微调：每个参数都参与更新，效果理论上最好，但训练慢、成本⾼，需要完整的优化器状态和梯度保存。适合
有充⾜算⼒且对性能极致要求的场景，⽐如公司级训练集群搞通⽤能⼒升级。
2）PEFT ⽅法：以 LoRA 为例，它在原始权重旁并⾏注⼊低秩矩阵，训练时只更新这些⼩矩阵。假设原始权重是 $W
\in \mathbb{R}^{768\times 768}$，LoRA 拆成两个⼩矩阵 $A \in \mathbb{R}^{768\times r}$ 和 $B \in
\mathbb{R}^{r\times 768}$，其中 $r$ 通常设为 8 或 16。这样可训练参数数量从⼏⼗亿骤降到百万级。
// 伪代码⽰意  LoRA 注⼊
Matrix W = loadOriginalWeight(); // 冻结
Matrix A = randomInit(768, 8);   // 可训练
Matrix B = randomInit(8, 768);   // 可训练
Matrix loraOutput = input * (W + A * B); // W 不更新
常⻅变体还有 Adapter Tuning、Preﬁx Tuning 等，但 LoRA 因其不改变推理结构、兼容性好成为主流。HuggingFace
的 peft  库已经⽀持多种模式，配合 transformers  ⼏⾏代码就能上。
效果上，多数任务中 LoRA 能达到全量微调 95% 以上的性能，但对强推理或复杂指令遵循任务可能略有差距。关键是
省了太多资源，让 7B/13B 模型在消费级 GPU 上也能快速迭代。

解。
2）再看模型规模和你的算⼒能不能对上。7B 参数的模型⽤单卡 A100 能跑，但如果是 70B，就得考虑模型并⾏或者直
接上云了。⼀般情况下，中⼩公司做垂直场景微调，7B 左右的模型性价⽐最⾼，显存压得下来，训练时间也扛得住。
3）还要看预训练数据的时间范围和领域。⽐如你要做⾦融研报⽣成，最好选那些在财经⽂本上持续预训练过的模型，
⽽不是通⽤语料训完就停的。有些开源模型会标明训练数据来源和时间，这点要仔细查⽂档。
4）最后看微调⽣态⽀不⽀持。像 Hugging Face 上有⼤量基于 LLaMA 系列的 LoRA 微调⼯具链，如果你选了⼀个冷⻔
模型，可能连适配的 Trainer 都没有，脏活累活都得⾃⼰写。

from transformers import AutoTokenizer, AutoModelForCausalLM

换模型时路径⼀改就⾏，但背后的数据和架构差异决定了效果上限。

### Q069
**分类：** 微调与 PEFT
**题目：** 微调中常⽤的优化器有哪些？
**参考答案：** 微调⼤模型时选优化器，关键得平衡收敛速度和训练稳定性。常⻅的⼏种其实都源⾃梯度下降的改进思路。
1）SGD with Momentum 算是⽼前辈了，加个动量项能让参数更新不那么容易在平坦区域震荡。但学习率敏感，调不
好容易不收敛，现在⼤模型微调⽤得少了。
2）Adam 最常⻅，⼏乎成了默认选项。它⾃适应调整每个参数的学习率，对稀疏梯度特别友好。BERT、RoBERTa 这
些预训练模型做下游任务微调时，基本都拿 Adam 打底。不过内存开销⼤，因为要存每个参数的⼀阶⼆阶梯度。
// PyTorch 伪代码⽰意，实际不⽤  Java 写训练
optimizer = torch.optim.Adam(model.parameters(), lr=2e-5)
3）AdamW 是 Adam 的升级版，把权重衰减从梯度更新⾥拆出来单独算，解决了 Adam 在正则化上的理论缺陷。
Hugging Face Transformers ⾥默认推荐的就是 AdamW，像微调 BART 或 T5 都这么配。
4）Lion 是较新的选择，Google 提出来的，只⽤⼀阶动量，内存⽐ Adam 少⼀半，⽽且在某些 NLP 任务上效果更
好。但它对学习率更敏感，需要更精细地调参。
总的来说，AdamW 因为稳定性和效果兼备，成了当前主流选择。⼩规模实验可以试试 Lion 看能不能提速，⽼项⽬可
能还在⽤ Adam。SGD 基本只出现在特定场景，⽐如某些视觉模型微调还会⽤。

### Q071
**分类：** 微调与 PEFT
**题目：** 什么是低秩适配（LoRA）技术？如何结合 LoRA 技术进⾏微调？
**参考答案：** 低秩适配（LoRA）的核⼼思路是，⼤模型微调时参数量太⼤，训练慢、显存吃紧，⼲脆不直接改原始权重，⽽是⽤低
秩矩阵去近似增量。
模型原本的权重矩阵 $W$ 是个⼤胖⼦，⽐如 768×768。LoRA 不动它，转⽽引⼊两个⼩矩阵 $A$ 和 $B$，维度分别是
$768×r$ 和 $r×768$，其中 $r$ 很⼩，⼀般取 8 到 64。训练时只更新这两个⼩矩阵，前向传播时等效注⼊ $\Delta W
= A \times B$。
这样显存和计算量就从和原矩阵成正⽐，变成和 $r$ 相关。⽐如 r=8，参数量直接降到原来的 1/96，训练速度翻倍都
不⽌。
结合 LoRA 做微调，流程也简单：
1）冻结原始模型所有权重
2）在指定层（⽐如注意⼒中的 Q、V 投影）插⼊ LoRA 模块
3）只训练新增的⼩矩阵，其余参数不动
4）推理时把 $A \times B$ 加到原权重上，或者直接合并进原权重
主流框架都有⽀持。Hugging Face 的 peft  库⼏⾏代码就能上：
from peft import LoraConfig, get_peft_model
lora_config = LoraConfig(
r=8,
lora_alpha=16,
target_modules=["q_proj", "v_proj"],
lora_dropout=0.1,
bias="none",
task_type="CAUSAL_LM"
)
model = get_peft_model(model, lora_config)
适合资源有限、多任务切换的场景。像微调⼀个 7B 模型，⽤ LoRA 能在单卡 24G 显存下搞定，否则根本搞不定。

### Q072
**分类：** 微调与 PEFT
**题目：** 微调的过拟合⻛险如何通过正则化缓解？
**参考答案：** 微调时模型已经⻅过⼤量训练数据，参数调整幅度⼩但密集，容易记住训练集的特定模式，导致在新数据上表现下
滑。正则化的核⼼作⽤就是在损失函数或训练过程中加⼊约束，让模型别学得太“死”。
1）L2 正则（权重衰减）是最常⽤的⼿段。它在损失函数⾥加上所有权重的平⽅和，迫使参数值趋向更⼩。⽐如
Hugging Face 的 Trainer  ⾥设置 weight_decay=0.01 ，就能防⽌某些权重变得过⼤，提升泛化能⼒。
training_args = TrainingArguments(
output_dir="output",
weight_decay=0.01
)
2）Dropout 在微调阶段依然有效。虽然预训练模型内部已经固化了结构，但在分类头甚⾄最后⼏层中间加 Dropout，
能让输出不依赖于少数神经元。⼀般保留率设在 0.1〜0.3 就够了。
3）早停（Early Stopping）特别适合微调场景。因为微调轮次少，可能只跑 3〜5 个 epoch，验证集性能⼀旦下降就
终⽌，能避免多⾛⼀两步掉进过拟合坑⾥。
4）还有像 标签平滑（Label Smoothing）这种软化监督信号的⽅法。它不让模型对真实标签过于⾃信，间接抑制过拟
合。在分类任务中设 label_smoothing_factor=0.1  是常⻅做法。
这些⽅法本质都是给学习过程加点“不确定性”或“惩罚”，让模型别把微调当背书，稍微收着点劲。

### Q073
**分类：** 微调与 PEFT
**题目：** 请详细讨论微调时如何防⽌灾难性遗忘问题？
**参考答案：** 微调⼤模型时，灾难性遗忘的本质是模型参数被新任务数据剧烈调整，导致旧知识的表征能⼒崩塌。关键不是“记住
所有旧数据”，⽽是让模型在适应新任务的同时，保留对原始分布的泛化能⼒。
1） 降低学习率是最直接的⼿段。预训练阶段⽤了海量数据，参数已经收敛到⼀个较优的全局位置。微调时如果⽤⼤学
习率，梯度更新会把参数推出这个区域。⼀般建议微调学习率设为预训练的 1/10 到 1/100，⽐如从 3e-4 降到 3e-5 或
1e-5。
2） 使⽤ Adapter 模块是⼀种参数隔离策略。只在 Transformer 层间插⼊⼩型神经⽹络，冻结原始主⼲参数。训练时
梯度只流经这些⼩模块，主⼲权重压根不更新。这样既适配了新任务，⼜完全保留了原模型的知识结构。
HuggingFace 的 PEFT 库就⽀持这类⽅法。
3） LoRA（低秩适应）更轻量。它不加新层，⽽是给注意⼒权重矩阵注⼊低秩分解矩阵。假设原始权重是 W，训练的
是 ΔW = BA（A∈R^{r×d}, B∈R^{d×r}），r 远⼩于 d（⽐如 r=8, d=4096）。更新时 W' = W + α * BA，主⼲参数不变，
只训练 A 和 B。这种⽅法显存占⽤少，推理还能合并权重，适合⼤规模部署。
4） 数据层⾯可以混合⼀定⽐例的通⽤语料。⽐如每 10 个微调样本中加⼊ 1 个来⾃原始预训练分布的句⼦，让梯度⽅
向不⾄于完全偏离。不过要⼩⼼引⼊噪声影响⽬标任务效果。
这些⽅法的核⼼思想都是限制参数搜索空间，不让优化过程肆意修改已学好的特征提取器。

### Q074
**分类：** 微调与 PEFT
**题目：** 在多模态微调（如图⽂⽣成）中，如何确保⽂本和图像数据的对⻬质量？
**参考答案：** 多模态微调⾥，⽂本和图像的对⻬质量直接决定模型能不能准确理解“图在说什么，话在描述谁”。如果数据没对⻬，

束，对⻬更稳。

### Q075
**分类：** 微调与 PEFT
**题目：** 参数⾼效微调（PEFT）如何减少计算成本？
**参考答案：** 参数⾼效微调的核⼼思路是，不碰预训练模型的⼤部分权重，只动其中⼀⼩撮关键参数。这样⼀来，显存占⽤和计算
量都⼤幅下降。
1）冻结主⼲：把⼤模型的绝⼤多数参数锁死，⽐如 LLaMA 的 70 亿参数⾥，可能只放开不到 1% 去更新。剩下的全当
静态特征提取器⽤。
2）引⼊可训练模块：像 LoRA 这种⽅法，会在注意⼒层的权重旁加个低秩矩阵做增量。训练时只优化这个⼩矩阵，原
始权重压根不参与梯度更新。假设原矩阵是 $ \mathbb{R}^{5120 \times 5120} $，LoRA 只学两个 $
\mathbb{R}^{5120 \times 8} $ 的⼩矩阵，参数量从千万级⼲到⼏万。
3）适配器注⼊：在 Transformer 层之间塞个⼩的前馈⽹络模块，叫 Adapter。它通常就⼀两层，宽度也⼩，训练时只
调这部分，其他照旧。像 BERT 这类模型⽤ Adapter 微调，能省下 70% 以上的训练资源。
4）前缀微调：给输⼊前⾯挂⼀串可学习的虚拟 token，叫 preﬁx。训练时只更新这些 token 的 embedding，模型主
体完全不动。T5、GPT 这类⾃回归模型常⽤这招。
这些⽅法都能做到单卡微调⼤模型。⽐如⽤ LoRA 调 LLaMA-2-13B，在 A100 上也能跑起来，显存直接从⼏⼗ GB 掉到
10G 内。

### Q076
**分类：** 微调与 PEFT
**题目：** 冻结层在微调中的作⽤是什么？
**参考答案：** 冻结层在微调时，其实就是把预训练模型⾥的⼀部分⽹络参数固定住，不让它们更新。你只让最后⼏层或者指定的层
去学习新任务的数据，这样能省下⼤量计算资源。
1）最常⻅的情况是⽤ BERT、ResNet 这类⼤模型做下游任务。⽐如你拿 ImageNet 上训好的 ResNet50 去做猫狗分

⼏⼗个 epoch 就能收敛，显存和时间都扛得住。
2）另⼀个关键是防⽌过拟合。你的⼩数据集可能就⼏千张图，如果放开所有层去更新，模型很容易记住噪声，泛化能
⼒直接崩掉。冻结⼤部分层相当于限制模型容量，逼它只调整⾼层语义适配新任务。
3）也有分阶段微调的做法：先冻结主⼲训 head，再解冻部分层⽤更⼩学习率细调。像 Hugging Face 的
Transformers 库⾥，.requires_grad_()  控制起来也就⼀⾏代码的事。
# 冻结  BERT base 层
model = BertForSequenceClassification.from_pretrained('bert-base-uncased')
for param in model.bert.parameters():
param.requires_grad = False  # 梯度不计算，节省内存
这种策略在 NLP 和 CV ⾥都成了标配，尤其是数据少、⼜想⽤⼤模型的时候，基本靠它稳住局⾯。

### Q077
**分类：** 微调与 PEFT
**题目：** 为什么需要混合精度训练？
**参考答案：** 混合精度训练解决的是深度学习训练中的计算效率与显存占⽤问题。现代GPU如A100、V100都配备了Tensor Core，
专⻔为半精度浮点数（FP16）运算优化，单次可处理更多数据，算得更快。
1）FP16相⽐FP32占⽤带宽和显存少⼀半，同样的batch size能跑更⼩的显存，或者在显存不变时增⼤batch size提升
训练稳定性。
2）但全⽤FP16会出问题，⽐如梯度下溢、权重更新精度不够导致模型不收敛。所以得保留⼀部分关键计算⽤FP32，
⽐如梯度累加、权重更新、loss缩放。
3）实际做法是：前向和反向传播⽤FP16加速，参数副本保持⼀份FP32的主拷⻉⽤于更新，这就是AMP（Automatic
Mixed Precision） 的核⼼机制。PyTorch⾥⼀⾏scaler = torch.cuda.amp.GradScaler() 就能开启。
像BERT、ResNet这类⼤模型训练，基本都默认开启混合精度。NVIDIA的Apex库和原⽣AMP都⽀持得很好。不⽤的
话，等于⽩⽩浪费30%以上的训练速度。

### Q079
**分类：** 微调与 PEFT
**题目：** SFT 指令微调数据如何构建？
**参考答案：** SFT（Supervised Fine-Tuning）指令微调数据的构建，本质是把模型要学的任务表达成“指令-输⼊-输出”的三元组

1）⼀条标准样本包含三个部分：instruction 是具体任务描述，⽐如“翻译成英⽂”；input 是待处理内容，可以为

3）质量⽐数量重要。避免重复、歧义或错误标注。建议做分层设计：通⽤能⼒、领域知识、业务特有逻辑各占⼀定⽐
例。⼀般⼏千到⼏万条⾼质量样本就能起效，上百万条反⽽可能引⼊噪声。
4）最后要做格式归⼀和去重。统⼀ prompt 模板⻛格，过滤低信息密度样本，⽐如全是停⽤词或模板化回复的数据。
整个过程其实是为模型“造考场”，题⽬得能考出真本事。

### Q080
**分类：** 微调与 PEFT
**题目：** 请解释⼤模型微调(Fine-tuning)的原理，并说明在什么业务场景下需要微调⽽不是直接使⽤基础模型？
**参考答案：** 微调的本质是参数⾼效迁移，不是从头训练。⼤模型已经在海量通⽤语料上学到了语⾔规律和世界知识，微调是在这
个基础上，⽤特定领域的数据去调整模型参数，让它适应新任务。
1）你拿到⼀个预训练好的⼤模型，⽐如 LLaMA 或 Qwen，它的“常识”已经很完整。但如果你要做⾦融研报⽣成，

2）微调就是拿⼀批⾦融报告数据，⽤有监督的⽅式继续训练。⽐如输⼊“某公司营收增⻓30%”，让模型输出对应的
研报段落。反向传播会更新权重，让模型在该任务上表现更好。
3）关键在于，并不是所有层都要重训。现在很多做法是只微调部分参数，⽐如 LoRA（Low-Rank Adaptation），冻结
原模型权重，只训练低秩矩阵。这样显存占⽤少，训练快，适合业务迭代。

输出格式/⻛格强约束，⽐如法律⽂书、医疗诊断记录
领域知识密集，基础模型没⻅过的专业术语体系
企业私有数据不能⾛ API，必须本地部署并定制
如果只是简单问答、摘要⽣成，且对表达⻛格要求不⾼，那其实⽤提⽰词+RAG就够了。微调成本⾼，周期⻓，真要上
得想清楚是不是⾮⽤不可。

### Q081
**分类：** 微调与 PEFT
**题目：** 指令微调的好处？
**参考答案：** 指令微调其实是让预训练模型更会“听⼈话”的关键⼀步。模型在⼤规模数据上预训练完，虽然学到了丰富的语⾔知
识，但并不清楚⽤户到底想让它⼲什么。这时候通过指令微调，⽤成对的“指令-期望输出”数据去微调，它就能学会
理解任务意图。
1）模型能准确理解任务形式。⽐如给⼀条指令“把下⾯这句话翻译成英⽂”，模型知道接下来要处理的是翻译，⽽不
是摘要或改写。这种泛化能⼒在多任务场景下特别有⽤。
2）减少对⽰例的依赖。没有指令微调的模型，往往需要上下⽂⾥给出⼏个例⼦（few-shot）才能⼲活，⽽微调后的模
型可以直接响应指令，输⼊更简洁。
3）⾏为更可控。你可以通过微调引导模型输出特定格式，⽐如 JSON、有序列表，或者避免⽣成某些敏感内容。像
Alpaca、Dolly 这些开源模型，都是基于 LLaMA 做了指令微调才变得可⽤。
举个简单例⼦，微调前你问“写个快排”，它可能只解释原理；微调后它⼤概率直接给你⼀段可运⾏的代码。
不过也别指望它能完全替代⼯程逻辑。指令微调解决的是“意图对⻬”，不是“能⼒增强”。模型本⾝不具备的能⼒，
⽐如数学推理、代码执⾏，光靠指令微调是搞不定的。

### Q082
**分类：** 未分类
**题目：** 如果一个GPU集群的LLM处理能力为1000tokens/s，那1000个用户同时并发访问，响应给每个用户的性能只有1 token/s吗？怎么分析性能瓶颈
**参考答案：** 不⼀定每个⽤户就只能分到 1 token/s，实际性能取决于调度策略、批处理能⼒和请求的负载特征。
⼤模型服务不像传统 Web 服务器那样“⼀个请求占⼀个线程”，它⾛的是 动态批处理（Dynamic Batching） 路⼦。
⽐如⽤ vLLM、TensorRT-LLM 这类推理框架时，多个⽤户的请求会被合并成⼀个 batch ⼀起跑，显存和计算单元利⽤
率更⾼。
**关键点**在于：
1）如果这 1000 个⽤户都是刚发请求，还没开始⽣成内容（preﬁll 阶段为主），那系统可以⼀次性把他们的 prompt 做
batch 处理，吞吐可能接近峰值 1000 tokens/s，平均下来每⼈⼏⼗甚⾄上百 token/s 都有可能。
2）但到了 decode 阶段，每个请求要逐 token ⽣成，且⻓度不⼀，这时候 batch size 会慢慢被“拖慢”。⻓⽂本、⾼
采样参数（如 temperature ⾼）的请求会拉低整体速度。
3）真正的瓶颈往往不是算⼒，⽽是显存带宽和 KV Cache 管理。每个并发请求都要保留 KV Cache，1000 个并发意味
着巨⼤的显存压⼒，搞不定就会 OOM 或被迫丢请求。
举个例⼦，A100 显存 80GB，跑 Llama3-8B，每个请求平均占 2GB KV Cache，理论上最多撑 40 个并发。超过就得靠
PagedAttention（像 vLLM 实现的）来分⻚管理，否则压根不经过计算就挂了。
所以性能瓶颈通常出现在：
KV Cache 显存占⽤过⼤
⼩ batch 或不规则请求导致 GPU 利⽤率上不去
请求排队时间远超⽣成时间
优化⽅向是：⽤ PagedAttention 降低显存压⼒，启⽤连续批处理（Continuous Batching），控制最⼤⽣成⻓度，避
免⼩请求拖垮整体延迟。

### Q083
**分类：** 未分类
**题目：** 向量数据库原理是什么？请简述下它的原理
**参考答案：** 向量数据库本质是为⾼维向量数据设计的存储与检索系统，核⼼⽬标是在亿级向量中快速找到和查询向量最相似的那
些。
传统数据库⽤B+树做精确匹配或范围查询，但向量相似性搜索要的是“近似”，⽐如余弦相似度、欧⽒距离。如果暴⼒

最近邻搜索（ANN）算法，在可接受精度损失下把查询从O(n)降到O(log n)甚⾄常数级。
常⻅实现⽅式有⼏种：
1）基于局部敏感哈希（LSH），把相近向量以更⾼概率映射到同⼀个桶⾥，适合⾼维稀疏场景，但召回率波动⼤；
2）基于树结构如随机投影树，划分空间，但⾼维下效果退化明显；
3）最主流的是图-based⽅法，⽐如HNSW（Hierarchical Navigable Small World），构建多层导航图，顶层稀疏快速
定位⽅向，底层精细搜索，能在毫秒内完成百万级查询，Milvus、Pinecone都在⽤。
另外，向量数据库通常会结合标量索引做混合查询，⽐如“找和这个图⽚相似且上传时间在最近7天的”。这就需要⽀
持结构化字段过滤 + 向量检索的联合优化。

清洗，再上⼀个 TinyBERT 做语义对⻬，转成标准表述“⽆法登录系统”。这步别上⼤模型，响应延迟扛不住。
2）核⼼分类环节看数据量和类⽬复杂度。如果只有⼏⼗个固定类别，RoBERTa + CRF ⾜够，推理快、好维护。要是类
⽬动态增⻓，⽐如每天新增业务线，就得考虑 Prompt Learning 配合 P-tuning v2，接通⼤模型如 ChatGLM-6B 做
zero-shot 分类，类⽬⼀改prompt跟着变，不⽤重新训练。
3）后置反馈闭环必须做。分错的⼯单要进⼈⼯复核池，这部分数据定期回流做增量训练。可以搭个轻量 Label Studio
+ Milvus 相似样本检索，⾃动推荐历史相似案例给标注员，效率翻倍。
4）⼯程部署上，⼩模型⾛ TensorFlow Lite 或 ONNX Runtime 嵌⼊服务内部，⼤模型⽤ vLLM 做批处理异步推理。别
把所有请求都怼到⼤模型上，成本直接炸。
