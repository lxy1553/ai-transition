# Day 25 - RAG 权限与敏感信息安全控制清单

这份清单用于检查 RAG、数据问答或 NL2SQL 项目是否具备基础安全控制能力。

## 1. 数据分级

| 检查项 | 要求 | 风险 |
|--------|------|------|
| 是否识别公开、内部、敏感、机密数据 | 每份文档和字段要有安全等级 | 不分级会导致所有资料被同等召回 |
| 是否识别 PII 信息 | 手机号、身份证、邮箱、地址等要单独标记 | 可能在答案或引用中泄露个人信息 |
| 是否识别业务敏感数据 | 薪酬、客户名单、合同、成本、利润等要标记 | 可能泄露公司核心经营信息 |

## 2. 入库前控制

| 检查项 | 要求 | 风险 |
|--------|------|------|
| 入库前是否脱敏 | 敏感字段进入知识库前要脱敏或不入库 | 检索命中后可能直接暴露原文 |
| 是否记录来源和版本 | 文档、chunk、向量都要保留来源 | 出错后无法追溯 |
| 是否记录权限标签 | chunk metadata 要包含部门、角色或数据域 | 检索时无法按用户权限过滤 |

## 3. 检索时控制

| 检查项 | 要求 | 风险 |
|--------|------|------|
| 是否先鉴权再检索 | 用户身份和角色要参与检索过滤 | 无权限资料可能进入上下文 |
| 是否按 metadata 过滤 | 根据部门、业务域、安全等级过滤 chunk | 模型可能看到不该看的内容 |
| 是否限制高敏资料召回 | 高敏资料默认不进入普通问答链路 | 引用和答案可能泄露敏感信息 |

## 4. 生成时控制

| 检查项 | 要求 | 风险 |
|--------|------|------|
| Prompt 是否限制只基于引用回答 | 不允许模型基于常识补全敏感信息 | 容易产生越权推断 |
| 是否对输出做脱敏扫描 | 答案返回前再次检查手机号、邮箱、密钥等 | 生成层可能拼出敏感内容 |
| 是否支持拒答和澄清 | 无权限、无依据、问题模糊时不能硬答 | 容易出现幻觉或合规风险 |

## 5. 审计与评测

| 检查项 | 要求 | 风险 |
|--------|------|------|
| 是否记录 request_id | 每次问答要能回放问题、用户、召回和答案 | 用户反馈后无法定位问题 |
| 是否记录命中的 citations | 记录文档、chunk、位置、版本和权限标签 | 无法排查越权或错误引用 |
| 是否有安全测试集 | 覆盖敏感、越权、无答案、危险操作样本 | 改规则后不知道是否误放行 |

## 面试一句话

RAG 安全不是只在 prompt 里写一句“不要泄露敏感信息”。
生产里要从数据分级、入库脱敏、权限标签、检索过滤、输出扫描、拒答策略和审计评测多层控制。
