# 代码注释规则

> 用途：以后写新代码、修改旧代码、复盘旧项目时，都按这个标准补注释。

> 原则：注释要帮助理解“为什么这样写、这段代码解决什么问题”，不能只写“怎么用”。

---

## 1. 总规则

代码注释必须用大白话，优先解释用途、业务含义和设计原因。

好的注释要回答：

- 这段代码是干什么的？
- 为什么这里要这样处理？
- 它在整个项目链路里起什么作用？
- 这里如果写错，可能会造成什么问题？

不要只写：

- 这个函数怎么调用
- 这个参数是什么类型
- 这一行代码表面上做了什么

代码本身已经能看出来的内容，不要重复注释。

---

## 2. 必须写注释的地方

以下位置必须补大白话注释：

| 位置 | 注释重点 |
|------|----------|
| 项目入口 | 说明这个脚本在项目里负责哪一步 |
| 核心函数 | 说明函数用途、输入输出和为什么需要它 |
| 业务规则 | 说明规则背后的业务原因 |
| SQL / RAG / NL2SQL 逻辑 | 说明这一步在数据问答链路里的作用 |
| 异常处理 | 说明为什么这样兜底，避免什么线上问题 |
| 配置项 | 说明配置影响什么行为 |
| 不直观的取舍 | 说明为什么先用简单方案，而不是复杂方案 |

---

## 3. 注释写法

### 推荐写法

```python
def normalize_question(question: str) -> str:
    """把用户问题统一成更稳定的格式，方便后面做检索和缓存。

    这里不是改变用户想问什么，而是去掉多余空格、统一大小写。
    如果不做这一步，同一个问题的不同写法可能命中不同缓存。
    """
```

```python
# 先按权限过滤，再把 chunk 交给模型。
# 这样可以避免用户无权访问的内容进入上下文，防止模型间接泄露敏感资料。
visible_chunks = filter_chunks_by_permission(chunks, user)
```

### 不推荐写法

```python
# 去掉空格
question = question.strip()
```

```python
# 调用函数
result = retrieve(query)
```

这些注释只重复代码表面动作，对理解用途帮助不大。

---

## 4. 旧代码补注释规则

旧代码不做无差别批量加注释。

以后只要修改旧文件，就顺手补以下内容：

1. 文件顶部补一句：这个脚本在项目里负责什么。
2. 核心函数补 docstring：说明用途、输入输出、为什么需要。
3. 复杂逻辑前补 1-3 行大白话注释：说明设计原因和风险。
4. 如果 README 只有运行命令，也要补“这个项目练什么、解决什么问题”。

---

## 5. 新代码提交前检查

每次新增代码前检查：

- [ ] 文件顶部能看出这个脚本用途。
- [ ] 核心函数有大白话 docstring。
- [ ] 业务规则说明了为什么要这样判断。
- [ ] 异常处理说明了要兜住什么问题。
- [ ] README 不只写用法，也写项目用途和生产映射。

---

## 6. 注释尺度

注释不是越多越好。

简单变量、普通赋值、明显的循环不用逐行解释。

真正需要解释的是：

- 业务规则
- 工程取舍
- 边界处理
- 安全控制
- RAG / NL2SQL 链路里的关键步骤

目标是让以后复盘时不用重新猜代码意图。
