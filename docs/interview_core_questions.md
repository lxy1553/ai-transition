## Q001：生产环境里的 RAG 系统一般怎么设计？
来源 Day：Day 15
### 回答
生产环境里的 RAG 通常不是一个简单的“向量检索 + 调模型”。
它一般会分成文档接入、解析清洗、切分、embedding、索引存储、在线检索、
权限过滤、重排、上下文构造、LLM 生成、引用返回和监控评测。
核心目标是让模型基于可信资料回答，并且答案可追溯、可控制、可优化。
我会把生产级 RAG 看成一套完整工程系统。离线侧负责知识库建设，
包括接入 PDF、Markdown、网页、数据库说明、指标文档等资料，然后做解析、清洗、
去重、脱敏、切分、embedding 和索引构建。在线侧负责用户问答，包括鉴权、
问题理解、query rewrite、向量检索、关键词检索、权限过滤、rerank、上下文拼装、
调用 LLM 生成答案，并返回引用来源。生产环境还要考虑监控和评测，
比如召回是否命中、回答是否基于资料、无答案率、延迟、成本和用户反馈。
所以 RAG 的难点不只是模型，而是资料质量、检索质量、权限安全和工程稳定性。

## Q002：为什么生产级 RAG 要分离线入库和在线问答两条链路？
来源 Day：Day 17
### 回答
因为文档处理和用户问答的节奏不同。离线入库负责提前把资料清洗、切分、
向量化并建好索引；在线问答负责快速检索、拼上下文和生成答案。
这样能降低在线延迟，也方便做知识库更新、权限管理和质量评估。
离线入库链路处理的是“知识怎么进入系统”，比如解析文档、抽取正文、
清洗噪声、去重、脱敏、按标题或语义切 chunk、生成 embedding，
并把向量、原文、来源、权限标签和版本信息写入索引。在线问答链路处理的是
“用户来了怎么回答”，比如鉴权、理解问题、检索候选片段、过滤无权限内容、
重排、构造 prompt、调用 LLM 并返回引用。如果每次用户提问时才临时处理文档，
系统会非常慢，也不好控制稳定性。分成两条链路后，离线侧可以批量加工和更新知识库，
在线侧可以专注低延迟、高质量和高可用。

## Q003：RAG 文档切分时怎么设计 chunk？
来源 Day：Day 16
### 回答
chunk 设计要平衡检索精度和上下文完整性。切得太大，召回不精准；
切得太小，容易丢上下文。生产里通常会按标题、段落、语义边界或业务对象切分，
并保留来源、层级、时间、权限等元数据，方便后续检索和引用。
chunk 不是简单按固定字数硬切。生产里要先看文档类型，比如制度文档适合按标题层级切，
接口文档适合按接口粒度切，指标文档适合按指标定义切，表结构文档适合按表或字段切。
还要考虑 overlap，避免答案跨段落时上下文断裂。每个 chunk 还应该带 metadata，
比如文档名、章节、更新时间、业务域、权限标签和原文位置。这样检索结果不仅能被模型使用，
也能返回给用户核对来源。对于数据问答和 NL2SQL，chunk 粒度尤其重要，
因为表、字段、指标口径如果被切散，模型就容易误解业务含义。

## Q004：向量检索、关键词检索和混合检索有什么区别？
来源 Day：Day 19
### 回答
向量检索适合找语义相近的内容，关键词检索适合找精确词、表名、字段名和编号。
混合检索会把两者结合起来，既能理解语义，又能保留精确匹配能力。
生产 RAG 一般更倾向混合检索，而不是只依赖一种检索方式。
向量检索通过 embedding 计算语义相似度，适合用户表达和文档表达不完全一致的场景。
比如用户问“授信通过率怎么算”，可以召回“approval_rate 指标定义”。
关键词检索更适合精确匹配，比如表名 `dws_credit_application_daily`、字段名 `dt`、错误码、
申请编号或合同编号。生产环境里只用向量检索可能漏掉精确实体，
只用关键词检索又可能处理不好同义表达。混合检索会同时跑向量检索和关键词检索，
再做融合排序或 rerank。这样在企业文档、数据字典、SQL 解释和 NL2SQL 场景里更稳。

## Q095：RAG 离线链路里的混合检索怎么设计，RRF 是什么？
来源 Day：Day 19
### 回答
RAG 里说混合检索，通常指同时使用向量检索和关键词检索。离线链路负责为两种检索方式准备好统一的知识索引，
在线链路再根据用户问题分别召回、融合和重排。这样设计是因为向量检索擅长语义相似，关键词检索擅长精确命中，
两者能互补。

离线侧一般会先做文档解析、清洗、去重、切分 chunk、抽取 metadata，然后给每个 chunk 分配稳定的 `chunk_id`。
同一批 chunk 会进入两类索引：一类是向量索引，用 embedding 做语义检索；另一类是 BM25 或倒排索引，
用关键词、表名、字段名、错误码、政策条款号做精确匹配。关键点是两边索引必须共享同一套 `chunk_id`、权限标签、
文档版本、业务域和来源信息，否则在线融合、去重、权限过滤和 citations 都会很难做。

为什么要混合，是因为只用向量检索容易漏掉精确实体。比如表名、字段名、合同号、错误码这类信息，
embedding 未必稳定；只用 BM25 又不懂同义表达，用户说法和文档说法不一致时容易漏召回。
生产里常见做法是：向量检索取 top 50，BM25 取 top 50，先按权限、版本和业务域过滤，再合并去重，
然后用融合算法或 reranker 得到最终 top-k。

混合方式主要有三种。第一种是分数加权，把向量相似度和 BM25 分数归一化后相加，例如
`final_score = 0.7 * dense_score + 0.3 * sparse_score`。但这个方法要小心，因为两种分数的尺度不同，
不归一化直接相加没有意义。第二种是候选集合并后交给 reranker，让 rerank 模型重新判断 query 和 chunk 的相关性。
第三种是 RRF，也就是 Reciprocal Rank Fusion，倒数排名融合。

RRF 不直接使用原始分数，而是使用每个文档在不同检索器里的排名。公式是：

```text
RRF_score(d) = sum(1 / (k + rank_i(d)))
```

其中 `rank_i(d)` 是文档 `d` 在第 `i` 个检索器里的排名，`k` 是平滑参数，常见取值是 60。
比如某个 chunk 在向量检索里排第 2，在 BM25 里排第 10，那么它的分数就是：

```text
1 / (60 + 2) + 1 / (60 + 10)
```

RRF 的好处是不用纠结向量分数和 BM25 分数是否可比，只看排名；如果一个 chunk 在多个检索器里都排得靠前，
它会被认为更可信。如果一个 chunk 只在某一种检索方式里靠前，也不会被直接丢掉。
所以在生产 RAG 里，一个比较稳的回答是：离线建统一 chunk、metadata、向量索引和 BM25 索引；
在线分别召回，用 RRF 或归一化加权做融合，再用 reranker 精排，最后取 top-k 拼上下文并返回 citations。

## Q005：rerank 在 RAG 里解决什么问题？
来源 Day：Day 19
### 回答
rerank 解决的是“召回结果不等于最终可用上下文”的问题。
第一阶段检索会拿到一批候选 chunk，但里面可能有噪声、重复或相关性不够的内容。
rerank 会重新排序，选出最适合回答当前问题的片段，减少无关上下文干扰模型。
RAG 一般会先用向量检索或混合检索召回一批候选结果，比如 top 20 或 top 50。
这个阶段追求召回率，宁可多拿一些，也不希望漏掉关键资料。但这些候选结果不一定都适合放进 prompt，
因为上下文窗口有限，无关内容还会干扰模型。rerank 的作用是基于用户问题和候选 chunk，
重新判断相关性，把真正能回答问题的内容排到前面。生产里可以用 rerank 模型，
也可以结合规则，比如优先同业务域、同权限范围、最新版本和高可信来源。
对于 RAG 效果优化来说，rerank 往往是提升回答质量的关键环节。

## Q007：RAG 为什么要做引用来源和可追溯？
来源 Day：Day 18
### 回答
因为企业场景里用户不只要答案，还要知道答案来自哪里。
引用来源可以让用户核对文档、章节、时间和版本，也方便排查错误。
没有引用的 RAG 回答很难建立信任，也很难定位是检索错、资料错还是模型生成错。
引用来源是生产 RAG 的基本能力。一个回答如果没有来源，用户很难判断它是基于公司文档，
还是模型自己推断出来的。引用可以包含文档名、章节、chunk 位置、更新时间和原文片段。
当回答错误时，团队可以根据引用判断问题在哪里：如果引用资料本身过期，
需要更新知识库；如果引用不相关，说明检索或 rerank 有问题；如果引用正确但回答错，
说明 prompt 或模型生成需要优化。在合规、财务、人事、数据口径和合同类场景里，
引用尤其重要，因为这些问题不能只给一个看起来合理的答案，必须能追溯依据。

## Q014：结构化输出为什么是 AI 应用里的关键能力？
来源 Day：Day 10
### 回答
因为生产系统不能只消费一段自然语言。下游程序需要稳定字段，
比如 `risk_level`、`tables`、`suggestions`、`can_publish`。
结构化输出让模型结果可以被校验、存储、路由、展示和继续处理。
真实 AI 应用里，模型输出通常不是给人直接看完就结束，而是要进入后续系统。
比如 SQL 解释助手需要输出表名、字段、风险等级、风险列表和修改建议；
RAG 系统需要输出答案、引用来源、置信度和是否拒答；客服系统需要输出意图、
工单类型和处理动作。如果只返回一段自然语言，程序很难稳定解析。
所以要通过 JSON schema、Pydantic 或业务协议约束输出结构。
结构化输出还能做自动校验，比如字段是否缺失、枚举值是否合法、类型是否正确。
这也是 Tool Use、工作流编排和前端展示的基础。

## Q015：JSON 合法和业务校验通过有什么区别？
来源 Day：Day 10
### 回答
JSON 合法只说明格式能被解析，比如括号、引号、逗号没错。
业务校验通过还要求字段完整、类型正确、枚举值合法，并且符合业务规则。
比如 `risk_level` 必须是固定枚举，`can_publish` 必须和风险等级一致。
JSON 合法是语法层面的概念，程序能 `json.loads` 解析就算合法。
但生产系统还需要业务层校验。比如 SQL 解释结果里必须有 `summary`、`tables`、
`risk_level`、`risks` 和 `suggestions`。其中 `risk_level` 不能随便写，
必须是 `low`、`medium`、`high` 这类固定枚举。`risks` 应该是数组，
每个风险最好有类型、说明和建议。更进一步，业务规则也要一致：
如果风险等级是 high，`can_publish` 就不能是 true。否则 JSON 虽然合法，
但下游系统会误判。生产里通常用 Pydantic、JSON Schema 或自定义规则做这层校验。

<a id="q016"></a>

## Q016：Tool Use 在生产 AI 应用里怎么设计？
来源 Day：Day 11
### 回答
Tool Use 的核心是让模型负责判断什么时候调用工具，让工具负责确定性执行。
生产里工具要有清晰入参、出参、权限、超时、错误处理和审计日志。
不能让模型直接操作数据库或系统，而是通过受控工具完成动作。
Tool Use 适合把 LLM 的理解能力和传统系统的确定性能力结合起来。
比如用户问 SQL 风险，模型可以调用 SQL parser 工具；问指标口径，
可以调用知识库检索工具；问数据结果，可以调用受控查询工具。
生产设计里，每个工具都要定义 schema，包括名称、用途、参数类型、返回结构和错误码。
工具执行前要做权限校验，执行时要有 timeout 和重试，执行后要记录日志。
LLM 不应该直接拼接任意 SQL 去数据库执行，而是只能调用经过限制的工具。
这样既能发挥模型的调度能力，又能把风险控制在工程边界内。

## Q017：Prompt 在生产项目里应该怎么管理？
来源 Day：Day 9
### 回答
生产里的 prompt 应该像配置和代码一样管理。要有版本、变量、模板、
输出格式约束、测试样例和变更记录。不能只把 prompt 写在代码里随便改，
否则出现回答变差时很难回滚和定位问题。
Prompt 在生产 AI 应用里是核心逻辑的一部分。一个好的 prompt 通常包含角色、
任务目标、上下文、约束、输出格式和异常处理要求。比如 RAG 场景要明确：
只能基于上下文回答，没有依据就说明无法判断，并返回引用来源。
NL2SQL 场景要明确：只能使用提供的表和字段，不能编造字段，生成后要输出解释。
工程上 prompt 要模板化，把用户问题、检索上下文、工具结果作为变量填入。
还要做版本管理和回归测试，避免一次 prompt 修改影响线上效果。
对于重要 prompt，最好保留变更记录、测试集结果和灰度发布策略。

## Q018：一次 LLM 调用需要关注哪些参数和工程问题？
来源 Day：Day 8
### 回答
要关注模型选择、messages、temperature、max tokens、timeout、重试、错误处理、
成本和日志。像 SQL 解释、RAG、NL2SQL 这类场景更重视准确和稳定，
通常会使用较低 temperature，并限制输出格式。
一次 LLM 调用不只是传一段 prompt。首先要选择合适模型，简单分类和结构化抽取可以用较小模型，
复杂推理或长上下文问答可能需要更强模型。messages 里要区分 system prompt、
user prompt 和检索上下文。temperature 控制随机性，生产里很多任务需要低 temperature，
避免输出漂移。max tokens 控制输出长度和成本。工程上还要设置 timeout、
重试、限流和异常处理，不能让模型调用拖垮接口。日志也很重要，
至少要记录请求 ID、模型、耗时、token、错误码和关键业务结果，
但不能把敏感数据明文打到日志里。

## Q019：FastAPI 在 AI 应用服务化里一般承担什么角色？
来源 Day：Day 5

比如提供 RAG 问答接口、SQL 解释接口、NL2SQL 接口和健康检查接口。
它结合 Pydantic 做请求校验和响应结构约束，适合快速构建 AI 应用后端。
在 AI 应用里，模型调用、检索、SQL 校验和工具执行通常都在 Python 生态里完成。

## Q020：AI API 服务为什么需要配置、日志和异常处理？
来源 Day：Day 6
### 回答
配置解决环境差异和密钥管理，日志解决排查和审计，异常处理解决服务稳定性。
AI 服务会调用模型、向量库、数据库和外部 API，出错点很多。
没有这些工程能力，服务很难稳定上线。
生产 AI API 服务不是只要本地能跑通。配置管理让不同环境使用不同参数，
比如模型名、API Key、数据库地址、向量库地址、timeout 和日志级别。
密钥不能写死在代码里，要通过环境变量或密钥管理系统注入。
日志用于排查问题和做审计，比如记录请求 ID、用户、耗时、调用链路、错误码和关键结果。
异常处理用于把底层错误转换成稳定响应，避免把堆栈、密钥或内部路径暴露给用户。
RAG 和 NL2SQL 的链路长，任何一个环节失败都要有清晰错误和降级策略。
这些能力决定服务能不能从脚本变成线上系统。

## Q021：调用外部 API 或 LLM API 时要怎么保证稳定性？
来源 Day：Day 4
### 回答
要设置 timeout、重试、限流、熔断、错误分类和日志。
还要处理 4xx、5xx、网络超时、返回格式异常和额度不足。
生产里不能假设外部 API 永远成功，要有失败兜底和可观测性。
调用外部 API 时，第一步是设置合理 timeout，避免请求一直卡住。
第二是做有限重试，通常只对网络抖动、超时、部分 5xx 做重试，
不要对参数错误或权限错误盲目重试。第三是错误分类，比如 401 是鉴权问题，
429 是限流问题，5xx 是服务端问题，JSON 解析失败是响应格式问题。
第四是日志和指标，记录请求耗时、状态码、错误类型和重试次数。
对于 LLM API，还要关注 token 成本、上下文长度、模型不可用和输出格式异常。
必要时要有降级策略，比如返回规则结果、提示稍后重试，或者切换备用模型。

## Q022：你的 SQL 解释助手在生产环境里能解决什么问题？

## Q023：为什么 SQL 风险检查不能完全依赖 LLM？
来源 Day：Day 13
### 回答
因为很多 SQL 风险是确定性规则，比如是否 `select *`、是否缺少 `where`、
是否缺少分区字段、是否包含危险语句。这些用程序判断更稳定、可测试。
LLM 更适合做解释和总结，不适合单独承担安全边界。
生产环境里，SQL 安全和资源风险不能完全交给 LLM 判断。
LLM 可能漏判、误判，也可能因为上下文不足编造字段含义。
而很多检查其实可以规则化，比如只允许 `select`，禁止 `drop`、`delete`、`update`，
检查是否有分区条件、是否有 limit、是否访问了黑名单表、是否跨域 join。
这些规则应该由确定性程序或 SQL parser 负责，并且可以写单元测试。
LLM 更适合基于规则结果生成自然语言解释，比如告诉用户为什么缺少分区会导致扫描成本高。
这种设计把安全边界放在规则和工具层，把表达能力交给模型，更符合生产要求。

## Q025：RAG 知识库更新和版本管理怎么做？
来源 Day：Day 17
### 回答
知识库要支持增量更新、版本记录、失效处理和回滚。
文档变更后要重新解析、切分、embedding，并更新索引。
同时要保留文档版本和更新时间，避免模型引用过期资料。

## Q027：RAG 知识入库和索引构建在生产环境里怎么设计？
来源 Day：Day 17
### 回答
生产级 RAG 入库不是简单把文档丢进向量库。
完整流程一般包括文档接入、解析、清洗去重、脱敏、chunk 切分、metadata 构建、
embedding 生成、向量索引构建和版本管理。这样在线检索时才能做权限过滤、
引用溯源、增量更新和质量排查。
我会把 RAG 入库看成离线数据处理链路。首先接入原始资料，比如飞书文档、
Confluence、Git README、数据字典、指标口径和历史 SQL。然后解析成正文，
清理页眉页脚、重复内容和废弃版本，并在入库前做脱敏和权限标记。
接着按标题、段落、问答、指标或表结构切成 chunk。每个 chunk 都要带 metadata，
比如 doc_id、source、business_domain、permission、version、updated_at 和 status。
然后生成 embedding，把向量写入向量库，把正文和元数据写入数据库。
生产里还要记录文档 hash，用于判断文档是否变化，支持增量更新和版本回滚。
这样设计的好处是，在线问答时不仅能检索相似内容，还能按权限和版本过滤，
并且回答可以追溯到原始文档。

## Q028：RAG 在线问答为什么必须返回引用来源？
来源 Day：Day 18
### 回答
企业 RAG 不能只返回一个看起来合理的答案，还要说明答案来自哪里。
引用来源能让用户核对文档、章节、版本和原文片段，也方便研发排查问题。
如果没有引用，很难判断错误是资料过期、检索不准，还是模型生成时编造。
我会把引用来源作为生产 RAG 的基础能力。在线问答时，系统先检索 top-k chunk，
再把这些片段作为上下文交给模型生成答案。最终返回时，除了答案文本，
还应该返回 source、chunk_id、标题、位置、更新时间和相关性分数等信息。
这样用户可以确认答案是否基于可信资料；研发也可以通过引用快速定位问题。
如果引用不相关，说明检索或 rerank 有问题；如果引用资料本身过期，
说明知识库更新和版本管理有问题；如果引用正确但答案错，才重点看 prompt 或模型生成。
在数据口径、财务、人事、合同和权限敏感场景里，引用不是加分项，而是上线门槛。

<a id="q029"></a>

## Q029：RAG 召回质量差时，你会怎么排查和优化？
来源 Day：Day 19
### 回答
我会先看检索结果，而不是先改 prompt。
排查顺序是：问题是否被正确理解，正确资料是否在知识库里，chunk 是否切得合理，
top-k 是否太小，metadata 过滤是否过严，关键词和向量检索是否命中，
以及是否需要 rerank。优化时会结合测试集和 bad case，而不是凭感觉调参数。
RAG 召回质量差通常有几类原因。第一，知识库没有正确资料，或者资料过期、重复、冲突。
第二，chunk 切分不合理，太大会导致相关内容被稀释，太小会丢上下文。
第三，用户问题和文档表达不一致，需要 query rewrite 或同义词扩展。
第四，只用向量检索可能漏掉表名、字段名、错误码这类精确词，
所以生产里常用混合检索。第五，metadata 过滤可能过严，比如业务域、
权限或版本过滤把正确资料排除了。第六，top-k 和 rerank 策略不合理，
正确资料虽然被召回，但排得太靠后，最终没有进入上下文。
生产里召回优化一般会围绕这些手段展开：

| 方法 | 作用 |
|------|------|
| 调整 chunk 策略 | 控制文档切分粒度，避免太碎或太大 |
| 调整 top-k | 平衡召回率、噪声和成本 |
| query rewrite | 把用户问题改写成更适合检索的表达 |
| 混合检索 | 同时使用向量检索和关键词检索 |
| metadata 过滤 | 按业务域、权限、版本、文档类型过滤 |
| rerank | 对候选结果重新排序，让最相关资料靠前 |
| bad case 分析 | 找出召回失败原因，反向优化知识库和检索策略 |

## Q030：RAG 问答 API 在生产环境里应该怎么设计？
来源 Day：Day 20
### 回答
RAG API 不能只设计成一个返回字符串的接口。生产里至少要包括请求校验、用户身份、
权限上下文、问题文本、召回参数、答案、引用来源、request_id、错误码和日志记录。
最小接口可以是 `POST /rag/ask`，请求里包含 `question`、`user_id`、`top_k`、
`business_domain` 等字段，响应里包含 `answer`、`citations`、`cannot_answer_reason`
和 `request_id`。引用来源要能追溯到文档、chunk、位置和版本，方便用户核对，也方便研发排查。
工程上还要有 timeout、异常处理、统一响应格式和回归测试。这样 RAG 才能从本地脚本变成可接入业务系统的服务。

<a id="q032"></a>

## Q032：Query rewrite 在 RAG 里解决什么问题，生产里怎么控制风险？
来源 Day：Day 22
### 回答
Query rewrite 解决的是用户表达和知识库表达不一致的问题。用户可能说得很口语、
省略上下文、中英文混用，或者一个问题里包含多个意图；而知识库通常使用标准术语、
表名、字段名和指标名。改写可以补全省略、加入同义词、保留精确实体，或者把复杂问题
拆成多个检索 query，从而提升召回质量。但生产里不能让改写随意改变用户意图。
工程上要保留原问题和改写问题同时检索，日志记录 original query、rewritten query、
召回结果和最终引用；对表名、字段名、错误码这类精确词要原样保留；多查询数量要受控，
避免延迟和噪声过高。是否启用改写要通过测试集验证，而不是凭感觉判断。

<a id="q033"></a>

## Q033：SQL 解释助手能检查哪些风险？
来源 Day：Day 20
### 回答
SQL 解释助手可以检查语法、结构和生产执行风险。常见风险包括没有 `where` 条件、
没有 `dt` 或其他分区限制、`select *` 读取过多字段、全局 `order by` 成本高、
缺少 `limit`、大表 join 条件不清晰、字段或表名不明确、可能扫描过大数据量，
以及出现 `delete`、`update`、`drop` 等危险语句。
生产里 SQL 风险检查不能只依赖模型自然语言判断，最好把高风险规则做成确定性校验。
LLM 可以负责解释和补充建议，但是否允许执行、是否缺少分区、是否包含危险操作，
应该由规则、SQL parser、权限系统和执行层限制共同控制。这样才能减少误判，
也能把金融信贷开发经验迁移到 NL2SQL、SQL 审核和数据问答场景里。

## Q035：为什么 RAG 项目演示不能只展示成功问题，还要展示 bad case？

## Q038：本地 RAG 项目当前有哪些限制，后续可以怎么优化？
来源 Day：Day 21
### 回答
当前本地 RAG 项目的主要限制是：embedding 使用关键词计数模拟，不是真实语义向量；
answer 还是基于召回 chunk 的答案草稿，没有接真实 LLM；user_id 和 business_domain
只是预留字段，还没有做真实权限过滤；评测集规模较小；也没有缓存、限流、监控、
服务部署和完整接口测试。后续可以从几层优化：替换真实 embedding 模型和向量数据库；
增加 query rewrite、混合检索和 rerank；接入 LLM 并要求只基于引用上下文回答；
增加权限过滤、敏感信息控制和访问审计；扩大评测集，加入拒答、权限、资料冲突和 bad case；
最后补 Docker、配置管理、日志监控和自动化接口测试。

<a id="q039"></a>

## Q039：用户原始问题为什么不一定适合直接检索？
来源 Day：Day 22
### 回答
用户原始问题通常是面向人的表达，不一定是面向检索系统的表达。用户可能省略主语、
使用口语、使用同义词、带上下文指代、中英文混用，或者一个问题里包含多个意图。
例如用户问“这个为什么要带来源”，知识库里可能写的是“RAG 必须返回 citations、
引用来源、chunk_id 和可追溯信息”。如果直接检索原话，向量和关键词都可能命中不稳。
query rewrite 的作用就是把问题补全成更接近知识库表达的检索 query，同时保留用户原意。
生产里通常会保留 original query 和 rewritten query，方便对比召回效果和排查改写偏移。

<a id="q040"></a>

## Q040：query rewrite 有哪些常见方式？
来源 Day：Day 22
### 回答
query rewrite 常见方式包括规则词典改写、同义词扩展、业务术语补全、上下文补全、
精确实体保留、多 query 拆分和 LLM 改写。规则词典适合高频、稳定的业务词，比如把
“来源”扩展成 citations、引用来源、可追溯；同义词扩展适合用户表达和文档表达不一致；
上下文补全适合“这个怎么算”这类指代问题；多 query 适合一个问题包含多个检索方向；
LLM 改写适合复杂口语表达。生产里通常先用可解释规则做第一版，再用测试集验证收益，
最后再考虑 LLM rewrite 和 rerank 配合。

<a id="q041"></a>

## Q041：rewrite drift 是什么，为什么危险？
来源 Day：Day 22
### 回答
rewrite drift 指 query rewrite 改变了用户原意，导致系统检索到看似相关但实际错误的资料。
它危险的地方在于，召回结果可能分数很高、答案看起来也合理，但回答的不是用户真正想问的问题。
例如用户问“接口怎么设计”，系统错误改写成“前端页面怎么设计”，就会把检索方向带偏。
生产里 rewrite drift 会造成错误回答、错误引用和用户信任下降，严重时还可能触发权限或合规风险。
控制方式是保留原始 query、限制扩展词数量、保留表名字段名等精确实体、记录 rewritten query、
用评测集比较改写前后效果，并在低置信度或意图不清时选择澄清或拒答。

<a id="q042"></a>

## Q042：怎么判断 query rewrite 是否真的改善了召回质量？
来源 Day：Day 22
### 回答
判断 query rewrite 是否有效，不能只看某一次答案是否变好，而要用固定测试集评估。
每条样本应该包含原始问题、期望来源、问题类型和必要的标准答案要点。评估时先看
expected source 是否进入 top-k，再看命中排名是否更靠前，同时观察无关 chunk 是否增加。
如果 rewritten query 提高了 hit_rate 和 avg_hit_rank，并且 citations 更相关，说明有正向收益。
如果命中率提高但噪声也明显增加，就要结合 rerank、限制扩展词或调整规则。生产里还要记录
original query、rewritten query、召回结果、最终引用和用户反馈，持续分析 bad case。

<a id="q043"></a>

## Q043：RAG 测试集应该怎么构建，评估哪些指标？
来源 Day：Day 23
### 回答
RAG 测试集要覆盖真实用户会问的问题，而不是只放几个简单样例。每条样本至少要包含
问题、期望答案要点、期望引用来源、问题类型和是否应该拒答。评估时要分层看：
检索层看正确资料是否进入 top-k、排名是否靠前；生成层看答案是否正确、是否基于上下文、
是否有幻觉；引用层看 citations 是否指向正确文档；工程层看延迟、失败率和成本。
测试集还要包含边界问题，比如知识库没有答案、权限不足、资料冲突、用户表达模糊和精确表字段匹配。
每次改 chunk、top-k、query rewrite、rerank 或 prompt 后都跑同一批样本，这样才能知道优化是否真的有效。

<a id="q044"></a>

## Q044：为什么评估 RAG 不能只看最终答案？
来源 Day：Day 23
### 回答
RAG 的最终答案是多个环节叠加后的结果，只看答案无法判断问题出在哪里。一个答案错误，
可能是知识库没有正确资料，也可能是 chunk 切分不合理、query rewrite 改写偏移、top-k 太小、
metadata 过滤误伤、rerank 排序错误，或者模型生成时没有忠实依据。评估时要拆开看：
先看检索层 expected source 是否进入 top-k，再看 citations 是否正确，最后看答案是否基于上下文。
这样才能把问题定位到资料、检索、引用、生成或工程稳定性，而不是盲目改 prompt。

<a id="q045"></a>

## Q045：expected_sources 在 RAG 测试集中有什么作用？
来源 Day：Day 23
### 回答
expected_sources 是每条测试问题应该命中的资料来源，用来判断检索是否找到了正确依据。
没有 expected_sources，系统即使返回了几个看起来相关的 chunk，也很难客观判断是否真的召回了正确资料。
它可以用于计算 hit_rate、hit_rank 和 citations 准确性。
当 expected source 没进入 top-k，
说明检索或知识库侧有问题；如果进入但排名靠后，说明排序或 rerank 需要优化；如果引用正确但答案错，
才重点看 prompt 或模型生成。生产里 expected_sources 是 RAG 评测从“凭感觉”走向“可回归”的关键字段。

<a id="q046"></a>

## Q046：RAG 系统什么时候应该拒答？
来源 Day：Day 24
### 回答
RAG 系统在没有可靠依据、用户无权限、问题涉及敏感信息、请求危险操作、
检索置信度过低或问题表达过于模糊时，应该拒答或先澄清。
拒答不是系统没能力，而是为了防止模型在没有资料支撑时编造答案。
生产里通常会结合规则拦截、权限过滤、检索分数、引用来源和测试集来判断是否可以回答。
如果拒答，也要返回清楚的原因，比如知识库没有足够依据、资料权限不足或问题需要补充时间范围。

<a id="q047"></a>

## Q047：生产环境里怎么设计 RAG 拒答策略？
来源 Day：Day 24
### 回答
生产 RAG 的拒答策略通常分层设计。
第一层用规则识别敏感信息、危险操作和越权请求；
第二层用权限系统过滤用户不可见资料；
第三层用检索结果判断 expected source 是否命中、top-k 分数是否足够、来源是否一致；
第四层在 prompt 中要求模型只基于 citations 回答，不能基于常识补全；
第五层用测试集评估拒答准确率、误拒率和幻觉率。
这样可以把拒答从“模型自己决定”变成可解释、可测试、可审计的工程能力。

<a id="q048"></a>

## Q048：RAG 幻觉通常怎么产生，怎么降低？
来源 Day：Day 24
### 回答
RAG 幻觉常见原因包括知识库资料缺失、chunk 切分不合理、检索没有命中正确资料、
query rewrite 改写偏移、无关上下文被塞进 prompt、模型没有被限制只基于引用回答。
降低幻觉要从链路上处理，而不是只改 prompt。
离线侧要保证资料可信、切分合理、版本清楚；
检索侧要提高 expected source 命中率，必要时使用混合检索和 rerank；
生成侧要要求答案必须有 citations 支撑；
边界侧要在无依据、低置信度、越权和敏感场景下拒答；
评测侧要长期维护 bad case 和拒答样本。

<a id="q049"></a>

## Q049：RAG 系统怎么做权限控制和敏感信息保护？
来源 Day：Day 25
### 回答
生产 RAG 的权限和敏感信息保护要贯穿全链路。
入库前要做数据分级、脱敏和权限标签，避免高敏资料直接进入普通知识库。
每个 chunk 要记录 security_level、allowed_roles、业务域、来源和版本。
在线检索时要基于用户身份、角色和业务域过滤 chunk，不能让无权限资料进入 LLM 上下文。
生成时要要求模型只基于授权 citations 回答，答案返回前还要做敏感信息扫描。
最后要记录 request_id、用户、召回结果、引用和答案，方便审计和 bad case 复盘。

<a id="q050"></a>

## Q050：为什么 RAG 不能只靠 prompt 防止敏感信息泄露？
来源 Day：Day 25
### 回答
prompt 只能约束模型行为，不能替代权限系统和数据治理。
如果敏感 chunk 已经进入上下文，模型就可能直接引用、改写或推断出敏感内容。
用户也可能通过 prompt 注入诱导模型忽略规则。
生产里应该把安全控制放在模型之前：
入库前脱敏，chunk 带权限标签，检索时按用户权限过滤，生成前只传授权上下文。
prompt 只是最后一层提醒，不应该成为唯一边界。

<a id="q051"></a>

## Q051：RAG 的 citations 也会有安全风险吗？
来源 Day：Day 25
### 回答
citations 也会有安全风险。
引用里可能包含敏感文档标题、路径、chunk 内容、业务域、版本或内部系统地址。
即使答案正文没有泄露，citation 也可能暴露用户无权知道的资料存在。
生产里 citations 要和答案正文一样做权限过滤和脱敏处理。
普通用户只看到允许访问的来源信息；
研发和审计人员可以在后台通过 request_id 查看更完整的召回链路。

<a id="q052"></a>

## Q052：生产 RAG 怎么控制 token 成本和响应延迟？
来源 Day：Day 26
### 回答
生产 RAG 控制成本和延迟，核心是减少无效上下文和重复计算。
检索阶段先用 metadata filter 缩小范围，避免无权限、无关文档进入候选。
重排阶段对候选 chunk 做 rerank、低分过滤和去重。
上下文组装阶段设置 token budget，只保留最相关、最有依据的少量 chunk。
模型调用阶段控制 max tokens，并对简单问题使用更便宜的模型。
对高频问题、固定口径和低风险 FAQ，可以缓存召回结果或最终答案。
同时每次请求都要记录 input token、output token、延迟、缓存命中率和 request_id，
这样才能判断优化有没有真实效果。

<a id="q053"></a>

## Q053：RAG 系统里缓存应该怎么设计，为什么不能只按问题文本缓存？
来源 Day：Day 26
### 回答
RAG 缓存不能只按问题文本缓存，因为同一个问题在不同用户、权限、业务域和知识库版本下，答案可能不同。
缓存 key 至少要考虑规范化后的问题、用户角色、业务域、知识库版本、prompt 版本和模型版本。
高风险或权限敏感问题可以只缓存召回结果，不直接缓存最终答案。
缓存还要有 TTL 和失效机制，文档更新、权限变更或 prompt 变更后必须失效。
否则缓存虽然降低成本，但可能返回旧答案、错答案或越权答案。

<a id="q054"></a>

## Q054：为什么 RAG 不是 top-k 越大越好？
来源 Day：Day 26
### 回答
top-k 越大，召回覆盖面可能更高，但进入上下文的噪声、token 成本和延迟也会增加。
无关 chunk 进入 prompt 后，模型可能被干扰，引用错误资料，甚至生成看似有依据但实际不准确的答案。
生产里通常会“召回多一点，最终使用少一点”：
先召回一批候选，再用权限过滤、rerank、去重和 token budget 选出少量高质量资料。
top-k 的选择要通过固定评测集验证，而不是凭感觉调大。

<a id="q065"></a>

## Q065：NL2SQL 为什么不能直接把用户问题交给模型生成 SQL？
来源 Day：Day 29
### 回答
生产环境里不能直接把用户问题交给模型生成 SQL。
因为模型可能编造不存在的表和字段，混淆指标口径，漏掉分区条件，生成高成本 SQL，
或者绕过权限查询敏感数据。
更稳的做法是先提供 Schema Catalog、指标口径、权限规则和 SQL 约束，
让模型只能在可用表字段和业务规则范围内生成 SQL。
生成后还要做 SQL 校验、权限检查和执行风险控制。

<a id="q066"></a>

## Q066：NL2SQL 的 Schema Catalog 应该包含哪些信息？
来源 Day：Day 29
### 回答
Schema Catalog 至少要包含表名、表描述、业务域、表粒度、字段名、字段类型、
字段含义、指标字段、维度字段、时间字段、分区字段、权限等级和常见问题示例。
这些信息能帮助系统判断用户问题应该查哪张表、用哪些字段、按什么维度聚合，
也能限制模型不要编造表字段。
生产里 Schema Catalog 通常来自元数据平台、指标平台、数据地图、Hive Metastore 或 dbt 文档。

<a id="q067"></a>

## Q067：NL2SQL 问题分类有什么价值？
来源 Day：Day 29
### 回答
NL2SQL 问题分类能降低 SQL 生成难度。
指标查询、维度分组、趋势查询、TopN、对比查询和明细查询，对应的 SQL 结构不一样。
先识别问题类型，可以决定是否需要聚合、group by、order by、limit、时间窗口或精确过滤。
如果问题涉及敏感信息或权限不足，还可以在生成 SQL 前拦截。
这比直接生成 SQL 更容易控制质量、成本和安全风险。

<a id="q068"></a>

## Q068：NL2SQL 里为什么要先做问题解析？
来源 Day：Day 30
### 回答
NL2SQL 先做问题解析，是为了把用户自然语言拆成 SQL 生成需要的结构化信息。
用户问题里通常混合了指标、维度、时间、过滤条件、排序、TopN 和业务口径。
如果直接让模型生成 SQL，模型容易漏掉时间范围、选错表字段、混淆指标口径或生成高成本查询。
问题解析会先抽取 metric、dimensions、time_range、filters、query_type 和 risk_flags，
再结合 Schema Catalog 选择候选表和字段。
这样后续 SQL 生成更可控，也更容易做权限校验、成本校验和错误排查。

<a id="q069"></a>

## Q069：自然语言时间在 NL2SQL 里怎么处理？
来源 Day：Day 30
### 回答
自然语言时间必须转成结构化时间条件。
比如“昨天”“上周”“最近 7 天”“本月”都要解析成 start_date、end_date、granularity 和 time_field。
生产里不能只把原始时间词塞进 prompt，因为不同业务可能有不同时间口径，
比如申请提交时间、审批完成时间、放款成功时间、还款到期日、分区日期和事件发生时间。
如果时间不清楚，要使用业务默认规则或向用户追问。
同时，大表查询必须限制时间范围，避免生成全表扫描 SQL。

<a id="q070"></a>

## Q070：指标抽取和维度抽取有什么区别？
来源 Day：Day 30
### 回答
指标抽取关注“要算什么”，比如授信申请量、授信通过率、放款金额、逾期率。
维度抽取关注“按什么看”，比如渠道、城市、信贷产品、日期、风险等级、逾期账龄。
在 SQL 里，指标通常对应聚合字段或聚合表达式，维度通常对应 group by、where 或展示字段。
两者不能混淆。比如“每个渠道的授信通过率”，授信通过率是指标，渠道是分组维度。
如果把维度当指标，SQL 聚合会错；如果把过滤条件当分组维度，结果粒度也会错。

<a id="q071"></a>

## Q071：NL2SQL 生成 SQL 时为什么必须加 Schema 约束？
来源 Day：Day 31
### 回答
NL2SQL 生成 SQL 时必须加 Schema 约束，因为模型本身不知道真实数据库里有哪些表、字段、指标口径和权限边界。
如果只让模型根据自然语言生成 SQL，它可能编造不存在的字段，选错事实表，漏掉分区条件，
或者访问敏感字段。

生产里应该把 Schema Catalog、指标定义、字段角色、时间字段和权限标签作为 SQL 生成的输入。
生成器只能使用白名单里的表字段，不能自由发明字段名。
生成后还要经过 SQL 校验，确认只读、字段合法、时间范围完整、权限允许、成本可控。

<a id="q072"></a>

## Q072：为什么 SQL 生成后不能直接执行？
来源 Day：Day 31
### 回答
SQL 生成后不能直接执行，因为生成正确语法不代表业务正确、安全或低成本。
它还可能存在口径错误、字段越权、缺少时间范围、扫描大表、没有 limit、危险关键字或 join 放大。

在金融信贷场景里，SQL 可能涉及客户隐私、授信审批、风控评级、逾期和贷后数据。
这些数据有明显权限边界和审计要求。
所以生成 SQL 后必须经过只读校验、表字段白名单校验、权限校验、时间范围校验、成本预估和审计记录。
只有通过校验的 SQL 才能进入执行层。

<a id="q073"></a>

## Q073：规则模板生成 SQL 和 LLM 生成 SQL 怎么取舍？
来源 Day：Day 31
### 回答
规则模板适合高频、结构清楚、口径稳定的问题，比如授信申请量、放款金额趋势、TopN 城市和逾期率对比。
它的优点是稳定、可解释、容易测试，也更适合作为生产 baseline。

LLM 适合复杂表达、跨表关联、模糊意图和非标准问法。
但 LLM 更容易出现字段编造、口径偏差和不稳定输出。

生产里通常不会二选一。
更稳的做法是简单问题走模板，复杂问题走 LLM，但两者都必须使用同一套 Schema Catalog、权限规则和 SQL 校验器。
最终看固定测试集上的准确率、拒答率、越权率、执行成本和可维护性。

<a id="q074"></a>

## Q074：NL2SQL 为什么必须有 SQL 校验层？
来源 Day：Day 32
### 回答
NL2SQL 必须有 SQL 校验层，因为生成 SQL 正确不代表可以安全执行。
模型或模板可能生成语法正确但业务错误、权限越界、缺少时间范围、扫描大表、
查询敏感字段或包含危险关键字的 SQL。
在金融信贷场景里，SQL 可能涉及客户隐私、授信审批、风控评级、逾期和贷后数据，
如果直接执行，可能造成数据泄露、错误决策或生产数据库压力。
SQL 校验层应该放在 SQL 生成之后、查询执行之前，统一检查只读、表字段白名单、
敏感字段、时间范围、limit、权限和成本风险。只有校验通过的 SQL 才能进入执行层。

---

<a id="q075"></a>

## Q075：SQL 校验层通常要检查哪些内容？
来源 Day：Day 32
### 回答
SQL 校验层至少要检查三类内容。
第一是安全约束，包括只读 SQL、危险关键字、敏感字段、权限边界和审计要求。
第二是业务和 Schema 约束，包括表是否在白名单里、字段是否存在、指标口径是否匹配、
明细查询是否有精确过滤。
第三是性能和成本约束，包括大表是否带时间范围或分区条件、是否有无限制 `order by`、
是否缺少 `limit`、是否存在高成本 join 或大范围扫描。
这些检查应该用代码层规则、SQL parser、权限系统和查询网关共同完成，不能只依赖 prompt。

<a id="q076"></a>

## Q076：为什么不能只靠 prompt 要求模型生成安全 SQL？
来源 Day：Day 32
### 回答
不能只靠 prompt 要求模型生成安全 SQL，因为 prompt 是软约束，模型可能理解错、遗漏规则，
也可能在复杂问题下生成越权字段、危险 SQL 或高成本查询。
生产安全必须用代码层硬约束实现。
比如敏感字段要有字段级权限标签，表字段要来自 Schema Catalog，危险关键字要静态拦截，
大表必须有时间范围，明细查询必须有 `limit` 和精确过滤。
prompt 可以帮助模型少犯错，但不能替代 SQL 校验器、权限系统、审计日志和查询网关。

<a id="q077"></a>

## Q077：NL2SQL 查询执行层为什么必须放在 SQL 校验之后？
来源 Day：Day 33
### 回答
查询执行层必须放在 SQL 校验之后，因为执行层一旦访问数据库，风险就从“生成错误”
变成了真实的数据泄露、性能冲击或误操作风险。
SQL 生成层可能产生字段越权、缺少时间范围、全表扫描、敏感字段查询或危险关键字。
这些问题必须先由 SQL Validator、权限系统和成本控制拦住。
执行层只应该接收已经通过校验的 SQL 或查询任务，使用只读连接执行，
并记录执行状态、耗时、返回行数和审计信息。

---

<a id="q078"></a>

## Q078：NL2SQL 查询执行层要控制哪些生产风险？
来源 Day：Day 33
### 回答
查询执行层要控制安全、性能、稳定性和可追溯四类风险。
安全上，要确保只读账号、权限校验和敏感字段拦截已经生效。
性能上，要限制超时、扫描量、返回行数、分页和大结果导出。
稳定性上，要处理数据库异常、连接失败、SQL 方言不兼容和空结果。
可追溯上，要记录 request_id、用户、SQL、参数、执行耗时、返回行数和失败原因。
在金融信贷场景里，这些控制能避免客户隐私泄露、生产库被拖慢和错误指标影响业务判断。

<a id="q079"></a>

## Q079：NL2SQL 返回结果为什么不能只返回数据库原始行？
来源 Day：Day 33
### 回答
不能只返回数据库原始行，因为业务用户通常需要的是可理解、可展示、可解释的答案。
同样是 SQL 结果，指标查询适合返回指标卡，趋势查询适合返回按日期排列的序列，
TopN 适合返回排序表，对比查询适合返回当前值、上期值和差值，明细查询要控制字段和行数。
如果只返回原始行，前端不知道如何展示，业务用户也不一定能理解口径。
生产里执行层应该返回结构化结果，包括 `response_type`、`columns`、`rows`、`row_count`、
`summary_text` 和必要的审计信息，再交给结果解释层转成业务语言。

<a id="q080"></a>

## Q080：NL2SQL 为什么需要结果解释层？
来源 Day：Day 34
### 回答
NL2SQL 需要结果解释层，是因为业务用户通常不只需要 SQL 返回的原始行列结果，
还需要理解这个结果代表什么、是否有风险、下一步该怎么追问。
查询执行层返回的是结构化数据，比如字段、行数、指标值、趋势序列和明细记录。
结果解释层要把这些内容转换成业务语言，例如说明哪个渠道通过率最高、逾期率比上周下降了多少、
某个申请编号当前审批状态是什么。
同时，解释层必须补充时间范围、指标口径、权限边界和样本限制，避免业务误读。
它不能编造 SQL 没查出来的原因，只能基于查询结果和元数据解释。

<a id="q081"></a>

## Q081：结果解释层怎么避免模型编造业务原因？
来源 Day：Day 34
### 回答
结果解释层要避免编造，核心是把解释建立在结构化查询结果上。
系统应该先由规则层生成事实性结论，比如最高值、最低值、环比差值、趋势方向和返回行数，
再把这些事实交给 LLM 做可选润色。
Prompt 里要明确要求模型只能引用提供的 rows、columns、指标口径和时间范围，
不能自行猜测原因。
如果需要原因分析，应该输出“建议继续按渠道、产品、风险等级或日期拆分”，
而不是直接说原因已经确定。
生产里还要保留原始查询结果、SQL、request_id 和审计日志，方便回溯解释是否忠于数据。

<a id="q082"></a>

## Q082：被 SQL 校验拦截的问题，结果解释层应该怎么回答？
来源 Day：Day 34
### 回答
被 SQL 校验拦截的问题，结果解释层不能回答成“没有数据”，也不能猜测敏感信息。
它应该明确说明该问题没有执行数据库查询，原因是权限、敏感字段、缺少时间范围、
危险 SQL 或高成本风险。
同时要给出安全替代路径，例如补充时间范围、改查汇总指标、走敏感数据审批流程，
或联系数据 owner。
这样业务用户能理解系统是在做安全控制，而不是查询失败或数据不存在。
在金融信贷场景里，这一点尤其重要，因为手机号、身份证号、客户明细和贷后敏感数据
都必须有权限和审计边界。

<a id="q083"></a>

## Q083：一个生产级 NL2SQL 助手应该包含哪些模块？
来源 Day：Day 35
### 回答
一个生产级 NL2SQL 助手不应该只有“自然语言生成 SQL”这一步。
它至少应该包含 Schema Catalog、问题解析、SQL 生成、SQL 校验、权限控制、查询执行、
结果解释、审计日志和评测回归。
Schema Catalog 提供真实表字段、指标口径和权限标签；问题解析负责抽取指标、维度、时间和风险；
SQL 生成负责在 Schema 约束下生成只读 SQL；SQL 校验负责拦截危险、越权和高成本查询；
查询执行层使用只读连接和超时限制查数；结果解释层把数据转成业务语言；
审计和评测则保证线上可追溯、可持续优化。

## Q084：你会怎么演示一个 NL2SQL 项目？

演示 NL2SQL 项目时，不能只展示一个成功生成 SQL 的例子。

演示顺序是先输入业务问题，再展示问题解析结果，包括指标、维度、时间和风险标记；

## Q085：NL2SQL 项目整合时，为什么要保留每一层的中间结果？
来源 Day：Day 35

NL2SQL 项目整合时必须保留每一层的中间结果，因为问题可能出在任何阶段。
如果最终答案错了，需要知道是问题解析漏了时间，Schema Router 选错表，
SQL Generator 生成错字段，SQL Validator 没拦住风险，Executor 执行方言不兼容，

保留中间结果可以支持调试、审计、回放和评测。

## Q086：为什么 AI 应用要从脚本重构成后端服务？
来源 Day：Day 36
### 回答
AI 应用从脚本重构成后端服务，是为了让能力能被前端、业务系统、机器人或其他服务稳定调用。
脚本适合本地验证，但不适合权限控制、接口复用、审计追踪、错误处理和多用户访问。
服务化后，需要把 API 层、service 层、storage 层、config 层和 errors 层拆清楚。
这样后续替换模型、数据库、SQL 生成器或权限系统时，不会影响接口契约。
在 NL2SQL 场景里，服务化还能统一处理 request_id、审计记录、安全阻断和响应格式。

<a id="q087"></a>

## Q087：NL2SQL API 为什么要区分安全阻断和系统错误？
来源 Day：Day 37
### 回答
NL2SQL API 必须区分安全阻断和系统错误，因为两者含义完全不同。
安全阻断是系统按预期拦截了敏感字段、危险 SQL、缺少时间范围或高成本查询，
应该返回清晰的业务状态，例如 `final_status=safely_blocked`。
系统错误才是服务异常、依赖缺失、数据库连接失败或代码 bug。
如果把安全阻断都返回 500，调用方会误以为系统故障，也无法向业务用户解释权限和合规边界。
区分状态能让前端、审计和运营处理更稳定。

## Q088：Docker 化解决了 AI 应用交付里的什么问题？
来源 Day：Day 38

如果只说“我本机能跑”，换机器、换环境或部署到服务器时很容易失败。
Dockerfile 把依赖安装、代码复制、启动命令和端口暴露写成标准步骤，
让开发、测试和部署使用同一套运行方式。

## Q089：为什么配置和密钥不能写死在代码里？
来源 Day：Day 39
### 回答
配置和密钥不能写死在代码里，因为不同环境的数据库地址、开关、路径、日志级别和权限策略都可能不同。

<a id="q093"></a>

## Q093：Agent 工作流为什么不能让模型完全自由调用工具？
来源 Day：Day 43
### 回答
生产环境里的 Agent 不能让模型完全自由调用工具，因为工具通常连接数据库、知识库、业务系统或外部接口。
如果模型没有受控流程，可能会选错工具、重复调用、绕过权限、执行高成本查询，
甚至把失败结果编造成确定答案。

更稳的做法是把 Agent 设计成受控工作流：先识别意图，再检索 schema 或知识，
生成候选操作，经过权限、安全、成本和格式校验后再执行。
每一步都要有结构化输入输出、最大调用次数、失败回退和审计记录。

在金融信贷 NL2SQL 场景里，Agent 不能直接执行模型生成的 SQL。
必须先检查敏感字段、时间范围、只读限制、表权限和扫描成本。
如果校验失败，就应该拒答或转人工，而不是继续生成看似合理的答案。
这种设计能在智能化和可控性之间取得平衡。

<a id="q094"></a>

## Q094：多工具 Agent 为什么必须有工具选择和失败回退策略？
来源 Day：Day 44
### 回答
多工具 Agent 必须有工具选择和失败回退策略，因为工具通常连接真实数据、知识库、业务系统或外部服务。
如果让模型自由调用工具，可能会选错工具、重复调用、绕过校验、增加成本，
甚至在工具失败后编造答案。

生产里应该先定义工具注册表，写清每个工具的用途、输入输出、权限要求、风险等级和前置条件。
编排层再根据用户意图、权限、风险标记和上一步结果选择工具路线。
例如金融信贷 NL2SQL 指标查询必须先查 Schema，再生成 SQL、校验 SQL，最后才能执行查询。
如果是政策规则解释，就应该走 RAG 检索和引用回答，而不是强行查数据库。

失败回退同样重要。
缺少时间范围时应该要求补充条件，敏感字段查询应该安全阻断，
工具超时应该返回系统错误或降级，检索无依据时应该拒答。
同时要设置最大步数和重复调用检测，防止 Agent 陷入循环。
这样多工具 Agent 才能做到可控、可审计、可评测，而不是一个不可解释的黑盒。

## Q096：生产级 AI 应用为什么需要端到端评测，而不是只测单个 Prompt 或工具？
来源 Day：Day 45
### 回答
生产级 AI 应用需要端到端评测，因为真实用户关心的是完整链路结果，
而不是某个 Prompt、检索器、SQL 生成器或实时工具单独表现。
一个金融信贷仓库 Agent 可能每个单点工具都能跑通，但整体仍然出错。
例如实时问题误走离线 SQL、缺少时间范围却继续查库、SQL 生成后跳过校验、
指标口径没有引用却回答、实时链路延迟却仍给出确定结论，都会导致生产风险。

端到端评测应该从用户问题开始，检查意图识别、上下文检索、工具路由、离线 SQL 生成、
SQL 校验、离线查询执行、实时指标查询、告警解释、口径引用、血缘追溯、安全阻断和审计记录。
每条样例都要有明确成功判定，包括预期最终状态、必须调用的工具、禁止调用的工具、
是否需要引用、是否允许执行 SQL、是否需要检查实时延迟和失败分类。
对敏感导出、越权访问、资料不足、实时延迟和系统异常来说，正确拒答、正确澄清或正确降级也应该算成功。

在金融信贷场景里，端到端评测尤其重要，因为系统可能接触授信、风控、放款、还款、
逾期、催收、实时反欺诈和客户敏感信息。
固定评测集可以持续回归，帮助团队判断每次 Prompt、规则、Schema Catalog、指标字典、
实时指标字典、血缘工具或工具路由改动是否真的提升质量，是否引入新的安全、合规或实时风险。

<a id="q100"></a>

## Q100：金融信贷仓库 Agent 的端到端评测集应该怎么设计？
来源 Day：Day 45
### 回答
金融信贷仓库 Agent 的端到端评测集不能只放“问指标并回答”的成功样例，
而要覆盖离线指标、实时指标、告警解释、指标口径、血缘追溯、权限阻断、缺参澄清和系统异常。
每条样例都应该从真实用户问题出发，写清业务类别、预期状态、必须调用的工具、禁止调用的工具、
是否需要引用、是否允许执行 SQL、是否要检查实时延迟，以及失败后应该归到哪类责任层。

离线指标样例要检查 Schema Catalog、分区字段、指标口径、SQL 生成、SQL 校验、执行和结果解释。
例如查询“昨日各渠道授信通过率”，必须走离线指标链路，不能查明细敏感字段，
也不能跳过 SQL Validator。
实时指标样例要检查窗口、事件时间、处理时间、延迟状态和实时工具路由，
例如“近 5 分钟实时拒绝率是否异常”不能用离线日报表回答。

评测集还必须包含应该失败的问题。
例如导出客户手机号、查询无权限明细、缺少时间范围、口径资料没有引用、分区缺失、实时链路延迟和数据库超时。
这些样例的正确结果不是硬答，而是安全阻断、要求补充条件、资料不足或系统异常兜底。
这样评测集才能真正保护生产边界，而不是只证明 Demo 能跑。

<a id="q101"></a>

## Q101：实时仓库问题进入 Agent 评测时要重点检查哪些风险？
来源 Day：Day 45
### 回答
实时仓库问题进入 Agent 评测时，首先要检查工具路由风险。
只要用户问题出现近 5 分钟、近 10 分钟、实时拒绝率、实时申请量、告警、事件流、链路延迟这类信号，
Agent 就应该优先走实时指标工具或告警工具，而不是生成离线 SQL 查日报表。
如果实时问题误走离线 SQL，即使最终答案看起来有数字，也应该判为失败。

其次要检查窗口和延迟风险。
实时指标必须说明统计窗口、事件时间、处理时间、延迟阈值和当前链路状态。
如果实时链路延迟 20 分钟，系统不能给出“当前正常”这类确定结论，
而应该提示数据延迟、返回 execution_failed 或降级到可解释状态。

还要检查告警解释边界和权限边界。
告警解释只能基于告警等级、触发规则、窗口指标和证据来源，不能编造“原因一定是某个策略上线”。
实时事件流里的客户明细、设备号、手机号、身份证号等敏感字段仍然要走权限校验或安全阻断。
所以实时仓库评测既要测回答准确性，也要测延迟、窗口、证据、权限和审计。

## Q097：AI Agent 出现 bad case 后，你会怎么做错误治理？
来源 Day：Day 46
### 回答
AI Agent 出现 bad case 后，我不会直接只改 Prompt，而是先做失败归因。
第一步是保留原始问题、工具路线、中间结果、最终状态、引用、SQL、校验结果和 request_id。
然后判断错误发生在哪一层：意图识别、工具路由、RAG 召回、SQL 生成、SQL 校验、
查询执行、结果解释还是异常处理。

如果是缺少时间范围却继续查询，就修工具路由和澄清规则；
如果是 RAG 没有引用仍回答，就修 citation 校验和无依据拒答；
如果是实时问题误走离线 SQL，就修意图识别和工具路由；
如果是实时链路延迟被包装成正常答案，就修实时状态校验和异常分类。
Prompt 只适合修任务表达、输出格式和解释约束，不能替代权限、安全和执行前置条件。

修复后必须把 bad case 加入固定评测集，并跑全量回归。
通过率、失败类型、拒答率和安全阻断结果都要对比修复前后。
在金融信贷场景里，这一步尤其重要，因为授信、风控、逾期和客户明细都涉及权限、合规和审计。
错误治理的目标不是让某一条样例看起来答对，而是让系统在同类问题上稳定、可控、可追溯。

## Q102：为什么 AI Agent bad case 必须按责任层归因？
来源 Day：Day 46
### 回答
AI Agent bad case 必须按责任层归因，因为不同层的问题修法完全不同。
如果实时问题误走离线 SQL，核心问题在意图识别和工具路由；
如果缺少时间范围还继续查询，问题在澄清策略和执行前置条件；
如果口径解释没有引用仍回答，问题在 RAG 召回、citation 校验和无依据拒答；
如果实时链路延迟还输出正常结论，问题在实时状态校验和异常分类。

如果不做责任层归因，团队很容易把所有问题都说成“模型不好”或“改 Prompt”。
Prompt 可以修表达、任务约束和输出格式，但不能替代权限控制、SQL 校验、工具前置条件、
实时延迟检测和审计记录。
这些生产边界必须由确定性工程规则兜住。

按责任层归因还能让修复可回归。
每个 bad case 都要记录原始问题、工具路线、中间结果、最终状态、失败类型、责任层、修复动作和回归样例。
修复后再跑固定评测集，确认该类问题稳定解决，同时没有让正常查询、安全阻断或实时告警解释退化。

## Q103：金融信贷离线仓库为什么要分 ODS、DWD、DWS、ADS？
来源 Day：Day 47
### 回答
金融信贷离线仓库分层，是为了把原始数据、明细事实、主题汇总和应用指标分开管理。
ODS 保留业务系统原始数据，DWD 负责清洗成统一粒度的明细事实，DWS 按主题和维度提前汇总，
ADS 面向日报、看板和高频业务查询提供稳定指标。

如果不分层，Agent 或数据分析人员会直接面对大量原始表，很容易选错表、混淆口径、扫大表、重复计数，
还可能访问客户手机号、身份证号、银行卡号等敏感字段。
分层后，经营总览优先查 ADS，维度趋势优先查 DWS，授权明细才查 DWD，ODS 默认只用于上游排查。

在 Agent 数据问答里，分层还能降低 NL2SQL 难度。
Schema Catalog 可以告诉 Agent 每张表的层级、粒度、分区、权限和适用问题，
让 Agent 先选对层，再生成 SQL、校验 SQL、执行查询和记录审计。

## Q104：Agent 查询离线指标时，为什么应该优先选择 ADS 或 DWS？
来源 Day：Day 47
### 回答
Agent 查询离线指标时优先选择 ADS 或 DWS，是因为这两层更接近业务指标。
ADS 面向日报、看板和经营总览，指标口径已经固定；DWS 面向主题汇总和维度分析，
适合按渠道、产品、日期、风险等级做趋势和对比。

如果 Agent 默认从 DWD 或 ODS 明细临时聚合，风险会明显增加。
第一，SQL 更复杂，容易漏分区、漏过滤条件或重复计数；
第二，扫描成本更高；
第三，更容易碰到敏感字段和权限边界；
第四，指标口径可能和业务看板不一致。

所以生产级离线指标 Agent 应该先通过意图识别和分层路由判断问题粒度：
总览走 ADS，维度趋势走 DWS，授权明细才走 DWD，敏感导出直接阻断。
之后再进入 Schema Catalog、SQL 生成、SQL 校验、执行、解释和审计。

## Q105：Agent 每一步为什么通常用 JSON 做结构化输入输出，字段应该怎么设计？
来源 Day：Agent 专题
### 回答
Agent 每一步通常用 JSON 做结构化输入输出，是因为 Agent 不是单轮聊天，而是一条可编排的工程链路。
自然语言适合给人看，但不适合直接驱动后续工具调用。
JSON 可以被代码稳定解析、Schema 校验、状态机路由、审计记录和评测回归。
如果模型只返回一段话，系统很难可靠判断下一步该查 Schema、走 RAG、生成 SQL、要求补充条件，还是安全阻断。

设计 JSON 字段时，核心原则是每一步只返回下一步真正需要的字段。
通常要有统一外壳：`request_id` 用来串联全链路，`step` 表示当前步骤，
`status` 表示处理结果，`data` 保存当前步骤的业务产物，`errors` 和 `warnings`
保存结构化错误与风险，`next_action` 决定下一步路由，`audit` 保存角色、时间和必要审计信息。
其中 `status` 和 `next_action` 必须使用固定枚举，不能让模型自由写“可以查”“好像不行”这类自然语言。

例如问题解析步骤的 `data` 可以包含 `intent`、`metric`、`dimensions`、`filters`、
`time_range`、`risk_flags` 和 `missing_fields`。
SQL 生成步骤要返回 `candidate_sql`、`used_tables`、`used_fields`、`is_readonly` 和生成依据。
SQL 校验步骤要返回 `safe_to_execute`、`blocked_reasons`、`checked_items` 和错误码。
查询执行步骤要返回 `execution_status`、`columns`、`rows`、`row_count` 和耗时。
最终回答步骤要返回 `answer`、`key_findings`、`citations`、`limitations` 和 `final_status`。

在金融信贷 NL2SQL 或 Agent 场景里，风险字段尤其重要。
例如 `risk_flags` 可以标记 `missing_time_range`、`sensitive_field`、`customer_detail`、
`bulk_export`、`write_operation`、`high_cost_query` 和 `permission_required`。
这些字段不能只藏在回答文本里，而要显式放在 JSON 中，由编排层决定是否继续执行。
命中敏感字段或写操作时，应进入 `safe_block`；缺少时间范围时，应进入 `clarification`；
SQL 校验失败时，不能进入 `query_executor`。

工程实现上，可以用 Pydantic、dataclass 或 JSON Schema 定义每一步的输入输出模型。
LLM 可以生成候选 JSON，但代码必须负责解析、Schema 校验和业务规则校验。
结构不合法就重试、降级、澄清或拒答，不能把不合格的自然语言结果继续传给工具。
高风险环节如权限、安全、SQL 校验、查询执行和审计，必须由确定性代码兜底。

一句面试总结是：JSON 的价值不是让输出看起来规范，而是让 Agent 每一步都能被解析、校验、路由、审计和评测。
字段设计要围绕下一步需要什么、风险如何拦截、失败如何归因来做。

## Q106：实时仓库和离线仓库在 Agent 数据问答里有什么区别？
来源 Day：Day 48
### 回答
离线仓库面向稳定的历史指标和经营分析，通常按天或批次加工数据。
Agent 查询昨日、上周、上月的授信通过率、放款金额、逾期率时，应该走离线 ADS 或 DWS。
实时仓库面向当前状态、分钟级窗口、实时特征和告警，适合回答近 5 分钟申请量、当前拒绝率是否异常、
实时还款失败数是否突增这类问题。

两者最大的区别不是表名不同，而是时间语义不同。
离线指标强调分区、批次、指标口径和稳定性；实时指标强调事件时间、处理时间、窗口、延迟和链路状态。
如果 Agent 把实时问题路由到离线 SQL，即使能查出数字，也可能业务上错误。

生产级 Agent 应该先识别问题是离线指标还是实时状态。
离线问题走 Schema Catalog、SQL 生成、SQL 校验和离线执行；
实时问题走实时指标工具或告警工具，并检查窗口和延迟状态。
敏感明细和实时事件流导出都必须经过权限控制和审计。

## Q107：Agent 查询实时指标时为什么必须检查窗口和延迟？
来源 Day：Day 48
### 回答
实时指标的含义依赖窗口。
“申请量是否异常”如果没有说明近 5 分钟、近 10 分钟、渠道或产品线，就无法判断应该查哪个实时指标。
所以缺少窗口或关键维度时，Agent 应该先澄清，而不是直接查询。

实时指标的可信度还依赖延迟。
如果实时链路延迟 20 分钟，那么当前窗口的数据可能不可用或已经过期。
这时 Agent 不能给出“当前正常”这类确定结论，而应该返回延迟说明、降级或 `execution_failed`。

工程上，实时工具的输出不能只有数值，还要包含窗口、事件时间、处理时间、延迟分钟数、延迟阈值和可用状态。
结果解释层只能基于这些结构化状态解释，不能自行编造实时原因。

## Q108：金融信贷 Agent 为什么要先做主题域识别，再做意图识别？
来源 Day：Day 49
### 回答
金融信贷 Agent 需要先做主题域识别，是因为信贷业务链路长，授信、额度、风控、放款、还款、逾期、催收对应的指标、
表、实时事件和权限边界都不同。
同样是“失败率”，在放款域可能指放款失败率，在还款域可能指扣款失败率，在风控域可能指策略拒绝率。
如果不先识别主题域，Agent 很容易选错指标或表。

识别主题域之后，还必须识别意图。
同一个主题域的问题可能是离线指标查询、实时状态查询、口径解释、血缘追溯、敏感导出或缺参澄清。
例如授信申请域里，“近 7 天各渠道通过率”走离线指标；
“近 5 分钟申请量是否异常”走实时指标；
“通过率怎么算”走 RAG 口径问答；
“通过率来自哪些表”走血缘工具。

所以生产级 Agent 的路由应该是：先识别主题域，再识别意图，再选择工具路线，并做权限、前置条件和审计。
这样才能避免把所有问题都粗暴交给 SQL 或模型自由选择工具。

## Q109：如何区分离线指标、实时状态、口径解释、血缘追溯和敏感导出问题？
来源 Day：Day 49
### 回答
可以先看用户问题里的触发信号。
出现昨天、近 7 天、上月、趋势、按渠道、按产品，通常是离线指标；
出现近 5 分钟、当前、实时、是否异常、突增，通常是实时状态；
出现怎么算、定义、口径、分子、分母，通常是口径解释；
出现来自哪些表、上游、下游、血缘，通常是血缘追溯；
出现导出、手机号、身份证、客户名单、全部明细，通常是敏感导出。

不同意图的处理路线完全不同。
离线指标走 ADS/DWS、Schema Catalog、SQL 生成、SQL 校验和离线执行；
实时状态走实时指标工具、延迟检查和告警工具；
口径解释走 RAG 并要求引用；
血缘追溯走 lineage_tool；
敏感导出走权限校验和安全阻断。

如果缺少必要上下文，例如“看一下是否异常”没有窗口、业务线或指标名，就应该先澄清。
意图分类的目标不是让模型显得聪明，而是让后续工具调用可控、可审计、可回归。

## Q098：离线/实时仓库如何支撑金融信贷 Agent 或 NL2SQL 数据问答？
来源 Day：仓库专题
### 回答
离线/实时仓库是金融信贷 Agent 和 NL2SQL 数据问答的数据底座。
如果没有仓库治理，Agent 只能看到一堆业务原始表和事件流，很容易选错表、混淆指标口径、
漏掉分区、忽略实时延迟、访问敏感字段，或者在离线任务失败、实时任务延迟时仍然给出错误结论。

生产里通常会把信贷离线数据按 ODS、DWD、DWS、ADS 分层。
ODS 保留原始业务数据，DWD 清洗成标准明细事实表，DWS 按主题和维度做轻度汇总，
ADS 面向日报、看板和经营分析沉淀应用指标。
Agent 做离线指标查询时，应该优先使用 DWS 或 ADS，而不是直接查 ODS 明细表。

实时仓库则面向实时风控、反欺诈、实时特征、告警和分钟级监控。
它需要管理实时事件、实时指标、窗口口径、事件时间、处理时间、延迟、乱序、重复和幂等。
例如近 5 分钟申请量、黑名单命中数、实时拒绝率突增和还款失败告警，都应该走实时状态查询工具，
不能和离线日报指标混在一起解释。

同时，还要把仓库元数据整理成 Schema Catalog、指标字典、实时指标字典、权限标签和血缘信息。
Schema Catalog 告诉 Agent 有哪些离线表、实时指标、业务粒度、分区字段和权限边界；
指标字典解释指标的分子、分母、过滤条件和适用范围；
实时指标字典解释窗口、延迟阈值和告警规则；
血缘信息说明指标来自哪些上游表、事件流和加工任务；
权限标签控制敏感字段和明细访问。

在工具编排上，查离线指标走 NL2SQL 链路，查实时状态走实时指标工具，
问口径走 RAG 链路，问来源走血缘工具，敏感明细查询则走安全阻断或权限校验。
这样 Agent 才能把“能回答”升级成“回答可信、可审计、可追溯、可控成本、能处理实时风险”。

<a id="q099"></a>

## Q099：生产级 AI 数据问答系统通常应该拆成哪些层？
来源 Day：Day 44
### 回答
生产级 AI 数据问答、NL2SQL 或 Agent 系统，不能只看成“用户问题进来，模型回答出去”。
我会把它拆成接入层、身份与权限层、编排层、意图识别层、上下文检索层、生成层、
安全校验层、工具执行层、结果解释层、审计日志层和评测回归层。

接入层负责 API、参数校验和统一响应；身份与权限层判断用户是谁、能访问哪些表字段和文档；
编排层决定下一步调用哪个工具，以及失败时是补充条件、降级、拒答还是转人工。
意图识别层判断用户是在查指标、问规则、查明细还是提出敏感导出请求。
上下文检索层包括 Schema 检索和 RAG 检索，负责找表字段、指标口径、政策规则和引用资料。

生成层负责生成候选 SQL、回答计划或结果解释草稿，但生成结果不能直接执行。
安全校验层要检查只读限制、敏感字段、权限边界、时间范围、limit 和成本风险。
工具执行层只执行已经满足前置条件的动作，例如执行 SQL、读取 RAG 资料或写审计。
结果解释层把结构化结果翻译成业务语言，但不能编造 SQL 没查出来的原因。
审计日志层记录 request_id、用户、问题、工具路线、SQL、校验结果、执行状态和最终回答。
评测回归层用固定测试集持续验证正常回答、安全阻断、权限不足、无答案和工具失败场景。

这套分层的核心思想是：模型可以参与理解、生成和表达，但权限、安全、执行和审计必须由确定性工程层控制。
在金融信贷场景里，这一点尤其重要，因为授信、风控、逾期、贷后和客户明细都涉及合规、成本和可追溯要求。
