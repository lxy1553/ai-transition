## 第1站：ODS 层 — 原始数据 1:1 镜像

**代码位置**: `src/data/warehouse/ods_layer.py`

ODS（Operational Data Store）的核心原则是**不做任何清洗、转换、过滤**——源系统存什么，ODS 就存什么，包括脏数据和明文敏感信息。

### 数据从哪来？

```python
# src/data/warehouse/ods_layer.py 第30-79行

@dataclass
class ODSTable:
    """ODS 表元数据"""
    name: str               # 表名
    source_system: str      # 来源系统
    ingest_method: str      # 接入方式: binlog / api_log / sdk / file

ODS_TABLES = {
    "ods_application": ODSTable(
        name="ods_application",
        source_system="mysql_credit_core",    # ← 核心业务库
        ingest_method="binlog",                # ← MySQL binlog 实时同步
    ),
    "ods_user_behavior": ODSTable(
        name="ods_user_behavior",
        source_system="sdk_analytics",         # ← 埋点SDK
        ingest_method="sdk",
    ),
    "ods_repayment": ODSTable(
        name="ods_repayment",
        source_system="mysql_credit_core",    # ← 还款系统
        ingest_method="binlog",
    ),
}
```

**为何这样设计**：用 `ODSTable` dataclass 存储每张表的元信息（来源系统、接入方式），而不是写成注释或散落在文档里。这样未来做数据血缘追溯时，可以直接读 `ODSTable.source_system` 知道数据源头，也可以做自动化校验（比如"标记为 binlog 接入的表必须包含 `dt` 分区键"）。

## 第2站：DWD 层 — 清洗 + 脱敏 + 质量评分

**代码位置**: `src/data/warehouse/dwd_layer.py`

DWD（Data Warehouse Detail）是**数据清洗工场**。ODS 的脏数据在这里被修正，敏感信息在这里脱敏，但**不聚合**——一行仍然对应一笔业务事件。

### 脱敏器

```python
# src/data/warehouse/dwd_layer.py 第48-78行

class DataMasker:
    """敏感信息脱敏器"""

    @staticmethod
    def mask_name(name: str) -> str:
        """姓名脱敏: 黄敏 → 黄*"""
        if not name or not isinstance(name, str):
            return "***"
        return name[0] + "*" * (len(name) - 1)

    @staticmethod
    def mask_id_card(id_card: str) -> str:
        """身份证脱敏: 保留前6位和后4位 → 934184********8691"""
        if not id_card or not isinstance(id_card, str) or len(id_card) < 15:
            return "INVALID_ID"
        return id_card[:6] + "********" + id_card[-4:]

    @staticmethod
    def mask_phone(phone: str) -> str:
        """手机号脱敏: 138****8795"""
        if not phone or not isinstance(phone, str) or len(phone) < 11:
            return "INVALID_PHONE"
        return phone[:3] + "****" + phone[-4:]
```

**为何这样写**：脱敏逻辑独立为 `DataMasker` 类，而不是内嵌在清洗函数里。原因：
1. **单一职责**：脱敏规则可能随监管要求变化（如 GDPR 要求更严格的脱敏），独立类便于修改
2. **可测试**：可以单独对 `DataMasker.mask_name("黄敏")` 写单元测试
3. **可复用**：多个 DWD 表都需要脱敏，不用重复代码

### 申请表清洗——逐字段扣分

```python
# src/data/warehouse/dwd_layer.py 第101-186行

def clean_application(self, ods_df: pd.DataFrame) -> tuple[pd.DataFrame, DQReport]:
    df = ods_df.copy()
    n_total = len(df)
    df['dq_score'] = 100  # ★ 初始满分100，每发现问题扣分

    # ── 1. 必填字段检查 ──
    required_fields = ['user_id', 'application_id', 'apply_amount']
    for field in required_fields:
        mask = df[field].isna() | (df[field].astype(str).str.strip() == '')
        df.loc[mask, 'dq_score'] -= 30  # 必填缺失 → 扣30分（致命）

    # ── 2. 金额合法性 ──
    amount_mask = pd.to_numeric(df['apply_amount'], errors='coerce')
    df.loc[amount_mask.isna() | (amount_mask <= 0), 'dq_score'] -= 20  # 异常金额 → 扣20分
    df['apply_amount'] = amount_mask.fillna(0).clip(lower=0)           # 修正：负数→0

    # ── 3. 产品类型标准化 ──
    valid_products = {'cash_loan': 'CASH_LOAN', 'installment': 'INSTALLMENT',
                     'revolving': 'REVOLVING'}
    df['product_type'] = df['product_type'].map(valid_products).fillna('UNKNOWN')
    df.loc[df['product_type'] == 'UNKNOWN', 'dq_score'] -= 10

    # ── 4. 敏感信息脱敏 ★ ──
    masker = DataMasker()
    df['user_name'] = df['user_name'].apply(masker.mask_name)   # 黄敏 → 黄*
    df['id_card'] = df['id_card'].apply(masker.mask_id_card)    # 934184****8691
    df['phone'] = df['phone'].apply(masker.mask_phone)          # 138****8795

    # ── 5. 隔离：dq_score < 60 的记录不入下游 ──
    quarantine_mask = df['dq_score'] < 60
    df_clean = df[~quarantine_mask].copy()
    df_quarantine = df[quarantine_mask].copy()

    # ── 6. 生成数据质量报告 ──
    report = DQReport(
        table_name='dwd_application',
        total_rows=n_total,
        passed_rows=len(df_clean),
        quarantined_rows=len(df_quarantine),  # 被隔离的数据条数
        null_rate_by_column=null_rates,
        invalid_rate_by_column=...,
    )
    return df_clean, report
```

**核心设计——扣分制**：
- 初始满分 `dq_score = 100`
- 必填缺失扣 30（致命，直接隔离）
- 金额异常扣 20（严重）
- 类型未知扣 10（轻微）
- 收入为 0 扣 5（可接受）

这样设计的好处是**量化数据质量**，而不是简单的"好/坏"二元判断。生产环境中，`dq_score` 进入监控体系，可以观察每天数据质量趋势。

## 第3站：DWS 层 — 用户风险特征宽表（核心）

**代码位置**: `src/data/warehouse/dws_layer.py`

这是整个数仓的**核心层**。DWS（Data Warehouse Summary）做了一件事：**把散落在多张 DWD 表中的同一用户的数据，聚合成一条记录——用户风险特征宽表**。一行 = 一个用户在某一天的完整风险画像。

### 宽表构建——三路聚合再合并

```python
# src/data/warehouse/dws_layer.py 第44-99行

def build_wide_table(self, dwd_application, dwd_behavior, dwd_repayment, dt):
    """
    构建用户风险特征宽表。
    输入: 各 DWD 表（明细粒度）
    输出: DWS 宽表（用户粒度，一行 = 一个用户在某天的风险画像）
    """

    # ── 1. 基础画像特征（从 dwd_application 聚合）──
    profile_features = self._build_profile_features(dwd_application)

    # ── 2. 行为衍生特征（从 dwd_behavior 滑动窗口聚合）──
    behavior_features = self._build_behavior_features(dwd_behavior, dt)

    # ── 3. 还款表现特征（从 dwd_repayment 聚合）──
    repayment_features = self._build_repayment_features(dwd_repayment)

    # ── 4. 三路合并为一张宽表 ──
    wide_table = profile_features.merge(
        behavior_features, on='user_id', how='left'
    ).merge(
        repayment_features, on='user_id', how='left'
    )

    # 填充缺失值（某些用户没有行为记录或还款记录）
    numeric_cols = wide_table.select_dtypes(include=[np.number]).columns
    wide_table[numeric_cols] = wide_table[numeric_cols].fillna(0)

    wide_table['dt'] = dt
    return wide_table
```

**为何用 left join**：不是所有用户都有行为日志或还款记录。新用户可能只有申请记录，用 left join 保证不丢数据。

### 特征 1：基础画像聚合

```python
# src/data/warehouse/dws_layer.py 第103-131行
```

```python
        night_hours = group_30d['event_time'].dt.hour.isin([22,23,0,1,2,3,4,5])
        night_ops_ratio = night_hours.mean() if len(group_30d) > 0 else 0

        page_view_7d = (group_7d['event_type'] == 'page_view').sum()
        input_7d = (group_7d['event_type'] == 'input').sum()
        error_7d = (group_7d['event_type'] == 'error').sum()

        result_rows.append({...})
```

```python

def _build_repayment_features(self, repay_df):
    agg = repay_df.groupby('user_id').agg(
        overdue_cnt_hist=('status', lambda x: (x == 'OVERDUE').sum()),  # 历史逾期次数
        total_due_amount=('due_amount', 'sum'),      # 总应还金额
        total_paid_amount=('paid_amount', 'sum'),    # 总实还金额
        repayment_cnt=('repayment_id', 'count'),     # 还款笔数
    ).reset_index()

    # ★ 衍生的按时还款率
    agg['on_time_rate'] = np.where(
        agg['repayment_cnt'] > 0,
        1 - agg['overdue_cnt_hist'] / agg['repayment_cnt'],
        1.0  # 无还款记录的新用户 → 默认为1
    )
    return agg
```

**`on_time_rate` 的含义**：如果一个用户过去 3 笔还款中有 2 笔逾期，on_time_rate = 0.33。这是还款意愿的直接量化指标。

```python
async def execute(self, request: InferenceRequest) -> DecisionResult:
    t_start = time.perf_counter()

    # asyncio.gather 真正的并发：特征服务+征信+多头+设备 同时请求
```

### 5.3 规则引擎——AST 安全求值

**代码位置**: `src/decision_engine/rule_engine.py`

```python
# src/decision_engine/rule_engine.py 第81-183行

class SafeExpressionEvaluator:
    """
    基于 Python AST 的安全表达式求值器。

    安全保证:
    1. 只允许 Compare, BoolOp, Name, Constant 节点
    2. 变量值只能来自传入的 context 字典
    3. 不支持函数调用、属性访问 → 杜绝注入攻击

    为什么不用 eval()？
      eval("__import__('os').system('rm -rf /')", context)
      → AST 解析会在第一步就拒绝 Import 节点，直接抛异常
    """

    def evaluate(self, condition: str, context: dict) -> bool:
        tree = self._compile(condition)           # "age >= 18 and age <= 65"
        return self._eval_node(tree.body, context) # 递归求值 AST

    def _eval_node(self, node: ast.AST, context: dict) -> Any:
        if isinstance(node, ast.Compare):          # a > b, a == b
            left = self._eval_node(node.left, context)
            for op_node, comparator in zip(node.ops, node.comparators):
                right = self._eval_node(comparator, context)
                op_func = self._OP_MAP.get(type(op_node))
                if op_func is None:
                    raise ValueError("不支持的操作符")
                if not op_func(left, right):
                    return False
            return True

        elif isinstance(node, ast.Name):           # 变量名
            if node.id not in context:
                raise ValueError(f"变量 '{node.id}' 未定义")  # ← 防止未定义变量
            return context[node.id]

        elif isinstance(node, ast.Constant):       # 字面量: 18, 0.8, "hello"
            return node.value

        raise ValueError(f"不支持的节点: {type(node).__name__}")  # ← 拒绝未知语法
```

**用 `ast` 模块而不用 `eval()` 的原因**：
- `eval()` 可以执行任意 Python 代码，包括 `__import__('os').system('rm -rf /')`
- AST 白名单模式只允许比较、布尔运算、变量引用、字面量，其他一切拒绝
- 规则条件写成 YAML 配置，运营人员可以修改而不担心注入

```python
# src/decision_engine/inference_pipeline.py 第375-389行

def _prob_to_score(self, prob: float) -> float:
    """
    违约概率 → 信用评分 (标准评分卡公式)

    score = base_score + factor * ln(odds)
    odds = (1-p)/p    → 好坏比率
    factor = PDO / ln(2)

    例如: p=0.05 → odds=19 → score=600 + 72.1*ln(19) ≈ 818
    """
    base_score = 600; pdo = 50; factor = pdo / np.log(2)
    prob_clipped = np.clip(prob, 1e-6, 1 - 1e-6)   # 防除零
    odds = (1 - prob_clipped) / prob_clipped
    score = base_score + factor * np.log(odds)
    return float(np.clip(score, 300, 900))
```

## 核心设计原则总结

| 原则 | 体现位置 | 说明 |
|------|----------|------|
| **数据不落地清洗** | ODS 层 | 1:1 镜像，脏数据原样保留 |
| **先脱敏再流转** | DWD 层 | PII 在 DWD 完成脱敏，下游永不见明文 |
| **扣分制质量评分** | DWD 层 | `dq_score=100` 逐项扣分，量化而非二元 |
| **PIT 正确性** | ADS 层 | `dt_特征 < dt_标签`，杜绝时间泄漏 |
| **硬规则短路** | 推理引擎 | 命中黑名单直接返回，不浪费模型算力 |
| **AST 安全求值** | 规则引擎 | 白名单模式，杜绝 `eval()` 注入 |
| **异步非阻塞** | 推理引擎 | 决策日志异步写 Kafka，不阻塞响应 |
| **降级兜底** | 推理引擎 | 特征服务超时 50ms → 缓存 → 默认值 |
