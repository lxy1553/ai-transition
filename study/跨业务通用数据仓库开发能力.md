## 基础背景：数据仓库解决的核心问题

### 为什么需要数据仓库？— 一个对比

```
┌─────────────────────────────────────────────────────────────────────┐
│                      没有数据仓库的痛苦                              │
│                                                                      │
│  运营问: "昨天注册用户多少？"                                        │
│                                                                      │
│  方案A: 直接查 MySQL 业务库                                          │
│    SELECT COUNT(*) FROM users WHERE created_at >= '2026-07-06'      │
│    ❌ 业务库承担高并发读写，跑分析查询会拖慢核心业务                  │
│    ❌ 业务库只保留当前状态，看不到历史变化                          │
│                                                                      │
│  方案B: 让后端导出 CSV，Excel 分析                                   │
│    ❌ 每天手动导出，效率低                                           │
│    ❌ 数据可能有重复、缺失，不可靠                                  │
│    ❌ 口径不一致：运营算的"新用户" vs 产品算的"新用户"不一样          │
│                                                                      │
│  方案C: 建数据仓库                                                   │
│    每天凌晨从业务库抽取 → 清洗 → 计算 → 统一口径                      │
│    ✅ 不拖慢业务库                                                   │
│    ✅ 每天自动产出，准时可靠                                         │
│    ✅ 全公司用同一套数据口径                                         │
└─────────────────────────────────────────────────────────────────────┘
```

### 一句话定义

> 数据仓库 = 把分散的业务数据集中到一个独立系统，按统一规则清理、组织、计算，为 BI 报表和 AI 模型提供**口径统一、可靠、可追溯**的数据。

## 能力 1：分层数据仓库架构设计

### 核心概念

行业标准四层架构（美团/阿里/字节通用）：

```
┌────────────────────────────────────────────────────────────────────────────┐
│                          数据仓库四层分层架构                                │
│                                                                             │
│  ┌──────────────────────────────────────────────────────────────────────┐  │
│  │                          ADS 层 (应用数据服务层)                       │  │
│  │  直接给前端/报表/模型用的表。每一个表对应一个具体业务场景。               │  │
│  │  例子: 风控日报表、模型监控大盘、用户信用报告、审批通过率看板            │  │
│  │  特点: 只做最终聚合和格式化，不做复杂计算。宽表为主。                    │  │
│  └──────────────────────────────────────────────────────────────────────┘  │
│                                    ↑                                        │
│  ┌──────────────────────────────────────────────────────────────────────┐  │
│  │                          DWS 层 (数据服务层)                           │  │
│  │  轻度汇总的公共指标，可被多个 ADS 表复用。                              │  │
│  │  例子: 用户行为特征宽表、渠道交易汇总表、风险信号聚合表                  │  │
│  │  特点: 纵向聚合(时间窗口 sum/avg/count)，不跨业务拼表                   │  │
│  └──────────────────────────────────────────────────────────────────────┘  │
│                                    ↑                                        │
│  ┌──────────────────────────────────────────────────────────────────────┐  │
│  │                          DWD 层 (数据明细层)                           │  │
│  │  清洗干净的原始数据，做标准化和维度关联。保留所有字段，一行 = 一个事件。  │  │
│  │  例子: 用户行为明细、交易明细、申请记录明细                             │  │
│  │  特点: 不做聚合，只做清洗(去重/脱敏/标准化)。粒度 = 业务事件粒度。      │  │
│  └──────────────────────────────────────────────────────────────────────┘  │
│                                    ↑                                        │
│  ┌──────────────────────────────────────────────────────────────────────┐  │
│  │                          ODS 层 (操作数据层)                           │  │
│  │  业务数据库的原样复制。不做任何处理，只做每日全量/增量抽取。             │  │
│  │  特点: 结构与源库完全一致，是"保险箱"而非"分析表"。                     │  │
│  └──────────────────────────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────────────────────────┘
```

### 1b. 通用实践：分层 DAG 编排器

```python
# ========== 四层 ETL DAG 编排 ==========
"""
  ODS (抽取/落地)
    ↓
  DWD (清洗/标准化)
    ↓
  DWS (聚合/特征化)
    ↓
  ADS (应用/报表化)

关键设计决策：
1. 同一层内的表可以并行跑 → 提高吞吐
2. 跨层必须串行 → 只有上游完成，下游才能用正确数据
3. 每个表独立定义 → 可插拔，新增表不影响已有表
"""

class LayerDAG:
    """四层依赖编排器"""

    def __init__(self, dt: str):
        self.dt = dt

    async def run_full_pipeline(self):
        # ====== Layer 1: ODS — 所有表并行 ======
        ods_tasks = [
            self._ods_mysql_table("users"),
            self._ods_mysql_table("applications"),
            self._ods_mongo_table("behavior_events"),
            self._ods_mongo_table("transaction_logs"),
        ]
        ods_results = await asyncio.gather(*ods_tasks)
        self._verify_layer("ODS", ods_results)

        # ====== Layer 2: DWD — 依赖 ODS ======
        dwd_tasks = [
            self._dwd_user_detail(),
            self._dwd_application_detail(),
            self._dwd_behavior_detail(),
            self._dwd_transaction_detail(),
        ]
        dwd_results = await asyncio.gather(*dwd_tasks)
        self._verify_layer("DWD", dwd_results)

        # ====== Layer 3: DWS — 依赖 DWD ======
        dws_tasks = [
            self._dws_user_risk_features(),
            self._dws_channel_daily_stats(),
            self._dws_transaction_aggregate(),
        ]
        dws_results = await asyncio.gather(*dws_tasks)
        self._verify_layer("DWS", dws_results)

        # ====== Layer 4: ADS — 依赖 DWS ======
        ads_tasks = [
            self._ads_risk_daily_report(),
            self._ads_model_monitor_dashboard(),
            self._ads_approval_rate_report(),
        ]
        ads_results = await asyncio.gather(*ads_tasks)
        self._verify_layer("ADS", ads_results)

        return {"status": "success", "dt": self.dt}

    def _verify_layer(self, layer_name: str, results: list):
        failures = [r for r in results if not r.get("success")]
        if failures:
            raise RuntimeError(
                f"{layer_name} 层失败 {len(failures)}/{len(results)} 个任务"
            )
```

### 每层的职责边界

| 层 | 做什么 | 不做什么 | 谁来做 | 为什么这样分 |
|----|--------|---------|--------|-------------|
| ODS | 原样抽取，不改结构 | 不做任何清洗/转换 | 数据工程师 | 保留原始数据，出了问题可回溯 |
| DWD | 清洗去重、标准化、维度关联 | 不做聚合(sum/avg) | 数据工程师 | 保证明细数据质量，下游可任意聚合 |
| DWS | 时间窗口聚合，特征计算 | 不做最终报表格式化 | AI工程师+数据工程师 | 公共指标只算一次，多个ADS复用 |
| ADS | 报表格式化，前端直出 | 不做复杂计算 | AI工程师 | 一个表=一个业务场景，清晰可控 |

**分层架构的核心设计决策**：

```python
# 为什么是四层而不是三层？

# 三层方案: ODS → DW(清洗+聚合) → ADS
#   问题: 清洗和聚合混在一起 → 清洗规则变了要重跑整个聚合
#         → 明细数据不可直接查询（被聚合吞掉了）

# 四层方案: ODS → DWD → DWS → ADS
#   优势: DWD 保留明细 → 聚合逻辑变更时从 DWD 重跑，不依赖 ODS
#         → 分析师可以直接查 DWD 明细
#         → 各层独立演化: DWD 改清洗规则不影响 DWS/ADS
```

### 2a. 星型模型 — 标准维度建模（适合 BI 分析）

```
┌─────────────────────────────────────────────────────────────────────────┐
│                          星型模型 (Star Schema)                          │
│                                                                          │
│                     ┌───────────────────────┐                           │
│                     │     dim_user           │                           │
│                     │   (用户维度表)          │                           │
│                     ├───────────────────────┤                           │
│                     │ user_id (PK)           │────┐                      │
│                     │ age_group / city       │    │                      │
│                     └───────────────────────┘    │                      │
│                                                   │                      │
│  ┌───────────────────────┐  ┌─────────────────┐  │  ┌────────────────┐  │
│  │     dim_product        │  │  FACT TABLE     │  │  │  dim_channel    │  │
│  │   (产品维度表)          │  │  (事实表)        │  │  │ (渠道维度表)    │  │
│  ├───────────────────────┤  ├─────────────────┤  │  ├────────────────┤  │
│  │ product_id (PK) ──────┼──│ product_id (FK) │  │  │ channel_id(PK)──┤  │

│  └───────────────────────┘  │ dt              │     └────────────────┘  │
│                              │ apply_amount    │                         │
│  ┌───────────────────────┐  │ is_approved     │                         │
│  │     dim_date           │  │ credit_score    │                         │
│  │   (日期维度表)          │  └─────────────────┘                         │
│  ├───────────────────────┤                                               │
│  │ dt (PK) ──────────────┼── dt (FK)                                     │
│  │ year, month, is_holiday│                                               │
│  └───────────────────────┘                                               │
│                                                                          │
│  核心原则: 事实表只存 ID 和度量值 → 维度表存描述信息 → JOIN 获取完整信息  │
└─────────────────────────────────────────────────────────────────────────┘
```

```sql
-- 星型模型 DDL
CREATE TABLE dim_user (
    user_id BIGINT PRIMARY KEY,
    user_name VARCHAR(100),
    age_group VARCHAR(20),       -- '18-25', '26-35', '36-45', '46+'
    city VARCHAR(50),
    register_date DATE
) COMMENT '用户维度表';

CREATE TABLE dim_date (
    dt DATE PRIMARY KEY,
    year SMALLINT, month TINYINT, day TINYINT,
    is_weekend BOOLEAN, is_holiday BOOLEAN
) COMMENT '日期维度表';

CREATE TABLE fact_credit_application (
    application_id BIGINT PRIMARY KEY,
    user_id BIGINT NOT NULL,              -- FK → dim_user
    dt DATE NOT NULL,                     -- FK → dim_date
    apply_amount DECIMAL(18,2),
    is_approved BOOLEAN,
    credit_score SMALLINT,
    INDEX idx_user (user_id), INDEX idx_dt (dt)
) COMMENT '信用申请事实表'
PARTITION BY RANGE (TO_DAYS(dt)) ();

-- 分析查询：各年龄段各城市审批通过率
SELECT u.age_group, u.city, d.year, d.month,
       COUNT(*) AS total_applications,
       SUM(CASE WHEN f.is_approved THEN 1 ELSE 0 END) AS approved,
       ROUND(SUM(CASE WHEN f.is_approved THEN 1 ELSE 0 END) * 100.0 / COUNT(*), 2) AS approval_rate
FROM fact_credit_application f
JOIN dim_user u ON f.user_id = u.user_id
JOIN dim_date d ON f.dt = d.dt
WHERE d.year = 2026 AND d.month = 7
GROUP BY u.age_group, u.city, d.year, d.month
ORDER BY approval_rate DESC;
```

### 2b. 宽表设计 — AI 模型的首选

```
为什么 AI/ML 场景用宽表而不是星型模型？

┌──────────────────────────────────────────────────────────────────────┐
│  星型模型 (适合 BI 分析):                                             │
│    每次查询都要 JOIN 维度表 → 分析师可以灵活切维度                   │
│                                                                       │
│  宽表 (适合 AI 模型):                                                 │
│    一行 = 一个样本的全部特征 → 不用 JOIN → 模型训练快 10 倍+          │
│                                                                       │
│  为什么模型不能用星型？                                               │
│  1. 模型需要几百列特征 → 每次训练 JOIN 5 个维度表 → 慢 10 倍         │
│  2. 特征快照: 需要 "2026-07-01 这个时刻的特征"                       │
│     → 维度表可能已经更新(改年龄、改城市) → 必须用历史快照            │
│  3. SHAP 可解释性: 宽表每一列 = 一个特征 → 直接对应 SHAP 贡献值      │
└──────────────────────────────────────────────────────────────────────┘
```

```sql
-- 宽表 DDL (供 AI 模型使用)
CREATE TABLE dws_user_risk_feature_wide (
    user_id   BIGINT,
    dt        DATE,                    -- 特征快照日期
    -- 基础维度
    age       TINYINT,  city_code VARCHAR(10),  register_days INT,
    -- 行为特征 (DWD聚合)
    apply_cnt_7d INT,  apply_cnt_30d INT,  night_ops_ratio_30d DOUBLE,
    page_view_cnt_7d INT,  error_event_cnt_7d INT,
    -- 交易特征
    total_transaction_amt_30d DECIMAL(18,2),  avg_transaction_amt_30d DECIMAL(18,2),
    -- 多头借贷
    multi_loan_platform_count INT,  total_debt_amount DECIMAL(18,2),
    -- 信用历史
    overdue_cnt_hist INT,  on_time_rate DOUBLE,
    PRIMARY KEY (user_id, dt)
) COMMENT '用户风险特征宽表 — 一行 = 一个样本'
PARTITION BY RANGE (TO_DAYS(dt)) ();

-- 宽表生成 SQL
INSERT INTO dws_user_risk_feature_wide
SELECT u.user_id, '${dt}' AS dt, u.age, u.city_code,
       DATEDIFF('${dt}', u.register_date) AS register_days,
       COALESCE(b.apply_cnt_7d, 0), COALESCE(b.apply_cnt_30d, 0),
       COALESCE(b.night_ops_ratio, 0.0),
       COALESCE(t.total_amt_30d, 0), COALESCE(t.avg_amt_30d, 0),
       COALESCE(m.platform_count, 0),
       COALESCE(c.overdue_cnt, 0), COALESCE(c.on_time_rate, 1.0)
FROM dim_user u
LEFT JOIN dws_user_behavior_agg b  ON u.user_id = b.user_id AND b.dt = '${dt}'
LEFT JOIN dws_transaction_agg t    ON u.user_id = t.user_id AND t.dt = '${dt}'
LEFT JOIN dws_multi_loan_agg m     ON u.user_id = m.user_id AND m.dt = '${dt}'
LEFT JOIN dws_credit_history_agg c ON u.user_id = c.user_id AND c.dt = '${dt}';
-- ★ LEFT JOIN: 全量用户都要有特征行。新用户特征为 0/默认值。
```

### 2c. 项目实践：宽表构建的三路聚合

```python
# src/data/warehouse/dws_layer.py 第44-99行

def build_wide_table(self, dwd_application, dwd_behavior, dwd_repayment, dt):
    """
    构建用户风险特征宽表。

    关键设计决策 1: 粒度 = 用户 × 日期
    为什么不是用户粒度？用户的状态在变（收入、行为、还款）。
    用户×日期保留时间维度的变化 → 模型可以学习"某人越来越差"的趋势。

    关键设计决策 2: left join 而非 inner join
    新用户可能只有申请记录，没有行为日志和还款记录。
    inner join → 新用户被丢弃 → 模型对新用户失效。

    关键设计决策 3: 先聚合再 Join（而非先 Join 再聚合）
    先 Join 再聚合会产生笛卡尔积：申请表 5行 × 行为表 100行 → 500行 → count 被放大。
    """

    # Step 1: 三路独立聚合（各自压缩到用户粒度）
    profile_features = self._build_profile_features(dwd_application)
    behavior_features = self._build_behavior_features(dwd_behavior, dt)
    repayment_features = self._build_repayment_features(dwd_repayment)

    # Step 2: left join 合并
    wide_table = (
        profile_features
        .merge(behavior_features, on='user_id', how='left')
        .merge(repayment_features, on='user_id', how='left')
    )

    # Step 3: 缺失值填充 ("无信息"用 0 表示，而非均值——均值 = 伪造信息)
    numeric_cols = wide_table.select_dtypes(include=[np.number]).columns
    wide_table[numeric_cols] = wide_table[numeric_cols].fillna(0)
    wide_table['dt'] = dt
    return wide_table


def _build_profile_features(self, app_df):
    """申请画像: 6维特征。monthly_income 取 max 而非 avg——取最诚实的填报值。"""
    return app_df.groupby('user_id').agg(
        apply_amount_avg=('apply_amount', 'mean'),
        apply_amount_max=('apply_amount', 'max'),
        monthly_income=('monthly_income', 'max'),       # ★ max
        total_apply_cnt=('application_id', 'nunique'),
        distinct_device_cnt=('device_id', 'nunique'),
        distinct_city_cnt=('city', 'nunique'),
    ).reset_index()


def _build_repayment_features(self, repay_df):
    """还款特征: 5维。新用户 on_time_rate 默认 1.0（无罪推定）"""
    agg = repay_df.groupby('user_id').agg(
        overdue_cnt_hist=('status', lambda x: (x == 'OVERDUE').sum()),
        total_due_amount=('due_amount', 'sum'),
        total_paid_amount=('paid_amount', 'sum'),
        repayment_cnt=('repayment_id', 'count'),
    ).reset_index()
    agg['on_time_rate'] = np.where(
        agg['repayment_cnt'] > 0,
        1 - agg['overdue_cnt_hist'] / agg['repayment_cnt'],
        1.0  # ★ 新用户无罪推定
    )
    return agg
```

### 3c. 重跑机制 — 面试高频考点

```python
class BackfillOrchestrator:
    """
    从指定日期开始重新计算整条链路的数据。

    需要重跑的场景:
    1. 源数据修正: 业务库发现脏数据 → 修复后重跑
    2. 逻辑变更: DWD 清洗规则改了 → 重刷历史
    3. 特征回填: 新加了 DWS 特征列 → 回溯 90 天
    4. 口径调整: "逾期率"统计口径变了 → 重算历史

    ★ 核心原则: 从上游到下游逐层重跑，不能跳过！
      ODS → DWD → DWS → ADS
      跳过 DWD 直接重跑 DWS → DWS 读的还是旧 DWD → 白跑
    """

    def __init__(self, start_dt: str, end_dt: str, layers: list[str] = None):
        self.start_dt, self.end_dt = start_dt, end_dt
        self.layers = layers or ['ODS', 'DWD', 'DWS', 'ADS']

    async def backfill(self) -> dict:
        results = {}
        for layer in self.layers:
            results[layer] = []
            tables = self._get_layer_tables(layer)
            for dt in self._get_date_range():
                for table in tables:
                    try:
                        self._verify_upstream_ready(layer, table, dt)
                        await self._rerun_table(layer, table, dt)
                        results[layer].append({"layer": layer, "table": table, "dt": dt, "status": "success"})
                    except Exception as e:
                        results[layer].append({"layer": layer, "table": table, "dt": dt, "status": "failed", "error": str(e)})
                        break  # 同表后续日期跳过
        return results

    def _verify_upstream_ready(self, layer, table, dt):
        """确保上游分区存在且数据完整"""
        for up_table in self._get_upstream_tables(layer, table):
            if not self._partition_exists(up_table, dt):
                raise RuntimeError(f"上游 {up_table} 分区 {dt} 不存在！请先重跑 {up_table}。")

# 场景1: 修复脏数据 - 全部四层
# BackfillOrchestrator('2026-07-01', '2026-07-03', layers=['ODS','DWD','DWS','ADS']).backfill()
# 场景2: 回溯新特征 - 只重跑聚合层
# BackfillOrchestrator('2026-04-08', '2026-07-06', layers=['DWS','ADS']).backfill()
```

## 能力 4：SQL 优化 — 从 30 分钟到 3 秒

### 十种经典优化模式

```
┌──────────────────────────────────────────────────────────────────────┐
│                  SQL 优化：先有数据量感知，再谈技术手段                │
│                                                                       │
│  数据量对照表:                                                        │
│    1 万行   → 怎么查都快，不用优化                                    │
│    10 万行  → JOIN 还行，子查询要小心                                 │
│    100 万行 → 必须用索引+分区，子查询大概率炸                          │
│    1000 万行→ 分区+索引是底线，JOIN 要评估                            │
│    1 亿行   → 每条 SQL 都要过优化 check list                          │
└──────────────────────────────────────────────────────────────────────┘
```

```sql
-- 优化 1: 分区裁剪 — 最重要，没有之一
-- ❌ 全表扫描 14 秒
SELECT COUNT(*) FROM fact_application WHERE create_date >= '2026-06-01';
-- ✅ 分区裁剪 0.3 秒（快 47 倍）
SELECT COUNT(*) FROM fact_application WHERE dt IN ('2026-06-01','2026-06-02','2026-06-03');

-- 优化 2: JOIN 替代子查询
-- ❌ 子查询 30 秒（先全量执行子查询，再逐行匹配）
SELECT * FROM fact WHERE user_id IN (SELECT user_id FROM dim_user WHERE age_group='36-45');
-- ✅ JOIN 0.5 秒（Hash Join，一次扫描）
SELECT f.* FROM fact f INNER JOIN dim_user u ON f.user_id=u.user_id WHERE u.age_group='36-45';

-- 优化 3: 窗口函数替代自连接
-- ❌ 自连接 15 秒（O(n²)）
SELECT a.*, (SELECT AVG(b.amount) FROM fact b WHERE b.user_id=a.user_id) FROM fact a;
-- ✅ 窗口函数 1 秒（O(n log n)，只扫描一次）
SELECT *, AVG(amount) OVER (PARTITION BY user_id) AS user_avg FROM fact;

-- 优化 4: UNION ALL 替代 OR（索引失效时）
-- ❌ OR 让索引失效 8 秒
SELECT * FROM fact WHERE (user_id=12345 OR apply_amount>50000);
-- ✅ UNION ALL 各自走索引 0.05 秒
SELECT * FROM fact WHERE user_id=12345 UNION ALL
SELECT * FROM fact WHERE apply_amount>50000 AND user_id!=12345;

-- 优化 5: COUNT(*) 替代 COUNT(column)
-- ❌ COUNT(column): 要检查每行的值是否为 NULL
SELECT COUNT(status) FROM fact;
-- ✅ COUNT(*): 只数行数，引擎最优化路径（除非你真要排除 NULL）

-- 优化 6: Broadcast Join (大表 JOIN 小表)
-- ❌ Shuffle Join 30 分钟
SELECT f.*, u.age_group FROM fact f JOIN dim_user u ON f.user_id=u.user_id;
-- ✅ Broadcast Join 2 分钟（小表广播到所有节点）
SELECT /*+ BROADCAST(u) */ f.*, u.age_group FROM fact f JOIN dim_user u ON f.user_id=u.user_id;

-- 优化 7: 避免 SELECT *
-- ❌ 50 列 → 500MB 网络传输
SELECT * FROM dws_user_risk_feature_wide WHERE dt='2026-07-01';
-- ✅ 5 列 → 50MB（快 10 倍）

-- 优化 8: GROUPING SETS 替代多次 UNION ALL
-- ❌ 三次全表扫描
SELECT age_group, NULL AS city, COUNT(*) FROM t GROUP BY age_group
UNION ALL SELECT NULL, city, COUNT(*) FROM t GROUP BY city
UNION ALL SELECT age_group, city, COUNT(*) FROM t GROUP BY age_group, city;
-- ✅ 一次扫描，同时产出三个聚合
SELECT age_group, city, COUNT(*) FROM t
GROUP BY GROUPING SETS ((age_group), (city), (age_group, city));

-- 优化 9: EXPLAIN 验证你的优化
EXPLAIN SELECT ...;
-- 关注: type: ALL→需优化 / range→走索引 / const→常量
--        rows: 扫描行数（优化后应大幅减少）
--        Extra: Using filesort→需优化 / Using index→最优

-- 优化 10: LIMIT + ORDER BY 走索引
-- ❌ 全表排序后取 Top 10 20 秒
SELECT * FROM fact ORDER BY apply_amount DESC LIMIT 10;
-- ✅ 有索引直接定位 0.001 秒
CREATE INDEX idx_amount ON fact(apply_amount);
```

## 能力 5：数据质量体系 — 扣分制 + 六维度校验

### 5a. 扣分制质量评分（项目实践）

数据质量管理不是简单的"通过/不通过"。生产中的真实情况是：数据可能部分损坏，但整体仍可用。扣分制比二元判定更精细。

```python
# src/data/warehouse/dwd_layer.py 第101-186行

def clean_application(self, ods_df: pd.DataFrame) -> tuple[pd.DataFrame, DQReport]:
    """
    核心设计: 扣分制 dq_score
    - 初始 100 分
    - 每发现一个问题扣分
    - dq_score >= 60 → 通过，进入下游
    - dq_score < 60  → 隔离，单独存储但不丢失

    为什么是扣分制而不是简单的 True/False？
    1. 数据质量是一个频谱，不是二元状态
       - 所有列OK(100分) vs 仅产品类型未知(90分) vs 必填缺失(40分)
         → 三种完全不同的处理策略
    2. 精细化运营: dq_score 趋势下降 → 源系统可能出问题 → 提前告警
    3. 下游可根据 dq_score 降级使用
    """
    df = ods_df.copy()
    df['dq_score'] = 100

    # 必填缺失 → 扣 30 分（致命）
    # 为什么 30？缺失任一关键字段→降到70→再有任意问题就隔离。10太宽松，50太严格。
    for field in ['user_id', 'application_id', 'apply_amount']:
        mask = df[field].isna() | (df[field].astype(str).str.strip() == '')
        df.loc[mask, 'dq_score'] -= 30

    # 金额异常 → 扣 20 分（严重）
    # 为什么 20？金额是核心字段但不是"无它就不可用"的主键。
    amount_mask = pd.to_numeric(df['apply_amount'], errors='coerce')
    df.loc[amount_mask.isna() | (amount_mask <= 0), 'dq_score'] -= 20
    df['apply_amount'] = amount_mask.fillna(0).clip(lower=0)

    # 产品类型未知 → 扣 10 分（轻微）
    # 为什么 10？产品类型不是核心风控字段，缺失了标记为 UNKNOWN 即可。
    valid_products = {'cash_loan': 'CASH_LOAN', 'installment': 'INSTALLMENT', 'revolving': 'REVOLVING'}
    df['product_type'] = df['product_type'].map(valid_products).fillna('UNKNOWN')
    df.loc[df['product_type'] == 'UNKNOWN', 'dq_score'] -= 10

    # 隔离: dq_score < 60 不入下游
    # 为什么 60？及格线——可接受1个严重(30)+1个中等(20)，或3个轻微(10×3)。
    quarantine_mask = df['dq_score'] < 60
    df_clean = df[~quarantine_mask].copy()

    report = DQReport(
        table_name='dwd_application', total_rows=len(df),
        passed_rows=len(df_clean), quarantined_rows=len(df) - len(df_clean),
        null_rate_by_column=null_rates, invalid_rate_by_column=invalid_rates,
    )
    return df_clean, report

    quarantined_rows(隔离率突增=源系统有问题)、null/invalid_rate_by_column(定位具体字段)
    """

    total_rows: int
    passed_rows: int
    quarantined_rows: int
    null_rate_by_column: dict[str, float]
    invalid_rate_by_column: dict[str, float]
```

### 5b. 六维度校验引擎（通用框架）

```
┌──────────────────────────────────────────────────────────────────────────┐
│                     数据质量六维度校验                                     │
│                                                                           │
│  1. 完整性 (Completeness)   — 该有的数据有没有？                          │
│  2. 唯一性 (Uniqueness)     — 主键是否唯一？                              │
│  3. 一致性 (Consistency)    — 跨表数据是否矛盾？                          │
│  4. 准确性 (Accuracy)       — 数值是否在合理范围？                        │
│  5. 及时性 (Timeliness)     — 数据是否按时产出？（T+1 每天8:00前就绪）     │
```

```python
class DataQualityEngine:
    """通用数据质量校验引擎 — 跨业务通用"""

    def __init__(self):
        self.checks = self._register_default_checks()

    def _register_default_checks(self) -> list[QualityCheck]:
        return [
            QualityCheck("row_count_not_zero", "completeness", "FATAL",
                lambda df: (len(df) > 0, f"行数 {len(df)}，预期 > 0")),
            QualityCheck("no_null_in_pk", "completeness", "ERROR",
                lambda df, pk: (df[pk].notna().all(), f"主键 {pk} 有 NULL")),
            QualityCheck("pk_unique", "uniqueness", "FATAL",
                lambda df, pk: (len(df) == len(df.drop_duplicates(subset=pk)), f"主键重复")),
            QualityCheck("value_in_range", "accuracy", "WARN",
                lambda df, col, lo, hi: (df[col].between(lo, hi).all(), f"{col} 超范围")),
            QualityCheck("date_format_valid", "validity", "ERROR",
                lambda df, col: (pd.to_datetime(df[col], errors='coerce').notna().all(), f"日期格式无效")),
        ]

    def validate(self, table_name, dt, df, pk_columns=None, value_ranges=None) -> QualityReport:
        """对一张表执行完整六维度校验"""
        report = QualityReport(table=table_name, dt=dt, total_checks=len(self.checks))
        for check in self.checks:
            try:
                # 根据 check.name 调度对应的校验逻辑
                if check.name == "row_count_not_zero":
                    passed, detail = check.check_fn(df)
                elif check.name == "pk_unique" and pk_columns:
                    passed, detail = check.check_fn(df, pk_columns)
                elif check.name == "value_in_range" and value_ranges:
                    for col, (lo, hi) in value_ranges.items():
                        if col in df.columns:
                            passed, detail = check.check_fn(df, col, lo, hi)
                            if not passed: break
                # ... 其他调度逻辑
                if not passed:
                    report.failed += 1
                    if check.severity == "FATAL":
                        raise RuntimeError(f"FATAL [{table_name}] {check.name}: {detail}")
            except RuntimeError: raise
            except Exception as e:
                report.failed += 1
        return report

# 使用: ETL + 质量门槛
async def etl_with_quality_gate(dt: str):
    qe = DataQualityEngine()
    dwd_df = await get_dwd_user_detail(dt)
    report = qe.validate("dwd_user_detail", dt, dwd_df,
        pk_columns=["user_id"], value_ranges={"age": (0, 120), "credit_score": (300, 900)})
    if report.pass_rate < 0.90:
        raise RuntimeError(f"质量不达标，通过率仅 {report.pass_rate:.0%}")
    await run_dws_layer(dt)
```

## 能力 6：PII 敏感数据脱敏与合规

### 核心方法论

脱敏不是简单的"全删"。好的脱敏策略需要平衡**隐私保护**和**数据可用性**。

### 核心代码：脱敏器设计

```python
# src/data/warehouse/dwd_layer.py 第48-78行

class DataMasker:
    """
    敏感信息脱敏器。

    设计原则:
    1. 每个方法独立 — 脱敏规则可能随监管要求单独变更
    2. 输入校验 — 空值、非字符串、异常格式 → 返回标记值而非崩溃
    3. 独立静态方法 — 可单独测试、可单独替换
    """

    @staticmethod
    def mask_name(name: str) -> str:
        """
        姓名脱敏: 张三 → 张*
        为什么保留首字？姓氏分布分析、地域分析仍可用。姓名不进入模型特征。
        """
        if not name or not isinstance(name, str):
            return "***"
        return name[0] + "*" * (len(name) - 1)

    @staticmethod
    def mask_id_card(id_card: str) -> str:
        """
        身份证脱敏: 934184196401168691 → 934184********8691
        为什么保留前6位和后4位？
        - 前6位=地区码 → 可衍生"户籍省份""城市等级"特征
        - 后4位=顺序码 → 可去重但不可反向定位
        - 脱敏后仍可用于模型: "北京人 vs 上海人的违约率差异"
        """
        if not id_card or not isinstance(id_card, str) or len(id_card) < 15:
            return "INVALID_ID"
        return id_card[:6] + "********" + id_card[-4:]

    @staticmethod
    def mask_phone(phone: str) -> str:
        """
        手机号脱敏: 13872128795 → 138****8795
        为什么保留前3位和后4位？
        - 前3位=运营商号段 → 可衍生"运营商类型"
        - 后4位=用户标识 → 可去重但不可反向定位
        """
        if not phone or not isinstance(phone, str) or len(phone) < 11:
            return "INVALID_PHONE"
        return phone[:3] + "****" + phone[-4:]

    @staticmethod
    def hash_user_id(user_id: str) -> str:
        """user_id 哈希化（不可逆）— 用于绝对不能暴露原始值的场景"""
        if not user_id or not isinstance(user_id, str):
            return "INVALID_UID"
        return hashlib.sha256(user_id.encode()).hexdigest()[:16]
```

**脱敏策略的四级选择矩阵**：

```python
脱敏策略矩阵 = {
    # 策略1: 掩码(Mask) — 保留部分结构
    #   适用于: 需要统计分析但不需要精准定位 → 姓名(黄*)、身份证(前6后4)
    #
    # 策略2: 哈希(Hash) — 完全不可逆但可去重
    #   适用于: 只需关联/去重 → user_id → SHA256
    #
    # 策略3: 泛化(Generalize) — 映射到更高层级
    #   适用于: 需要分析但不需要精确值 → IP 192.168.x.x、年龄 25-30
    #
    # 策略4: 加密(Encrypt) — 可逆但需密钥
    #   适用于: 需要保留原始值但有权限控制 → 银行卡号加密，只有结算部门可解密
}
```

```python
class SchemaRegistry:
    """
    表结构注册中心 — 数据仓库的"目录服务"。

    职责:
    1. 从 config/schemas/*.yaml 加载表结构定义
    2. 写入时将 _TABLE_SCHEMA.json 存入数据目录（schema随数据走）
    3. 提供统一的表查询接口

    为什么需要 SchemaRegistry？
    1. NL2SQL 可程序化读取表结构构造 Prompt
    2. 写入时校验列名、类型是否与 schema 一致
    3. Schema 文件和代码一起 git 管理，变更可追溯
    4. 数据被复制/迁移时，schema 文件跟随 → 知识不丢失
    """

    def __init__(self, config_dir: str = "config"):
        self.schemas_dir = Path(config_dir) / "schemas"
        self._tables: dict[str, TableSchema] = {}
        self._load_all()

    def write_schema_to_data_dir(self, data_dir, table_name, layer="ods") -> Path:
        """将表结构 JSON 写入数据目录 → 数据自带文档"""
        schema = self._tables.get(f"{layer}.{table_name}")
        out_dir = Path(data_dir)
        schema_file = out_dir / f"_TABLE_SCHEMA_{table_name}.json"
        with open(schema_file, 'w', encoding='utf-8') as f:
            json.dump(schema.to_dict(), f, indent=2, ensure_ascii=False)
        return schema_file

    def get_table(self, layer: str, table_name: str) -> Optional[TableSchema]:
        return self._tables.get(f"{layer}.{table_name}")

    def list_tables(self, layer: Optional[str] = None) -> list[TableSchema]:
        tables = list(self._tables.values())
        return [t for t in tables if t.layer == layer] if layer else tables
```

### 8b. 通用实践：LineageRegistry — 程序化血缘追踪

```python
@dataclass
class ColumnLineage:
    """单列血缘"""
    table_name: str; column_name: str; column_type: str; description: str
    source_table: str | None = None; source_column: str | None = None
    transform_logic: str | None = None

@dataclass
class TableLineage:
    """一张表的完整血缘"""
    table_name: str; layer: str; owner: str; update_frequency: str; description: str
    upstream_tables: list[str] = field(default_factory=list)
    columns: list[ColumnLineage] = field(default_factory=list)


class LineageRegistry:
    """血缘注册中心"""

    def __init__(self):
        self._tables: dict[str, TableLineage] = {}
        self._init_defaults()  # 注册 ODS/DWD/DWS/ADS 所有表

    def get_forward_lineage(self, table_name: str) -> list[str]:
        """正向血缘：从源头往下 → 影响哪些下游？"""
        affected, to_visit = [], [table_name]
        while to_visit:
            current = to_visit.pop(0)
            for downstream in self._get_downstream(current):
                if downstream not in affected:
                    affected.append(downstream); to_visit.append(downstream)
        return affected

    def get_backward_lineage(self, table_name: str) -> list[str]:
        """反向血缘：从报表往回 → 数据怎么来的？"""
        upstream, to_visit = [], [table_name]
        while to_visit:
            current = to_visit.pop(0)
            for up in self._tables.get(current, TableLineage(...)).upstream_tables:
                if up not in upstream:
                    upstream.append(up); to_visit.append(up)
        return upstream

    def get_column_lineage_chain(self, table: str, column: str) -> str:
        """列级血缘：一个列从上到下的完整计算链路"""
        for col in self._tables.get(table, TableLineage(...)).columns:
            if col.column_name == column:
                chain = [f"{col.table_name}.{col.column_name}: {col.transform_logic}"]
                if col.source_table and col.source_column:
                    chain.append(self.get_column_lineage_chain(col.source_table, col.source_column))
                return " ← ".join(chain)
        return f"列 {table}.{column} 无血缘记录"

# 使用示例:
reg = LineageRegistry()
reg.get_forward_lineage("ods_mysql_applications")   # → ['dwd_application_detail', 'dws_user_risk_features']
reg.get_backward_lineage("dws_user_risk_features")   # → ['dwd_application_detail', 'ods_mysql_applications']
reg.get_column_lineage_chain("dws_user_risk_features", "apply_cnt_7d")
# → dws_user_risk_features.apply_cnt_7d: COUNT(...) ← dwd_application_detail.application_id
```

## 能力 9：数据产品设计 — 从宽表到可交付的数据产品

### 核心方法论

ADS 层不是简单的"把 DWS 表改个名"。好的数据产品需要：明确的消费者、明确的指标口径、自包含（消费者不需再关联其他表）。

```python
# src/data/warehouse/ads_layer.py — 三种数据产品

class ADSLayer:
    """ADS 层 — 面向应用的数据集市。"""

    # ═══ 数据产品 1: 训练样本集 (Parquet) ═══
    def build_training_samples(self, dws_wide_table, label_df):
        """
        为什么训练样本是 ADS 产品而不是 DWS 直接输出？
        1. 标签来源独立(DWS 不包含 label)
        2. 训练样本需要 PIT 约束(时间窗口参数)
        3. 不同模型需要不同样本(不同窗口、不同标签定义)
        4. ADS 隔离: 模型训练人员只消费 ADS，不需了解 DWD/DWS 细节
        """
        return dws_wide_table.merge(label_df, on='user_id', how='inner')

    # ═══ 数据产品 2: 模型监控日报 (CSV → Grafana) ═══
    def build_model_monitor_daily(self, predictions, dws_wide_table, dt):
        """
        为什么预聚合为每日一条？
        Grafana 每次刷新如果扫描百万行 → 5s；预聚合后只查30行 → 50ms。

        为什么包含 score_p10/p50/p90 三个分位数？
        avg_score 只能反映"平均水平"，掩盖分布变化。
        如果评分从正态变为两极分化，avg 不变但风控效果巨变。
        """
        n_total = len(predictions)
        return pd.DataFrame([{
            'dt': dt,
            'total_applications': n_total,
            'approval_rate': round(len(predictions[predictions['decision']=='APPROVE'])/n_total, 4),
            'reject_rate': round(len(predictions[predictions['decision']=='REJECT'])/n_total, 4),
            'avg_score': round(predictions['score'].mean(), 2),
            'score_std': round(predictions['score'].std(), 2),
            'score_p10': round(predictions['score'].quantile(0.10), 2),
            'score_p50': round(predictions['score'].quantile(0.50), 2),
            'score_p90': round(predictions['score'].quantile(0.90), 2),
            'avg_credit_limit': round(predictions[predictions['credit_limit']>0]['credit_limit'].mean(), 2),
        }])

    # ═══ 数据产品 3: 资产组合分析 (JSON → 风控报表/BI) ═══
    def build_portfolio_analysis(self, decisions, dws_wide_table):
        """
        为什么用 JSON 而不是 CSV/Parquet？
        - 消费者是 Web 前端或 BI 工具
        - 包含嵌套结构(score_distribution 是一个 dict)
        - JSON: 人类可读 + 程序可解析 + 前端可直接渲染
        """
        def score_bucket(s):
            if s>=750: return 'A+'
            if s>=700: return 'A'
            if s>=650: return 'B+'
            if s>=600: return 'B'
            if s>=500: return 'C'
            return 'D'

        merged['bucket'] = merged['score'].apply(score_bucket)
        return {
            'total_portfolio': int(len(merged)),
            'total_credit_exposure': round(float(merged['credit_limit'].sum()), 2),
            'score_distribution': {k: int(v) for k,v in merged['bucket'].value_counts().to_dict().items()},
            'avg_score_by_bucket': {k: round(float(v),1) for k,v in merged.groupby('bucket')['score'].mean().to_dict().items()},
        }
```

```python
    source_table: str         # 数据来源表


class MetricRegistry:
    """指标口径注册中心 — 全公司唯一的数据字典"""

    def __init__(self):
        self.metrics: dict[str, MetricDefinition] = {}
        self._init_defaults()

    def _init_defaults(self):
        self.register(MetricDefinition(
            metric_id="approval_rate", metric_name="审批通过率",
            owner="风控策略组", version="v2.1",
            business_definition="审批通过数 / 已完成审批总数",
            numerator_formula="COUNT(DISTINCT CASE WHEN final_status='APPROVED' THEN app_id END)",
            denominator_formula="COUNT(DISTINCT CASE WHEN final_status IN ('APPROVED','REJECTED') THEN app_id END)",
            exclusions=["final_status='MANUAL_REVIEW'", "apply_amount<100", "user_type='TEST'"],
            source_table="dwd_application_detail"
        ))
        self.register(MetricDefinition(
            metric_id="overdue_rate_30d", metric_name="30天逾期率",
            owner="风控模型组", version="v1.0",
            business_definition="放款后30天内逾期订单数 / 总放款订单数",
            numerator_formula="COUNT(DISTINCT CASE WHEN overdue_days>=30 THEN loan_id END)",
            denominator_formula="COUNT(DISTINCT loan_id)",
            exclusions=["loan_status='PREPAID'"],
            source_table="dwd_loan_performance"
        ))

    def register(self, metric: MetricDefinition):
        self.metrics[metric.metric_id] = metric

    def generate_sql(self, metric_id: str, dt: str) -> str:
        """根据口径自动生成 SQL — 确保所有系统用同一口径"""
        m = self.metrics[metric_id]
        exclusion_clause = ""
        if m.exclusions:
            exclusion_clause = "\n  AND " + "\n  AND ".join(f"NOT ({e})" for e in m.exclusions)
        return f"""
-- 指标: {m.metric_name} ({m.metric_id}) v{m.version}
-- 口径: {m.business_definition}
SELECT '{dt}' AS dt,
       {m.numerator_formula} AS numerator,
       {m.denominator_formula} AS denominator,
       ROUND({m.numerator_formula} * 100.0 / NULLIF({m.denominator_formula}, 0), 2) AS {m.metric_id}
FROM {m.source_table} WHERE dt = '{dt}'{exclusion_clause}
"""

# 所有 BI 报表、AI 模型、运营后台都调用同一个 generate_sql → 数字永远一致
```

## 面试高频考点 & 深度回答

### Q1: ODS、DWD、DWS、ADS 四层分别是什么？为什么要分层？

> ODS 是原样复制（保险箱），DWD 是清洗标准化（数据质量层），DWS 是轻度聚合（公共指标层），ADS 是应用报表（业务场景层）。
>
> 为什么是四层而不是三层？三层方案(ODS→DW→ADS)把清洗和聚合混在一起——清洗规则变了要重跑整个聚合，明细数据被聚合吞掉不可直接查询。四层方案保留 DWD 明细层，聚合变更从 DWD 重跑不依赖 ODS，各层独立演化。
>
> 分层的核心原因是**解耦**：源头变动不影响下游、中间层可复用、问题可追溯、权限可隔离。

### Q2: 增量抽取 vs 全量抽取怎么选？

> 决策三要素：
> 1. **有没有更新时间列？** 有 → 增量；没有 + 表小 → 全量；没有 + 表大 → 协商加列。
> 2. **数据量多大？** < 100 万行 → 全量省心；> 1000 万行 → 必须增量。
> 3. **需不需要历史快照？** 需要每天用户状态快照 → 全量；只需最新状态 → 增量。
>
> ODS 层大部分用增量（减少源库压力），DWS/ADS 层用 INSERT OVERWRITE 覆盖写（保证幂等性）。

### Q3: SQL 慢查询怎么排查？

> 四步诊断法：
> 1. **EXPLAIN 看执行计划**：type=ALL? rows 很大? → 缺索引或分区。
> 2. **检查分区裁剪**：WHERE 里有没有 dt？没有 → 90% 慢查询的原因。
> 3. **检查 JOIN**：JOIN 列有没有索引？是不是笛卡尔积？
> 4. **检查数据倾斜**：某个 key 数据量特别大 → 加盐打散。
>
> 优化优先级：分区裁剪 > 加索引 > JOIN替代子查询 > 窗口函数替代自连接 > Broadcast Join > 列裁剪。

### Q4: 数据质量怎么保证？

> 三层保证：
> **事前**：ETL 嵌入扣分制(dq_score) + 六维度校验，FATAL 直接阻断，ERROR 阻断下游。
> **事中**：每天产出后自动跑质量报告，核心指标波动 > 20% 自动告警。
> **事后**：BackfillOrchestrator 从源头逐层重跑。核心原则：**不能跨层跳过**，ODS→DWD→DWS→ADS 严格按序。
>
> 扣分制的设计哲学：数据质量是一个频谱，不是二元状态。必填缺失扣30(致命)、金额异常扣20(严重)、类型未知扣10(轻微)，60分及格线刚好是"可接受一个严重+一个中等"的边界。

### Q5: 如何保证跨部门数据口径一致？

> 建立公司级指标字典（MetricRegistry），每个指标有唯一 metric_id + 明确的分子分母公式 + 排除条件清单。所有 BI 报表、AI 模型、运营后台都从同一个 MetricRegistry.generate_sql() 拿 SQL。口径变更需审批，改公式要有版本号。

### Q6: 数据血缘有什么用？怎么实现？

> 当 ADS 报表数字异常时，通过反向血缘几秒钟定位到源头是哪张 ODS 表出了问题。
>
> 实现方式：YAML 手工维护（灵活但依赖人工）、SQL 解析自动提取（无人工负担）、SchemaRegistry 程序化管理（schema 随数据走，NL2SQL 可直接消费）。

### Q7: 怎么保护敏感数据？

> DWD 层统一脱敏，四层策略分级：掩码(Mask，保留部分结构→黄\*)、哈希(Hash，完全不可逆但可去重→SHA256)、泛化(Generalize，映射到更高层级→192.168.x.x)、加密(Encrypt，可逆但需密钥→银行卡号)。
>
> 核心原则：DWD 是 PII 脱敏的唯一入口——所有下游不应再看到明文。

### Q8: 你怎么处理新用户（冷启动）？

> 宽表构建用 left join + fillna(0)：保证新用户不丢数据。还款特征 on_time_rate 默认 1.0（无罪推定），行为特征默认 0（无信息用 0 表示，而非均值——均值 = 伪造信息）。新用户在模型中的表现完全基于其申请画像，行为/还款维度无信息即为中性。
