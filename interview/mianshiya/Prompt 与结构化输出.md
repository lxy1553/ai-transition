## 1. 什么是 GPT Structured Outputs？

> ID: `Q043`

GPT Structured Outputs 是⼀种让⼤模型输出结构化数据的能⼒，⽐如 JSON 格式的结果，⽽不是⾃由⽂本。这在需
要程序直接解析模型输出的场景⾥特别有⽤。
你告诉模型你想要的 schema，它就会按这个格式返回数据。这样⼀来，前端或后端代码就能直接消费输出，不⽤再
写⼀堆正则或逻辑去提取信息。
⽐如你在调⽤ GPT 时启⽤ response_format  参数，并指定为 JSON Schema，模型就会严格按字段输出。OpenAI
从 2023 年底开始⽀持这个功能，适⽤于像 gpt-3.5-turbo 和 gpt-4 这样的模型。
{
"name": "Alice",
"age": 30,
"city": "Beijing"
}
这种机制适合做配置⽣成、表单填充、API 数据抽取等任务。和以前“靠运⽓”解析⽂本不同，现在结果可预测、可验
证。
不过得注意，模型不会⾃动知道什么时候该输出结构，必须显式设置参数，否则还是⾛⾃由⽣成。另外复杂 schema
可能增加 token 消耗，设计时要权衡清晰度与成本。
如果原题解中的某句话使⽤了超链接，不要对这句话做任何改动。

## 2. ⼤模型的结构化输出指的是什么？

> ID: `Q044`

⼤模型的结构化输出，说⽩了就是让模型不光吐⽂字，还能按指定格式返回数据，⽐如 JSON、XML 或特定 schema
的内容。你给个指令，它能⽣成符合接⼝契约的响应，⽽不是⼀堆⾃由发挥的⾃然语⾔。
这在实际系统集成⾥特别关键。⽐如你让⼤模型从⼀段⽂本⾥抽⽤户信息，直接回“张三，35岁，北京”还得再解
析，但如果要求结构化输出，它就能直接给你：
{
"name": " 张三 ",
"age": 35,
"city": " 北京 "
}
这种能⼒⼀般靠提⽰词⼯程（prompt engineering）引导，或者⽤像 JSON mode 这类机制强制约束输出格式。
OpenAI 的 API ⽀持 response_format={ "type": "json_object" } ，这时候模型就必须输出合法 JSON，否
则调⽤⽅解析要出问题。
更进⼀步的做法是结合 function calling 或 tool use，把结构化输出当成调⽤外部⼯具的中间协议。⽐如你让模型判
断是否需要查天⽓，它就返回⼀个带 function_call  字段的 JSON，⾥⾯是函数名和参数，下游系统直接执⾏。
这类技术减少了后处理成本，让⼤模型能真正嵌⼊到软件流程⾥，⽽不是只当个聊天玩具。不过对模型本⾝的推理⼀
致性和格式遵从度要求更⾼，⼩模型容易格式错乱，得反复校验。

## 3. 什么是护栏技术？

> ID: `Q045`

护栏技术本质是⼀种在系统运⾏时动态控制流量、防⽌雪崩的保护机制。它不追求彻底解决问题，⽽是快速⽌损，让
系统能继续对外提供部分服务。
1）核⼼思路是“宁可限流，不可拖垮”。⽐如⼀个电商系统的下单接⼝突然变慢，⼤量请求堆积，数据库连接被打
满，这时候如果不加⼲预，整个应⽤可能直接宕机。护栏技术就是在这个临界点介⼊，把超出处理能⼒的请求直接拒
绝或降级。
2）常⻅的实现⽅式有熔断、限流、降级。熔断像电路保险丝，当错误率超过 50% 就直接切断请求，避免连锁故障；
限流控制每秒最多放⾏ 1000 个请求，多余的就地拦截；降级则是返回兜底数据，⽐如商品详情⻚的推荐模块挂了，就
直接返回空列表，保证主流程可⽤。
3）实际项⽬⾥，Sentinel 和 Hystrix 都是典型的护栏⼯具。拿 Sentinel 来说，你可以给某个接⼝配置 QPS 超过 50
就⾃动限流，规则可以实时推送，不⽤重启服务。
// 定义资源
SphU.entry("getOrder")
// 业务逻辑
这种机制特别适合⽤在⾼并发场景，⽐如秒杀系统。你不可能靠堆机器扛住所有流量，得靠这些“安全策略”来兜
底。真正的稳定性不是不出问题，⽽是出问题时系统还能活着。
