# 如果一个GPU集群的LLM处理能力为1000tokens/s，那1000个用户同时并发访问，响应给每个用户的性能只有1 token/s吗？怎么分析性能瓶颈

> 来源: 面试鸭题库 | 分类: 未分类

不⼀定每个⽤户就只能分到 1 token/s，实际性能取决于调度策略、批处理能⼒和请求的负载特征。
⼤模型服务不像传统 Web 服务器那样“⼀个请求占⼀个线程”，它⾛的是 动态批处理（Dynamic Batching） 路⼦。
⽐如⽤ vLLM、TensorRT-LLM 这类推理框架时，多个⽤户的请求会被合并成⼀个 batch ⼀起跑，显存和计算单元利⽤
率更⾼。
关键点在于：
1）如果这 1000 个⽤户都是刚发请求，还没开始⽣成内容（preﬁll 阶段为主），那系统可以⼀次性把他们的 prompt 做
batch 处理，吞吐可能接近峰值 1000 tokens/s，平均下来每⼈⼏⼗甚⾄上百 token/s 都有可能。
2）但到了 decode 阶段，每个请求要逐 token ⽣成，且⻓度不⼀，这时候 batch size 会慢慢被“拖慢”。⻓⽂本、⾼
采样参数（如 temperature ⾼）的请求会拉低整体速度。
3）真正的瓶颈往往不是算⼒，⽽是显存带宽和 KV Cache 管理。每个并发请求都要保留 KV Cache，1000 个并发意味
着巨⼤的显存压⼒，搞不定就会 OOM 或被迫丢请求。
举个例⼦，A100 显存 80GB，跑 Llama3-8B，每个请求平均占 2GB KV Cache，理论上最多撑 40 个并发。超过就得靠
PagedAttention（像 vLLM 实现的）来分⻚管理，否则压根不经过计算就挂了。
所以性能瓶颈通常出现在：
KV Cache 显存占⽤过⼤
⼩ batch 或不规则请求导致 GPU 利⽤率上不去
请求排队时间远超⽣成时间
优化⽅向是：⽤ PagedAttention 降低显存压⼒，启⽤连续批处理（Continuous Batching），控制最⼤⽣成⻓度，避
免⼩请求拖垮整体延迟。
