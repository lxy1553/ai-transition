# 你有多个知识库，做 RAG 的时候，怎么保证查询效率和准确性兼容，并尽可能减少幻觉？

> 来源: 面试鸭题库 | 分类: RAG 检索增强

多个知识库做 RAG，关键是在检索阶段就控制好数据的“来源质量”和“语义相关性”，否则后⾯⽣成再怎么优化也压
根不经过。
1） 先做知识库的预分类或打标。⽐如⽤ Milvus 或 Elasticsearch 存向量时，给每个⽂档加 metadata 标记来源类型
（如产品⽂档、客服记录、API ⼿册）。查询时先根据 query 的意图路由到最可能相关的库，避免全量检索拖慢速度。
2） ⽤ 混合检索 策略。单纯靠向量可能召回语义相近但事实错误的内容，容易引发幻觉。结合关键词匹配（BM25）和
向量相似度，像 Weaviate 或 Vespa 都⽀持这种 hybrid search，能同时抓住字⾯匹配和深层语义，准确率提升明显。
3） 检索后做重排序（rerank）。别拿到 top-k 就直接喂给 LLM，⽤⼀个轻量级模型⽐如 BGE-reranker 对结果按相关
性重新打分，把真正靠谱的段落往前排。这⼀步能过滤掉不少“看起来像对其实不对”的内容。
4） 在 prompt 中显式标注信息来源。把检索到的⽂本块加上出处标签，⽐如 [来⾃：API ⼿册 v3.2]，让⼤模型知道依
据从哪来，⽣成时更倾向引⽤原⽂，减少脑补。
5） 最后加⼀层验证机制。对关键回答，可以⽤⼩模型做⼀致性校验，⽐如把⽣成结果和原始⽂档⽚段做⼀次
entailment 判断，不⼀致就触发⼈⼯审核或 fallback 回答。
