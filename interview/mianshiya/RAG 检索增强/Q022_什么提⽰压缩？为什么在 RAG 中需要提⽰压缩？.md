# 什么提⽰压缩？为什么在 RAG 中需要提⽰压缩？

> 来源: 面试鸭题库 | 分类: RAG 检索增强

提⽰压缩其实就是把 RAG 中过⻓的上下⽂输⼊给“瘦⾝”。你想想，⼤模型的上下⽂⻓度是有限的，⽐如 32K 或
128K，但检索回来的⽂档⽚段⼀多，加上原始问题和模板，很容易就超了。
1）最直接的问题就是 token 数爆掉，请求直接被拒，压根不经过模型推理。
2）就算没超，⻓上下⽂推理成本⾼，响应慢，延迟上去⽤户体验就差了。
3）⽽且很多检索出的内容其实跟问题关系不⼤，纯属噪⾳，反⽽⼲扰⽣成质量。
所以得做提⽰压缩，把那些冗余信息、重复段落、⽆关句⼦给剔除或简化。不是简单删减，⽽是保留关键事实和语
义，让输⼊更紧凑⾼效。
常⻅的做法有⼏种：
⽤⼩模型先做⼀轮重排序或摘要，⽐如⽤ BGE 或 Cohere reranker 挑最相关的 top-k。
在 prompt 组装时加逻辑，只保留⾼分 chunk，甚⾄对每个 chunk 做句⼦级裁剪。
也可以引⼊类似 HyDE 的思路，先⽣成假设答案再反向匹配，过滤低相关度内容。
代码上没啥复杂语法，核⼼是控制组装后的总 token 数：
# 伪代码⽰意
context = "\n".join([chunk["text"] for chunk in relevant_chunks[:3]])  # 限制数量
prompt = f" 基于以下内容回答： \n{context}\n\n 问题：{query}"
要不要压缩，其实看场景。如果检索精度⾼、返回少，可能不需要。但⾯对海量⽂档库，⽐如⽤在企业知识库搜寻，
就得靠这招扛住⻓上下⽂压⼒。
