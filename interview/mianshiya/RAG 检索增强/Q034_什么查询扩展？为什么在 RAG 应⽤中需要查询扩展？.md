# 什么查询扩展？为什么在 RAG 应⽤中需要查询扩展？

> 来源: 面试鸭题库 | 分类: RAG 检索增强

查询扩展本质是把⽤户那句简单的原始问题，变形成⼀个或多个更完整、信息更丰富的查询语句。你别⼩看这⼀步，
它直接决定了后续检索能不能捞出真正相关的内容。
1） ⽤户搜“苹果⼿机续航差”，其实他关⼼的是 iPhone 15 Pro 的电池表现和省电技巧。原始 query 太短，关键词稀
疏，直接去向量库搜索容易匹配到⽆关的“苹果⽔果”或者泛泛的“⼿机耗电”。这时候就得靠查询扩展来补全意图。
2） 常⻅做法有⼏种。⼀种是同义词/近义词扩写，“续航”换成“电池寿命”“使⽤时间”；⼀种是基于⼤模型做多跳推
理，让 LLM 把原始问题拆成⼏个⼦问题，⽐如“iPhone 15 Pro 满电能⽤多久”“哪些设置能延⻓待机”；还有⼀种是
查询重写，把⼝语化表达转成更标准的检索语⾔。
3） 不做查询扩展的 RAG 系统，召回率通常很低。特别是⾯对缩写、俚语、模糊表达时，向量检索压根不经过语义深
化，直接就去算相似度，结果就是 top-k ⽂档⾥没有命中关键段落。像 LangChain、LlamaIndex 都内置了 query
transformation 模块，就是为了解决这个断层。
这步看着是脏活累活，但它是连接“⼈话”和“机器可检索语义”的关键桥梁。搞不定查询扩展，RAG 效果就很难稳
定。
如何进⾏ RAG 调优后的效果评估？请给出真实应⽤场景中采⽤的效果评估标准与⽅法
RAG 系统调优后不能只看⽣成结果“好不好”，得⽤量化指标和真实场景反馈来判断是否真的提升了效果。重点是结合
⾃动评估指标和⼈⼯评估，覆盖相关性、准确性和可⽤性。
1）先看召回阶段，⽤传统信息检索指标。⽐如在企业知识库问答场景中，我们拿 1000 条历史⽤户问题做测试集，计
算 top-5 ⽂档的 Recall@5 和 MRR。如果 Recall@5 从 0.6 提升到 0.78，说明检索更准了，这是硬指标。
2）再看整个 RAG 流程的输出质量。可以⽤ Rouge-L 衡量⽣成答案与标准答案的⽂本相似度，但别太依赖它。更关键
的是 BERTScore 或 BARTScore，它们能更好捕捉语义匹配程度。⽐如在客服机器⼈中，我们发现 Rouge-L 变化不
⼤，但 BERTScore 提⾼了 12%，实际回答更贴切了。
3）引⼊事实⼀致性检测。⽤ Faithfulness 指标判断⽣成内容是否基于检索到的⽂档。⽐如模型说“报销周期是7
天”，但⽂档写的是“5个⼯作⽇”，这就是幻觉。可以⽤专⻔打分模型如 RAGAS 来量化。
4）最终绕不开⼈⼯评估。设计评分卡，让业务⼈员从相关性、准确性、完整性、可读性四个维度打分。⽐如在⾦融投
研场景，分析师对调优前后的回答盲评，发现新版本在“数据来源清晰”这⼀项得分提升明显。
真实项⽬⾥，像⽤ LangChain + Pinecone 做法律咨询系统时，就是靠这套组合拳验证调优有效，上线后⼀次解决率提
⾼了 23%。
