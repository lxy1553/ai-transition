## 1. 什么是⼤模型微调？与预训练的核⼼区别是什么？

> ID: `Q001`

⼤模型微调是在预训练好的模型基础上，⽤特定领域的数据继续训练，让它适应具体任务。⽐如你有个通⽤语⾔模
型，想让它擅⻓写法律⽂书，就拿⼀堆法律⽂本去微调它。
1）预训练是打基础，模型在海量⽆标注数据上学习语⾔规律，⽬标是理解“什么是语⾔”。这个阶段通常要⽤上千张
GPU卡跑⼏周，参数量动不动上百亿。
2）微调是定向优化，⽤的是⼩规模、有标注的领域数据，⽐如客服对话、医疗报告。训练轮次少，计算资源消耗低，
⼀般⼏张GPU就能搞定。它的⽬标不是从零学语⾔，⽽是调整已有知识去解决“某个具体问题”。
3）两者的区别不在代码实现，⽽在数据和⽬标。预训练像通识教育，什么都学；微调像职业培训，专精⼀项。
Hugging Face 上很多 bert-base-chinese  是预训练模型，⽽ bert-wwm-ext  就是在此基础上做了额外微调的
版本。
微调压根不重新训练整个⽹络，只是在原有权重上做⼩幅更新。如果任务差异太⼤，⽐如从⽂本⽣成跳到图像识别，
那微调就搞不定了，得换架构重来。

## 2. 常⻅的微调任务有哪些？

> ID: `Q002`

微调任务其实就是在预训练模型基础上，针对特定场景做适配。最常⻅的⼏种类型，基本覆盖了⼤部分 NLP 需求。
1）⽂本分类
把句⼦或段落分到预定义类别⾥。⽐如情感分析判断评论是正⾯还是负⾯，⽤ BERT 接⼀个分类头就⾏。
// 伪代码⽰意
model = BERT.from_pretrained("bert-base-uncased")
classifier = nn.Linear(768, num_labels) // 接在最后⼀层向量上
2）命名实体识别（NER）
找出⽂本⾥的实体，像⼈名、地名、组织名。⼀般在每个 token 上做序列标注，CRF 层常⽤来优化标签间转移逻辑。
3）问答任务（QA）
最典型的是 SQuAD 这类抽取式问答。模型要从给定段落⾥标出答案的起⽌位置。输⼊是“问题+段落”，输出两个指
针。
4）⽂本⽣成
⽐如摘要⽣成、翻译。⽤ T5 或 BART 这类 seq2seq 模型，输⼊原⽂，输出压缩后的摘要。训练时⽤交叉熵算 loss。
5）句⼦对关系判断
判断两句话是什么关系。⽐如⾃然语⾔推理（MNLI），看第⼆句是⽀持、⽭盾还是中⽴于第⼀句。STS 任务则是算语
义相似度。
这些任务的数据格式和输出头不⼀样，但训练⽅式都差不多：冻结部分底层参数，放开顶层微调，学习率通常设得⽐
较⼩，3e-5 到 5e-5 就够了。数据量⼀般⼏千到⼏万条就能⻅效，迁移学习在这⾥真能省下⼤量训练成本。

## 3. 常⻅的微调⽅法有哪些？

> ID: `Q003`

微调本质上是拿预训练模型在特定任务上做⼆次训练，让模型更适配具体场景。常⻅的⽅法选择得看数据量、算⼒和
任务复杂度。
1）全量微调就是把整个模型的参数都放开训练。效果通常最好，但显存消耗⼤，8 张 A100 都不⼀定扛住 Llama 2 7B
的全量微调。适合数据质量⾼、预算充⾜的场景。
2）LoRA（低秩⾃适应）现在成了主流⽅案。它冻结原模型权重，在旁边加个⼩的低秩矩阵做增量，训练时只更新这
部分。显存能省下 70%，训练速度也快，Hugging Face 上⼤部分开源微调项⽬都在⽤。
# LoRA 核⼼思想： A 下降维度， B 升维，中间做低秩分解
lora_weights = (x @ A) @ B
output = original_output + lora_weights
3）Adapter ⽅法是在 Transformer 层之间插⼊⼩的神经⽹络模块，训练时固定主⼲，只调这些⼩模块。结构改动⽐
LoRA ⼤，但现在⽤得少⼀些。
4）P-Tuning 和 Prompt Tuning 属于提⽰微调，不改模型本⾝，⽽是学⼀组虚拟 token 的 embedding。适合少样本
场景，但效果波动⼤，对初始化敏感。
选哪个得权衡。数据多、资源⾜，可以试全量微调；⼤多数情况下，LoRA 是性价⽐之选，像 Qwen、ChatGLM 的社
区微调基本都靠它打底。

## 4. PEFT 是什么？为什么需要 PEFT？

> ID: `Q004`

PEFT 全称是 Parameter-Eﬃcient Fine-Tuning，直⽩点说就是“只动⼀⼩部分参数来微调⼤模型”。现在⼤模型动不
动就上百亿、上千亿参数，⽐如 Llama、ChatGLM 这种，直接全量微调成本太⾼了，显存、算⼒、时间都吃不消。
其实我们发现，微调的时候没必要更新所有参数。PEFT 的思路是冻结原模型绝⼤部分权重，只训练少量额外引⼊或选
中的参数，就能达到接近全量微调的效果。这样显存能省下 70% 以上，训练速度也快很多，普通⼏张卡甚⾄单卡也能
搞。
常⻅的 PEFT ⽅法⾥，LoRA 是最⽕的。它在原始层旁边加个低秩矩阵做增量，训练时只更新这个⼩矩阵。⽐如⼀个
7B 模型，⽤ LoRA 可能只改 0.1% 的参数，效果却不差。HuggingFace 的 peft  库已经⽀持得⾮常好，配合
transformers 能轻松上⼿。
代码⻓这样：
from peft import LoraConfig, get_peft_model
lora_config = LoraConfig(
r=8,
lora_alpha=16,
target_modules=["q_proj", "v_proj"],
lora_dropout=0.1,
bias="none",
task_type="CAUSAL_LM"
)
model = get_peft_model(model, lora_config)
除了 LoRA，还有 Adapter、Preﬁx-tuning 等⽅法，但 LoRA 因为⽆推理延迟、实现简单，成了主流选择。特别是资源
有限⼜想定制⼤模型的场景，⽐如企业私有模型微调，PEFT ⼏乎是标配⽅案。
参数⾼效微调（PEFT）的核⼼思路是什么？列举 3 种典型⽅法
参数⾼效微调的核⼼思路是只微调模型中极⼩⼀部分参数，冻结绝⼤部分原始参数，从⽽在保持下游任务性能的同
时，⼤幅降低计算和存储开销。
⼤模型动辄上百亿参数，全量微调需要⼏⼗甚⾄上百 GB 显存，普通团队根本搞不定。PEFT 的做法是“以⼩博⼤”，
通过引⼊少量可训练参数来适配新任务，⽐如只调 0.1%~1% 的参数就能达到接近全量微调的效果。
1） LoRA（Low-Rank Adaptation）
在原始权重旁注⼊低秩矩阵，训练时只更新这些⼩矩阵。推理时可以合并到原权重中，完全不增加推理延迟。
HuggingFace Transformers 已深度集成，现已成为主流选择。
2） Adapter
在 Transformer 模块间插⼊⼩型前馈⽹络（通常两层 MLP），原模型冻结，只训练插⼊的模块。虽然效果好，但会增
加推理延迟，因为每次前向都要⾛额外⽹络。
3） Prompt Tuning / Preﬁx Tuning
通过可学习的“软提⽰”向量引导模型输出，本质是把任务建模成“填提⽰”的过程。Preﬁx Tuning 把这些向量拼接
在每⼀层输⼊前，Prompt Tuning 只加在第⼀层。节省参数，但对初始化敏感，⼩模型上容易不稳定。

## 5. PEFT 和全量微调的区别？

> ID: `Q005`

PEFT 本质上是解决⼤模型微调成本太⾼的问题。全量微调要更新整个模型所有参数，显存和计算资源消耗巨⼤，⽐如
⼀个 13B 的模型，光梯度存储就得上百 GB 显存。
⽽ PEFT 的思路是冻结原模型⼤部分参数，只训练⼀⼩部分新增或改造的模块。这样⼀来，显存占⽤能降到原来的
1/10 甚⾄更低，普通单卡也能跑。
1）全量微调：每个参数都参与更新，效果理论上最好，但训练慢、成本⾼，需要完整的优化器状态和梯度保存。适合
有充⾜算⼒且对性能极致要求的场景，⽐如公司级训练集群搞通⽤能⼒升级。
2）PEFT ⽅法：以 LoRA 为例，它在原始权重旁并⾏注⼊低秩矩阵，训练时只更新这些⼩矩阵。假设原始权重是 $W
\in \mathbb{R}^{768\times 768}$，LoRA 拆成两个⼩矩阵 $A \in \mathbb{R}^{768\times r}$ 和 $B \in
\mathbb{R}^{r\times 768}$，其中 $r$ 通常设为 8 或 16。这样可训练参数数量从⼏⼗亿骤降到百万级。
// 伪代码⽰意  LoRA 注⼊
Matrix W = loadOriginalWeight(); // 冻结
Matrix A = randomInit(768, 8);   // 可训练
Matrix B = randomInit(8, 768);   // 可训练
Matrix loraOutput = input * (W + A * B); // W 不更新
常⻅变体还有 Adapter Tuning、Preﬁx Tuning 等，但 LoRA 因其不改变推理结构、兼容性好成为主流。HuggingFace
的 peft  库已经⽀持多种模式，配合 transformers  ⼏⾏代码就能上。
效果上，多数任务中 LoRA 能达到全量微调 95% 以上的性能，但对强推理或复杂指令遵循任务可能略有差距。关键是
省了太多资源，让 7B/13B 模型在消费级 GPU 上也能快速迭代。

## 6. 在进⾏ Fine-Tuning 时，如何选择适合的预训练模型？

> ID: `Q006`

这个问题其实是在问怎么给特定任务挑⼀个合适的预训练模型来微调，关键不是模型越⼤越好，⽽是看匹配度和资源
约束。
1）先看任务类型对不对⼝。⽐如做中⽂⽂本⽣成或理解，那肯定优先选在中⽂语料上预训练过的模型，像 Qwen、
ChatGLM 这类原⽣⽀持中⽂的系列就⽐纯英⽂的 LLaMA 更合适。语⾔不匹配的话，再⼤的参数量也搞不定语义理
解。
2）再看模型规模和你的算⼒能不能对上。7B 参数的模型⽤单卡 A100 能跑，但如果是 70B，就得考虑模型并⾏或者直
接上云了。⼀般情况下，中⼩公司做垂直场景微调，7B 左右的模型性价⽐最⾼，显存压得下来，训练时间也扛得住。
3）还要看预训练数据的时间范围和领域。⽐如你要做⾦融研报⽣成，最好选那些在财经⽂本上持续预训练过的模型，
⽽不是通⽤语料训完就停的。有些开源模型会标明训练数据来源和时间，这点要仔细查⽂档。
4）最后看微调⽣态⽀不⽀持。像 Hugging Face 上有⼤量基于 LLaMA 系列的 LoRA 微调⼯具链，如果你选了⼀个冷⻔
模型，可能连适配的 Trainer 都没有，脏活累活都得⾃⼰写。
代码上其实就是加载不同 checkpoint 的区别：
from transformers import AutoTokenizer, AutoModelForCausalLM
tokenizer = AutoTokenizer.from_pretrained("qwen-7b")
model = AutoModelForCausalLM.from_pretrained("qwen-7b")
换模型时路径⼀改就⾏，但背后的数据和架构差异决定了效果上限。

## 7. 微调中常⽤的优化器有哪些？

> ID: `Q007`

微调⼤模型时选优化器，关键得平衡收敛速度和训练稳定性。常⻅的⼏种其实都源⾃梯度下降的改进思路。
1）SGD with Momentum 算是⽼前辈了，加个动量项能让参数更新不那么容易在平坦区域震荡。但学习率敏感，调不
好容易不收敛，现在⼤模型微调⽤得少了。
2）Adam 最常⻅，⼏乎成了默认选项。它⾃适应调整每个参数的学习率，对稀疏梯度特别友好。BERT、RoBERTa 这
些预训练模型做下游任务微调时，基本都拿 Adam 打底。不过内存开销⼤，因为要存每个参数的⼀阶⼆阶梯度。
// PyTorch 伪代码⽰意，实际不⽤  Java 写训练
optimizer = torch.optim.Adam(model.parameters(), lr=2e-5)
3）AdamW 是 Adam 的升级版，把权重衰减从梯度更新⾥拆出来单独算，解决了 Adam 在正则化上的理论缺陷。
Hugging Face Transformers ⾥默认推荐的就是 AdamW，像微调 BART 或 T5 都这么配。
4）Lion 是较新的选择，Google 提出来的，只⽤⼀阶动量，内存⽐ Adam 少⼀半，⽽且在某些 NLP 任务上效果更
好。但它对学习率更敏感，需要更精细地调参。
总的来说，AdamW 因为稳定性和效果兼备，成了当前主流选择。⼩规模实验可以试试 Lion 看能不能提速，⽼项⽬可
能还在⽤ Adam。SGD 基本只出现在特定场景，⽐如某些视觉模型微调还会⽤。

## 8. 如何判断微调效果是否达到预期？

> ID: `Q008`

微调效果能不能扛住业务，得靠数据说话，不能拍脑袋。
1）先看基础指标，⽐如分类任务的准确率、F1 值，⽣成任务的 BLEU、ROUGE。这些是硬指标，训练集和验证集上
的提升得稳定，⼀般要求验证集指标⽐基线模型⾼ 5% 以上才算有显著收益。
2）光看数字不够，得做⼈⼯评估。抽样⼀批模型输出，让业务⽅或标注⼈员打分，重点关注⽣成内容是否符合语义、
逻辑通顺、不胡说⼋道。像⽤ Qwen 或 Llama 微调后，常出现“套话连篇但答⾮所问”的问题，这种机器指标发现不
了。
3）上线前必须⾛ A/B 测试。把⽼模型和新模型同时跑在线上，按流量切分，对⽐点击率、⽤户停留时⻓、转化率这些
业务指标。⽐如在客服场景⽤微调模型，⽬标是降低⼈⼯转接率，那这个值下降 10% 以上才叫真有效。
4）还得看泛化性。拿⼀些边缘 case 或分布外数据测试，⽐如新增的⽤户 query 类型，模型能不能兜住。如果⼀遇到
没⻅过的句式就崩，说明过拟合了，得回炉加数据增强或者正则。
代码层⾯可以写个简单的评估脚本：
from sklearn.metrics import f1_score
f1 = f1_score(y_true, y_pred, average='weighted')
print(f" 微调后 F1: {f1:.3f}")
最后记住，过拟合是微调最⼤的坑，训完训练集接近 100% 准确率，验证集却卡在 70%，基本就是背答案了。数据质
量差、样本太少、学习率太⾼都可能引发这问题。
介绍⼏种常⻅的微调策略的优缺点
微调⼤模型不是上来就全量参数⼀把梭，得看资源和场景做选择。
1）全量微调把所有参数放开训练，效果最好，像 Llama 系列在特定任务上刷榜基本都靠它。但代价也⼤，⼀张 A100
训个 7B 模型就得跑好⼏天，显存轻松⼲到 80G 以上，⼩团队根本搞不定。
2）LoRA（低秩适配） 是现在最⽕的轻量⽅案。它冻结原模型，在输⼊层加两个低秩矩阵做增量，训练时只更新这两
块。⽐如 HuggingFace 上很多基于 Qwen、ChatGLM 的衍⽣模型都是⽤ LoRA 微出来的。显存能压到 20G 内，训练
快，还能多任务共⽤⼀个底模型。缺点是超参敏感，r 秩选不好容易⽋拟合。
3）Adapter Tuning 在 Transformer 层中间插⼊⼩型神经⽹络模块，训练时只动这些⼩模块。结构改动⽐ LoRA ⼤，
推理有延迟，但现在基本被 LoRA 吊打，除⾮你对模块化设计有强需求。
4）Preﬁx Tuning 和 Prompt Tuning 都是往输⼊前⾯塞可学习的向量。前者改 attention key/value，后者直接拼接
embedding。适合少样本场景，但⻓序列下效果不稳定，⼀般只⽤于探索性实验。

## 9. 什么是低秩适配（LoRA）技术？如何结合 LoRA 技术进⾏微调？

> ID: `Q009`

低秩适配（LoRA）的核⼼思路是，⼤模型微调时参数量太⼤，训练慢、显存吃紧，⼲脆不直接改原始权重，⽽是⽤低
秩矩阵去近似增量。
模型原本的权重矩阵 $W$ 是个⼤胖⼦，⽐如 768×768。LoRA 不动它，转⽽引⼊两个⼩矩阵 $A$ 和 $B$，维度分别是
$768×r$ 和 $r×768$，其中 $r$ 很⼩，⼀般取 8 到 64。训练时只更新这两个⼩矩阵，前向传播时等效注⼊ $\Delta W
= A \times B$。
这样显存和计算量就从和原矩阵成正⽐，变成和 $r$ 相关。⽐如 r=8，参数量直接降到原来的 1/96，训练速度翻倍都
不⽌。
结合 LoRA 做微调，流程也简单：
1）冻结原始模型所有权重
2）在指定层（⽐如注意⼒中的 Q、V 投影）插⼊ LoRA 模块
3）只训练新增的⼩矩阵，其余参数不动
4）推理时把 $A \times B$ 加到原权重上，或者直接合并进原权重
主流框架都有⽀持。Hugging Face 的 peft  库⼏⾏代码就能上：
from peft import LoraConfig, get_peft_model
lora_config = LoraConfig(
r=8,
lora_alpha=16,
target_modules=["q_proj", "v_proj"],
lora_dropout=0.1,
bias="none",
task_type="CAUSAL_LM"
)
model = get_peft_model(model, lora_config)
适合资源有限、多任务切换的场景。像微调⼀个 7B 模型，⽤ LoRA 能在单卡 24G 显存下搞定，否则根本搞不定。

## 10. 微调的过拟合⻛险如何通过正则化缓解？

> ID: `Q010`

微调时模型已经⻅过⼤量训练数据，参数调整幅度⼩但密集，容易记住训练集的特定模式，导致在新数据上表现下
滑。正则化的核⼼作⽤就是在损失函数或训练过程中加⼊约束，让模型别学得太“死”。
1）L2 正则（权重衰减）是最常⽤的⼿段。它在损失函数⾥加上所有权重的平⽅和，迫使参数值趋向更⼩。⽐如
Hugging Face 的 Trainer  ⾥设置 weight_decay=0.01 ，就能防⽌某些权重变得过⼤，提升泛化能⼒。
training_args = TrainingArguments(
output_dir="output",
weight_decay=0.01
)
2）Dropout 在微调阶段依然有效。虽然预训练模型内部已经固化了结构，但在分类头甚⾄最后⼏层中间加 Dropout，
能让输出不依赖于少数神经元。⼀般保留率设在 0.1〜0.3 就够了。
3）早停（Early Stopping）特别适合微调场景。因为微调轮次少，可能只跑 3〜5 个 epoch，验证集性能⼀旦下降就
终⽌，能避免多⾛⼀两步掉进过拟合坑⾥。
4）还有像 标签平滑（Label Smoothing）这种软化监督信号的⽅法。它不让模型对真实标签过于⾃信，间接抑制过拟
合。在分类任务中设 label_smoothing_factor=0.1  是常⻅做法。
这些⽅法本质都是给学习过程加点“不确定性”或“惩罚”，让模型别把微调当背书，稍微收着点劲。

## 11. 请详细讨论微调时如何防⽌灾难性遗忘问题？

> ID: `Q011`

微调⼤模型时，灾难性遗忘的本质是模型参数被新任务数据剧烈调整，导致旧知识的表征能⼒崩塌。关键不是“记住
所有旧数据”，⽽是让模型在适应新任务的同时，保留对原始分布的泛化能⼒。
1） 降低学习率是最直接的⼿段。预训练阶段⽤了海量数据，参数已经收敛到⼀个较优的全局位置。微调时如果⽤⼤学
习率，梯度更新会把参数推出这个区域。⼀般建议微调学习率设为预训练的 1/10 到 1/100，⽐如从 3e-4 降到 3e-5 或
1e-5。
2） 使⽤ Adapter 模块是⼀种参数隔离策略。只在 Transformer 层间插⼊⼩型神经⽹络，冻结原始主⼲参数。训练时
梯度只流经这些⼩模块，主⼲权重压根不更新。这样既适配了新任务，⼜完全保留了原模型的知识结构。
HuggingFace 的 PEFT 库就⽀持这类⽅法。
3） LoRA（低秩适应）更轻量。它不加新层，⽽是给注意⼒权重矩阵注⼊低秩分解矩阵。假设原始权重是 W，训练的
是 ΔW = BA（A∈R^{r×d}, B∈R^{d×r}），r 远⼩于 d（⽐如 r=8, d=4096）。更新时 W' = W + α * BA，主⼲参数不变，
只训练 A 和 B。这种⽅法显存占⽤少，推理还能合并权重，适合⼤规模部署。
4） 数据层⾯可以混合⼀定⽐例的通⽤语料。⽐如每 10 个微调样本中加⼊ 1 个来⾃原始预训练分布的句⼦，让梯度⽅
向不⾄于完全偏离。不过要⼩⼼引⼊噪声影响⽬标任务效果。
这些⽅法的核⼼思想都是限制参数搜索空间，不让优化过程肆意修改已学好的特征提取器。

## 12. 在多模态微调（如图⽂⽣成）中，如何确保⽂本和图像数据的对⻬质量？

> ID: `Q012`

多模态微调⾥，⽂本和图像的对⻬质量直接决定模型能不能准确理解“图在说什么，话在描述谁”。如果数据没对⻬，
模型学到的就是错配关系，⽐如把“猫坐在沙发上”跟⼀张狗的照⽚强⾏关联，后续⽣成或推理全崩。
1） ⾸先靠⾼质量数据清洗。原始图⽂对往往来⾃⽹⻚，alt-text 和图⽚内容不⼀致太常⻅。可以⽤预训练的 CLIP 模
型做初筛，计算图⽂相似度，低于阈值（⽐如 0.2）的直接过滤。CLIP 本⾝是跨模态编码器，能给出语义匹配分数，
这⼀步能⼲掉明显噪声。
2） 然后是增强对⻬信号。单纯⽤原始图⽂对微调，监督信号弱。可以在输⼊侧构造负样本，⽐如同⼀个 batch ⾥拿
其他样本的⽂本当负例，做对⽐学习（ITC loss），逼模型分辨“哪个⽂本真配这张图”。BLIP、ALBEF 这些模型就这
么⼲的。
3） 微调策略上，不要只微调分类头，要端到端微调图像编码器和⽂本编码器。尤其是 ViT 的⾼层特征和⽂本 token 要
能 cross-attention 对上。可以⽤ attention 可视化检查：⽣成⽂本时，模型是不是真的关注图中对应区域。
4） 最后加点⽣成式损失，⽐如在 decoder 阶段⽤语⾔模型⽬标重建描述，让模型不仅判别还得会说。这样双重约
束，对⻬更稳。

## 13. 参数⾼效微调（PEFT）如何减少计算成本？

> ID: `Q013`

参数⾼效微调的核⼼思路是，不碰预训练模型的⼤部分权重，只动其中⼀⼩撮关键参数。这样⼀来，显存占⽤和计算
量都⼤幅下降。
1）冻结主⼲：把⼤模型的绝⼤多数参数锁死，⽐如 LLaMA 的 70 亿参数⾥，可能只放开不到 1% 去更新。剩下的全当
静态特征提取器⽤。
2）引⼊可训练模块：像 LoRA 这种⽅法，会在注意⼒层的权重旁加个低秩矩阵做增量。训练时只优化这个⼩矩阵，原
始权重压根不参与梯度更新。假设原矩阵是 $ \mathbb{R}^{5120 \times 5120} $，LoRA 只学两个 $
\mathbb{R}^{5120 \times 8} $ 的⼩矩阵，参数量从千万级⼲到⼏万。
3）适配器注⼊：在 Transformer 层之间塞个⼩的前馈⽹络模块，叫 Adapter。它通常就⼀两层，宽度也⼩，训练时只
调这部分，其他照旧。像 BERT 这类模型⽤ Adapter 微调，能省下 70% 以上的训练资源。
4）前缀微调：给输⼊前⾯挂⼀串可学习的虚拟 token，叫 preﬁx。训练时只更新这些 token 的 embedding，模型主
体完全不动。T5、GPT 这类⾃回归模型常⽤这招。
这些⽅法都能做到单卡微调⼤模型。⽐如⽤ LoRA 调 LLaMA-2-13B，在 A100 上也能跑起来，显存直接从⼏⼗ GB 掉到
10G 内。

## 14. 冻结层在微调中的作⽤是什么？

> ID: `Q014`

冻结层在微调时，其实就是把预训练模型⾥的⼀部分⽹络参数固定住，不让它们更新。你只让最后⼏层或者指定的层
去学习新任务的数据，这样能省下⼤量计算资源。
1）最常⻅的情况是⽤ BERT、ResNet 这类⼤模型做下游任务。⽐如你拿 ImageNet 上训好的 ResNet50 去做猫狗分
类，前⾯的卷积层已经学到了边缘、纹理这些通⽤特征，压根不需要重学。把这些层冻住，只微调最后的全连接层，
⼏⼗个 epoch 就能收敛，显存和时间都扛得住。
2）另⼀个关键是防⽌过拟合。你的⼩数据集可能就⼏千张图，如果放开所有层去更新，模型很容易记住噪声，泛化能
⼒直接崩掉。冻结⼤部分层相当于限制模型容量，逼它只调整⾼层语义适配新任务。
3）也有分阶段微调的做法：先冻结主⼲训 head，再解冻部分层⽤更⼩学习率细调。像 Hugging Face 的
Transformers 库⾥，.requires_grad_()  控制起来也就⼀⾏代码的事。
# 冻结  BERT base 层
model = BertForSequenceClassification.from_pretrained('bert-base-uncased')
for param in model.bert.parameters():
param.requires_grad = False  # 梯度不计算，节省内存
这种策略在 NLP 和 CV ⾥都成了标配，尤其是数据少、⼜想⽤⼤模型的时候，基本靠它稳住局⾯。

## 15. 为什么需要混合精度训练？

> ID: `Q015`

混合精度训练解决的是深度学习训练中的计算效率与显存占⽤问题。现代GPU如A100、V100都配备了Tensor Core，
专⻔为半精度浮点数（FP16）运算优化，单次可处理更多数据，算得更快。
1）FP16相⽐FP32占⽤带宽和显存少⼀半，同样的batch size能跑更⼩的显存，或者在显存不变时增⼤batch size提升
训练稳定性。
2）但全⽤FP16会出问题，⽐如梯度下溢、权重更新精度不够导致模型不收敛。所以得保留⼀部分关键计算⽤FP32，
⽐如梯度累加、权重更新、loss缩放。
3）实际做法是：前向和反向传播⽤FP16加速，参数副本保持⼀份FP32的主拷⻉⽤于更新，这就是AMP（Automatic
Mixed Precision） 的核⼼机制。PyTorch⾥⼀⾏scaler = torch.cuda.amp.GradScaler() 就能开启。
像BERT、ResNet这类⼤模型训练，基本都默认开启混合精度。NVIDIA的Apex库和原⽣AMP都⽀持得很好。不⽤的
话，等于⽩⽩浪费30%以上的训练速度。

## 16. 模型输出重复和幻觉如何微调解决？

> ID: `Q016`

模型输出重复和幻觉，本质是训练数据、⽣成策略和微调⽅式共同作⽤的结果。想靠微调彻底解决不现实，但能显著
缓解。
1）重复问题，常⻅于解码阶段贪⼼搜索。beam search 如果宽度太⼩，容易陷⼊循环。可以在推理时引⼊ n-gram 重
复惩罚，⽐如 Hugging Face 的 transformers  库⾥设置 no_repeat_ngram_size=2 ，让模型避开已⽣成的词
组。微调时也可以在 loss 上做⽂章，对重复 token 加权惩罚。
2）幻觉更棘⼿，模型把参数⾥的统计规律当事实输出。指令微调（Instruction Tuning）是个办法，⽤⾼质量问答对
训练，⽐如 Alpaca 数据集格式，教会模型“不知道就说不知道”。关键是数据质量，10 万条精准样本⽐ 100 万条噪声
数据管⽤得多。
3）RLHF 也能压幻觉，InstructGPT 就这么⼲的。⽤⼈类偏好打标数据训练奖励模型，再⽤ PPO 调整⽣成策略，让输
出更可信。但这套流程成本⾼，⼀般公司搞不定。
代码上，Hugging Face 的 Trainer  配合 DataCollatorForLanguageModeling  做微调很顺⼿。推理时控制
top_p （0.9）和 temperature （0.7），别让模型太放⻜。
说到底，微调能改善，但没法根治。⼯程上得配合检索增强（RAG），让模型查实时数据，⽽不是全靠记忆。

## 17. SFT 指令微调数据如何构建？

> ID: `Q017`

SFT（Supervised Fine-Tuning）指令微调数据的构建，本质是把模型要学的任务表达成“指令-输⼊-输出”的三元组
结构。关键不是堆数据量，⽽是让样本覆盖多样任务类型和表达⽅式。
1）⼀条标准样本包含三个部分：instruction 是具体任务描述，⽐如“翻译成英⽂”；input 是待处理内容，可以为
空；output 是期望的模型输出。例如 instruction="摘要⽣成"，input="⼀段新闻⽂本"，output="⽣成的简短摘
要"。
2）数据来源主要有⼏种路径。可以⽤公开指令数据集如 Alpaca、Flan-v2 做基础，再结合业务场景构造垂直领域样
本。⽐如客服场景，收集真实⽤户问题和⼈⼯回复对，包装成指令格式。也可以⽤⼤模型⾃动⽣成指令样本，通过
prompt 让 GPT-4 产出“问题+答案+上下⽂”，控制多样性和质量。
3）质量⽐数量重要。避免重复、歧义或错误标注。建议做分层设计：通⽤能⼒、领域知识、业务特有逻辑各占⼀定⽐
例。⼀般⼏千到⼏万条⾼质量样本就能起效，上百万条反⽽可能引⼊噪声。
4）最后要做格式归⼀和去重。统⼀ prompt 模板⻛格，过滤低信息密度样本，⽐如全是停⽤词或模板化回复的数据。
整个过程其实是为模型“造考场”，题⽬得能考出真本事。

## 18. 请解释⼤模型微调(Fine-tuning)的原理，并说明在什么业务场景下需要微调⽽不是直接使⽤基础模型？

> ID: `Q018`

微调的本质是参数⾼效迁移，不是从头训练。⼤模型已经在海量通⽤语料上学到了语⾔规律和世界知识，微调是在这
个基础上，⽤特定领域的数据去调整模型参数，让它适应新任务。
1）你拿到⼀个预训练好的⼤模型，⽐如 LLaMA 或 Qwen，它的“常识”已经很完整。但如果你要做⾦融研报⽣成，
它可能写得不够专业。这时候直接 prompt ⼯程搞不定，因为领域术语、⾏⽂逻辑差异太⼤。
2）微调就是拿⼀批⾦融报告数据，⽤有监督的⽅式继续训练。⽐如输⼊“某公司营收增⻓30%”，让模型输出对应的
研报段落。反向传播会更新权重，让模型在该任务上表现更好。
3）关键在于，并不是所有层都要重训。现在很多做法是只微调部分参数，⽐如 LoRA（Low-Rank Adaptation），冻结
原模型权重，只训练低秩矩阵。这样显存占⽤少，训练快，适合业务迭代。
需要微调的场景通常是：
输出格式/⻛格强约束，⽐如法律⽂书、医疗诊断记录
领域知识密集，基础模型没⻅过的专业术语体系
企业私有数据不能⾛ API，必须本地部署并定制
如果只是简单问答、摘要⽣成，且对表达⻛格要求不⾼，那其实⽤提⽰词+RAG就够了。微调成本⾼，周期⻓，真要上
得想清楚是不是⾮⽤不可。

## 19. 指令微调的好处？

> ID: `Q019`

指令微调其实是让预训练模型更会“听⼈话”的关键⼀步。模型在⼤规模数据上预训练完，虽然学到了丰富的语⾔知
识，但并不清楚⽤户到底想让它⼲什么。这时候通过指令微调，⽤成对的“指令-期望输出”数据去微调，它就能学会
理解任务意图。
1）模型能准确理解任务形式。⽐如给⼀条指令“把下⾯这句话翻译成英⽂”，模型知道接下来要处理的是翻译，⽽不
是摘要或改写。这种泛化能⼒在多任务场景下特别有⽤。
2）减少对⽰例的依赖。没有指令微调的模型，往往需要上下⽂⾥给出⼏个例⼦（few-shot）才能⼲活，⽽微调后的模
型可以直接响应指令，输⼊更简洁。
3）⾏为更可控。你可以通过微调引导模型输出特定格式，⽐如 JSON、有序列表，或者避免⽣成某些敏感内容。像
Alpaca、Dolly 这些开源模型，都是基于 LLaMA 做了指令微调才变得可⽤。
举个简单例⼦，微调前你问“写个快排”，它可能只解释原理；微调后它⼤概率直接给你⼀段可运⾏的代码。
不过也别指望它能完全替代⼯程逻辑。指令微调解决的是“意图对⻬”，不是“能⼒增强”。模型本⾝不具备的能⼒，
⽐如数学推理、代码执⾏，光靠指令微调是搞不定的。
