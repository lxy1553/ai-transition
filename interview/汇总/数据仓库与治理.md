## 1. 金融信贷离线仓库为什么要分 ODS、DWD、DWS、ADS？

> 来源: 核心题库

金融信贷离线仓库分层，是为了把原始数据、明细事实、主题汇总和应用指标分开管理。
ODS 保留业务系统原始数据，DWD 负责清洗成统一粒度的明细事实，DWS 按主题和维度提前汇总，
ADS 面向日报、看板和高频业务查询提供稳定指标。

如果不分层，Agent 或数据分析人员会直接面对大量原始表，很容易选错表、混淆口径、扫大表、重复计数，
还可能访问客户手机号、身份证号、银行卡号等敏感字段。
分层后，经营总览优先查 ADS，维度趋势优先查 DWS，授权明细才查 DWD，ODS 默认只用于上游排查。

在 Agent 数据问答里，分层还能降低 NL2SQL 难度。
Schema Catalog 可以告诉 Agent 每张表的层级、粒度、分区、权限和适用问题，
让 Agent 先选对层，再生成 SQL、校验 SQL、执行查询和记录审计。

## 2. 如何区分离线指标、实时状态、口径解释、血缘追溯和敏感导出问题？

> 来源: 核心题库

可以先看用户问题里的触发信号。
出现昨天、近 7 天、上月、趋势、按渠道、按产品，通常是离线指标；
出现近 5 分钟、当前、实时、是否异常、突增，通常是实时状态；
出现怎么算、定义、口径、分子、分母，通常是口径解释；
出现来自哪些表、上游、下游、血缘，通常是血缘追溯；
出现导出、手机号、身份证、客户名单、全部明细，通常是敏感导出。

不同意图的处理路线完全不同。
离线指标走 ADS/DWS、Schema Catalog、SQL 生成、SQL 校验和离线执行；
实时状态走实时指标工具、延迟检查和告警工具；
口径解释走 RAG 并要求引用；
血缘追溯走 lineage_tool；
敏感导出走权限校验和安全阻断。

如果缺少必要上下文，例如“看一下是否异常”没有窗口、业务线或指标名，就应该先澄清。
意图分类的目标不是让模型显得聪明，而是让后续工具调用可控、可审计、可回归。

## 3. 离线指标血缘和实时告警血缘有什么区别？

> 来源: 核心题库

离线指标血缘关注批处理链路，通常是 ODS 到 DWD、DWS、ADS，再到指标、报表或下游应用。
它重点看分区日期、调度任务、依赖表、产出表、指标口径版本和报表影响范围。
比如授信通过率异常，要排查 ADS 日报指标表、DWS 渠道汇总、DWD 申请明细、ODS 原始申请单以及对应日批任务。

实时告警血缘关注流处理链路，通常是实时事件、消息流、实时计算任务、窗口指标、延迟状态、告警规则和通知渠道。
它重点看事件时间、处理时间、窗口大小、乱序、重复、延迟阈值和告警规则。
比如实时风控拒绝率突增，要追溯 `risk_decision_made` 事件、10 分钟窗口计算任务、实时拒绝率指标和告警规则。

一句面试**总结**是：离线血缘重在批次、表依赖和报表影响；实时血缘重在事件、窗口、延迟和告警证据。

## 4. 日报 + 实时告警摘要为什么必须区分离线日期和实时窗口？

> 来源: 核心题库

日报 + 实时告警摘要必须区分离线日期和实时窗口，因为两类数据的时间口径完全不同。
离线日报通常来自 ADS 分区表，表示昨天或某个批次的经营结果，重点是 `biz_date`、分区产出和指标口径。
实时告警来自近 5 分钟、近 10 分钟或近 1 小时窗口，重点是事件时间、处理时间、延迟状态和告警规则。

如果把昨天日报指标当成当前状态，就会误导业务；如果把实时 10 分钟波动当成全天经营结果，也会夸大风险。
所以摘要里必须明确：哪些结论来自离线日报，统计日期是什么；哪些结论来自实时告警，窗口和延迟状态是什么。

一句面试**总结**是：离线日期决定历史经营口径，实时窗口决定当前风险口径，不能混在一起。

## 5. 如何设计空分区、实时延迟、告警误报和口径冲突这类 bad case？

> 来源: 核心题库

这类 bad case 的设计要模拟真实生产风险，每个场景写清预期行为：

1. 空分区样例：模拟离线 ADS 表当日分区尚未产出。
用户问"今天各渠道授信通过率"，Agent 查不到数据。
预期行为：返回 no_data 状态 + 分区状态说明，
不能编造指标，也不能用昨天数据冒充今天。
must_contain：分区状态（not_ready/empty）、建议稍后查询。
forbidden：返回数值 0、返回昨日数据作为"今天"。

2. 实时延迟样例：模拟 Kafka 积压 30 分钟，链路延迟 1800 秒。
用户问"近 5 分钟拒绝率是否异常"。
预期行为：返回 degraded 状态 + 延迟说明 + 查看最近可信窗口的建议。
forbidden：直接给出确定结论（"当前正常"或"当前异常"）。

3. 告警误报样例：模拟告警因规则错误而触发（如阈值设置过低）。
Agent 查到告警已触发，但检查指标值和规则版本后，
发现指标值和快照时间不匹配，判断为误报。
预期行为：返回 alarm_false_positive + 证据分析 + 建议调整规则。
forbidden：把告警当真实风险直接报告。

4. 口径冲突样例：模拟同一指标在离线日报和实时窗口的计算结果
不一致。预期行为：Agent 显式区分两个数据源的时间口径和统计范围，
分别说明，不强行合并一个数字。

每个 bad case 必须写清 expected_status、expected_route、
must_contain 和 forbidden_behavior。修复后跑全量回归，
确认新规则没有破坏正常查询、安全阻断和告警解释。

## 6. 为什么实时窗口和离线日报指标不能混着解释？

> 来源: 核心题库

因为框架只是实现手段，业务价值来自数据边界、工具路由、安全控制、
审计和评测。

如果开口就是"我用 LangChain 搭了个 Agent"，面试官听到的是：
你关注的是用什么工具，而不是解决了什么问题。

正确的讲法是：先用业务问题开头（金融信贷场景为什么需要同时处理
离线指标和实时告警），再讲系统如何通过意图识别、工具路由、
SQL 校验和审计来保证安全和可控。框架只是"我们在哪些环节用它
加速开发"的补充说明，出现在最后。

比如："在工具注册和编排层我们用了 LangGraph 的 StateGraph
来管理多步调用和状态流转，因为它的状态管理和节点跳转机制
比较适合我们 topic_router → intent_router → tool_executor
的链式编排。但对于权限校验、SQL Validator 和审计落库这些
确定性逻辑，我们是代码层独立实现的，不依赖任何框架。"

一句话：框架是手段，不是项目价值本身。
面试官更关心你解决了什么问题、做了哪些工程取舍。

## 7. AI 应用上线后需要监控哪些指标？

> 来源: 核心题库

要监控业务效果、模型效果和工程稳定性。比如请求量、成功率、延迟、错误率、
token 成本、召回命中率、回答正确率、无答案率、用户反馈和工具调用失败率。
这些指标用于发现问题和持续优化。
AI 应用上线后的监控要分层看。工程层看 QPS、平均延迟、P95 延迟、错误率、
超时率、模型调用失败率和外部依赖状态。成本层看 token 消耗、单次请求成本、
缓存命中率和不同模型使用占比。效果层看 RAG 召回是否命中正确资料、
引用是否准确、回答是否基于上下文、NL2SQL 是否通过校验、SQL 执行是否成功。
业务层看用户满意度、问题解决率、人工转接率和 bad case 类型。
有了这些指标，团队才能知道问题出在知识库、检索、prompt、模型、工具还是接口稳定性上。

## 8. 指标抽取和维度抽取有什么区别？

> 来源: 核心题库

指标抽取关注“要算什么”，比如授信申请量、授信通过率、放款金额、逾期率。
维度抽取关注“按什么看”，比如渠道、城市、信贷产品、日期、风险等级、逾期账龄。
在 SQL 里，指标通常对应聚合字段或聚合表达式，维度通常对应 group by、where 或展示字段。
两者不能混淆。比如“每个渠道的授信通过率”，授信通过率是指标，渠道是分组维度。
如果把维度当指标，SQL 聚合会错；如果把过滤条件当分组维度，结果粒度也会错。

## 9. 指标字典和 Schema Catalog 有什么区别？

> 来源: 核心题库

指标字典和 Schema Catalog 解决的是两个不同问题。
指标字典回答“这个数怎么算”，Schema Catalog 回答“这个数从哪些表字段查”。
前者管业务口径，后者管数据结构。

指标字典关注指标含义、计算公式、分子、分母、过滤条件、时间口径、统计粒度、可用维度、适用范围和负责人。
例如审批通过率的指标字典要写清：分子是审批通过申请数，分母是进入审批流程申请数，
时间口径按申请完成时间，过滤条件要剔除测试渠道和撤销申请。
它主要用于口径解释、指标校验、结果解释和 RAG 问答，防止 Agent 把指标讲错或混用。

Schema Catalog 关注表名、表描述、字段名、字段类型、主键、分区字段、表粒度、表关系、权限标签和可查询范围。
例如 `ads_credit_daily_metrics` 这张表要写清字段包括 `biz_date`、`channel`、`product_code`、
`apply_cnt`、`approved_cnt`、`approval_rate`，分区字段是 `biz_date`，不包含客户手机号和身份证号。
它主要用于 NL2SQL 的表选择、字段选择、SQL 生成、SQL 校验和权限控制，防止 Agent 选错表、编造字段、漏分区或越权查询。

在金融信贷 Agent 链路里，用户问“审批通过率怎么算”，应该优先查指标字典或 RAG 口径文档；
用户问“近 7 天各渠道审批通过率是多少”，才需要先根据指标字典确认口径，再通过 Schema Catalog 找到合适的 ADS/DWS 表和字段，
生成 SQL 并校验执行。

一句面试**总结**是：指标字典保证“算得对”，Schema Catalog 保证“查得对”。
生产级数据问答必须同时具备这两类元数据，否则要么口径错，要么 SQL 查错。
