## 1. 当⼤模型API响应延迟超过1秒时，前端可以采取哪些优化策略保证⽤户体验？

> 来源: 面试鸭题库

前端能做的其实很有限，核⼼是“让⽤户感觉快”，⽽不是真的让模型变快。⼤模型API本⾝耗时主要在服务端推理，
前端更多是体验层的兜底。
1）请求发出去之后，⽴刻给⽤户反馈，别⼲等。⽐如⻢上展⽰⼀个“思考中...”的动画或者⻣架屏，告诉⽤户系统已
经在处理了，避免误操作重复提交。
2）如果接⼝超过800ms还没回来，可以主动提⽰“⽣成时间较⻓，请稍候”。这种预期管理很重要，⽤户知道要等，
就不会觉得卡死了。
3）优先考虑流式响应（streaming）。让后端以SSE或WebSocket⽅式分段返回结果，前端拿到⼀段就渲染⼀段。就像
ChatGPT那样逐字输出，视觉上⽐等1秒再刷出来舒服得多。
4）结合loading状态和取消按钮。允许⽤户在等待时⼿动中断请求，提升控制感。特别是移动端，⻓时间⽆响应容易
引发焦虑。
5）缓存历史问答对。对于常⻅问题，⽐如产品FAQ，可以直接⾛本地缓存秒回，根本不⽤⾛⼤模型。
6）降级策略。检测到连续超时，可以⾃动切换到轻量模型接⼝或静态回答模板，保证基础可⽤性。

## 2. 当⼤模型上下⽂窗⼝扩展到100万token时，哪些现有业务场景可能发⽣质变？

> 来源: 面试鸭题库

上下⽂窗⼝突破百万 token，最直接的影响是模型能“看⻅”和处理的信息⻓度远超以往。以前处理⻓⽂本得切⽚、
摘要、丢信息，现在可以直接喂完整⽂档，很多场景的实现⽅式会彻底改变。
1）⾦融领域的财报分析、合同审查会变得精准得多。⽐如分析⼀份上百⻚的跨国并购协议，模型能⼀次性理解所有条
款的关联性，找出潜在⻛险点，⽽不⽤像过去那样分段处理导致上下⽂断裂。类似 Kensho 这类系统在做法律与⾦融
⽂档分析时，准确率和效率会跃升。
2）代码⼯程的理解和维护进⼊新阶段。整个⼤型项⽬的源码可以作为上下⽂输⼊，让模型真正理解模块间的依赖关
系。像 GitHub Copilot 这种⼯具，不再只是函数级补全，⽽是能做跨⽂件重构建议、架构优化提⽰，相当于有个全局
视⻆的资深架构师辅助。
3）科研⽂献综述和知识发现会发⽣质变。研究⼈员可以把⼏千篇论⽂全⽂丢给模型，让它⾃动梳理领域发展脉络、发
现研究空⽩。这在⽣物医药、材料科学这类⽂献密集型领域尤其关键。
4）客服和企业知识库的体验升级。传统 RAG ⾯对复杂问题经常答⾮所问，因为检索⽚段有限。百万上下⽂下，可以
直接加载整个产品⼿册或历史⼯单，回答更完整准确。

## 3. 假设请你设计⼀个医疗问诊系统，如何平衡AI幻觉带来的⻛险与效率提升？需要哪些技术⼿段？

> 来源: 面试鸭题库

医疗问诊系统⾥，AI 效率是真⾼，⼀个模型能同时看⼏⼗路问诊，但幻觉问题可不敢⻢⻁。说错症状、开错建议，那
是要出事的。
关键在于不能让⼤模型直接输出诊断结论。得给它套上⼏层“安全锁”。
1）所有⽤户输⼊先过⼀遍意图识别和实体抽取，⽤微调过的⼩模型把“我头疼三天了”转成结构化数据，⽐如
{symptom: headache, duration: 3 days}。这步很稳，不靠⽣成。
2）结构化数据进知识图谱推理。咱们有医学本体库，像 UMLS 或⾃建的疾病-症状-检查关系⽹。模型查“头痛+发
烧”可能关联流感、脑膜炎，但不会瞎编个“⽕星病毒”。
3）⽣成回复时，强制⾛检索增强（RAG）。答案必须来⾃权威指南，⽐如 UpToDate 或卫健委诊疗规范，向量库⾥只
放审核过的⽂档。⼤模型只负责把检索到的内容⽤⼝语化组织出来，不准⾃由发挥。
4）最后加⼀层规则引擎做合规校验。⽐如涉及处⽅药，必须提⽰“请线下就诊”，涉及急症关键词⾃动转⼈⼯。
说⽩了，⼤模型只⼲“语⾔润⾊”的活，诊断逻辑全由确定性系统驱动。效率归AI，安全归规则，各司其职。
设计智能客服系统时，如何通过知识库构建解决⻓尾问题？请描述具体实现步骤
智能客服的⻓尾问题指的是那些发⽣频率低、但种类繁多的⽤户提问，⽐如“怎么退订海外会员的⾃动续费”。这类问
题单个看很少⻅，加起来能占到 30%-40% 的咨询量，靠⼈⼯维护规则搞不定。
解决思路是把散落的问题沉淀成结构化知识库，让系统能⾃动匹配相似问法。第⼀步，收集历史⼯单、客服聊天记
录，⽤ NLP 做聚类分析，把语义相近的问题归到⼀起，⽐如“取消订阅”、“退会员”、“停⾃动扣费”都归为同⼀类。
第⼆步，定义知识条⽬结构，每个条⽬包含标准问法、同义问法扩展、答案内容、适⽤业务场景标签。可以⽤
Elasticsearch 建索引，⽤户提问进来后，先做分词和语义向量化，再在知识库中找最相似的标准问法。
第三步，引⼊反馈闭环。线上每次匹配结果被客服采纳或⽤户点击“有帮助”，就记⼀次正向反馈；如果没命中或被修
正，就触发⼈⼯复核机制，定期更新知识条⽬。
1）聚类原始问题发现潜在⻓尾
2）构建带同义词扩展的知识条⽬
3）语义检索 + 向量相似度匹配
4）通过⽤户⾏为持续优化覆盖
关键是别指望⼀步全覆盖，⽽是靠这个闭环让知识库⾃⼰“⻓⼤”。

## 4. 模拟面试里为什么一定要准备 bad case 和安全问题？

> 来源: 核心题库

因为生产系统最容易在异常和风险场景暴露问题。面试官也常用这类问题区分"会搭 Demo"和"理解生产约束"的候选人。

成功路径大家都能讲——查指标、返回答案、展示引用。
但 bad case 才体现工程成熟度：数据延迟时如何降级而不是给错误结论、
敏感字段查询如何阻断而不是婉拒、口径冲突时如何标注而不是强行合并、
实时问题误走离线 SQL 时如何从工具路由层面修复。

安全问题的准备同样关键，因为金融信贷 Agent 涉及授信、风控、逾期和客户明细，
面试官一定会追问权限控制、敏感信息保护和审计追溯。如果能讲清楚这些边界的工程实现，
会显著区别于只会调 API 的候选人。

## 5. FastAPI 在 AI 应用服务化里一般承担什么角色？

> 来源: 核心题库

FastAPI 通常负责把 Python 里的 AI 能力包装成 HTTP API。
比如提供 RAG 问答接口、SQL 解释接口、NL2SQL 接口和健康检查接口。
它结合 Pydantic 做请求校验和响应结构约束，适合快速构建 AI 应用后端。
在 AI 应用里，模型调用、检索、SQL 校验和工具执行通常都在 Python 生态里完成。
FastAPI 可以把这些能力服务化，让前端、业务系统或调度系统通过 HTTP 调用。
比如 `POST /rag/ask` 接收用户问题，返回答案和引用；
`POST /sql/explain` 接收 SQL，返回风险等级和建议；
`POST /nl2sql` 接收自然语言问题，返回 SQL、校验结果和解释。
FastAPI 的优势是开发效率高，自带 OpenAPI 文档，和 Pydantic 结合紧密。
生产里还要补齐鉴权、限流、日志、异常处理、配置管理和部署监控。

## 6. AI API 服务为什么需要配置、日志和异常处理？

> 来源: 核心题库

配置解决环境差异和密钥管理，日志解决排查和审计，异常处理解决服务稳定性。
AI 服务会调用模型、向量库、数据库和外部 API，出错点很多。
没有这些工程能力，服务很难稳定上线。
生产 AI API 服务不是只要本地能跑通。配置管理让不同环境使用不同参数，
比如模型名、API Key、数据库地址、向量库地址、timeout 和日志级别。
密钥不能写死在代码里，要通过环境变量或密钥管理系统注入。
日志用于排查问题和做审计，比如记录请求 ID、用户、耗时、调用链路、错误码和关键结果。
异常处理用于把底层错误转换成稳定响应，避免把堆栈、密钥或内部路径暴露给用户。
RAG 和 NL2SQL 的链路长，任何一个环节失败都要有清晰错误和降级策略。
这些能力决定服务能不能从脚本变成线上系统。

## 7. 调用外部 API 或 LLM API 时要怎么保证稳定性？

> 来源: 核心题库

要设置 timeout、重试、限流、熔断、错误分类和日志。
还要处理 4xx、5xx、网络超时、返回格式异常和额度不足。
生产里不能假设外部 API 永远成功，要有失败兜底和可观测性。
调用外部 API 时，第一步是设置合理 timeout，避免请求一直卡住。
第二是做有限重试，通常只对网络抖动、超时、部分 5xx 做重试，
不要对参数错误或权限错误盲目重试。第三是错误分类，比如 401 是鉴权问题，
429 是限流问题，5xx 是服务端问题，JSON 解析失败是响应格式问题。
第四是日志和指标，记录请求耗时、状态码、错误类型和重试次数。
对于 LLM API，还要关注 token 成本、上下文长度、模型不可用和输出格式异常。
必要时要有降级策略，比如返回规则结果、提示稍后重试，或者切换备用模型。

## 8. 为什么 AI 应用要从脚本重构成后端服务？

> 来源: 核心题库

AI 应用从脚本重构成后端服务，是为了让能力能被前端、业务系统、机器人或其他服务稳定调用。
脚本适合本地验证，但不适合权限控制、接口复用、审计追踪、错误处理和多用户访问。
服务化后，需要把 API 层、service 层、storage 层、config 层和 errors 层拆清楚。
这样后续替换模型、数据库、SQL 生成器或权限系统时，不会影响接口契约。
在 NL2SQL 场景里，服务化还能统一处理 request_id、审计记录、安全阻断和响应格式。

## 9. Docker 化解决了 AI 应用交付里的什么问题？

> 来源: 核心题库

Docker 化解决的是运行环境可复现和交付一致性问题。
AI 应用常依赖 Python 版本、系统库、模型 SDK、环境变量和项目文件结构。
如果只说“我本机能跑”，换机器、换环境或部署到服务器时很容易失败。
Dockerfile 把依赖安装、代码复制、启动命令和端口暴露写成标准步骤，
让开发、测试和部署使用同一套运行方式。
但 Docker 化不等于生产完备，生产还需要配置管理、密钥管理、日志、监控、健康检查和发布流程。

## 10. 为什么配置和密钥不能写死在代码里？

> 来源: 核心题库

配置和密钥不能写死在代码里，因为不同环境的数据库地址、开关、路径、日志级别和权限策略都可能不同。
如果写死在代码里，切换开发、测试、生产环境就必须改代码，容易引入误配置。
密钥更不能提交到 Git，因为一旦泄露，可能导致模型账户、数据库或内部服务被滥用。
生产里应该用环境变量、配置中心或密钥系统管理真实配置，
仓库里只保留 `.env.example` 说明需要哪些变量和示例值。
