## 1. 当需要处理超⻓⼤模型上下⽂窗⼝限制时，有哪些可⾏的⼯程解决⽅案？

> 来源: 面试鸭题库

处理超⻓上下⽂，本质是绕过或优化模型的 token ⻓度硬限制。直接喂满动辄 32K、128K 上下⽂，成本⾼且效果不⼀
定好，得靠⼯程⼿段拆解。
1）分块与索引检索
把⻓⽂档切⽚存⼊向量数据库，⽐如⽤ LangChain + Chroma/Pinecone，查询时只召回相关 chunk。关键在分块策
略，按段落切可能割裂语义，结合滑动窗⼝重叠（overlap）能缓解。但太碎了会丢失全局逻辑。
2）摘要增强（Summary Augmentation）
对历史对话或⽂档提前⽣成摘要，新请求到来时，把摘要当上下⽂注⼊。像 LLM-powered summarizer 在对话系统⾥
很常⻅，能把前 10 轮压缩成 1-2 句，省下⼤量 token。
3）位置插值（Position Interpolation）
针对 Transformer 架构的位置编码做⼿脚。原⽣ RoPE ⽀持外推但衰减快，采⽤ NTK-by-parts 或 YaRN 等⽅法，在推
理时动态调整位置频率，让模型“脑补”超出训练⻓度的位置信息。实测能在 32K 模型上跑 64K 输⼊，效果接近原⽣
⻓上下⽂。
4）侧边缓存（Side Cache）
Google 提出的思路，把已计算的 KV 缓存复⽤起来。⽐如⽤户修改了 prompt 中⼀⼩段，不需要全量 recompute，命
中部分直接从 cache 读。适合交互式场景，降低延迟和算⼒消耗。
5）分层上下⽂路由
复杂系统如 Agent ⼯作流，⽤调度层决定哪些信息进⻓期记忆（vector store），哪些放短期上下⽂。AutoGPT 类项⽬
就这么⼲，核⼼状态放 context，细节查数据库。

## 2. 提⽰词优化考虑哪些维度？你们提⽰词模板有哪些字段？

> 来源: 面试鸭题库

这问题其实是在问提⽰词⼯程的系统化设计思路，不只是随便写句话。
1）⾸先看维度。⼀个有效的提⽰词必须明确任务⽬标、输⼊约束和输出格式。任务类型决定结构，⽐如分类、⽣成、
抽取需要的模板完全不同。上下⽂信息也得给⾜，模型才能理解语义边界。还要考虑安全合规要求，像敏感词过滤、
价值观对⻬这些都得前置控制。
2）再说模板字段。我们常⽤的提⽰词模板包含这⼏个关键部分：⻆⾊定义，说明这个 prompt 是给谁⽤的；背景描
述，交代清楚业务场景；指令主体，这是核⼼，告诉模型具体要做什么；⽰例样本，few-shot 的例⼦能显著提升准确
率；输出约束，⽐如 JSON 格式、字数限制、枚举值范围；最后是异常处理机制，定义模糊输⼊时的降级策略。
代码⽰例⻓这样：
prompt = """
⻆⾊：电商客服助⼿
背景：⽤户在商品详情⻚发起咨询
指令：根据商品信息和⽤户问题，⽣成专业且友好的回复
输⼊：
- 商品标题： {title}
- ⽤户问题： {query}
输出格式： JSON ，包含  reply 字段，不超过  150 字
"""
这种结构能让⼤模型稳定输出符合预期的结果，⽽不是靠运⽓。

## 3. 如果一个GPU集群的LLM处理能力为1000tokens/s，那1000个用户同时并发访问，响应给每个用户的性能只有1 token/s吗？怎么分析性能瓶颈

> 来源: 面试鸭题库

不⼀定每个⽤户就只能分到 1 token/s，实际性能取决于调度策略、批处理能⼒和请求的负载特征。
⼤模型服务不像传统 Web 服务器那样“⼀个请求占⼀个线程”，它⾛的是 动态批处理（Dynamic Batching） 路⼦。
⽐如⽤ vLLM、TensorRT-LLM 这类推理框架时，多个⽤户的请求会被合并成⼀个 batch ⼀起跑，显存和计算单元利⽤
率更⾼。
**关键点**在于：
1）如果这 1000 个⽤户都是刚发请求，还没开始⽣成内容（preﬁll 阶段为主），那系统可以⼀次性把他们的 prompt 做
batch 处理，吞吐可能接近峰值 1000 tokens/s，平均下来每⼈⼏⼗甚⾄上百 token/s 都有可能。
2）但到了 decode 阶段，每个请求要逐 token ⽣成，且⻓度不⼀，这时候 batch size 会慢慢被“拖慢”。⻓⽂本、⾼
采样参数（如 temperature ⾼）的请求会拉低整体速度。
3）真正的瓶颈往往不是算⼒，⽽是显存带宽和 KV Cache 管理。每个并发请求都要保留 KV Cache，1000 个并发意味
着巨⼤的显存压⼒，搞不定就会 OOM 或被迫丢请求。
举个例⼦，A100 显存 80GB，跑 Llama3-8B，每个请求平均占 2GB KV Cache，理论上最多撑 40 个并发。超过就得靠
PagedAttention（像 vLLM 实现的）来分⻚管理，否则压根不经过计算就挂了。
所以性能瓶颈通常出现在：
KV Cache 显存占⽤过⼤
⼩ batch 或不规则请求导致 GPU 利⽤率上不去
请求排队时间远超⽣成时间
优化⽅向是：⽤ PagedAttention 降低显存压⼒，启⽤连续批处理（Continuous Batching），控制最⼤⽣成⻓度，避
免⼩请求拖垮整体延迟。

## 4. 向量数据库原理是什么？请简述下它的原理

> 来源: 面试鸭题库

向量数据库本质是为⾼维向量数据设计的存储与检索系统，核⼼⽬标是在亿级向量中快速找到和查询向量最相似的那
些。
传统数据库⽤B+树做精确匹配或范围查询，但向量相似性搜索要的是“近似”，⽐如余弦相似度、欧⽒距离。如果暴⼒
遍历所有向量算距离，1亿个向量每个128维，⼀次查询就得⼏⼗毫秒甚⾄上百毫秒，根本扛不住。所以关键在于近似
最近邻搜索（ANN）算法，在可接受精度损失下把查询从O(n)降到O(log n)甚⾄常数级。
常⻅实现⽅式有⼏种：
1）基于局部敏感哈希（LSH），把相近向量以更⾼概率映射到同⼀个桶⾥，适合⾼维稀疏场景，但召回率波动⼤；
2）基于树结构如随机投影树，划分空间，但⾼维下效果退化明显；
3）最主流的是图-based⽅法，⽐如HNSW（Hierarchical Navigable Small World），构建多层导航图，顶层稀疏快速
定位⽅向，底层精细搜索，能在毫秒内完成百万级查询，Milvus、Pinecone都在⽤。
另外，向量数据库通常会结合标量索引做混合查询，⽐如“找和这个图⽚相似且上传时间在最近7天的”。这就需要⽀
持结构化字段过滤 + 向量检索的联合优化。

## 5. 解释LangChain框架中的Chain和Agent概念，并举例说明各自的应用场景

> 来源: 面试鸭题库

LangChain ⾥的 Chain 和 Agent 是两个关键抽象，解决的是不同复杂度的任务编排问题。
Chain 其实就是⼀个函数式的组合，把多个处理步骤像链条⼀样串起来。每个环节的输出⾃动作为下⼀个环节的输
⼊，中间可以穿插对 LLM 的调⽤、数据处理或外部⼯具。⽐如你做客服机器⼈，⽤户问“今天北京天⽓怎么样”，你
可以先⽤⼀个 PromptTemplate 构造查询语句，接着调⽤ LLM 解析出地点和需求，再交给⼀个 WeatherTool 获取真
实数据，最后⽤另⼀个模板⽣成回复。这个流程就是典型的 Chain，顺序执⾏，逻辑固定。
// 伪代码⽰意
String result = chain.execute(" 今天北京天⽓如何？ ");
⽽ Agent 更灵活，它让 LLM ⾃⼰决定“下⼀步⼲什么”。LLM 不只是回答问题，⽽是充当“决策⼤脑”，根据当前状态
选择调⽤哪个⼯具、是否需要查数据库、要不要反复推理。⽐如做⼀个智能数据分析助⼿，⽤户说“帮我分析上周销
售额下降的原因”，Agent 可能先查销售表，再对⽐营销活动记录，发现某区域⼴告暂停后销量下滑，于是建议检查推
⼴策略。整个过程路径不固定，由 LLM 动态规划。
1）Chain 适合流程明确、步骤固定的场景，⽐如内容⽣成流⽔线、结构化信息提取
2）Agent 适合开放性任务，需要动态决策，⽐如智能客服、⾃动化运维诊断
3）Agent 的代价是性能开销⼤，因为每⼀步都要⾛ LLM 推理，容易陷⼊循环或误判

## 6. 什么是大模型的"涌现能力"？列举三种典型表现并解释其可能成因

> 来源: 面试鸭题库

⼤模型的“涌现能⼒”指的是当模型参数规模达到某个临界点后，突然展现出⼩模型不具备的能⼒，这种能⼒⽆法通
过简单外推得到。
1）上下⽂学习（In-Context Learning）
模型仅凭输⼊中的⼏个⽰例就能完成新任务，不需要额外微调。⽐如给 GPT-3 输⼊“苹果→⽔果，胡萝⼘→蔬菜，番
茄→？”它能推理出“蔬菜”。这背后其实是模型在推理时动态构建了内部计算路径，把提⽰词当成了临时指令模板。
2）思维链推理（Chain-of-Thought Reasoning）
模型能输出中间推理步骤，⽐如数学题先列式再求解。实验发现，只要在提⽰中加⼊“让我们⼀步步思考”，PaLM、
GPT-3.5 等模型就能激活这种⾏为。这说明⼤模型内部已经形成了类似推理的计算模式，但需要外部触发。
3）多任务泛化
同⼀个模型可以同时处理翻译、代码⽣成、逻辑判断等完全不同的任务，且表现接近专⽤模型。像 Llama 系列在训练
时混⼊了海量多源数据，使得其权重空间⾃然分化出多个功能⼦空间。
可能成因在于：参数量突破临界点（⼀般认为要超 60B）后，模型表达能⼒发⽣质变，⾼维空间中出现新的结构化组
织⽅式。训练数据的多样性让模型学会了“元学习”⸺不是记答案，⽽是学怎么学。

## 8. 请描述使用LangChain构建一个文档问答系统的关键技术组件及实现步骤

> 来源: 面试鸭题库

构建⽂档问答系统，本质是把⾮结构化⽂本转成向量，再结合⼤模型做⾃然语⾔交互。LangChain 提供了⼀整套⼯具
链来简化这个过程。
1）⽂档加载与分割
先⽤ DocumentLoaders 从 PDF、Word 或⽹⻚抓取原始内容。拿到⽂本后得切分，⼀般按段落或固定⻓度（⽐如 500
token），太⻓会超出模型上下⽂，太短⼜丢失语义。这⾥常⽤ RecursiveCharacterTextSplitter。
2）向量化与检索
切好的⽂本块喂给嵌⼊模型（Embedding Model），⽐如 OpenAI 的 text-embedding-ada-002 或开源的 BGE。⽣成
的向量存进向量数据库，像 Chroma、Pinecone 或 Milvus。⽤户提问时，系统先把问题向量化，去库⾥找最相似的
top-k ⽂本块，这叫相关性检索。
3）提⽰⼯程与模型调⽤
检索到的上下⽂拼上⽤户问题，组装成 prompt，丢给⼤语⾔模型（LLM）。LangChain 提供 PromptTemplate 来规范
格式，避免模型胡说。典型的模板就是“根据以下内容回答问题：{context} 问题：{question}”。
4）链式调⽤（Chain）
把上述步骤串起来⽤ LLMChain 或 RetrievalQA 链。RetrievalQA 是现成的⾼级封装，⼀⾏代码就能连起检索器和
LLM，适合快速搭建原型。
整个流程⾥，数据预处理和检索质量直接决定效果上限。模型再强，喂进去的上下⽂不对，也搞不定准确回答。

## 9. 假设要开发一个智能工单分类系统，请拆解AI可参与的环节并说明技术选型思路

> 来源: 面试鸭题库

智能⼯单分类不是简单打个标签，得把AI嵌进整个流转链条⾥，哪⾥能省⼈⼒、提准确率就往哪插。
1）⼯单预处理阶段，⽂本清洗和标准化可以⽤规则+轻量模型。⽐如⽤户输⼊“登不录”，先⽤正则和纠错词典做基础
清洗，再上⼀个 TinyBERT 做语义对⻬，转成标准表述“⽆法登录系统”。这步别上⼤模型，响应延迟扛不住。
2）核⼼分类环节看数据量和类⽬复杂度。如果只有⼏⼗个固定类别，RoBERTa + CRF ⾜够，推理快、好维护。要是类
⽬动态增⻓，⽐如每天新增业务线，就得考虑 Prompt Learning 配合 P-tuning v2，接通⼤模型如 ChatGLM-6B 做
zero-shot 分类，类⽬⼀改prompt跟着变，不⽤重新训练。
3）后置反馈闭环必须做。分错的⼯单要进⼈⼯复核池，这部分数据定期回流做增量训练。可以搭个轻量 Label Studio
+ Milvus 相似样本检索，⾃动推荐历史相似案例给标注员，效率翻倍。
4）⼯程部署上，⼩模型⾛ TensorFlow Lite 或 ONNX Runtime 嵌⼊服务内部，⼤模型⽤ vLLM 做批处理异步推理。别
把所有请求都怼到⼤模型上，成本直接炸。

## 10. JSON 合法和业务校验通过有什么区别？

> 来源: 核心题库

JSON 合法只说明格式能被解析，比如括号、引号、逗号没错。
业务校验通过还要求字段完整、类型正确、枚举值合法，并且符合业务规则。
比如 `risk_level` 必须是固定枚举，`can_publish` 必须和风险等级一致。
JSON 合法是语法层面的概念，程序能 `json.loads` 解析就算合法。
但生产系统还需要业务层校验。比如 SQL 解释结果里必须有 `summary`、`tables`、
`risk_level`、`risks` 和 `suggestions`。其中 `risk_level` 不能随便写，
必须是 `low`、`medium`、`high` 这类固定枚举。`risks` 应该是数组，
每个风险最好有类型、说明和建议。更进一步，业务规则也要一致：
如果风险等级是 high，`can_publish` 就不能是 true。否则 JSON 虽然合法，
但下游系统会误判。生产里通常用 Pydantic、JSON Schema 或自定义规则做这层校验。

## 11. 一次 LLM 调用需要关注哪些参数和工程问题？

> 来源: 核心题库

要关注模型选择、messages、temperature、max tokens、timeout、重试、错误处理、
成本和日志。像 SQL 解释、RAG、NL2SQL 这类场景更重视准确和稳定，
通常会使用较低 temperature，并限制输出格式。
一次 LLM 调用不只是传一段 prompt。首先要选择合适模型，简单分类和结构化抽取可以用较小模型，
复杂推理或长上下文问答可能需要更强模型。messages 里要区分 system prompt、
user prompt 和检索上下文。temperature 控制随机性，生产里很多任务需要低 temperature，
避免输出漂移。max tokens 控制输出长度和成本。工程上还要设置 timeout、
重试、限流和异常处理，不能让模型调用拖垮接口。日志也很重要，
至少要记录请求 ID、模型、耗时、token、错误码和关键业务结果，
但不能把敏感数据明文打到日志里。

## 12. 演示 citations 时，应该向面试官或同事说明什么？

> 来源: 核心题库

演示 citations 时不能只说“这里有引用”，要说明引用解决的是可信度、追溯和排查问题。

## 13. query rewrite 有哪些常见方式？

> 来源: 核心题库

query rewrite 常见方式包括规则词典改写、同义词扩展、业务术语补全、上下文补全、
精确实体保留、多 query 拆分和 LLM 改写。规则词典适合高频、稳定的业务词，比如把
“来源”扩展成 citations、引用来源、可追溯；同义词扩展适合用户表达和文档表达不一致；
上下文补全适合“这个怎么算”这类指代问题；多 query 适合一个问题包含多个检索方向；
LLM 改写适合复杂口语表达。生产里通常先用可解释规则做第一版，再用测试集验证收益，
最后再考虑 LLM rewrite 和 rerank 配合。

## 14. rewrite drift 是什么，为什么危险？

> 来源: 核心题库

rewrite drift 指 query rewrite 改变了用户原意，导致系统检索到看似相关但实际错误的资料。
它危险的地方在于，召回结果可能分数很高、答案看起来也合理，但回答的不是用户真正想问的问题。
例如用户问“接口怎么设计”，系统错误改写成“前端页面怎么设计”，就会把检索方向带偏。
生产里 rewrite drift 会造成错误回答、错误引用和用户信任下降，严重时还可能触发权限或合规风险。
控制方式是保留原始 query、限制扩展词数量、保留表名字段名等精确实体、记录 rewritten query、
用评测集比较改写前后效果，并在低置信度或意图不清时选择澄清或拒答。

## 21. 结果解释层怎么避免模型编造业务原因？

> 来源: 核心题库

结果解释层要避免编造，核心是把解释建立在结构化查询结果上。
系统应该先由规则层生成事实性结论，比如最高值、最低值、环比差值、趋势方向和返回行数，
再把这些事实交给 LLM 做可选润色。
Prompt 里要明确要求模型只能引用提供的 rows、columns、指标口径和时间范围，
不能自行猜测原因。
如果需要原因分析，应该输出“建议继续按渠道、产品、风险等级或日期拆分”，
而不是直接说原因已经确定。
生产里还要保留原始查询结果、SQL、request_id 和审计日志，方便回溯解释是否忠于数据。

## 23. 生产级 AI 数据问答系统通常应该拆成哪些层？

> 来源: 核心题库

生产级 AI 数据问答、NL2SQL 或 Agent 系统，不能只看成“用户问题进来，模型回答出去”。
我会把它拆成接入层、身份与权限层、编排层、意图识别层、上下文检索层、生成层、
安全校验层、工具执行层、结果解释层、审计日志层和评测回归层。

接入层负责 API、参数校验和统一响应；身份与权限层判断用户是谁、能访问哪些表字段和文档；
编排层决定下一步调用哪个工具，以及失败时是补充条件、降级、拒答还是转人工。
意图识别层判断用户是在查指标、问规则、查明细还是提出敏感导出请求。
上下文检索层包括 Schema 检索和 RAG 检索，负责找表字段、指标口径、政策规则和引用资料。

生成层负责生成候选 SQL、回答计划或结果解释草稿，但生成结果不能直接执行。
安全校验层要检查只读限制、敏感字段、权限边界、时间范围、limit 和成本风险。
工具执行层只执行已经满足前置条件的动作，例如执行 SQL、读取 RAG 资料或写审计。
结果解释层把结构化结果翻译成业务语言，但不能编造 SQL 没查出来的原因。
审计日志层记录 request_id、用户、问题、工具路线、SQL、校验结果、执行状态和最终回答。
评测回归层用固定测试集持续验证正常回答、安全阻断、权限不足、无答案和工具失败场景。

这套分层的核心思想是：模型可以参与理解、生成和表达，但权限、安全、执行和审计必须由确定性工程层控制。
在金融信贷场景里，这一点尤其重要，因为授信、风控、逾期、贷后和客户明细都涉及合规、成本和可追溯要求。

## 24. 数据质量问题为什么要接入错误治理和回归测试？

> 来源: 核心题库

因为数据质量问题不是一次性故障，而是会重复出现的生产风险。
如果只靠人工排查，下次空分区、重复写入或补偿延迟发生时，Agent 还会犯同样的错。

生产级的处理方式是把质量问题按类型（空值、重复、乱序、延迟、补偿、幂等）分类，
每条关联一个具体的质量规则 ID，定义触发条件（如 NULL 比例 > 5%、同一 event_id 出现超过 1 次）、
风险等级和质量状态（quality_warning 或 quality_blocked）。

然后将质量异常做成规则、审计字段和固定 bad case，接入端到端回归测试集。
每次代码或规则变更后自动跑全量回归，验证 quality_status 和回答行为是否符合预期断言。
这样才能从"每次人工排查"变成"规则化 + 审计化 + 回归化"的工程闭环。

## 25. 为什么综合项目必须提供统一入口，而不是保留多个独立脚本？

> 来源: 核心题库

因为面试和生产都看重系统边界，而不是零散能力清单。

多个独立脚本的问题在于：无统一入口和复用，配置散落难以维护，工具重复实现导致审计和错误码不一致，
用户（或面试官）也无法直观理解这些能力如何协同工作。

统一入口意味着意图识别、权限校验、工具路由、结果校验和审计逻辑被真正整合成一个主流程。
这才是"一个系统"而不是"一堆脚本拼成的故事"。

**统一入口的最小主流程包含哪些节点？**
用户问题 → 意图识别 → 权限校验 → 工具路由 → 工具执行 → 结果校验 → 回答生成 →
审计落库 → trace/eval。每个节点只做一件事，做完交给下一个，不能越界。
工具执行负责实际调用（查 SQL、读 RAG、查实时指标），回答生成负责把结构化结果转成业务语言，
trace/eval 负责记录链路供回归评测。

**模块边界不清晰会带来什么问题？**
第一，无法排查——出错了不知道是哪一步的问题，因为大家都在干活。
第二，无法复用——校验逻辑写在 SQL 生成器里，以后实时工具想复用还得再写一遍。
第三，安全形同虚设——SQL 生成器说"我顺便校验了权限"，但模型生成的校验不可靠。
第四，评测无从下手——边界混在一起，不知道每层该测什么。

**集成 README 至少应该讲清楚哪些内容？**
综合项目 README 至少要包含：项目目标和适用场景、架构流程和主链路图、启动方式和环境依赖、
接口请求/响应示例（含字段含义）、固定测试问题（成功样例+bad case）、常见失败场景和排查方式、
当前限制和后续优化方向。它的作用是让面试官能在 30 秒内理解这个项目解决什么问题。

**Day 50-58 的哪些能力应该优先整合进统一入口？**
应该整合的具体能力包括：指标字典 RAG 口径问答（Day 50）、Schema Catalog + 工具注册（Day 51）、
离线 NL2SQL 指标查询（Day 53）、实时告警解释工具（Day 54）、日报 + 告警摘要 Agent（Day 55）、
血缘追溯工具（Day 52）、端到端评测回归（Day 56）、数据质量状态检查（Day 57）和审计存储（Day 58）。
统一入口按"意图识别→权限校验→工具路由→工具执行→结果校验→回答生成→审计落库"的流程编排它们。

## 26. 为什么综合项目要保留本地 fallback？

> 来源: 核心题库

因为这个项目既是学习仓库，也是作品集。本地 fallback 有三个核心价值：

第一，保证演示稳定。如果所有能力都强依赖外部模型和在线环境，演示时网络抖动或模型不可用就会导致翻车。
第二，保证回归可重复。确定性规则的结果不会因为模型版本变化而漂移，评测集每次跑出来的结果一致。
第三，把确定性规则和模型能力分开讲清楚。对于 SQL 安全、权限拦截、延迟判断这类关键逻辑，
本来就不应该完全依赖模型——它们应该走确定性代码，模型只负责理解和表达。

保留本地 fallback 也能让你在面试时说清楚"哪些用规则兜底、哪些交给模型"的工程取舍。

## 27. 3 分钟项目演示最应该讲什么？

> 来源: 核心题库

最应该讲业务问题、系统边界、核心链路和工程取舍。
下面是我建议的完整讲法，完全从业务视角展开，先带面试官看两个真实场景，再倒推模块设计。
①业务问题——为什么金融信贷场景同时需要离线仓库和 实时仓库；
②系统架构——RAG、NL2SQL、实时工具、血缘和审计各自负责 什么；
③成功路径——给一个正常查询或日报摘要样例；
④风险路径—— 给一个延迟、越权或口径冲突样例；
⑤工程价值——为什么这不是普通聊天机器人，而是有数据边界和审计能力的 Agent。
**为了方便，把这个项目称为"信贷智能分析体（Credit Agent）"。**

 *第一部分：先看两个真实的业务场景（带出业务问题）
**场景一（实时决策 - 贷中拦截）**
业务人员小李在后台监控大屏上，突然看到一个用户正在极速申请多笔贷款。
小李输入："实时查一下用户ID_7788，当前在途申请单量和近5分钟登录IP是否异常？"
系统在800毫秒内回复："该用户当前有3笔在途申请，近5分钟登录IP跨越3个省份，
触发实时反欺诈规则，建议暂缓授信。"

**场景二（离线分析 - 日报摘要）**
第二天早上9:00，风控总监张总输入："生成昨日全渠道的授信通过率日报，
并对比上周同期，附带简要分析。"
系统在15秒后回复："昨日总申请量12,345笔，通过率34.5%（环比上周同期+1.2%）。
主要增长源于公积金渠道……（附详细图表链接）"

**为什么必须两仓并存？**
实时仓库支撑场景一：它必须持有毫秒级的流式特征（滑动窗口内的行为），
确保拦截在用户点击"提款"前生效。离线仓库支撑场景二：它必须保证T+1精确性，
涉及全量聚合、同环比计算和监管报送要求的强一致性（一分钱都不能差）。
如果只用实时仓库算昨日全量，成本极高且易丢数；如果只用离线仓库，
反欺诈滞后一小时，资金可能已被洗走。这就是业务倒逼技术，必须采用离在线双轨制。

---

* 第二部分：系统边界——拆解五大模块分工
在明确两仓数据底座后，Agent的上层模块各司其职，边界极其清晰：

| 模块 | 系统边界（只做什么，不做什么） | 金融信贷场景下的职责 |
| RAG（检索增强） | 只处理非结构化知识，不碰任何数值型敏感数据 | 挂载《授信审批操作手册》《反洗钱法规》。当问"利率上限是多少"时，从向量库召回条款原文，不做任何推理计算 |
| NL2SQL（离线查询） | 只读访问离线数仓，禁止DML操作 | 将"昨日通过率"翻译成标准SQL，查询离线宽表。返回确定性的批量统计结果 |
| 实时工具（Function Call） | 只对接实时特征API网关，禁止全表扫描，必须带Key | 封装"查近5分钟IP""查当前在贷余额"等原子工具，每个工具必须在200ms内超时熔断 |
| 数据血缘（Lineage） | 只记录"数据从哪里来"，不记录模型思考过程 | 确保任何返回的指标都能追溯到来源，监管审计必查项 |
| 审计日志（Audit Log） | 只记录"谁、何时、问了什么、查了什么" | 记录每条SQL和工具调用，符合银保监会数据安全要求 |

---

* 第三部分：成功路径——正常查询的系统响应
**用户问题**："汇总一下近3天优质客户的授信额度使用率，
并结合最新的风险偏好文件，**总结**是否需收紧额度。"

**系统响应（按时间线）**：
1. **0~500ms（意图识别与路由）**：Agent判断需要离线统计（额度使用率）+ RAG检索
（风险偏好文件）。
2. **500ms~3s（并行执行）**：左侧线程NL2SQL生成SQL提交给离线数仓；
右侧线程RAG在向量库中召回《2026年风险偏好声明书》相关章节。
3. **3s~5s（结果融合与生成）**：离线数仓返回45.6%，RAG返回"集中度红线为60%"。
Agent生成结论："当前优质客户使用率45.6%，距红线尚有空间，无需收紧。附政策原文链接。"

这个链路中，实时工具未参与，血缘自动记录了查询的表名，审计存储了本次问答全文。
每个模块只在自己边界内工作，Agent充当"大脑"仅做编排和**总结**，不瞎算数。

---

*第四部分：风险路径——延迟、越权、口径冲突处理

**风险1：实时工具延迟超标**
场景：小李查实时IP归属地，依赖的第三方IP库接口因网络抖动超过800ms未返回。
工程取舍：不等待，立即降级。Agent返回：
"实时IP库查询超时（已尝试2次），本次决策采用本地缓存快照（5分钟前的归属地），
请人工复核。"背后的权衡：在金融信贷中，可用性大于极致的准确性，
不能让整个页面卡死，必须给业务人员一个"有损但有据可查"的结果。

**风险2：越权访问**
场景：普通客户经理输入"查一下总裁办王总的授信额度明细。"
Agent在执行实时工具前先鉴权，发现无权访问executive_level数据，直接拦截：
"权限不足，您无权查看该级别数据。本次行为已记录至审计日志。"
背后的设计：Agent虽然是AI，但执行层必须继承RBAC，LLM本身没有权限，
它只是根据权限返回结果。

**风险3：离线与实时口径冲突**
场景：用户问"当前在贷余额是多少？"
实时数仓显示1.23亿（含今日未结算的实时放款），离线仓库显示1.20亿（昨日日终）。
Agent必须显式区分，绝不模棱两可，回复为：
"截至当前实时滚动在贷余额为1.23亿（预估，含今日未正式记账放款）。
截至昨日日终法定口径余额为1.20亿。请您根据使用场景选用——运营监控看实时，
财务对账看离线。"这直接定义了Agent的数据解释边界——不强行弥合差异，
而是把差异产生的时间窗口原因解释清楚。

---

* 第五部分：工程价值——为什么不是普通聊天机器人？

1. **确定性隔离（消除幻觉）**：普通聊天机器人可能乱编数字。我们强制所有数值
必须来自工具调用（SQL或API），LLM只负责编排和生成自然语言结论。
不信任LLM算数，只信任它说话。

2. **数据边界与审计闭环**：它知道自己"该查什么"和"不该查什么"（越权拦截）。
同时，血缘+审计构成了监管的"黑匣子"——遇到客户投诉"为什么额度被降"，
可以一键拉出当时的Trace链路：用户提问→查了哪张离线表→实时接口返回了什么
特征→Agent如何拼接答案。这在银保监会检查中是**一票否决项**。

3. **成本与性能的精细治理**：通过模块边界，把高成本的RAG向量检索留给低频政策
问答，把高消耗的NL2SQL留给T+1离线低峰期，把实时计算资源留给核心的毫秒级拦截。
没有让实时工具去扫全表，也没有让RAG去读财务数字。这种分层治理使得单次对话
的云资源成本控制在0.02元以内，具备大规模商业化落地的可行性。

**收尾总结**：这个项目本质上是为金融信贷风控场景定制的一层"可信执行层"。
它左手拉着离线数仓做"账房先生"，右手拉着实时数仓做"哨兵"，
由Agent当"翻译官"，通过清晰的边界划分和工程化的降级策略，
把大模型的不确定性锁在了业务允许的笼子里。

## 31. 什么是特征

> 来源: 学习复习计划

**特征 = 从原始数据中提取的、能刻画事物某个侧面的数值或类别标签。**  
例如：
- 信贷：年龄、近30天申请次数、征信分、收入负债比。
- 推荐：用户过去7天点击最多的品类、商品价格、当前时段。
- 图像识别：边缘直方图、纹理特征、或者深度学习中间层的 embedding 向量。
- 自然语言处理：词袋向量、TF-IDF 权重、BERT 输出的语义向量。

特征可以是**人工定义**的（传统 ML），也可以是**自动学习**出来的（深度学习），但在生产系统中，它们都必须被可靠地计算、存储和管理。

### 1. 信贷风控

### 3. 广告点击率预估

### 4. 图像识别（工业缺陷检测）
- **传统 ML 特征**：边缘长度、面积、圆形度、灰度共生矩阵纹理特征。

- **实现**：特征平台可统一管理“图像 embedding”这种特征，离线批量用 GPU 提取并存入特征存储，在线推理时直接调用。

### 5. 自然语言处理（智能客服）
- **特征**：对话历史嵌入、用户意图分类结果、FAQ 匹配相似度分数、对话轮次、客户情绪分数。
- **实现**：使用 BERT 等模型将文本转换为固定维度的语义特征向量，离线为所有 FAQ 生成 embedding 存入向量数据库，在线时实时编码用户问题并检索相似 FAQ。

## 33. 请谈谈特征工程的实现：从设计到服务怎么设计？

> 来源: 学习复习计划

特征不仅仅是算出来就完了，在生产系统中需要一套完整的工程链路：

### 1. 特征设计
由数据科学家和业务专家一起，根据业务理解和数据分析，设计出具有区分度和稳定性的特征。例如：
- 基础画像：年龄、性别、学历。
- 行为统计：近 30 天登录次数、夜间操作占比。
- 交叉特征：用户品类偏好 × 折扣敏感度。
- 序列特征：最近 50 次点击的商品 ID 序列。

### 2. 离线特征计算（批量）
- **技术**：使用 Spark SQL 或 PySpark，从数仓（Iceberg/Hive）中读取 ODS/DWD 明细数据，按天或小时进行聚合计算。
- **例子**：计算“用户过去 90 天的累计借款金额”，Spark 作业每天凌晨扫描交易表，按 `user_id` 分组聚合，结果写入 Iceberg 的特征表。
- **特征回填**：定义新特征后，通过指定起始时间，Spark 可以重新计算过去任意一段时间的历史特征值，并覆盖写入特征表。Iceberg 的时间旅行功能保证了随时可回溯历史快照。

### 3. 实时特征计算（流式）
- **技术**：Flink 或 Kafka Streams 消费 Kafka 中的埋点流、CDC 变更流，在滑动窗口内计算聚合指标。
- **例子**：实时统计“过去5分钟内同一IP的申请次数”，Flink 使用 `TUMBLE` 窗口，每次新事件到来都会更新 Redis 中的计数器，用于反欺诈模型。

### 4. 特征存储与服务
采用**双存储架构**：
- **离线存储**：Iceberg/Hive 上的 Parquet 表，用于大规模训练样本的批量拉取。
- **在线存储**：Redis/Cassandra，用于毫秒级低延迟查询。实时特征由 Flink 直接写入，离线特征由 Feast materialize 命令批量推送。
- **特征服务接口**：通过 Feast 的 `get_online_features` API 或自定义的 Feature Service（gRPC），统一提供在线特征查询。

### 5. 特征注册与管理（特征平台）
使用 **Feast** 作为元数据层，它定义了：
- **Entity**：特征关联的主体（如 `user_id`、`item_id`）。
- **FeatureView**：一组特征的计算逻辑和数据源（离线表、实时流）。
- **OnDemandFeatureView**：在线动态计算的特征（如请求参数转换）。
- **Point-in-Time Join**：自动保证训练样本的时间正确性。
