## 1. Tool Use 在生产 AI 应用里怎么设计？

> ID: `Q016`

Tool Use 的核心是让模型负责判断什么时候调用工具，让工具负责确定性执行。
生产里工具要有清晰入参、出参、权限、超时、错误处理和审计日志。
不能让模型直接操作数据库或系统，而是通过受控工具完成动作。
Tool Use 适合把 LLM 的理解能力和传统系统的确定性能力结合起来。
比如用户问 SQL 风险，模型可以调用 SQL parser 工具；问指标口径，
可以调用知识库检索工具；问数据结果，可以调用受控查询工具。
生产设计里，每个工具都要定义 schema，包括名称、用途、参数类型、返回结构和错误码。
工具执行前要做权限校验，执行时要有 timeout 和重试，执行后要记录日志。
LLM 不应该直接拼接任意 SQL 去数据库执行，而是只能调用经过限制的工具。
这样既能发挥模型的调度能力，又能把风险控制在工程边界内。

## 2. Agent 工作流为什么不能让模型完全自由调用工具？

> ID: `Q093`

生产环境里的 Agent 不能让模型完全自由调用工具，因为工具通常连接数据库、知识库、业务系统或外部接口。
如果模型没有受控流程，可能会选错工具、重复调用、绕过权限、执行高成本查询，
甚至把失败结果编造成确定答案。

更稳的做法是把 Agent 设计成受控工作流：先识别意图，再检索 schema 或知识，
生成候选操作，经过权限、安全、成本和格式校验后再执行。
每一步都要有结构化输入输出、最大调用次数、失败回退和审计记录。

在金融信贷 NL2SQL 场景里，Agent 不能直接执行模型生成的 SQL。
必须先检查敏感字段、时间范围、只读限制、表权限和扫描成本。
如果校验失败，就应该拒答或转人工，而不是继续生成看似合理的答案。
这种设计能在智能化和可控性之间取得平衡。

## 3. 多工具 Agent 为什么必须有工具选择和失败回退策略？

> ID: `Q094`

多工具 Agent 必须有工具选择和失败回退策略，因为工具通常连接真实数据、知识库、业务系统或外部服务。
如果让模型自由调用工具，可能会选错工具、重复调用、绕过校验、增加成本，
甚至在工具失败后编造答案。

生产里应该先定义工具注册表，写清每个工具的用途、输入输出、权限要求、风险等级和前置条件。
编排层再根据用户意图、权限、风险标记和上一步结果选择工具路线。
例如金融信贷 NL2SQL 指标查询必须先查 Schema，再生成 SQL、校验 SQL，最后才能执行查询。
如果是政策规则解释，就应该走 RAG 检索和引用回答，而不是强行查数据库。

失败回退同样重要。
缺少时间范围时应该要求补充条件，敏感字段查询应该安全阻断，
工具超时应该返回系统错误或降级，检索无依据时应该拒答。
同时要设置最大步数和重复调用检测，防止 Agent 陷入循环。
这样多工具 Agent 才能做到可控、可审计、可评测，而不是一个不可解释的黑盒。

## 4. 生产级 AI 应用为什么需要端到端评测，而不是只测单个 Prompt 或工具？

> ID: `Q096`

生产级 AI 应用需要端到端评测，因为真实用户关心的是完整链路结果，
而不是某个 Prompt、检索器、SQL 生成器或实时工具单独表现。
一个金融信贷仓库 Agent 可能每个单点工具都能跑通，但整体仍然出错。
例如实时问题误走离线 SQL、缺少时间范围却继续查库、SQL 生成后跳过校验、
指标口径没有引用却回答、实时链路延迟却仍给出确定结论，都会导致生产风险。

端到端评测应该从用户问题开始，检查意图识别、上下文检索、工具路由、离线 SQL 生成、
SQL 校验、离线查询执行、实时指标查询、告警解释、口径引用、血缘追溯、安全阻断和审计记录。
每条样例都要有明确成功判定，包括预期最终状态、必须调用的工具、禁止调用的工具、
是否需要引用、是否允许执行 SQL、是否需要检查实时延迟和失败分类。
对敏感导出、越权访问、资料不足、实时延迟和系统异常来说，正确拒答、正确澄清或正确降级也应该算成功。

在金融信贷场景里，端到端评测尤其重要，因为系统可能接触授信、风控、放款、还款、
逾期、催收、实时反欺诈和客户敏感信息。
固定评测集可以持续回归，帮助团队判断每次 Prompt、规则、Schema Catalog、指标字典、
实时指标字典、血缘工具或工具路由改动是否真的提升质量，是否引入新的安全、合规或实时风险。

## 5. 金融信贷仓库 Agent 的端到端评测集应该怎么设计？

> ID: `Q100`

金融信贷仓库 Agent 的端到端评测集不能只放“问指标并回答”的成功样例，
而要覆盖离线指标、实时指标、告警解释、指标口径、血缘追溯、权限阻断、缺参澄清和系统异常。
每条样例都应该从真实用户问题出发，写清业务类别、预期状态、必须调用的工具、禁止调用的工具、
是否需要引用、是否允许执行 SQL、是否要检查实时延迟，以及失败后应该归到哪类责任层。

离线指标样例要检查 Schema Catalog、分区字段、指标口径、SQL 生成、SQL 校验、执行和结果解释。
例如查询“昨日各渠道授信通过率”，必须走离线指标链路，不能查明细敏感字段，
也不能跳过 SQL Validator。
实时指标样例要检查窗口、事件时间、处理时间、延迟状态和实时工具路由，
例如“近 5 分钟实时拒绝率是否异常”不能用离线日报表回答。

评测集还必须包含应该失败的问题。
例如导出客户手机号、查询无权限明细、缺少时间范围、口径资料没有引用、分区缺失、实时链路延迟和数据库超时。
这些样例的正确结果不是硬答，而是安全阻断、要求补充条件、资料不足或系统异常兜底。
这样评测集才能真正保护生产边界，而不是只证明 Demo 能跑。

## 6. 实时仓库问题进入 Agent 评测时要重点检查哪些风险？

> ID: `Q101`

实时仓库问题进入 Agent 评测时，首先要检查工具路由风险。
只要用户问题出现近 5 分钟、近 10 分钟、实时拒绝率、实时申请量、告警、事件流、链路延迟这类信号，
Agent 就应该优先走实时指标工具或告警工具，而不是生成离线 SQL 查日报表。
如果实时问题误走离线 SQL，即使最终答案看起来有数字，也应该判为失败。

其次要检查窗口和延迟风险。
实时指标必须说明统计窗口、事件时间、处理时间、延迟阈值和当前链路状态。
如果实时链路延迟 20 分钟，系统不能给出“当前正常”这类确定结论，
而应该提示数据延迟、返回 execution_failed 或降级到可解释状态。

还要检查告警解释边界和权限边界。
告警解释只能基于告警等级、触发规则、窗口指标和证据来源，不能编造“原因一定是某个策略上线”。
实时事件流里的客户明细、设备号、手机号、身份证号等敏感字段仍然要走权限校验或安全阻断。
所以实时仓库评测既要测回答准确性，也要测延迟、窗口、证据、权限和审计。

## 7. AI Agent 出现 bad case 后，你会怎么做错误治理？

> ID: `Q097`

AI Agent 出现 bad case 后，我不会直接只改 Prompt，而是先做失败归因。
第一步是保留原始问题、工具路线、中间结果、最终状态、引用、SQL、校验结果和 request_id。
然后判断错误发生在哪一层：意图识别、工具路由、RAG 召回、SQL 生成、SQL 校验、
查询执行、结果解释还是异常处理。

如果是缺少时间范围却继续查询，就修工具路由和澄清规则；
如果是 RAG 没有引用仍回答，就修 citation 校验和无依据拒答；
如果是实时问题误走离线 SQL，就修意图识别和工具路由；
如果是实时链路延迟被包装成正常答案，就修实时状态校验和异常分类。
Prompt 只适合修任务表达、输出格式和解释约束，不能替代权限、安全和执行前置条件。

修复后必须把 bad case 加入固定评测集，并跑全量回归。
通过率、失败类型、拒答率和安全阻断结果都要对比修复前后。
在金融信贷场景里，这一步尤其重要，因为授信、风控、逾期和客户明细都涉及权限、合规和审计。
错误治理的目标不是让某一条样例看起来答对，而是让系统在同类问题上稳定、可控、可追溯。

## 8. 为什么 AI Agent bad case 必须按责任层归因？

> ID: `Q102`

AI Agent bad case 必须按责任层归因，因为不同层的问题修法完全不同。
如果实时问题误走离线 SQL，核心问题在意图识别和工具路由；
如果缺少时间范围还继续查询，问题在澄清策略和执行前置条件；
如果口径解释没有引用仍回答，问题在 RAG 召回、citation 校验和无依据拒答；
如果实时链路延迟还输出正常结论，问题在实时状态校验和异常分类。

如果不做责任层归因，团队很容易把所有问题都说成“模型不好”或“改 Prompt”。
Prompt 可以修表达、任务约束和输出格式，但不能替代权限控制、SQL 校验、工具前置条件、
实时延迟检测和审计记录。
这些生产边界必须由确定性工程规则兜住。

按责任层归因还能让修复可回归。
每个 bad case 都要记录原始问题、工具路线、中间结果、最终状态、失败类型、责任层、修复动作和回归样例。
修复后再跑固定评测集，确认该类问题稳定解决，同时没有让正常查询、安全阻断或实时告警解释退化。

## 9. 实时仓库和离线仓库在 Agent 数据问答里有什么区别？

> ID: `Q106`

离线仓库面向稳定的历史指标和经营分析，通常按天或批次加工数据。
Agent 查询昨日、上周、上月的授信通过率、放款金额、逾期率时，应该走离线 ADS 或 DWS。
实时仓库面向当前状态、分钟级窗口、实时特征和告警，适合回答近 5 分钟申请量、当前拒绝率是否异常、
实时还款失败数是否突增这类问题。

两者最大的区别不是表名不同，而是时间语义不同。
离线指标强调分区、批次、指标口径和稳定性；实时指标强调事件时间、处理时间、窗口、延迟和链路状态。
如果 Agent 把实时问题路由到离线 SQL，即使能查出数字，也可能业务上错误。

生产级 Agent 应该先识别问题是离线指标还是实时状态。
离线问题走 Schema Catalog、SQL 生成、SQL 校验和离线执行；
实时问题走实时指标工具或告警工具，并检查窗口和延迟状态。
敏感明细和实时事件流导出都必须经过权限控制和审计。

## 10. 金融信贷 Agent 为什么要先做主题域识别，再做意图识别？

> ID: `Q108`

金融信贷 Agent 需要先做主题域识别，是因为信贷业务链路长，授信、额度、风控、放款、还款、逾期、催收对应的指标、
表、实时事件和权限边界都不同。
同样是“失败率”，在放款域可能指放款失败率，在还款域可能指扣款失败率，在风控域可能指策略拒绝率。
如果不先识别主题域，Agent 很容易选错指标或表。

识别主题域之后，还必须识别意图。
同一个主题域的问题可能是离线指标查询、实时状态查询、口径解释、血缘追溯、敏感导出或缺参澄清。
例如授信申请域里，“近 7 天各渠道通过率”走离线指标；
“近 5 分钟申请量是否异常”走实时指标；
“通过率怎么算”走 RAG 口径问答；
“通过率来自哪些表”走血缘工具。

所以生产级 Agent 的路由应该是：先识别主题域，再识别意图，再选择工具路线，并做权限、前置条件和审计。
这样才能避免把所有问题都粗暴交给 SQL 或模型自由选择工具。

## 11. 金融信贷 Agent 的 Schema Catalog 应该包含哪些信息？

> ID: `Q113`

生产级 Schema Catalog 不能只记录表名和字段名。
它应该同时记录表描述、业务主题域、仓库层级、表粒度、主键、分区字段、时间字段、字段类型、字段含义、
可用维度、指标字段、敏感字段、权限等级、数据更新频率、owner、查询约束和下游使用场景。

在金融信贷 Agent 里，Catalog 的作用是约束 NL2SQL 和工具路由。
例如 `ads_credit_daily_metrics` 适合日报总览，`dws_credit_apply_channel_1d` 适合渠道趋势，
`dwd_credit_apply_detail_di` 含客户手机号和身份证号，只能在授权排查时访问。
Agent 生成 SQL 前要根据 Catalog 判断应该选 ADS、DWS 还是 DWD，必须带时间范围和分区条件，
不得编造字段，不得访问无权限的敏感字段。

一句面试总结是：Schema Catalog 保证 Agent “查得对、查得安全、查得可控成本”。

## 12. 为什么工具调用前必须检查前置条件和风险等级？

> ID: `Q115`

工具调用前检查前置条件和风险等级，是为了把“模型建议”变成“可执行动作”之前先过一层工程闸门。
金融信贷数据问答涉及经营指标、实时风控、客户明细和合规审计，如果工具调用没有前置检查，
Agent 可能在缺少时间范围时扫描大表，在缺少实时窗口时给出无意义结果，在链路延迟时误判正常，
或者把客户手机号、身份证号导出给无权限用户。

前置条件通常包括：意图是否匹配、必填参数是否齐全、时间范围是否明确、分区条件是否存在、
是否命中敏感字段、用户权限是否足够、实时延迟是否可接受、是否有可靠引用来源。
风险等级用于决定是否自动执行、是否需要二次确认、是否必须审计、失败后是澄清、降级还是阻断。

一句面试总结是：Agent 可以负责理解和编排，但工具执行必须由权限、Catalog、注册表和校验器共同控制。

## 13. 数据血缘在金融信贷 Agent 里解决什么问题？

> ID: `Q116`

数据血缘在金融信贷 Agent 里主要解决来源追溯、影响分析、异常排查和审计解释。
它回答的不是“指标怎么算”，也不是“指标当前是多少”，而是“这个数据从哪里来、经过哪些任务、会影响哪些下游”。

例如授信通过率的血缘可以从 ODS 授信申请订单表追到 DWD 明细表、DWS 渠道汇总表、ADS 日报指标表，
再追到授信通过率指标和信贷经营日报。
如果 DWS 汇总任务异常，Agent 可以基于血缘判断哪些 ADS 表、指标和报表可能受影响。
实时场景里，风控拒绝率告警可以追溯到风控决策事件、实时窗口计算任务、实时指标和告警规则。

生产级 Agent 回答血缘问题时，应该返回节点、路径、加工任务、下游影响和证据，而不是只用自然语言猜测。
同时血缘工具也要遵守权限和敏感字段控制，不能因为查询血缘就暴露客户手机号、身份证号或明细名单。

## 14. Agent 做血缘追溯时，为什么必须返回证据和影响范围？

> ID: `Q118`

血缘追溯通常服务生产排查和审计，不能只回答“可能是某张表的问题”。
如果没有证据，排查人员无法确认 Agent 的结论来自真实血缘图还是模型猜测。
如果没有影响范围，数据团队也不知道要通知哪些业务方、暂停哪些报表、回补哪些任务。

生产级血缘回答至少要包含上游节点、下游节点、依赖关系、加工任务、指标或报表 ID、告警规则和必要的版本信息。
例如 `dws_credit_apply_channel_1d` 异常时，要说明它会影响 `ads_credit_daily_metrics`、
`metric_credit_approval_rate` 和 `report_credit_operation_daily`，并给出对应调度任务或依赖边。

在金融信贷场景里，错误报表可能影响授信经营分析、风控策略复盘、贷后逾期判断和管理层日报。
因此 Agent 的血缘回答必须可追溯、可复核、可审计。

## 15. 实时链路延迟超阈值时，Agent 应该怎么回答？

> ID: `Q124`

实时链路延迟超阈值时，Agent 绝对不能忽略延迟给出确定结论。
正确回答必须包含四部分：

1. 数据延迟说明：当前链路的实际延迟是多少（如 720 秒）、
阈值是多少（如 120 秒），解释为什么数据不可信。

2. 数据可用性状态：标记为 degraded 或 delayed，
不是 ok 也不是 empty，让调用方明确感知到数据质量问题。

3. 建议动作：查看最近可信窗口（15 分钟前）、
检查实时任务和消息积压状态、查看上游事件源是否正常。

4. 降级方案：如果用户坚持要看数据，提供最近可信窗口的
指标作为参考，但加注"此数据为快照，非当前实时状态"。

生产里不能为了给用户一个结论而忽略延迟状态——
把延迟数据包装成确定答案，比直接说"不知道"危险得多。
延迟数据可能误导风控判断、告警处置和运营决策。

## Q125：如何设计一个金融信贷日报 + 实时告警 Agent？
来源 Day：Day 55
重要程度：5/5
### 回答
我会把金融信贷日报 + 实时告警 Agent 设计成多工具编排系统，而不是一个单纯的 SQL 查询。
入口先识别用户是要“经营日报 + 当前风险摘要”，然后分别调用离线日报工具、实时告警工具、指标口径工具和审计工具。

离线日报工具查询 ADS 日报指标，例如授信申请量、授信通过率、放款金额、M1 逾期率，并强制日期分区。
实时告警工具查询近 1 小时实时告警，例如风控拒绝率突增、还款失败链路延迟，并返回窗口、阈值、延迟和证据。
指标口径工具负责解释关键指标，避免 Agent 把通过率、放款率、逾期率讲错。
最后由结果解释层生成摘要、风险提示和建议排查项，同时写审计日志。

这个 Agent 的关键边界是：可以解释查到的指标和告警证据，但不能编造根因；
可以返回聚合指标和告警证据，但不能返回客户手机号、身份证号或客户名单。

## 16. 多工具 Agent 为什么必须写审计日志？

> ID: `Q127`

多工具 Agent 必须写审计日志，因为它的回答不是一次模型生成，而是由多个工具调用共同组成。
如果日报摘要出错，团队需要知道它查了哪个 ADS 分区、用了哪个实时告警窗口、引用了哪个指标口径、有没有触发安全阻断。
没有审计日志，就无法复盘、回归和追责。

审计日志至少应记录 request_id、用户问题、工具名称、输入参数、输出摘要、执行状态、失败原因、证据 ID、时间戳和最终回答。
在金融信贷场景里，审计尤其重要，因为指标口径、风控告警、客户隐私和经营报表都涉及合规和可追溯要求。

一句面试总结是：Agent 的可信不是只靠回答文本，而是靠可追溯的工具链路和审计记录。

## 17. 金融信贷仓库 Agent 的端到端评测应该覆盖哪些场景？

> ID: `Q128`

金融信贷仓库 Agent 的端到端评测不能只覆盖成功查询。
它应该同时覆盖离线指标查询、实时状态查询、口径解释、血缘追溯、日报告警摘要、权限阻断、空分区、实时延迟、工具异常、告警误报和口径冲突。

每条样例要明确用户问题、预期意图、预期工具路线、预期状态、必须包含的证据和不允许出现的行为。
例如近 7 天授信通过率应该走离线 NL2SQL；实时风控拒绝率应该走实时工具；授信通过率怎么算应该走指标字典 RAG；
要求导出手机号和身份证号应该走安全阻断。

生产级评测还要检查审计记录、引用来源、SQL 安全、延迟状态和拒答类型。
只有最终回答、工具路线、证据和安全状态都符合预期，才算真正通过。

## 18. 为什么 Agent 端到端评测不能只看最终回答文字？

> ID: `Q129`

Agent 端到端评测不能只看最终回答文字，因为最终回答可能看起来合理，但内部路线是错的。
例如实时问题误走离线日报，口径问题误走 SQL，敏感导出问题中间已经查了 DWD 明细，最终文本即使没有泄露数据，也说明安全边界已经被突破。

金融信贷数据 Agent 的正确性包括多层：意图是否识别正确、工具是否路由正确、前置条件是否满足、SQL 是否安全、实时延迟是否检查、证据是否返回、审计是否记录。
这些都不是只看最终自然语言能判断的。

所以端到端评测要检查结构化 trace，包括 route、tool_calls、status、evidence、audit、refusal_reason 和 final_answer。
一句面试总结是：生产 Agent 评测的是链路正确，不只是回答像样。

## 19. 为什么金融信贷仓库 Agent 必须感知数据质量状态？

> ID: `Q131`

金融信贷仓库 Agent 的回答直接依赖四个数据来源：离线分区、实时窗口、补偿任务和上游事件质量。
如果 Agent 不知道当前数据是否完整、是否重复、是否延迟，就可能把暂时错误的数据包装成确定结论。

具体来说：离线分区可能为空或未产出，Agent 查不到数据却可能误报为"指标为零"；
实时窗口可能因 Kafka 积压导致数据不完整，Agent 给出的指标偏低；
补偿任务未完成时，逾期率等关键指标在短时间内是偏的；
上游事件重复写入或幂等失败，会导致实时指标虚高。

**数据质量规则至少应该包含的字段：**
每条质量规则需要定义规则 ID、规则名称、表名或事件名、检查字段、检查类型
（空值率、重复率、延迟分钟数、分区状态）、触发阈值（如 NULL 比例 > 5%）、
风险等级（warning / blocked）、检查频率和通知方式。

**质量状态要结构化输出，而不是只写一段自然语言说明。**
原因是编排层需要根据状态自动决策：`quality_ok` 就走正常回答，
`quality_warning` 就走降级说明，`quality_blocked` 就走安全阻断。
如果状态是自然语言，程序无法稳定判断该走哪条路由。

**什么情况下降级，什么情况下阻断？**
- 降级：数据有轻微延迟但可参考、补偿任务正在执行中、小比例空值不影响趋势判断。
  降级时要说明"当前数据可能不完整，建议稍后核实"。
- 阻断：离线分区完全未产出、实时延迟超过阈值 2 倍以上、关键指标空值率 > 30%、
  上游事件明确的重复写入。阻断时要给出排查建议而非强行回答。

**实时质量问题和离线质量问题的处理区别：**
离线质量问题通常可以等补偿或重跑，Agent 返回"数据正在恢复，请稍后查询"。
实时质量问题不能等，必须立即降级或阻断，因为用户问的是当前状态，
延迟的数据比没有数据更危险——它会把错误结论包装成确定答案。

在金融信贷场景里，这些错误会直接影响经营判断（如放款决策）、风险研判
（如逾期率趋势）和告警处置（如误报或漏报）。数据质量问题还会影响 Agent
的可信度——如果用户多次得到错误回答，会丧失对 Agent 的信任，
即使后续数据恢复了用户也不信了。

## 20. 为什么金融信贷 Agent 服务化必须先定 request_id 和审计结构？

> ID: `Q133`

Agent 不能只停留在本地脚本，因为本地脚本无法接入前端、调试平台或演示接口，
没有 request_id 和 trace_id，bad case 无法定位和回放。
服务化的核心是先定审计字段骨架，再逐步扩展接口能力。

**request_id 和 trace_id 分别解决什么问题？**
request_id 是一次业务请求的唯一标识，对外暴露给调用方，用于请求定位和状态查询。
trace_id 是分布式链路追踪 ID，在 Agent、工具、数据源之间传播，用于串联所有节点。
一个 request_id 下可以有多个 span，trace_id 负责把它们串成完整调用链。
request_id 定位"哪个请求出了问题"，trace_id 定位"问题出在哪个节点"。

**审计需要分类记录，通常分为四类：**
- 查询审计：记录用户问了什么、查了哪张表、SQL 是什么、返回了多少行
- 告警审计：记录告警是否正常触发、延迟状态、告警规则版本和证据快照
- 任务审计：记录调度任务执行状态、产出分区、失败原因和补偿执行情况
- 权限审计：记录用户访问了哪些表字段、是否触发安全阻断、阻断原因是什么

**审计字段必须结构化，不能只存自然语言日志。**
原因有三：第一，结构化字段可以被程序稳定解析、校验和路由，status 和 error_code
必须用固定枚举；第二，可以触发告警，比如 safely_blocked 比例超过阈值自动报警；
第三，回归测试可以按结构化字段精准匹配预期结果。

**服务化接口至少应该固定这些状态和错误码：**
- `success`：正常回答
- `safely_blocked`：安全阻断（权限不足、敏感字段、危险 SQL）
- `clarification_required`：缺少必要参数（时间范围、窗口、维度）
- `execution_failed`：工具执行失败（超时、数据库异常、模型不可用）
- `system_error`：系统异常（配置错误、依赖缺失）

如果每次请求没有 request_id、trace_id 和统一状态，后续即使发现错误答案，
也很难知道问题出在模型、工具还是数据链路。先定审计字段骨架，
包括 request_id、trace_id、user_id、scenario、intent、tool_route、model_id、status、
latency_ms 和 error_code，再逐步扩展接口能力。

## 21. 为什么金融信贷 Agent 项目演示必须同时展示成功路径和风险路径？

> ID: `Q137`

因为成功路径只能说明系统能工作，风险路径才能说明系统在错误、越权、延迟和口径冲突场景下仍然可控。
金融信贷 Agent 的价值不只是回答问题，还包括知道**什么时候不该回答、什么时候要降级、什么时候要阻断**。

只展示成功样例，项目看起来像精心挑选的 Demo；主动展示 bad case 并说明系统的处理逻辑，
面试官才能看到你理解的是生产系统约束，而不只是模型调用。

---

**风险场景一：权限阻断**

普通客户经理输入："查一下总裁办王总的授信额度明细，顺便说说他最近的交易记录。"

系统处理流程：
1. 意图识别发现请求包含"额度明细"+"交易记录"，属于 DWD 明细查询
2. 权限校验发现该用户 role=customer_manager，无权访问 executive_level 标签的数据
3. 安全阻断触发，Agent 返回：
   "权限不足，您无权查看该级别数据。如需高级别数据查询，请在 OA 系统中提交审批申请，
   审批通过后由数据管理员在受控环境下执行。本次行为已记录至审计日志。"

生产价值：说明 Agent 懂得执行层必须继承 RBAC，LLM 本身没有权限判断能力，
权限必须由确定性代码在工具调用前拦截。不是所有请求都应该被执行。

---

**风险场景二：实时链路延迟降级**

风控运营人员输入："近 5 分钟实时拒绝率是不是突增了？"

系统处理流程：
1. Agent 判断为实时状态查询，路由到实时指标工具
2. 实时工具检查当前链路延迟：720 秒（12 分钟），超过 180 秒阈值
3. 系统进入降级模式，Agent 返回：
   "当前实时链路延迟 12 分钟（阈值 3 分钟），近 5 分钟窗口数据可能不完整，
   无法确定拒绝率是否真正突增。建议：①查看最近可信窗口（15 分钟前）的指标；
   ②检查实时任务和消息积压状态；③如需立即处置，建议人工查看监控大盘原始事件。"

生产价值：说明 Agent 不是给一个看起来合理的数字就完事，
而是在数据不可信时**宁可说不确定，也不给错误结论**。
金融信贷里，延迟数据被包装成确定答案，可能误导风控策略。

---

**风险场景三：口径冲突，Agent 主动解释差异**

管理层输入："当前在贷余额是多少？"

系统处理流程：
1. Agent 判断可能匹配两个数据源：离线 ADS 日报表（昨日日终）+ 实时窗口指标（当前滚动）
2. 系统决定同时查询两路，发现数值不一致：实时 1.23 亿 vs 离线 1.20 亿
3. Agent 不强行给出一个数字，而是**显式区分**：
   "截至当前实时滚动在贷余额约为 1.23 亿元（含今日未正式记账放款）。
   截至昨日日终（T+1）法定口径余额为 1.20 亿元。
   请根据您的使用场景选用：运营监控看实时，财务对账看离线。"

生产价值：Agent 不强行弥合差异，而是把差异产生的时间窗口原因解释清楚。
这在金融信贷里是关键能力——口径混淆比回答错误更难被发现，
但长期会破坏数据信任。

---

**风险场景四：资料不足，应该有依据地拒答**

用户问："这个客户的欺诈概率是多少？"

系统处理流程：
1. Agent 判断意图为风控评分查询
2. 查找指标字典和工具注册表，发现该指标依赖的反欺诈模型仅在支付链路实时调用，
   不对外暴露给 Agent 查询
3. Agent 返回：
   "当前系统未接入客户级别的欺诈评分查询能力。欺诈评分仅用于支付链路的实时决策，
   不在 Agent 可查询范围内。如需客户风控分析，建议查询：①该客户近 90 天申请通过率；
   ②该客户命中黑名单的次数。是否需要我查询这些替代指标？"

生产价值：Agent 不是"不知道就编一个"，而是清楚地说明"为什么不能查，
什么可以替代"。这在金融信贷里是合规基本要求——不能给用户一个看起来合理
但无法追溯依据的风控结论。

---

**演示时的讲法建议**

展示 bad case 时要按三步讲：
1. 这个场景是什么风险类型（权限阻断/数据延迟/口径冲突/资料不足）
2. 系统如何识别和处理（权限校验→阻断？延迟检查→降级？口径冲突→区分说明？）
3. 这个设计在生产里保护了什么（合规安全？数据可信度？用户信任？）

这样面试官看到的是一个**可复盘、可迭代、有生产判断**的工程闭环，
而不是一个只会跑通成功样例的 Demo。 这类治理能力比单纯答对一个问题更接近生产价值。

## 22. 为什么金融信贷 Agent 项目不能只强调用了 LangChain / LangGraph / LangSmith？

> ID: `Q141`

因为框架只是实现手段，业务价值来自数据边界、工具路由、安全控制、审计和评测。

如果只强调框架名，面试官会认为你是在堆技术词，而不是解决真实场景问题。
正确的讲法是：先用业务问题开头（金融信贷场景为什么需要同时处理离线指标和实时告警），
再讲系统如何通过意图识别、工具路由、SQL 校验和审计来保证安全和可控，
框架只是"我们在哪些环节用它加速开发"的补充说明。

一句话：框架是手段，不是项目价值本身。面试官更关心你解决了什么问题、做了哪些取舍。
