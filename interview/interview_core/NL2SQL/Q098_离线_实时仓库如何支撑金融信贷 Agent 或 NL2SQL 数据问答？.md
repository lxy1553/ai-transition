# 离线/实时仓库如何支撑金融信贷 Agent 或 NL2SQL 数据问答？

> 来源: 核心题库 | 分类: NL2SQL

离线/实时仓库是金融信贷 Agent 和 NL2SQL 数据问答的数据底座。
如果没有仓库治理，Agent 只能看到一堆业务原始表和事件流，很容易选错表、混淆指标口径、
漏掉分区、忽略实时延迟、访问敏感字段，或者在离线任务失败、实时任务延迟时仍然给出错误结论。

生产里通常会把信贷离线数据按 ODS、DWD、DWS、ADS 分层。
ODS 保留原始业务数据，DWD 清洗成标准明细事实表，DWS 按主题和维度做轻度汇总，
ADS 面向日报、看板和经营分析沉淀应用指标。
Agent 做离线指标查询时，应该优先使用 DWS 或 ADS，而不是直接查 ODS 明细表。

实时仓库则面向实时风控、反欺诈、实时特征、告警和分钟级监控。
它需要管理实时事件、实时指标、窗口口径、事件时间、处理时间、延迟、乱序、重复和幂等。
例如近 5 分钟申请量、黑名单命中数、实时拒绝率突增和还款失败告警，都应该走实时状态查询工具，
不能和离线日报指标混在一起解释。

同时，还要把仓库元数据整理成 Schema Catalog、指标字典、实时指标字典、权限标签和血缘信息。
Schema Catalog 告诉 Agent 有哪些离线表、实时指标、业务粒度、分区字段和权限边界；
指标字典解释指标的分子、分母、过滤条件和适用范围；
实时指标字典解释窗口、延迟阈值和告警规则；
血缘信息说明指标来自哪些上游表、事件流和加工任务；
权限标签控制敏感字段和明细访问。

在工具编排上，查离线指标走 NL2SQL 链路，查实时状态走实时指标工具，
问口径走 RAG 链路，问来源走血缘工具，敏感明细查询则走安全阻断或权限校验。
这样 Agent 才能把“能回答”升级成“回答可信、可审计、可追溯、可控成本、能处理实时风险”。
