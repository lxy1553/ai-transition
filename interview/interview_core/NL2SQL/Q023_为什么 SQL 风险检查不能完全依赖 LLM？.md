# 为什么 SQL 风险检查不能完全依赖 LLM？

> 来源: 核心题库 | 分类: NL2SQL

因为很多 SQL 风险是确定性规则，比如是否 `select *`、是否缺少 `where`、
是否缺少分区字段、是否包含危险语句。这些用程序判断更稳定、可测试。
LLM 更适合做解释和总结，不适合单独承担安全边界。
生产环境里，SQL 安全和资源风险不能完全交给 LLM 判断。
LLM 可能漏判、误判，也可能因为上下文不足编造字段含义。
而很多检查其实可以规则化，比如只允许 `select`，禁止 `drop`、`delete`、`update`，
检查是否有分区条件、是否有 limit、是否访问了黑名单表、是否跨域 join。
这些规则应该由确定性程序或 SQL parser 负责，并且可以写单元测试。
LLM 更适合基于规则结果生成自然语言解释，比如告诉用户为什么缺少分区会导致扫描成本高。
这种设计把安全边界放在规则和工具层，把表达能力交给模型，更符合生产要求。
