# NL2SQL 为什么不能直接把用户问题交给模型生成 SQL？

> 来源: 核心题库 | 分类: NL2SQL

生产环境里不能直接把用户问题交给模型生成 SQL，有五个确定的风险：

**1. 模型可能编造不存在的表和字段。** 模型不知道真实数据库的 schema，
可能生成看似合理但实际不存在的表名和字段名。比如用户问"授信通过率"，
模型可能编一个 `credit_approval_rate_table`，实际应该是 `ads_credit_daily_metrics`。

**2. 混淆指标口径。** 同样是"通过率"，授信通过率和放款通过率的分子分母完全不同，
如果模型凭常识生成 SQL，口径大概率是错的。授信通过率的分母是进入审批流程的申请数，
分子是审批通过数，这两个字段在不同表里。

**3. 漏掉分区条件。** 离线仓库表按 dt 或 biz_date 分区，SQL 没有分区条件就会
全表扫描数年的历史数据，拖垮数仓。模型不会主动加 `WHERE dt = '2026-07-01'`，
需要工程约束强制检查。

**4. 生成高成本 SQL。** 缺少 limit、无条件 order by、大表 join 或笛卡尔积，
都可能在生产数据库上造成性能灾难。

**5. 绕过权限查询敏感数据。** 模型不知道字段的权限等级，可能生成包含手机号、
身份证号的 SQL，绕过业务系统的权限校验。

**正确做法**：先提供 Schema Catalog（真实表字段白名单）、指标口径（分子分母定义）、
权限规则（敏感字段标记）和 SQL 约束（强制时间范围、强制 limit），
让模型只能在可用范围内生成 SQL。生成后还必须经过 SQL Validator 校验后才执行。
