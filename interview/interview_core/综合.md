## 1. JSON 合法和业务校验通过有什么区别？

> ID: `Q015`

JSON 合法只说明格式能被解析，比如括号、引号、逗号没错。
业务校验通过还要求字段完整、类型正确、枚举值合法，并且符合业务规则。
比如 `risk_level` 必须是固定枚举，`can_publish` 必须和风险等级一致。
JSON 合法是语法层面的概念，程序能 `json.loads` 解析就算合法。
但生产系统还需要业务层校验。比如 SQL 解释结果里必须有 `summary`、`tables`、
`risk_level`、`risks` 和 `suggestions`。其中 `risk_level` 不能随便写，
必须是 `low`、`medium`、`high` 这类固定枚举。`risks` 应该是数组，
每个风险最好有类型、说明和建议。更进一步，业务规则也要一致：
如果风险等级是 high，`can_publish` 就不能是 true。否则 JSON 虽然合法，
但下游系统会误判。生产里通常用 Pydantic、JSON Schema 或自定义规则做这层校验。

## 2. 一次 LLM 调用需要关注哪些参数和工程问题？

> ID: `Q018`

要关注模型选择、messages、temperature、max tokens、timeout、重试、错误处理、
成本和日志。像 SQL 解释、RAG、NL2SQL 这类场景更重视准确和稳定，
通常会使用较低 temperature，并限制输出格式。
一次 LLM 调用不只是传一段 prompt。首先要选择合适模型，简单分类和结构化抽取可以用较小模型，
复杂推理或长上下文问答可能需要更强模型。messages 里要区分 system prompt、
user prompt 和检索上下文。temperature 控制随机性，生产里很多任务需要低 temperature，
避免输出漂移。max tokens 控制输出长度和成本。工程上还要设置 timeout、
重试、限流和异常处理，不能让模型调用拖垮接口。日志也很重要，
至少要记录请求 ID、模型、耗时、token、错误码和关键业务结果，
但不能把敏感数据明文打到日志里。

## 3. 演示 citations 时，应该向面试官或同事说明什么？

> ID: `Q037`

演示 citations 时不能只说“这里有引用”，要说明引用解决的是可信度、追溯和排查问题。
应该讲清楚每条 citation 至少包含文档来源、chunk_id、doc_id、标题、位置和相关性分数。
用户可以根据 citations 核对答案依据；研发可以根据 citations 判断错误来自哪里：
如果引用不相关，说明检索或 rerank 有问题；如果引用资料过期，说明知识库版本管理有问题；
如果引用正确但答案错，才重点看 prompt 或模型生成。生产场景里，citations 也是审计、
合规和用户信任的基础能力，尤其适用于指标口径、SQL 规范、财务、人事和合同类问答。

## 4. query rewrite 有哪些常见方式？

> ID: `Q040`

query rewrite 常见方式包括规则词典改写、同义词扩展、业务术语补全、上下文补全、
精确实体保留、多 query 拆分和 LLM 改写。规则词典适合高频、稳定的业务词，比如把
“来源”扩展成 citations、引用来源、可追溯；同义词扩展适合用户表达和文档表达不一致；
上下文补全适合“这个怎么算”这类指代问题；多 query 适合一个问题包含多个检索方向；
LLM 改写适合复杂口语表达。生产里通常先用可解释规则做第一版，再用测试集验证收益，
最后再考虑 LLM rewrite 和 rerank 配合。

## 5. rewrite drift 是什么，为什么危险？

> ID: `Q041`

rewrite drift 指 query rewrite 改变了用户原意，导致系统检索到看似相关但实际错误的资料。
它危险的地方在于，召回结果可能分数很高、答案看起来也合理，但回答的不是用户真正想问的问题。
例如用户问“接口怎么设计”，系统错误改写成“前端页面怎么设计”，就会把检索方向带偏。
生产里 rewrite drift 会造成错误回答、错误引用和用户信任下降，严重时还可能触发权限或合规风险。
控制方式是保留原始 query、限制扩展词数量、保留表名字段名等精确实体、记录 rewritten query、
用评测集比较改写前后效果，并在低置信度或意图不清时选择澄清或拒答。

## 6. 为什么演示时要主动讲 bad case？

> ID: `Q057`

主动讲 bad case 能体现你理解的是生产系统，而不是只会展示顺利样例。
RAG 项目一定会遇到资料缺失、召回不准、引用错误、权限边界和成本问题。
如果只展示成功样例，面试官很难判断你是否知道系统边界。
主动展示 bad case，并说明它是资料问题、检索问题、prompt 问题还是权限问题，
再给出 query rewrite、rerank、评测集、拒答和监控等改进方案，会更像真实工程复盘。

## 8. 为什么试投要先小批量验证，而不是一开始海投？

> ID: `Q060`

试投初期的目标不是立刻投很多，而是验证简历表达、项目包装和岗位匹配度。
如果一开始海投，简历问题、项目讲法问题和岗位筛选问题会被放大，反馈也难分析。
先投 5-10 个高匹配岗位，可以观察是否有简历查看、沟通、约面或卡点反馈。
根据反馈再调整简历关键词、项目描述、薪资预期和投递方向。
这比盲目扩大数量更容易提高命中率。

## 9. AI 转型学习环境为什么要先保证可复现？

> ID: `Q061`

AI 转型不是只学一个模型调用，而是要持续做项目、复盘问题、沉淀代码和文档。
如果环境不可复现，后面每次换机器、换依赖或重跑项目都会浪费时间，问题也很难定位。
生产里更看重可复现的工程习惯：
固定目录结构、虚拟环境、依赖清单、运行命令、日志输出和结果文件。
这些基础能力能保证一个 Demo 不是“只在自己电脑上跑过”，
而是别人也能理解、运行和排查。

## 10. Python 模块化为什么是 AI 应用工程的基础？

> ID: `Q062`

AI 应用通常会同时包含数据加载、Prompt 构造、模型调用、结果解析、日志记录和异常处理。
如果所有逻辑都写在一个文件里，后面很难测试、复用和排查问题。
模块化的价值是把职责拆清楚：
配置归配置，数据处理归数据处理，模型调用归模型调用，接口层归接口层。
生产项目里这样做可以降低修改风险，
也方便把本地脚本逐步升级成 API 服务、批处理任务或可复用组件。

## 12. 结果解释层怎么避免模型编造业务原因？

> ID: `Q081`

结果解释层要避免编造，核心是把解释建立在结构化查询结果上。
系统应该先由规则层生成事实性结论，比如最高值、最低值、环比差值、趋势方向和返回行数，
再把这些事实交给 LLM 做可选润色。
Prompt 里要明确要求模型只能引用提供的 rows、columns、指标口径和时间范围，
不能自行猜测原因。
如果需要原因分析，应该输出“建议继续按渠道、产品、风险等级或日期拆分”，
而不是直接说原因已经确定。
生产里还要保留原始查询结果、SQL、request_id 和审计日志，方便回溯解释是否忠于数据。

这类交付方式能证明项目具备工程化能力，而不是只在本地临时跑通。

## 14. 生产级 AI 数据问答系统通常应该拆成哪些层？

> ID: `Q099`

生产级 AI 数据问答、NL2SQL 或 Agent 系统，不能只看成“用户问题进来，模型回答出去”。
我会把它拆成接入层、身份与权限层、编排层、意图识别层、上下文检索层、生成层、
安全校验层、工具执行层、结果解释层、审计日志层和评测回归层。

接入层负责 API、参数校验和统一响应；身份与权限层判断用户是谁、能访问哪些表字段和文档；
编排层决定下一步调用哪个工具，以及失败时是补充条件、降级、拒答还是转人工。
意图识别层判断用户是在查指标、问规则、查明细还是提出敏感导出请求。
上下文检索层包括 Schema 检索和 RAG 检索，负责找表字段、指标口径、政策规则和引用资料。

生成层负责生成候选 SQL、回答计划或结果解释草稿，但生成结果不能直接执行。
安全校验层要检查只读限制、敏感字段、权限边界、时间范围、limit 和成本风险。
工具执行层只执行已经满足前置条件的动作，例如执行 SQL、读取 RAG 资料或写审计。
结果解释层把结构化结果翻译成业务语言，但不能编造 SQL 没查出来的原因。
审计日志层记录 request_id、用户、问题、工具路线、SQL、校验结果、执行状态和最终回答。
评测回归层用固定测试集持续验证正常回答、安全阻断、权限不足、无答案和工具失败场景。

这套分层的核心思想是：模型可以参与理解、生成和表达，但权限、安全、执行和审计必须由确定性工程层控制。
在金融信贷场景里，这一点尤其重要，因为授信、风控、逾期、贷后和客户明细都涉及合规、成本和可追溯要求。

## 15. 数据质量问题为什么要接入错误治理和回归测试？

> ID: `Q132`

因为数据质量问题不是一次性故障，而是会重复出现的生产风险。
如果只靠人工排查，下次空分区、重复写入或补偿延迟发生时，Agent 还会犯同样的错。

生产级的处理方式是把质量问题按类型（空值、重复、乱序、延迟、补偿、幂等）分类，
每条关联一个具体的质量规则 ID，定义触发条件（如 NULL 比例 > 5%、同一 event_id 出现超过 1 次）、
风险等级和质量状态（quality_warning 或 quality_blocked）。

然后将质量异常做成规则、审计字段和固定 bad case，接入端到端回归测试集。
每次代码或规则变更后自动跑全量回归，验证 quality_status 和回答行为是否符合预期断言。
这样才能从"每次人工排查"变成"规则化 + 审计化 + 回归化"的工程闭环。

## 16. 为什么综合项目必须提供统一入口，而不是保留多个独立脚本？

> ID: `Q135`

因为面试和生产都看重系统边界，而不是零散能力清单。

多个独立脚本的问题在于：无统一入口和复用，配置散落难以维护，工具重复实现导致审计和错误码不一致，
用户（或面试官）也无法直观理解这些能力如何协同工作。

统一入口意味着意图识别、权限校验、工具路由、结果校验和审计逻辑被真正整合成一个主流程。
这才是"一个系统"而不是"一堆脚本拼成的故事"。

**统一入口的最小主流程包含哪些节点？**
用户问题 → 意图识别 → 权限校验 → 工具路由 → 工具执行 → 结果校验 → 回答生成 →
审计落库 → trace/eval。每个节点只做一件事，做完交给下一个，不能越界。
工具执行负责实际调用（查 SQL、读 RAG、查实时指标），回答生成负责把结构化结果转成业务语言，
trace/eval 负责记录链路供回归评测。

**模块边界不清晰会带来什么问题？**
第一，无法排查——出错了不知道是哪一步的问题，因为大家都在干活。
第二，无法复用——校验逻辑写在 SQL 生成器里，以后实时工具想复用还得再写一遍。
第三，安全形同虚设——SQL 生成器说"我顺便校验了权限"，但模型生成的校验不可靠。
第四，评测无从下手——边界混在一起，不知道每层该测什么。

**集成 README 至少应该讲清楚哪些内容？**
综合项目 README 至少要包含：项目目标和适用场景、架构流程和主链路图、启动方式和环境依赖、
接口请求/响应示例（含字段含义）、固定测试问题（成功样例+bad case）、常见失败场景和排查方式、
当前限制和后续优化方向。它的作用是让面试官能在 30 秒内理解这个项目解决什么问题。

**Day 50-58 的哪些能力应该优先整合进统一入口？**
应该整合的具体能力包括：指标字典 RAG 口径问答（Day 50）、Schema Catalog + 工具注册（Day 51）、
离线 NL2SQL 指标查询（Day 53）、实时告警解释工具（Day 54）、日报 + 告警摘要 Agent（Day 55）、
血缘追溯工具（Day 52）、端到端评测回归（Day 56）、数据质量状态检查（Day 57）和审计存储（Day 58）。
统一入口按"意图识别→权限校验→工具路由→工具执行→结果校验→回答生成→审计落库"的流程编排它们。

## 17. 为什么综合项目要保留本地 fallback？

> ID: `Q136`

因为这个项目既是学习仓库，也是作品集。本地 fallback 有三个核心价值：

第一，保证演示稳定。如果所有能力都强依赖外部模型和在线环境，演示时网络抖动或模型不可用就会导致翻车。
第二，保证回归可重复。确定性规则的结果不会因为模型版本变化而漂移，评测集每次跑出来的结果一致。
第三，把确定性规则和模型能力分开讲清楚。对于 SQL 安全、权限拦截、延迟判断这类关键逻辑，
本来就不应该完全依赖模型——它们应该走确定性代码，模型只负责理解和表达。

保留本地 fallback 也能让你在面试时说清楚"哪些用规则兜底、哪些交给模型"的工程取舍。

## 18. 3 分钟项目演示最应该讲什么？

> ID: `Q138`

最应该讲业务问题、系统边界、核心链路和工程取舍。
下面是我建议的完整讲法，完全从业务视角展开，先带面试官看两个真实场景，再倒推模块设计。
①业务问题——为什么金融信贷场景同时需要离线仓库和 实时仓库；
②系统架构——RAG、NL2SQL、实时工具、血缘和审计各自负责 什么；
③成功路径——给一个正常查询或日报摘要样例；
④风险路径—— 给一个延迟、越权或口径冲突样例；
⑤工程价值——为什么这不是普通聊天机器人，而是有数据边界和审计能力的 Agent。
**为了方便，把这个项目称为"信贷智能分析体（Credit Agent）"。**

 *第一部分：先看两个真实的业务场景（带出业务问题）
**场景一（实时决策 - 贷中拦截）**
业务人员小李在后台监控大屏上，突然看到一个用户正在极速申请多笔贷款。
小李输入："实时查一下用户ID_7788，当前在途申请单量和近5分钟登录IP是否异常？"
系统在800毫秒内回复："该用户当前有3笔在途申请，近5分钟登录IP跨越3个省份，
触发实时反欺诈规则，建议暂缓授信。"

**场景二（离线分析 - 日报摘要）**
第二天早上9:00，风控总监张总输入："生成昨日全渠道的授信通过率日报，
并对比上周同期，附带简要分析。"
系统在15秒后回复："昨日总申请量12,345笔，通过率34.5%（环比上周同期+1.2%）。
主要增长源于公积金渠道……（附详细图表链接）"

**为什么必须两仓并存？**
实时仓库支撑场景一：它必须持有毫秒级的流式特征（滑动窗口内的行为），
确保拦截在用户点击"提款"前生效。离线仓库支撑场景二：它必须保证T+1精确性，
涉及全量聚合、同环比计算和监管报送要求的强一致性（一分钱都不能差）。
如果只用实时仓库算昨日全量，成本极高且易丢数；如果只用离线仓库，
反欺诈滞后一小时，资金可能已被洗走。这就是业务倒逼技术，必须采用离在线双轨制。
---
* 第二部分：系统边界——拆解五大模块分工
在明确两仓数据底座后，Agent的上层模块各司其职，边界极其清晰：

| 模块 | 系统边界（只做什么，不做什么） | 金融信贷场景下的职责 |
| RAG（检索增强） | 只处理非结构化知识，不碰任何数值型敏感数据 | 挂载《授信审批操作手册》《反洗钱法规》。当问"利率上限是多少"时，从向量库召回条款原文，不做任何推理计算 |
| NL2SQL（离线查询） | 只读访问离线数仓，禁止DML操作 | 将"昨日通过率"翻译成标准SQL，查询离线宽表。返回确定性的批量统计结果 |
| 实时工具（Function Call） | 只对接实时特征API网关，禁止全表扫描，必须带Key | 封装"查近5分钟IP""查当前在贷余额"等原子工具，每个工具必须在200ms内超时熔断 |
| 数据血缘（Lineage） | 只记录"数据从哪里来"，不记录模型思考过程 | 确保任何返回的指标都能追溯到来源，监管审计必查项 |
| 审计日志（Audit Log） | 只记录"谁、何时、问了什么、查了什么" | 记录每条SQL和工具调用，符合银保监会数据安全要求 |
---
* 第三部分：成功路径——正常查询的系统响应
**用户问题**："汇总一下近3天优质客户的授信额度使用率，
并结合最新的风险偏好文件，总结是否需收紧额度。"

**系统响应（按时间线）**：
1. **0~500ms（意图识别与路由）**：Agent判断需要离线统计（额度使用率）+ RAG检索
（风险偏好文件）。
2. **500ms~3s（并行执行）**：左侧线程NL2SQL生成SQL提交给离线数仓；
右侧线程RAG在向量库中召回《2026年风险偏好声明书》相关章节。
3. **3s~5s（结果融合与生成）**：离线数仓返回45.6%，RAG返回"集中度红线为60%"。
Agent生成结论："当前优质客户使用率45.6%，距红线尚有空间，无需收紧。附政策原文链接。"

这个链路中，实时工具未参与，血缘自动记录了查询的表名，审计存储了本次问答全文。
每个模块只在自己边界内工作，Agent充当"大脑"仅做编排和总结，不瞎算数。
---
*第四部分：风险路径——延迟、越权、口径冲突处理

**风险1：实时工具延迟超标**
场景：小李查实时IP归属地，依赖的第三方IP库接口因网络抖动超过800ms未返回。
工程取舍：不等待，立即降级。Agent返回：
"实时IP库查询超时（已尝试2次），本次决策采用本地缓存快照（5分钟前的归属地），
请人工复核。"背后的权衡：在金融信贷中，可用性大于极致的准确性，
不能让整个页面卡死，必须给业务人员一个"有损但有据可查"的结果。

**风险2：越权访问**
场景：普通客户经理输入"查一下总裁办王总的授信额度明细。"
Agent在执行实时工具前先鉴权，发现无权访问executive_level数据，直接拦截：
"权限不足，您无权查看该级别数据。本次行为已记录至审计日志。"
背后的设计：Agent虽然是AI，但执行层必须继承RBAC，LLM本身没有权限，
它只是根据权限返回结果。

**风险3：离线与实时口径冲突**
场景：用户问"当前在贷余额是多少？"
实时数仓显示1.23亿（含今日未结算的实时放款），离线仓库显示1.20亿（昨日日终）。
Agent必须显式区分，绝不模棱两可，回复为：
"截至当前实时滚动在贷余额为1.23亿（预估，含今日未正式记账放款）。
截至昨日日终法定口径余额为1.20亿。请您根据使用场景选用——运营监控看实时，
财务对账看离线。"这直接定义了Agent的数据解释边界——不强行弥合差异，
而是把差异产生的时间窗口原因解释清楚。
---
* 第五部分：工程价值——为什么不是普通聊天机器人？

1. **确定性隔离（消除幻觉）**：普通聊天机器人可能乱编数字。我们强制所有数值
必须来自工具调用（SQL或API），LLM只负责编排和生成自然语言结论。
不信任LLM算数，只信任它说话。

2. **数据边界与审计闭环**：它知道自己"该查什么"和"不该查什么"（越权拦截）。
同时，血缘+审计构成了监管的"黑匣子"——遇到客户投诉"为什么额度被降"，
可以一键拉出当时的Trace链路：用户提问→查了哪张离线表→实时接口返回了什么
特征→Agent如何拼接答案。这在银保监会检查中是**一票否决项**。

3. **成本与性能的精细治理**：通过模块边界，把高成本的RAG向量检索留给低频政策
问答，把高消耗的NL2SQL留给T+1离线低峰期，把实时计算资源留给核心的毫秒级拦截。
没有让实时工具去扫全表，也没有让RAG去读财务数字。这种分层治理使得单次对话
的云资源成本控制在0.02元以内，具备大规模商业化落地的可行性。

**收尾总结**：这个项目本质上是为金融信贷风控场景定制的一层"可信执行层"。
它左手拉着离线数仓做"账房先生"，右手拉着实时数仓做"哨兵"，
由Agent当"翻译官"，通过清晰的边界划分和工程化的降级策略，
把大模型的不确定性锁在了业务允许的笼子里。

## 19. 简历里最值得保留的项目亮点应该是什么？

> ID: `Q144`

最值得保留的是系统边界、数据底座、工具分工、安全与审计、评测与回归这五类内容。

系统边界说明这个 Agent 能处理什么（离线指标、实时告警、口径问答、血缘追溯），
也知道什么时候该拒答、降级或澄清。数据底座说明你理解 ODS/DWD/DWS/ADS 分层、
实时事件流和窗口语义，不只是调模型。工具分工说明 RAG、NL2SQL、实时工具和血缘工具各司其职，
通过注册表和前置条件控制调用。安全与审计说明敏感字段拦截、权限校验、request_id 全链路追踪
和审计日志留存。评测与回归说明端到端评测集覆盖正常查询、安全阻断、空分区、实时延迟和口径冲突，
每次改动后跑全量回归。

这五类内容能说明这个项目不是随便拼起来的聊天应用，而是有真实工程取舍的准生产 Agent。

## 20. 作品集目录至少应该包含哪些内容？

> ID: `Q145`

作品集目录不是把学习笔记堆在一起，而要让面试官 30 秒内理解你做了什么、怎么运行、怎么判断效果。
至少应包含：项目 README（一句话目标、启动方式、核心接口）、架构图和主链路说明、离线入库和在线问答流程、
固定测试样例和 bad case、端到端评测集说明、Docker 或本地运行方式、配置说明、审计日志示例和后续优化 backlog。
作品集目录的目标是让面试官感觉这个项目可以被接手、被运行、被评估，
而不是一个"只有作者能跑通"的学习记录。

## 21. 哪些回答信号能让面试官觉得你更接近生产工程师？

> ID: `Q146`

面试官判断候选人是否接近生产工程师，通常看几个信号。
第一，能不能主动讲出系统边界：不是所有问题都能回答，什么时候该拒答、降级或澄清。
第二，能不能讲出安全与权限的工程实现：不是只靠 prompt 约束，而是字段级权限、SQL 校验、审计日志这些确定性控制。
第三，能不能讲出评测和回归：不是"我觉得效果还行"，而是有固定测试集、bad case 归因、全量回归。
第四，能不能讲出成本与延迟取舍：不是参数越大越好，而是如何在准确率、延迟和成本之间做 engineering tradeoff。
第五，能不能先讲业务问题再讲技术方案：先讲金融信贷场景为什么需要离线日报 + 实时告警 Agent，
再讲 RAG、NL2SQL、血缘和审计各自解决什么问题，最后才说用了什么框架。
第六，能不能在 30 秒内给出分层短答案，然后被追问时展开，而不是一上来就背长篇回答。
这些信号共同说明你理解的是生产系统约束，而不只是模型调用。
