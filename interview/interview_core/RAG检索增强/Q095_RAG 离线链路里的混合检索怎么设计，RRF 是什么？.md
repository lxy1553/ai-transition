# RAG 离线链路里的混合检索怎么设计，RRF 是什么？

> 来源: 核心题库 | 分类: RAG检索增强

RAG 里说混合检索，通常指同时使用向量检索和关键词检索。离线链路负责为两种检索方式准备好统一的知识索引，
在线链路再根据用户问题分别召回、融合和重排。这样设计是因为向量检索擅长语义相似，关键词检索擅长精确命中，
两者能互补。

离线侧一般会先做文档解析、清洗、去重、切分 chunk、抽取 metadata，然后给每个 chunk 分配稳定的 `chunk_id`。
同一批 chunk 会进入两类索引：一类是向量索引，用 embedding 做语义检索；另一类是 BM25 或倒排索引，
用关键词、表名、字段名、错误码、政策条款号做精确匹配。关键点是两边索引必须共享同一套 `chunk_id`、权限标签、
文档版本、业务域和来源信息，否则在线融合、去重、权限过滤和 citations 都会很难做。

为什么要混合，是因为只用向量检索容易漏掉精确实体。比如表名、字段名、合同号、错误码这类信息，
embedding 未必稳定；只用 BM25 又不懂同义表达，用户说法和文档说法不一致时容易漏召回。
生产里常见做法是：向量检索取 top 50，BM25 取 top 50，先按权限、版本和业务域过滤，再合并去重，
然后用融合算法或 reranker 得到最终 top-k。

混合方式主要有三种。第一种是分数加权，把向量相似度和 BM25 分数归一化后相加，例如
`final_score = 0.7 * dense_score + 0.3 * sparse_score`。但这个方法要小心，因为两种分数的尺度不同，
不归一化直接相加没有意义。第二种是候选集合并后交给 reranker，让 rerank 模型重新判断 query 和 chunk 的相关性。
第三种是 RRF，也就是 Reciprocal Rank Fusion，倒数排名融合。

RRF 不直接使用原始分数，而是使用每个文档在不同检索器里的排名。公式是：

```text
RRF_score(d) = sum(1 / (k + rank_i(d)))
```

其中 `rank_i(d)` 是文档 `d` 在第 `i` 个检索器里的排名，`k` 是平滑参数，常见取值是 60。
比如某个 chunk 在向量检索里排第 2，在 BM25 里排第 10，那么它的分数就是：

```text
1 / (60 + 2) + 1 / (60 + 10)
```

RRF 的好处是不用纠结向量分数和 BM25 分数是否可比，只看排名；如果一个 chunk 在多个检索器里都排得靠前，
它会被认为更可信。如果一个 chunk 只在某一种检索方式里靠前，也不会被直接丢掉。
所以在生产 RAG 里，一个比较稳的回答是：离线建统一 chunk、metadata、向量索引和 BM25 索引；
在线分别召回，用 RRF 或归一化加权做融合，再用 reranker 精排，最后取 top-k 拼上下文并返回 citations。
