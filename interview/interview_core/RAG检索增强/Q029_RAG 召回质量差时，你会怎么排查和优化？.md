# RAG 召回质量差时，你会怎么排查和优化？

> 来源: 核心题库 | 分类: RAG检索增强

我会先看检索结果，而不是先改 prompt。
排查顺序是：问题是否被正确理解，正确资料是否在知识库里，chunk 是否切得合理，
top-k 是否太小，metadata 过滤是否过严，关键词和向量检索是否命中，
以及是否需要 rerank。优化时会结合测试集和 bad case，而不是凭感觉调参数。
RAG 召回质量差通常有几类原因。第一，知识库没有正确资料，或者资料过期、重复、冲突。
第二，chunk 切分不合理，太大会导致相关内容被稀释，太小会丢上下文。
第三，用户问题和文档表达不一致，需要 query rewrite 或同义词扩展。
第四，只用向量检索可能漏掉表名、字段名、错误码这类精确词，
所以生产里常用混合检索。第五，metadata 过滤可能过严，比如业务域、
权限或版本过滤把正确资料排除了。第六，top-k 和 rerank 策略不合理，
正确资料虽然被召回，但排得太靠后，最终没有进入上下文。
生产里召回优化一般会围绕这些手段展开：

| 方法 | 作用 |
|------|------|
| 调整 chunk 策略 | 控制文档切分粒度，避免太碎或太大 |
| 调整 top-k | 平衡召回率、噪声和成本 |
| query rewrite | 把用户问题改写成更适合检索的表达 |
| 混合检索 | 同时使用向量检索和关键词检索 |
| metadata 过滤 | 按业务域、权限、版本、文档类型过滤 |
| rerank | 对候选结果重新排序，让最相关资料靠前 |
| bad case 分析 | 找出召回失败原因，反向优化知识库和检索策略 |
