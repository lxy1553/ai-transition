## 1. 结构化输出为什么是 AI 应用里的关键能力？

> ID: `Q014`

因为生产系统不能只消费一段自然语言。下游程序需要稳定字段，
比如 `risk_level`、`tables`、`suggestions`、`can_publish`。
结构化输出让模型结果可以被校验、存储、路由、展示和继续处理。
真实 AI 应用里，模型输出通常不是给人直接看完就结束，而是要进入后续系统。
比如 SQL 解释助手需要输出表名、字段、风险等级、风险列表和修改建议；
RAG 系统需要输出答案、引用来源、置信度和是否拒答；客服系统需要输出意图、
工单类型和处理动作。如果只返回一段自然语言，程序很难稳定解析。
所以要通过 JSON schema、Pydantic 或业务协议约束输出结构。
结构化输出还能做自动校验，比如字段是否缺失、枚举值是否合法、类型是否正确。
这也是 Tool Use、工作流编排和前端展示的基础。

## 2. Prompt 在生产项目里应该怎么管理？

> ID: `Q017`

生产里的 prompt 应该像配置和代码一样管理。要有版本、变量、模板、
输出格式约束、测试样例和变更记录。不能只把 prompt 写在代码里随便改，
否则出现回答变差时很难回滚和定位问题。
Prompt 在生产 AI 应用里是核心逻辑的一部分。一个好的 prompt 通常包含角色、
任务目标、上下文、约束、输出格式和异常处理要求。比如 RAG 场景要明确：
只能基于上下文回答，没有依据就说明无法判断，并返回引用来源。
NL2SQL 场景要明确：只能使用提供的表和字段，不能编造字段，生成后要输出解释。
工程上 prompt 要模板化，把用户问题、检索上下文、工具结果作为变量填入。
还要做版本管理和回归测试，避免一次 prompt 修改影响线上效果。
对于重要 prompt，最好保留变更记录、测试集结果和灰度发布策略。
