**“我们的数据来源主要分三大类：业务系统数据库、还款系统的binlog同步、以及埋点SDK上报。这三者覆盖了信贷业务所需的全部核心数据，并分别通过不同的技术链路进入数据平台。”**

### 1. 业务系统库（核心交易数据）
这是指我们的信贷核心系统、用户中心等 **OLTP 数据库**（通常为 MySQL）。里面存储着用户信息、借款申请、合同、放款记录等强一致性数据。  
对于这类数据，我们通常采用**批量离线抽取 + CDC实时补全**的方式：
- 离线部分：通过 Spark JDBC 连接备库，按天全量或增量抽取到数据湖（Iceberg）的 ODS 层。
- 实时部分：同样通过 binlog 同步实时变更，保证数仓中的用户状态与业务库基本实时一致。

### 2. 还款系统 binlog 同步（重点讲实现）
还款系统也是基于 MySQL 的，我们需要**实时**捕获每笔还款、逾期状态变更，驱动贷后监控和特征更新。实现方式是经典的 **CDC（变更数据捕获）** 链路：

- **原理**：MySQL 开启 binlog（ROW 模式），使用 **Debezium** 作为 CDC 工具，它伪装成 MySQL 的从库，实时读取 binlog 流。
- **消息封装**：Debezium 将 INSERT/UPDATE/DELETE 事件解析成标准化的 JSON/Avro 消息，包含 `before/after` 数据快照、操作类型、时间戳等，并自动注册 Schema 到 **Schema Registry**。
- **进入 Kafka**：Debezium 将捕获的变更推送到 Kafka 的专用 Topic（如 `cdc_repay`），保留操作顺序。
- **入仓处理**：我们部署 **Flink** 作业消费该 Topic，进行数据清洗、与维表关联、格式化，然后写入 **实时数仓层（Doris/ClickHouse）** 供实时查询，同时准实时写入 **Iceberg** 表供离线分析。  
  这里的关键设计：利用 Flink 的 **Checkpoint 机制** 保证端到端精确一次，即使宕机也不会丢数据或重复；同时 Iceberg 支持 ACID，可安全并发写入。

### 3. 埋点 SDK 上报（结合之前的设计回答）
用户行为数据（浏览、点击、申请流程中断等）全部来自自研的**埋点 SDK**。实现细节如下：

- **客户端 SDK**：集成在 App/Web 端，提供简洁 API（如 `track('loan_apply', {...})`）。内部做了几件事：
  - 自动组装公共字段（user_id、设备指纹、会话 ID、时间戳等），**强制过滤明文 PII**（正则匹配身份证、手机号等，只发哈希值）。
  - 事件先写入**本地队列**（SQLite/IndexedDB）持久化，防止 App 崩溃丢失。
  - 后台定时或达到阈值（如 20 条）时，批量 **gzip 压缩** 后通过 HTTPS 发送到**采集网关**，失败自动重试且带退避。
- **采集网关**：基于 Go/FastAPI 的高性能服务，负责：
  - 验证 AppKey、流量限流。
  - **二次清洗**：再次用正则扫描整条 JSON，确保无明文 PII 流入后端。
  - 校验 JSON Schema，转换数据格式，然后写入 Kafka（如 Topic `user_behavior_view`），key 为 user_id 保证有序。
- **数据流**：Kafka 中的埋点数据会被两条链路消费：
  - **实时链路**（Flink）：计算 5 分钟/1 小时窗口内的行为计数、频次等特征，直接写入 Redis 供在线推理使用。
  - **离线链路**（Spark Streaming 或 Flink）：准实时入湖到 Iceberg ODS 层，用于 T+1 的离线特征回填和模型训练。

### 总结
这三个数据源在数仓中通过 user_id、apply_id 等标识符关联，共同构建了覆盖贷前、贷中、贷后的完整数据视图。整个过程中，我们通过**Schema Registry 管理格式演进**、**Kafka 解耦上下游**、**Flink 保证实时计算的一致性**，以及**多层 PII 防护**，确保了数据的实时性、准确性和合规性。
