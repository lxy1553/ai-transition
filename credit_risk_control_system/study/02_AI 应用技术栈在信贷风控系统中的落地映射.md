### 场景 2：拒贷原因智能生成（嵌入推理流水线）

这是 **RAG + 规则引擎 + SHAP** 的结合：

```python
# src/decision_engine/explanation_generator.py（新增模块）

class RejectExplanationGenerator:
    """
    拒贷文案智能生成器。

    将"机器可读的拒绝原因"转化为"用户可理解的自然语言"。

    输入:
      - 规则引擎触发的 reason_codes: ["RC_BL001", "RC_DT001"]
      - SHAP 特征贡献: {"debt_to_income_ratio": 0.15, "multi_head_cnt_7d": 0.08}
      - 用户画像: {"age": 28, "income_level": "8000-15000"}

    输出:
      "您的申请暂未通过，主要原因：
       1. 当前负债率偏高（负债/收入 = 65%，建议控制在50%以下）
       2. 近期多头借贷记录较多（近7天在3个平台有申请记录）
       改善建议：结清部分现有贷款后30天可重新申请。"
    """

    def __init__(self):
        # RAG 检索库: 拒贷原因码 → 文案模板 + 改善建议
        self.template_store = VectorStore("reject_reason_templates")

    async def generate(self, reason_codes, shap_values, user_profile):
        # Step 1: 检索每个 reason_code 对应的文案模板（RAG）
        templates = await self.template_store.search(reason_codes, top_k=5)

        # Step 2: 用 SHAP 值补充具体数字
        context = self._build_context(templates, shap_values, user_profile)

        # Step 3: LLM 生成个性化文案
        prompt = f"""
        你是信贷审批解释助手。基于以下信息生成友好的拒贷说明：

        拒绝原因码: {reason_codes}
        SHAP特征贡献: {shap_values}
        用户画像: {user_profile}
        参考模板: {templates}

        要求: 1) 语气温和 2) 给出具体数字 3) 提供改善建议 4) 不超过200字
        """
        return await llm.agenerate(prompt)
```

## 三、NL2SQL（自然语言转 SQL）

### 在项目中的位置

NL2SQL 直接对应你 `day29-42` 学的完整 NL2SQL 系统。在风控场景中，它服务于**业务人员的自助数据分析**：

┌──────────────────────────────────────────────────────────────────┐

│  │ Step 1: Question Parser (day30)                              │ │

│  │   实体: 时间=上周, 地区=广东, 指标=通过率                     │ │
│  │                                                              │ │
│  │ Step 2: Schema Router (day29)                                │ │
│  │   匹配表: dws.user_risk_feature_wide (DWS层风险宽表)         │ │
│  │         ads.approval_rate_daily (ADS层通过率日报)             │ │
│  │                                                              │ │
│  │ Step 3: SQL Generator (day31)                                │ │

│  │ Step 4: SQL Validator (day32)                                │ │
│  │   ✓ 语法检查 · ✓ 权限检查 · ✓ 敏感字段脱敏                    │ │

│  │ Step 5: Query Executor (day33) → ClickHouse 执行             │ │

│  │ Step 6: Result Interpreter (day34)                           │ │
│  │   "上周广东地区通过率为 72.3%，较上月同期上升 3.1 个百分点。"    │ │

## 五、LangGraph

### 在项目中的位置

LangGraph 解决**有状态的、多分支的复杂工作流**，这在风控场景非常关键：

### 场景 1：信贷审批全生命周期状态机

```python
"""
LangGraph 构建的信贷审批工作流

这个工作流比简单的"规则→模型→额度"流水线复杂得多，
因为它涉及人工审核、补充材料、多次回调等状态转换。

对应你学过的 LangGraph 知识:
- StateGraph: 定义审批状态
- conditional_edges: 不同条件下的分支
- checkpoint: 状态持久化（审批进度可恢复）
"""

from langgraph.graph import StateGraph, END
from typing import TypedDict, Literal

class ApplicationState(TypedDict):
    application_id: str
    user_id: str
    status: str              # pending / review / approved / rejected
    rule_results: list
    model_score: float
    required_documents: list[str]
    reviewer_notes: str
    retry_count: int

def create_approval_workflow():
    workflow = StateGraph(ApplicationState)

    # 节点定义
    workflow.add_node("submit", submit_application)         # 提交申请
    workflow.add_node("auto_check", auto_rule_check)        # 自动规则检查
    workflow.add_node("model_score", model_scoring)         # 模型打分
    workflow.add_node("manual_review", manual_review)       # 人工审核
    workflow.add_node("request_docs", request_documents)    # 要求补充材料
    workflow.add_node("final_approve", final_approve)       # 最终批准
    workflow.add_node("final_reject", final_reject)         # 最终拒绝

    # 条件边（LangGraph 的核心优势）
    workflow.add_conditional_edges(
        "auto_check",
        route_after_check,  # 根据规则结果决定下一步
        {
            "reject": "final_reject",       # 硬拒绝 → 直接拒绝
            "manual": "manual_review",      # 疑似风险 → 人工审核
            "pass": "model_score",          # 通过 → 进入模型打分
        }
    )

    workflow.add_conditional_edges(
        "model_score",
        route_after_scoring,
        {
            "approve": "final_approve",     # 高分 → 批准
            "manual": "manual_review",      # 边缘分 → 人工
            "reject": "final_reject",       # 低分 → 拒绝
        }
    )

    workflow.add_conditional_edges(
        "manual_review",
        route_after_review,
        {
            "approve": "final_approve",
            "reject": "final_reject",
            "request_docs": "request_docs", # 需要补材料 → 回到等待状态
        }
    )

    workflow.set_entry_point("submit")
    return workflow.compile()
```

### 场景 2：贷后催收策略状态机

```python
"""
催收策略是风控系统的贷后环节，LangGraph 天然适合:

      正常还款 ──→ END
    /
  放款 ──→ D+1提醒
           \
            逾期D+1 ──→ 短信
              \
              逾期D+7 ──→ 电话
                \

每个节点根据用户画像、金额、历史行为选择不同的催收策略。
"""

class CollectionState(TypedDict):
    loan_id: str
    days_past_due: int
    overdue_amount: float
    user_risk_level: str
    collection_action: str
    contact_result: str

def create_collection_workflow():
    workflow = StateGraph(CollectionState)

    workflow.add_node("check_dpd", check_days_past_due)
    workflow.add_node("sms_reminder", send_sms)
    workflow.add_node("ai_call", ai_phone_call)
    workflow.add_node("manual_collect", assign_to_collector)
    workflow.add_node("legal_action", initiate_legal)

    workflow.add_conditional_edges(
        "check_dpd",
        lambda s: "sms" if s["days_past_due"] <= 7 else
                  "call" if s["days_past_due"] <= 30 else
                  "manual" if s["days_past_due"] <= 90 else "legal",

    )

    return workflow.compile()
```

### LangSmith 监控的具体内容

```python
# LangSmith 在这个项目中的追踪配置

import langsmith as ls
from langsmith import traceable

# 1. RAG 检索质量追踪
@traceable(name="credit_policy_rag", tags=["rag", "credit"])
async def retrieve_policy_docs(query: str):
    """追踪每次政策文档检索：query → retrieved_docs → 相关性评分"""
    docs = await vector_store.asearch(query, top_k=5)
    # LangSmith 自动记录: 输入query, 输出文档列表, 延迟, token数
    return docs

# 2. NL2SQL 全链路追踪
@traceable(name="nl2sql_risk_query", tags=["nl2sql", "clickhouse"])
async def nl2sql_pipeline(natural_query: str):
    """追踪: NL问题 → Schema路由 → SQL生成 → 执行 → 结果解释"""
    # 每一步都可被 LangSmith 单独追踪
    intent = await parse_question(natural_query)        # Trace span 1
    sql = await generate_sql(intent)                    # Trace span 2
    result = await execute_sql(sql)                     # Trace span 3
    explanation = await interpret_result(result)         # Trace span 4
    return explanation

# 3. 拒贷文案生成质量追踪
@traceable(name="reject_explanation_gen", tags=["explanation", "llm"])
async def generate_reject_explanation(reason_codes, shap_values, user_profile):
    """
    追踪每次拒贷文案生成，并收集用户反馈:
    - 用户是否理解了？
    - 文案是否准确反映了真实原因？
    - 用户是否在30天内重新申请？（间接衡量解释是否有用）
    """
    prompt = build_explanation_prompt(reason_codes, shap_values, user_profile)
    response = await llm.agenerate(prompt)

    # LangSmith feedback: 可关联后续用户行为
    ls.feedback.create_feedback(
        key="user_reapplied_in_30d",
        score=1 if user_reapplied else 0,
        comment="用户在收到解释后是否在30天内重新申请"
    )

    return response
```

## 八、关键面试回答

面试官问："你在风控项目中怎么用 LangChain/LangGraph？"

> **标准回答结构**：
>
> 这个风控系统采用**双引擎架构**——传统ML引擎做确定性决策（毫秒级、可解释、可审计），AI应用引擎做智能增强。
>
> 我负责的 AI 应用层有四个落地场景：
>
> 1. **RAG + 拒贷文案生成**：将规则引擎的 reason_code + SHAP 值 → LLM 生成个性化拒贷解释（"您负债率65%偏高，建议结清后30天再试"），替代了原来的固定模板。
>
> 2. **NL2SQL 风控分析助手**：业务人员用自然语言问"上周广东通过率"，系统自动转SQL查ClickHouse，这直接复用了我之前做的 NL2SQL 六步流水线。
>
> 3. **LangGraph 审批状态机**：信贷审批不只是"规则→模型→额度"，还有人工审核、补材料、驳回重审这些复杂分支。LangGraph 的状态图 + 条件边完美建模了这个流程。
>
> 4. **LangSmith 全链路追踪**：所有 LLM 调用（RAG检索、文案生成、NL2SQL）都有 trace，可以回溯任何一次 AI 辅助决策的完整过程，满足金融监管对"可解释AI"的要求。
>
> **核心原则是 AI 辅助而非替代**——审批决策始终由规则引擎+模型做出，AI只负责解释、分析和流程编排。
